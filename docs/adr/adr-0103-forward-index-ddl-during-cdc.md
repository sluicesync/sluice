# ADR-0103: Forwarding secondary-index (and index-only) DDL during continuous CDC sync

## Status

**Proposed** — a planning sketch, NOT yet implemented. This ADR records the design for forwarding a secondary-index (or any index-only) DDL added to a source AFTER cold-copy onto the target during continuous sync. It is **demand-gated**: the gap it closes is a schema-fidelity gap, **not a value-fidelity / silent-loss one** (data flows correctly across the gap — only the index does not cross), so it does not gate any release. Nothing here is built today; the file documents the detection-per-source / apply-per-target shape so the chunk can be picked up self-contained when an operator demands it. It promotes roadmap [item 24](../dev/roadmap.md) from a roadmap write-up into a proper ADR and expands its detection/apply tiers, the PlanetScale deploy-request apply path (the headline target path), and a recommended phasing.

## Context

### The confirmed gap (live, 2026-06-18) — and why it spans ALL three sources, not just VStream

Secondary-index DDL added to a live source table AFTER cold-copy does NOT propagate to the target during continuous CDC sync, on ANY source. Confirmed live 2026-06-18 (Track B / B2): an `ALTER TABLE wide_payload ADD INDEX idx_lst_ts (ts)` issued via Vitess online DDL (`ddl_strategy=vitess`) cut over on both shards — the index was present on the source — but appeared on NEITHER the PS-MySQL target (`lst-mysql-b-v2`) nor the PS-PG target (`lst-pg-b2`). Data kept flowing correctly throughout. The binlog-source and PG-source conclusions are code-confirmed (the wire protocols structurally cannot carry the index without re-introspection).

**No silent loss.** DATA is unaffected; this is purely a schema-fidelity gap. A missing secondary index degrades query performance on the target but never corrupts or drops a row. That is why the item is demand-gated rather than release-gating.

### Root cause — sluice already has the APPLY arm; what is missing is DETECTION

sluice HAS an apply-side classifier + forwarder for index shapes. The single-stream forward intercept (`internal/pipeline/schema_forward_intercept.go`) routes `ShapeKindCreateIndex` / `ShapeKindDropIndex` through `applyShapeForward` → `ir.ShapeDeltaApplier` (the same per-shape dispatch the Shape-A boundary router uses; ADR-0091 §1). The cross-engine index *translation* + creation already exists too — cold-copy creates every source index on the target via deferred index creation. So the apply machinery is built and proven.

The missing half is **detection**: no CDC reader populates secondary-index metadata into the `ir.SchemaSnapshot` it emits, so `ClassifyShape` on a live CDC snapshot never sees an index shape. The CDC wire protocols are all column-oriented:

- **VStream (Vitess/PlanetScale).** `projectVStreamFields` (`internal/engines/mysql/field_to_ir.go:135`) returns `&ir.Table{Schema, Name, Columns}` — no `Indexes`, not even the PK — because vtgate `FIELD` events are columns-only.
- **binlog (vanilla MySQL).** The reader DOES re-introspect on a DDL event (`internal/engines/mysql/cdc_reader.go`, `tableFor` → `loadTableSchema`), but its cached `tableSchema` is **columns + PK only** by deliberate design — the type comment at `cdc_reader.go:220-242` states: *"Indexes (other than the PK) and foreign keys are not consulted on a row event so we don't pay to load them on every cache miss."* The lazily-rebuilt IR in `maybeSnapshotSchemaB1` (`cdc_reader.go:907`) carries `Columns` + a projected `PrimaryKey` only — `irTbl.Indexes` is never set.
- **PG logical replication.** pgoutput Relation messages carry no secondary-index metadata, and `NormalizeForCDCComparison` explicitly sets `out.Indexes = nil` (`internal/engines/postgres/cdc_normalize.go:156`), pinned by a test asserting *"pgoutput carries no secondary index metadata"*. The comment there (`cdc_normalize.go:143-156`) spells out the consequence: *"CREATE/DROP INDEX therefore cannot be forwarded on PG via pgoutput — a documented limitation; MySQL's information_schema re-read CDC projection carries indexes and does forward them"* — except, as the binlog bullet above shows, the MySQL re-read deliberately drops secondary indexes too, so today NO source forwards them.

The result is the same on every source: an index-only DDL leaves the projected `SchemaSignature` unchanged (the signature is name + ordered type — ADR-0049 — which indexes do not touch) ⇒ no `SchemaSnapshot` delta is emitted (and on PG the normalizer would zero `Indexes` even if one were) ⇒ `ClassifyShape` never sees `ShapeKindCreateIndex` / `ShapeKindDropIndex` ⇒ nothing is forwarded. The `ShapeKindCreateIndex` / `ShapeKindDropIndex` arms are reachable today only via the migrate-path full-introspection diff, never the live CDC forward.

### Why detection is the genuinely hard half (and source-specific)

Each source hides the index DDL differently, so there is no one detection hook:

- **Vitess/PlanetScale online DDL never emits a literal `ALTER`.** A `ddl_strategy=vitess` online DDL cuts over by renaming a `_vt_` shadow table over the logical table — ADR-0073(c) already FILTERS that shadow-table churn out of COPY + CDC and survives the cutover zero-loss. The VStream event stream therefore carries no `ALTER … ADD INDEX`; the only post-cutover signal is a column-identical `FIELD` re-emit (same columns, so even a re-introspection of *columns* would be a no-op). The authoritative record of the completed DDL lives out-of-band in `SHOW VITESS_MIGRATIONS` (the completed statement text + a completion timestamp), which the reader does not poll today.
- **vanilla MySQL emits a literal DDL query event** for a direct `ALTER … ADD INDEX` in the binlog — the reader already sees it (the generic-DDL branch in `cdc_reader.go` clears the schema cache and captures the DDL position). The gap is only that the re-introspection it then does drops secondary indexes. (gh-ost / pt-osc shadow-table DDLs on vanilla MySQL behave like the Vitess case — the literal index DDL runs against a shadow table, not the logical one, and needs the same cutover-re-introspect handling.)
- **PostgreSQL logical replication streams NO DDL at all.** pgoutput carries DML + Relation messages only; there is no event a reader could even key off. The signal has to come from an event-trigger-based DDL capture (the `postgres-trigger` engine already captures DDL via event triggers — ADR-0066) or a periodic index re-introspection.

This is the asymmetry that makes the chunk worth an ADR: the apply side is uniform and mostly built, but detection is three different mechanisms with three different cost/credential profiles.

## Decision

Forward index-only DDL during continuous sync via a **detection tier per source** that lands index metadata into the live `ir.SchemaSnapshot`, feeding the EXISTING `ShapeKindCreateIndex` / `ShapeKindDropIndex` apply arms, which dispatch to an **apply tier per target** that uses each target's NON-BLOCKING index primitive. Detection is the new code; apply mostly exists. Because an index is not data, **the apply ordering relative to the CDC data stream does not matter for correctness** (an index added "late" relative to the rows it indexes still indexes them once it builds) — this is what makes an asynchronous, out-of-band apply (notably the PlanetScale deploy request) a clean fit.

The stance is **opt-in or report, never silent auto-apply**, per the "index lifecycle is explicit, not silently auto-handled" tenet (CLAUDE.md § Contain Postgres complexity, generalized). The default for a target whose non-blocking apply couples sluice to a control plane (PlanetScale) is to REPORT the detected index + the deploy request to apply; a self-hosted target with an in-band primitive (`ALGORITHM=INPLACE` / `CREATE INDEX CONCURRENTLY`) MAY auto-apply behind an explicit `--forward-index-ddl` opt-in.

### Detection tiers (the new, hard half)

**(a) vanilla MySQL source — extend the existing on-DDL re-introspection (cheapest; Phase 1).** The binlog reader already re-introspects on a DDL event; widen `loadTableSchema` / `tableSchema` to ALSO load secondary indexes (`information_schema.statistics`) and add an index-vector signal to `maybeSnapshotSchemaB1` analogous to the ADR-0091 F7a nullability signal (`forwardNullSig`) — a SEPARATE forward-delta tracker (`forwardIndexSig`) kept OUT of the value-fidelity-critical `snapshotSig` (the ADR-0049 decode contract must stay name + ordered type only). When `schemaForward` is on and the index vector changes, emit a boundary even though the decode signature did not. This reuses the entire F7a GAP-#2 pattern; it is the lowest-risk, highest-value slice because a direct `ALTER … ADD INDEX` is the common case and the DDL is already in-band.

**(b) Vitess/PlanetScale source — primary-routed re-introspection on the post-cutover FIELD re-emit, with `SHOW VITESS_MIGRATIONS` as the authoritative cross-check.** Two options, to be settled in Phase 2:

- **(b-in-band) Re-introspect indexes on the post-cutover FIELD re-emit.** When ADR-0073(c) observes an online-DDL cutover (the shadow-table rename completes and the logical table's `FIELD` re-emits), issue a primary-routed `SHOW INDEX FROM <table>` (vtgate routes a `SHOW INDEX` to a shard primary) out-of-band on the schema-DB connection — mirroring the degrade-don't-refuse primary-routed probe `verifyVStreamPositionReachable` already does. Diff against the cached index vector; emit the boundary on a delta. In-band with the stream, no extra credential surface (reuses the data-plane DSN).
- **(b-authoritative) Poll `SHOW VITESS_MIGRATIONS`.** It records the completed online-DDL statement text + a completion signal the VStream event stream does not cleanly surface. More authoritative (it names the exact DDL) but needs an out-of-band primary-routed poll loop the reader does not run today, and parsing the statement text reintroduces a regex-over-DDL surface the tenets discourage. **Recommendation: prefer (b-in-band) re-introspection** (structured `SHOW INDEX` output, no DDL-string parsing) and keep `SHOW VITESS_MIGRATIONS` as an optional confirmation / completion-timing signal, not the primary detector.

**(c) PostgreSQL source — event-trigger DDL capture or periodic re-introspection (hardest; later phase).** Logical replication carries no DDL. Two routes: reuse the `postgres-trigger` engine's event-trigger DDL capture (ADR-0066) to record `CREATE INDEX` / `DROP INDEX` statements in a capture table the streamer reads (works for the trigger engine; the slot-based pgoutput engine would need the trigger installed alongside, an explicit extra surface), or a periodic index re-introspection on a cadence (cheap, lossy on timing, no DDL-string parsing). Either is strictly more invasive than (a)/(b); deferred until PG-source index forwarding is demanded specifically. Note PG `CREATE INDEX CONCURRENTLY` does not write to WAL in a way pgoutput surfaces, so re-introspection is the only protocol-level option without the trigger.

### Apply tiers (mostly built — use each target's NON-BLOCKING index primitive)

An index build on a large table must not lock the table, so each target apply uses its non-blocking primitive:

**(a) PlanetScale MySQL target — issue a PlanetScale DEPLOY REQUEST (the headline path) via `planetscale-go`.** PlanetScale blocks direct online `ALTER` over the data-plane MySQL protocol (large-table index builds go through the platform's own online-schema-change machinery), so the correct non-blocking apply is to drive the **control-plane Public API**. The concrete `planetscale-go` surface (verified against the package, 2026-06-18):

- **Client:** `planetscale.NewClient(opts ...ClientOption) (*Client, error)`; auth via `WithServiceToken(name, token string)` (or `WithAccessToken(token string)`), plus `WithBaseURL` / `WithHTTPClient` / `WithUserAgent`.
- **Branches:** `client.DatabaseBranches` (`DatabaseBranchesService`): `Create(ctx, *CreateDatabaseBranchRequest)`, `Get`, `Diff(ctx, *DiffBranchRequest) (*Diff, error)`, `Delete`, `RefreshSchema`. `CreateDatabaseBranchRequest` carries `{Organization, Database, Name, ParentBranch, …}`.
- **Deploy requests:** `client.DeployRequests` (`DeployRequestsService`): `Create(ctx, *CreateDeployRequestRequest) (*DeployRequest, error)`, `Get`, `List`, `Apply(ctx, *ApplyDeployRequestRequest)`, `AutoApply`, `Review`, `Queue` / `Unqueue`, `Cancel` / `Close`, `Revert`. `CreateDeployRequestRequest` carries `{Organization, Database, Branch, IntoBranch, Notes, AutoCutover, AutoDeleteBranch}`.

**The load-bearing model finding (verified against the pscale CLI + docs):** a PlanetScale deploy request carries the **schema DIFF between two branches**, NOT an inline `ALTER` statement. The workflow is: create a development branch off the production branch → connect to that dev branch over the MySQL protocol and run the `ALTER … ADD INDEX` / `CREATE INDEX` there → open a deploy request (`DeployRequests.Create`, which captures the dev-vs-production diff) → deploy it (`Apply`, or auto-apply), at which point PlanetScale runs its own online schema change (copy table + sync + quick cutover; revertable within 30 minutes). So sluice's PS-MySQL apply path is necessarily: *(1) create/reuse an ephemeral dev branch; (2) apply the detected index DDL to it over the data-plane MySQL DSN; (3) `DeployRequests.Create`; (4) report or `Apply`.* This is several control-plane steps, not one statement — which strongly reinforces the **opt-in/report** stance: by default sluice REPORTS "detected `CREATE INDEX idx_lst_ts (ts)` on `wide_payload`; here is the deploy request to apply" rather than silently mutating the operator's production branch through a sequence of branch + deploy-request operations.

**(b) self-hosted MySQL target — `ALTER … ALGORITHM=INPLACE, LOCK=NONE`** (non-blocking on InnoDB for secondary-index add), or a gh-ost-style helper for engines/versions where INPLACE is unavailable. In-band over the data-plane DSN; no control plane. The existing `ir.ShapeDeltaApplier` MySQL path can carry this with the algorithm/lock clause added.

**(c) PostgreSQL target — `CREATE INDEX CONCURRENTLY`** (non-blocking, no table lock). Caveats to handle: it cannot run inside a transaction (so it must be issued outside the apply tx — which is fine, since index apply ordering does not matter for correctness), and it can leave an `INVALID` index behind on failure that needs an explicit `DROP INDEX` + retry. The applier must detect the invalid-index state and clean up loudly rather than leave a half-built index.

### Recommended phasing

- **Phase 1 — vanilla MySQL source → self-hosted MySQL target (and PG target via the existing CONCURRENTLY path).** Cheapest detection (extend the in-band on-DDL re-introspection, reuse the F7a GAP-#2 forward-signal pattern) + the simplest apply (in-band `ALGORITHM=INPLACE, LOCK=NONE` / `CREATE INDEX CONCURRENTLY`, no new credential surface). Highest value per unit of risk; closes the common direct-`ALTER … ADD INDEX` case.
- **Phase 2 — Vitess/PlanetScale source detection** (primary-routed re-introspection on the post-cutover FIELD re-emit) **+ PlanetScale MySQL target apply via the deploy request** (`planetscale-go`), shipped as an opt-in capability / report. This is the path with the live finding behind it but the largest new surface (control-plane creds + the branch→deploy-request model).
- **Phase 3 — PostgreSQL source detection** (event-trigger capture or periodic re-introspection), demand-gated on a real PG-source index-forwarding need.

## Consequences

- **New dependency surface: `planetscale-go`** (the control-plane SDK) enters the dependency graph for the PS-MySQL apply tier. This is a NEW credential surface distinct from the target MySQL/VStream data-plane DSN: a PlanetScale **service token** + the org/db/branch identifiers. Sluice must surface those as explicit config (mirroring how it already surfaces the data-plane DSN), and must NEVER infer or silently reuse data-plane creds for control-plane operations. The deploy-request path also introduces a dev-branch lifecycle (create → mutate → deploy → optionally auto-delete) that sluice would own — a meaningful new operational concept, another argument for opt-in/report rather than silent auto-apply.
- **Opt-in / report stance, not silent auto-apply.** Default behavior for any target is to REPORT the detected index delta + the concrete apply action (for PS: the deploy request; for PG/self-hosted MySQL: the `CREATE INDEX CONCURRENTLY` / `ALGORITHM=INPLACE` statement). Auto-apply is behind an explicit `--forward-index-ddl` opt-in and, for PS, additionally requires the control-plane token to be configured. This keeps index lifecycle explicit per the tenet.
- **Apply is mostly already built; cross-engine index translation already exists.** The `ShapeKindCreateIndex` / `ShapeKindDropIndex` arms, `ir.ShapeDeltaApplier`, and the cold-copy deferred-index-creation translation are all in place — Phase 1's apply is largely wiring the detected boundary to the existing path with the right non-blocking clause. The genuinely new code is detection + the PS deploy-request driver.
- **Ordering does not matter for correctness.** Because an index is not row data, the apply can lag the CDC data stream arbitrarily without a correctness consequence — no position-handoff guard (ADR-0007) is needed around it, unlike a data-bearing schema change. This is why the asynchronous deploy-request apply is clean and why a periodic-re-introspection detector (which is timing-lossy) is acceptable for indexes specifically.
- **Testing approach.** Unit: the per-source index-vector forward signal fires on a `CREATE INDEX` / `DROP INDEX` delta and is independent of the ADR-0049 decode signature (a pure index change emits a boundary but does NOT perturb `snapshotSig`). Integration: a live `ALTER … ADD INDEX` on a source mid-sync lands the index on the target (per source × target pair); for PS-MySQL, against the local Vitess cluster (`vitesscluster` tag) for detection and a mocked/recorded `planetscale-go` client for the deploy-request driver (real PlanetScale control-plane calls stay behind a credentialed opt-in tag, like `psverify`). Pin the invalid-index cleanup on the PG `CONCURRENTLY` failure path.
- **Explicitly out of scope.** Foreign-key DDL forwarding (a separate, harder shape — FKs are data-integrity constraints, not performance indexes, and cross-engine FK semantics differ); primary-key / unique-key changes (these move the decode/identity contract and are NOT index-only — they go through the existing column/identity shape paths, loudly); and any DDL-string parsing of the `SHOW VITESS_MIGRATIONS` statement text (the design prefers structured `SHOW INDEX` re-introspection over regex-over-DDL).

## Alternatives considered

- **Status quo — operator adds the index on the target out-of-band.** For PS-MySQL via a deploy request the operator issues themselves; for PG via `CREATE INDEX CONCURRENTLY`. This is the documented workaround today and is entirely safe (no silent loss). It is the correct default until demand appears — which is exactly why this ADR is Proposed/demand-gated rather than scheduled.
- **Full schema re-introspection on EVERY CDC DDL boundary (load all indexes, all constraints, all FKs).** Would catch index DDL as a side effect, but pays the full `information_schema` introspection cost on every boundary (the exact cost the binlog reader's `tableSchema` design deliberately avoids: *"so we don't pay to load them on every cache miss"*), and on PG there is no DDL boundary to hang it off at all. **Rejected** in favor of a targeted index-vector signal that loads only what the index shape needs, only when an index delta is plausible.
- **Direct blocking `ALTER … ADD INDEX` over the data-plane DSN.** Simplest to wire, but on a large table it locks for the duration of the build, and on PlanetScale/Vitess vtgate it hits the ~30 s statement cap we observed live (the migration would be killed mid-build). **Rejected** — the whole point is a non-blocking primitive per target.
- **PlanetScale deploy request vs a direct `ALGORITHM=INPLACE` against the PS data plane.** PlanetScale blocks direct online `ALTER` for large-table schema changes (they must go through the platform's online-schema-change machinery), so `ALGORITHM=INPLACE` is not the PS path — the deploy request IS the non-blocking primitive on that target. `ALGORITHM=INPLACE` remains the right primitive only for **self-hosted** MySQL. This is why the apply tier is target-specific rather than one MySQL path.
- **`pscale api` / raw Public API calls as the PS apply driver instead of `planetscale-go`.** PlanetScale exposes a `pscale api` escape hatch and a documented REST Public API. Usable as a lower-level fallback, but `planetscale-go` is the typed, maintained SDK that the pscale CLI itself is built on (`client.DeployRequests.Create` / `Apply`, `client.DatabaseBranches.Create` / `Diff`), so it is the primary integration; the raw API is the documented fallback if a needed operation is missing from the SDK.
