# ADR-0104: Pipelined / concurrent CDC apply for the MySQL target

## Status

**Accepted (2026-06-19) — the concurrent key-hash apply path is implemented, shipped in v0.99.77, and live-validated; the ineffective Phase-1 commit-pipeline was REMOVED in v0.99.78.** The journey: Phase 1 ("in-order pipelined COMMIT", behind `--apply-pipeline-depth`) was deployed to the live Track-B cross-region PlanetScale-MySQL link at depth=8 and **measured INEFFECTIVE** — it overlaps only the commit RTT, not the per-batch data execs, so throughput stayed ~serial (~21/s vs ~44/s source; 7 of 8 backends idle). The corrected design replaces it: **concurrent DISPATCH across W key-hash lanes** (`--apply-concurrency W`; merged stream fanned by primary-key hash so same-key ops stay in one in-order lane — the dependent-row hazard a naive concurrent dispatch would hit is structurally impossible — lanes committing concurrently on a dedicated pool, resume position advanced only to a source boundary durable across all lanes via the seq-frontier). **Live-validated on Track B: serial ~52 rows/s → `--apply-concurrency=4` ~206 rows/s (~4.0×), behind shard drains, wedge closes**, and the rig regression cycle confirmed exactly-once + same-key ordering + the serial-vs-W byte-identical differential. The Phase-1 commit-pipeline (`--apply-pipeline-depth`, `mysqlPipeline`) was removed wholesale in v0.99.78 (zero users; the flag is gone). Remaining follow-up before graduating `--apply-concurrency` out of PREVIEW: per-lane AIMD + tx-killer (#54) convergence (lanes currently use static batch sizing, so a tx-killer abort stops + warm-resumes rather than shrinking in place). The MySQL-target counterpart of [ADR-0092](adr-0092-pipelined-cdc-apply.md) (Postgres pipelined CDC apply). Realizes roadmap [item 23(c)](../dev/roadmap.md) — "Cross-region CDC apply to PlanetScale-MySQL: apply throughput". Extends — does not supersede — [ADR-0017](adr-0017-batched-cdc-apply.md) (batched apply), [ADR-0052](adr-0052-aimd-apply-batch-size-controller.md) (AIMD controller), [ADR-0081](adr-0081-applier-control-plane-extraction.md) (the shared `appliershared.RunBatchLoop` seam), and [ADR-0089](adr-0089-default-adaptive-apply-batch-size.md) (adaptive batch size by default). It composes with the ADR-0089 keyless guard, the ADR-0010 idempotency contract, the ADR-0007 position-then-commit atomicity, and the ADR-0089/§"tx-killer" (`ir.TransactionKilledError` / `OnShrink`, #54) machinery — every one of those is preserved verbatim. **Concurrency chunk** → the `-race` integration gate gates the eventual implementation before any tag is cut.

## Context

### The throughput deficit and the item-23 wedge it causes

On a cross-region multi-shard Vitess/PlanetScale → PlanetScale-**MySQL** `sync`, steady-state CDC apply does NOT keep up with the source write rate. The load-forced `--log-level=debug` capture (2026-06-19, 7 parallel writers: source ~700 rows/s vs cross-region apply ~50–60 rows/s) established the causal chain that makes this more than a performance footnote — it is the root cause of the item-23 per-shard wedge:

1. MySQL apply < source write rate, so one shard's events accumulate a large backlog.
2. vtgate's `MinimizeSkew` (which keeps the merged multi-shard stream commit-time ordered) delivers the MORE-behind shard and **HOLDS** the other.
3. Because apply never catches up, the hold is effectively permanent — a stalled shard on an otherwise-"healthy" stream (no error, heartbeats flowing, the peer shard applying). Observed frozen 12+ minutes with lag growing; it does not self-resolve even after the load is removed.
4. The capture also REFUTED the auto-reopen plan (item 23 B-2): the freeze is delivery-side (vtgate stops delivering the held shard), and a clean warm-resume re-forms the identical skew hold from the persisted positions — so a reopen thrashes uselessly. **The durable fix is apply throughput**: keep apply ≥ source so no large asymmetric backlog forms and `MinimizeSkew` never holds a shard.

The decisive contrast is the same Vitess source pointed at a **PostgreSQL** target (Track B2): it keeps BOTH shards caught up, because PG already has ADR-0092 pipelined apply (~90 rows/s → ~4,400 rows/s, ~48×). MySQL never got the equivalent — ADR-0092 explicitly scoped itself "Postgres only … the `mysql` driver's pipelining story is different and out of scope here." This ADR closes that.

### Why the cold-copy fan-out does not transfer

The cold-copy throughput arc — ADR-0095 auto-shard-by-table, [ADR-0097](adr-0097-parallel-writer-fanout-vstream-snapshot-copy.md) per-table write fan-out, [ADR-0099](adr-0099-cross-table-vstream-copy-concurrency.md) cross-table read concurrency, [ADR-0100](adr-0100-cross-table-vstream-write-concurrency.md) cross-table write pipelines, [ADR-0101](adr-0101-native-mysql-concurrent-cold-copy.md)/[ADR-0102](adr-0102-native-mysql-per-table-write-fanout.md) native W×D — all rest on the fact that a SNAPSHOT has **no ordering constraint**: a row read at the consistent cut can be written in any order, by any worker, because the final target state is the snapshot regardless of application order; PK-hash routing + idempotent upsert make cross-worker interleaving irrelevant. **CDC apply is the opposite**: the change stream is an ordered log of mutations (an UPDATE must land after the INSERT that created its row; a DELETE after the UPDATE), and exactly-once requires the position to advance monotonically and atomically with the data ([ADR-0007](adr-0007-position-persistence.md)). So the fan-out levers cannot be lifted wholesale — any apply concurrency must carry explicit per-key/per-table/per-shard ordering guarantees, which is what this ADR designs.

### Why batching alone hit a ceiling (the ADR-0092 RTT argument, MySQL flavor)

The shared batch loop ([ADR-0081] `appliershared.RunOneBatch`) applies a batch of N changes as **N serial `tx.ExecContext` round trips inside one transaction**, then one position write, then one commit. Batching ([ADR-0017]/[ADR-0089]) amortizes the commit fsync over N changes — the >10× Bug-141 win (240 → ~6,500 rows/s) — but does nothing for the N data round trips: they stay serial, so apply throughput is bounded by `1 / per_row_exec_latency`, and on a cross-region link that latency is dominated by the network RTT. On the live Track-B link, AIMD churned p95 toward ~10 s and shrank the batch toward 1 (the `MinimizeSkew`-feeding apply-deficit), the exact MySQL counterpart of ADR-0092's measured `1 / 0.011 ≈ 90 rows/s` PG ceiling. The wire is the bottleneck, not the target database.

### The MySQL mechanism gap (the load-bearing constraint on the design)

ADR-0092's PG win came from `pgx.Batch` + `SendBatch`: pgx's wire protocol lets the client **queue many statements and flush them in one round trip**, with the server executing them in order and returning all results together. **`go-sql-driver/mysql` has no equivalent.** The MySQL client/server protocol is request-response per statement: each `tx.ExecContext` is a full round trip, and there is no client-side pipeline-flush API on the driver. The driver's `multiStatements=true` (semicolon-joined `Exec` in one round trip) exists, but — exactly like ADR-0092's rejected "simple query protocol" alternative — it forces literal-inlined values, discarding the parameterized `prepareApplierValue` codec path and reintroducing escaping / type-fidelity risk, which is unacceptable for a value-fidelity-critical tool (and it does not pipeline prepared statements, only inlined text). So the *direct* ADR-0092 port — "collapse N serial execs into one pipelined flush on one connection" — is **not mechanically available on MySQL**. The MySQL levers are different, and the design must say so explicitly rather than pretend the PG mechanism ports.

## Decision

Adopt a **phased** MySQL-target apply-concurrency design. The order is chosen so the lowest-risk, highest-RTT-leverage win lands first and later phases are gated on measurement, not speculation.

### The cross-region RTT math (the lever, common to all phases)

Per-batch apply cost on a high-RTT link is roughly `BEGIN_rtt + N·exec_rtt + position_rtt + COMMIT_rtt`. Batching already folds the per-row fsync into the one commit. The remaining serial cost is dominated by the round trips, and the only way to beat `1 / RTT` when you cannot pipeline statements on one connection is to **overlap RTTs across connections** — run K independent in-flight transactions on K dedicated backends so that while transaction A is blocked on its commit RTT, transactions B…K are doing useful work. K connections each at `1 / RTT` give an aggregate `K / RTT` ceiling, until the source rate or the target's execution rate becomes the new bound. The phasing differs only in *what unit* the K transactions are partitioned over — and therefore in *how ordering is preserved*.

### Phase 1 (primary, ship first): in-order pipelined commit on a dedicated pool

Keep apply **strictly ordered**, but overlap the cross-region commit RTTs by allowing a **bounded window of in-flight transactions** committed **in submission order**. The single ordered change stream is sliced into consecutive batches (the existing `RunOneBatch` units); instead of `begin→dispatch→position→commit` blocking the loop end-to-end before the next batch starts, batch *i* is dispatched onto its own pinned backend from a **dedicated pipeline pool** and handed to a commit pipeline that commits batches in order while batch *i+1* is already dispatching. A bounded in-flight window W (and a byte/inflight cap, reusing the ADR-0028 byte accounting) bounds memory and the crash-replay window.

Why this is the primary: it is the most direct analogue of ADR-0092 that MySQL's protocol *does* support (MySQL cannot pipeline statements within one tx, but nothing stops K independent transactions from overlapping their RTTs across K connections), it preserves the global apply order trivially (commit-in-submission-order — a single ordered queue of completions), and it requires no per-key/per-table reasoning. It directly attacks the wedge: a higher aggregate apply rate keeps both shards drained so `MinimizeSkew` never holds one.

The ordering subtlety this phase MUST get right: two consecutive batches can contain dependent rows (an INSERT in batch *i*, its UPDATE in batch *i+1*). Committing batch *i+1* before batch *i* would apply the UPDATE before the INSERT. Therefore the pipeline commits **strictly in submission order** — a backend that finishes its dispatch waits for all lower-numbered batches to commit before it commits. This still overlaps the *dispatch* and *RTT* of later batches with the *commit-wait* of earlier ones (the win), while the commit *linearization point* stays in source order (the correctness). The position written is always the position of the highest *contiguously-committed* batch — never a gap.

### Phase 2 (if Phase 1 is insufficient): per-shard apply pipelines

If the merged single-stream Phase-1 pipeline still cannot keep both shards drained (e.g. one shard alone out-writes the cross-region single-stream apply rate), split apply by **source shard**. Cross-shard rows are disjoint by the hash-vindex (a given PK lives on exactly one shard), so per-shard apply streams touch disjoint row sets and can run as independent in-order Phase-1 pipelines, one per shard. This directly attacks the `MinimizeSkew`-hold causation at its root: each shard drains at its own full rate, no shard's backlog forces a hold on a peer.

Phase 2 is deferred behind Phase 1 because it is structurally heavier (it requires the merged VStream channel to be **demultiplexed by shard** before the applier — today the streamer hands the applier ONE merged ordered channel, and the merge is what makes the single position monotone), and because Phase 1's W-way overlap may already lift aggregate apply above the worst single-shard rate. The cross-shard disjointness that makes per-shard ordering sufficient must be a verified property, not an assumption (see *Exactly-once + ordering plan*).

### Phase 3 (only if measurement demands, likely never for this workload): per-key / per-table apply concurrency

Parallelize across independent keys/tables within a stream, serializing only within a key (so all changes to one PK stay ordered), composing with [ADR-0010] idempotency. This is the most general lever but also the most invasive — it needs a per-key in-flight tracker so a second change to an in-flight key blocks behind the first, and it interacts with foreign-key ordering across tables. It is recorded here as the escape hatch for a workload whose write skew is *within a shard* (so per-shard pipelines don't help) and whose hot keys are few, but it is explicitly **out of scope for the first implementation** — Phases 1 and 2 attack the measured (cross-region, multi-shard) cause directly.

### What ships first

Phase 1 only. It is the smallest diff that closes the measured wedge, it mirrors the proven ADR-0092 precedent as closely as MySQL's protocol allows, and Phases 2–3 are deferred behind live measurement on the Phase-1 build (the same "design-gated on the Phase-A finding" discipline item 23 already follows).

### Mechanism — generalize the [ADR-0081] seam, mirror ADR-0092's pool + fallback shape

ADR-0092 already generalized the seam's transaction handle to `appliershared.BatchTx interface{ Rollback() error }` and proved the dedicated-pool + raw-conn-escape + one-time-WARN-fallback pattern on PG. Phase 1 reuses that shape:

- A **dedicated MySQL pipeline pool** (`pipelineDB`, lazy-opened on first pipelined `BeginTx`, mirroring PG's `pipelineDB`/`pipelineCfg`/`pipelinePool`) sized to W backends, separate from the applier's primary `db` (which keeps serving the per-change `Apply` path and the metadata probes — `pkFor`/`colTypesFor`/`tableIsKeyless` — exactly as today). Crucially, on MySQL "pipelining" means **W concurrent ordered transactions across W backends**, not a within-tx statement flush, so the pool needs W connections, not one pinned backend per batch like PG.
- The SQL builders (`buildInsertSQL` / `buildUpdateSQL` / `buildDeleteSQL` / `buildTruncateSQL`) and the `prepareApplierValue` / `prepareValue` codec path are reused **byte-for-byte**. This is the load-bearing correctness invariant ADR-0092 names: the change is *when/where* statements are committed, never *how a value is encoded*. The Bug-6 JSON `CAST(? AS JSON)` WHERE form, the `_binary`-charset avoidance, the row-alias UPSERT (`ON DUPLICATE KEY UPDATE`), and the keyed/keyless dispatch all carry over unchanged.
- The position write rides each batch's own transaction (ADR-0007), `writePositionTx` unchanged. The Bug-56 per-exec / commit watchdog (`txExec` / `commitWithTimeout` / `execTimeoutCtx`) wraps every pipelined exec and commit just as it does serial ones.
- If the pipeline pool cannot be opened (or W resolves to 1), the path falls back to the existing serial `*sql.Tx` batch loop with a **one-time WARN** — loud, never silent, no throughput claim made — identical to ADR-0092's `warnPipelineFallbackOnce`.

The seam stays engine-neutral: the in-order commit-pipeline machinery (the W-deep submission-ordered completion queue) is a candidate for `appliershared` so a future engine can reuse it, but the first cut may live MySQL-side and be hoisted later (the ADR-0081 tiered-extraction precedent).

## Exactly-once + ordering plan (the load-bearing section)

This is where the design earns its keep. Apply concurrency that weakens exactly-once is a non-starter (the project's cardinal tenet). The invariants and how each survives:

- **Same-row sequence + global commit order (Phase 1).** Changes are dispatched in source order and committed **strictly in submission order** (a batch waits for all lower-numbered batches to commit before it commits). So the *linearization point* of every change is in source order — a dependent UPDATE/DELETE can never commit before the INSERT it depends on, even though their dispatch RTTs overlap. The only concurrency is between *dispatch of batch i+1* and *commit-wait of batch i*; the commit order itself is serial. For Phase 2, each shard's stream is independently in-order, and cross-shard rows are disjoint (hash-vindex), so there is no cross-shard ordering dependency to preserve — verified, not assumed (next bullet).

- **Position + data in ONE transaction, monotone across pipelined/per-shard commits (ADR-0007).** Each pipelined batch writes its own position token on its own transaction and commits both atomically. For the **merged multi-shard position** (a `[]shardGtid` compared by the per-shard GTID-superset `PositionOrderer` — every anchor shard must appear in the candidate with a GTID ⊇ the anchor's), the persisted position must never regress and must reflect only **contiguously-committed** work: Phase 1 advances the persisted position to the highest contiguously-committed batch's token (a batch that committed out from under an un-committed predecessor — impossible under strict-order commit, but the invariant is stated so a future relaxation cannot silently break it). For **Phase 2 per-shard pipelines**, each shard advances only its own component of the merged `[]shardGtid` token; the merged token is the per-shard product, and because shards are disjoint the product is monotone iff each component is — which a single-writer-per-shard pipeline guarantees. A per-shard component that would regress, or a cross-shard token that is not a superset of the last persisted one, is a hard error (loud), never a silent overwrite. The `PositionOrderer` superset check is the existing oracle for this and gets a concurrency-stress pin.

- **Crash-and-resume: no double-apply, no skip, no position regression.** On crash, only contiguously-committed batches (Phase 1) / per-shard-committed positions (Phase 2) are durable; the in-flight window's un-committed batches rolled back with their data (ADR-0007). Resume re-streams from the persisted position; the overlap re-applies through [ADR-0010] idempotent upsert (keyed tables) — exactly-once across rollback+resume. Keyless tables stay at-least-once and keep their [ADR-0089] flush-boundary clamp (next bullet). The crash-replay *window* grows from one batch to W batches (Phase 1) — benign under ADR-0010 idempotency for keyed/unique tables, the same trade ADR-0089 already accepted for larger batches.

- **Composes with ADR-0010 idempotency and the ADR-0089 keyless guard.** The keyless guard (`isKeylessInsert` → flush boundary for truly-keyless Inserts) is preserved: a keyless Insert forces its batch to commit as a batch of 1 AND must additionally **drain the in-flight pipeline window to empty before and after** so the keyless single-row commit linearizes exactly as it does under `--apply-batch-size=1` — the pipeline never widens a keyless table's at-least-once replay window past the serial baseline. (Keyless CDC remains at-least-once per Bug 143; this design neither cures nor worsens that.) Keyed/unique tables (Bug-125 class 1 & 2) pipeline normally.

- **Composes with AIMD + the tx-killer convergence (ADR-0052/0089, #54).** See the dedicated subsection below — the AIMD control model changes from "latency vs batch size" to "latency vs (batch size × in-flight depth)", and the `ir.TransactionKilledError` immediate-shrink + `OnShrink`-persist machinery (#54) keeps working per-pipeline-transaction. A tx-killer abort of one in-flight batch rolls back that batch (Vitess `Aborted` = rolled back, position never advanced past it), the pipeline stalls the window at that point, AIMD shrinks immediately, and the warm-resume re-applies the rolled-back batch's source changes at the smaller size — the existing #54 "no explicit split path; the shrink + idempotent re-apply IS the split" argument holds per pipeline.

- **Per-shard lag stays OBSERVABLE (item 23 B-1).** The `shardProgressWatchdog` per-shard source-vs-applied gap WARN must keep firing — the design must not blind it. Phase 1 keeps the merged single channel, so the watchdog is unchanged. Phase 2 demultiplexes by shard, which makes the per-shard applied position *more* directly observable (each pipeline's committed token is its shard's applied GTID); the watchdog wiring must follow the demux so it still compares per-shard source-emitted vs per-shard applied. The whole point is to make apply ≥ source so the wedge never forms — but if it ever does, it must stay visible.

## Consequences

- **A dedicated MySQL pipeline pool** (W backends) is new connection-budget surface. Unlike PG, MySQL has no `TargetConnectionBudgetProber` (the `--max-target-connections` ceiling bounds the *cold-start* copy product, not the apply phase). So the pipeline depth W must be bounded by an explicit, operator-honest limit — an opt-in knob (e.g. an apply-pipeline-depth, zero-value-safe per the v0.99.51 trap: `0`/`1` ⇒ serial, byte-identical to today) — with a WARN naming the W backends it will open, never a silent auto-clamp claim. On PlanetScale-MySQL specifically, vttablet connection limits and the per-tx tx-killer interact with W, so the default W should be conservative (small) and AIMD-aware.

- **The AIMD control model changes.** Today AIMD adapts batch size against p95 commit latency. With a pipeline of depth W, the throughput is `W × (batch / latency)`, so the controller's signal must distinguish "latency too high" (shrink batch) from "in-flight depth saturated by RTT" (the lever pipelining adds). The first cut can keep AIMD governing batch size per-transaction and treat W as a separate operator-set depth (simplest, composes cleanly with #54's per-tx shrink); a later refinement could let the controller adapt W. Either way the p95 measurement must be **per-transaction commit latency**, not wall-time-since-window-open, so the pipeline overlap does not poison the controller (the same item-18 "measure apply work, not the blocked wait" discipline the shared loop already enforces).

- **A second code path through the apply hot path** (pipelined vs serial), mitigated by reusing the identical builders/codecs and keeping the loud serial fallback — the exact ADR-0092 mitigation. The differential oracle (pipelined vs serial produce byte-identical target state) is the primary value-fidelity defense.

- **Testing — pin the class, not the representative** (the Bug-74 corollary applies because this is an apply-path change, even though it does not change value encoding):
  - **Ordering under concurrency:** a change stream with dependent rows (INSERT→UPDATE→DELETE on the same PK split across batch boundaries and across the in-flight window) produces the correct final state; an out-of-order commit is structurally impossible (strict submission-order commit) and is pinned by an instrumented commit-order assertion.
  - **Crash-resume / idempotent-replay:** a hard kill at every point in the in-flight window (mid-dispatch of batch i+1, between commits of i and i+1, after the position write but before the source ack) resumes with no double-apply (keyed), no skip, no position regression; keyless tables stay clamped to the `=1` window.
  - **Merged-position monotonicity:** the `[]shardGtid` persisted token is monotone under the per-shard `PositionOrderer` across a concurrency-stressed run (Phase 2 especially), and a would-be regression is refused loudly.
  - **Value-fidelity differential:** the full type-family × shape matrix (JSON, ENUM/SET, geometry, temporal, decimal, the Bug-6 JSON WHERE form) applied through the pipelined path equals the serial path byte-for-byte.
  - **`-race` integration before tag** — goroutines, shared in-flight window state, a completion queue, crash-recovery → unambiguously a concurrency chunk; the `-race` integration gate MUST be green on CI before the tag is cut (push-first-tag-after, per CLAUDE.md).

- **Out of scope:** Phase 3 (per-key/per-table concurrency) — recorded as a future lever, not built first. Source-side fixes (Vitess throttler / `MinimizeSkew` tuning — infra, item 19) — orthogonal; this ADR fixes the *consumer* side. Changing the keyless at-least-once guarantee (Bug 143) — unchanged. Auto-reopen (item 23 B-2) — refuted; not revived here.

## Alternatives considered

- **Status quo (serial batch + AIMD).** The current ceiling. AIMD amortizes the commit fsync but not the per-row/per-batch RTT, so on a cross-region link it churns toward batch=1 under the apply deficit and the wedge forms. The measured cause of item 23. **Rejected** — it is exactly the problem.

- **Direct ADR-0092 port (one pipelined statement-flush per tx on one connection).** Not mechanically available: `go-sql-driver/mysql` has no `pgx.Batch`/`SendBatch` statement-pipelining API. MySQL's wire protocol is request-response per statement; you cannot queue N parameterized statements and flush them in one round trip on one backend. **Rejected as infeasible on MySQL** — which is why ADR-0092 scoped itself PG-only and this ADR exists.

- **`multiStatements=true` (semicolon-joined `Exec`, one round trip).** Like ADR-0092's rejected "simple query protocol" alternative: it forces literal-inlined values, discarding the parameterized `prepareApplierValue` codec path and reintroducing escaping / type-fidelity risk (and it does not pipeline prepared statements). Unacceptable for a value-fidelity-critical tool. **Rejected.**

- **Bigger batches.** The first thing AIMD already tries, and it is bounded above by the PlanetScale vttablet 20 s tx-killer (a large cross-region batch is killed — the #54 finding) and by in-flight memory (ADR-0028 byte cap). Past the tx-killer ceiling, bigger batches *reduce* throughput (kill → retry → shrink). **Rejected** as the lever — the lever is overlapping more transactions, each small enough to commit.

- **Auto-reopen the wedged stream (item 23 B-2).** REFUTED by the 2026-06-19 load-forced capture: the freeze is delivery-side, and a reopen re-forms the identical `MinimizeSkew` hold from the persisted positions, so it thrashes (reopen → re-wedge → reopen). The real lever is apply throughput so the hold never forms. **Rejected** (see roadmap item 23 B-2).

- **Do nothing / document the ceiling.** Leaves every cross-region multi-shard MySQL `sync` exposed to the per-shard wedge — the common PlanetScale topology (source and target in different regions). The decisive contrast (the same source → PG keeps both shards caught up *because* of ADR-0092) shows the deficit is sluice's to close, not the operator's to work around. **Rejected.**

## Live validation (2026-06-19) — Phase-1 commit-only overlap is INEFFECTIVE; corrected design

Phase 1 was implemented (commit-pipeline on a dedicated W-backend pool, exactly-once-correct — incl. the value-fidelity-review halt-on-first-failure fix — and `-race`-green) and deployed to the live Track-B cross-region Vitess→PlanetScale-MySQL link at `--apply-pipeline-depth=8`. The measurement REFUTES the Phase-1 design:

- **Apply throughput did not rise.** At baseline (~44 rows/s source) the depth-8 pipeline applied **~21 rows/s** — *below* the source rate, lag growing — i.e. it did not even keep up at baseline, let alone close the wedge. Re-measured over multiple windows (steady, not a post-restart transient).
- **The W backends are idle.** Sampling `information_schema.PROCESSLIST` for active (`COMMAND='Query'`) connections to the target showed **0–1 of the 8** pipeline backends doing work at any instant. Seven sit in `Sleep`.

**Root cause.** Phase 1 overlaps only the COMMIT, not the DISPATCH. The shared `RunOneBatch` loop runs on a SINGLE producer goroutine and issues every batch's N data `INSERT`/`UPDATE`/`DELETE` execs **synchronously** on that goroutine (each a cross-region round trip); only the trailing `Commit` is handed to the pipeline worker. So per batch the producer still pays `BEGIN + N·exec + position` round trips serially, and overlapping the one commit RTT with the next batch's dispatch saves ~1 RTT out of ~`N+3` — marginal, and zero once the data execs dominate. The "aggregate `W/RTT` ceiling" in §"The cross-region RTT math" is therefore **wrong for a single-producer dispatch**: W transactions are *in flight* for commit, but their *work* (the data execs) is not concurrent — only ~1 backend is ever executing. Depth=8 ≈ depth=1.

**Corrected design (supersedes Phase 1's "in-order pipelined commit").** The throughput lever requires **concurrent DISPATCH**, not just concurrent commit. But a *naive* concurrent dispatch — hand consecutive batches to W dispatch-worker goroutines that each run `BEGIN → N execs → position-write` on their own backend with only the COMMIT linearized in source order — is **UNSAFE**, a silent-loss hazard, and is explicitly rejected. The reason: dependent same-key operations in consecutive batches. Suppose batch *i* carries `INSERT pk=5` and batch *i+1* carries `DELETE pk=5` (or `UPDATE pk=5`). Under naive concurrent dispatch their two transactions exec at the same time, before either commits. Batch *i+1*'s `DELETE pk=5` runs a current-read against the latest *committed* state — which does not yet contain *i*'s uncommitted insert — so it either (a) matches **0 rows** and the applier's zero-rows-affected tolerance (ADR-0010) swallows it, then *i*'s INSERT commits afterward and the row that should have been deleted **survives** (silent loss), or (b) blocks on *i*'s row lock and, if *i* in turn waits on a lock *i+1* holds, **deadlocks**. In-order *commit* does not save this: the damage is done at *exec/snapshot* time, before the commit ordering ever applies. So **concurrent dispatch is only safe when same-key operations are guaranteed to land on a single in-order lane** — i.e. the concurrency must partition the change stream such that all changes for a given key are dispatched serially, in source order, by one worker. That is precisely what Phase 2 (per **source shard** — the hash-vindex puts a given PK on exactly one shard, so a shard is a same-key-closed partition) and Phase 3 (per **key** — an explicit per-key in-flight lane) provide. Concurrent dispatch is therefore **not a standalone Phase**; it is the mechanism *inside* a correct partitioning. Each partition lane runs its own in-order dispatch→commit pipeline (reusing the already-built FIFO commit worker, halt-on-first-failure, and keyless drain unchanged); lanes run concurrently with each other, which is where the `W/RTT` overlap actually comes from. The substantive additions the re-implementation carries: (1) the **partition router** (shard for Phase 2 / key-hash for Phase 3) that keeps same-key ops on one lane; (2) **per-lane position coordination** — the persisted resume point must reflect only work durable across *all* lanes (Phase 2 exploits the `[]shardGtid` structure: each shard-lane advances only its own component and the merged token is the per-shard product, monotone iff each component is; a key-hash router instead needs a source-order *contiguous-sequence frontier* — the persisted position is the merged VGTID at the highest seq for which every lane has committed all changes ≤ that seq, the slowest lane gating the frontier); (3) the applier's metadata caches (`pkCache` / `colTypeCache` / `keylessCache`) and any per-dispatch shared state — single-goroutine-safe today — made concurrency-safe (a `sync.RWMutex` or per-worker copies). The already-merged commit-pipeline scaffolding is the foundation all of this builds on.

**Plumbing constraint discovered during Phase 2 design (2026-06-19).** Per-shard routing needs each change's *source shard*, but `ir.Change` carries no shard — the VStream reader stamps each change with the **full merged `[]shardGtid`** position (`positionFor` returns `r.currentVgtid`), not the originating shard, and `stampShardChange` / `redactChange` / the dispatch closures all type-`switch` on the *concrete* `ir.Insert`/`ir.Update`/`ir.Delete`. So a mysql-internal wrapper type that carried the shard would silently bypass every one of those switches, and adding a `Shard` field to the engine-neutral `ir.Change` violates the IR-first tenet (a Vitess concept in the shared contract). Net: per-shard routing has **no clean home for the shard identity**. A **key-hash router needs no shard identity at all** — it derives the lane from the change's own primary-key values (already available via `pkCache`), and same key → same hash → same lane gives the same same-key-closed guarantee a shard does, while also generalizing to an *unsharded* source (where per-shard degenerates to a single lane and buys nothing). The cost it adds over per-shard is the contiguous-sequence frontier position machinery (item 2 above) and two ordering interactions to handle explicitly: cross-table **foreign-key** order (a child row may hash to a different lane than its parent) and **source-transaction cohesion** (ADR-0027 — a single source tx's rows must not be split across lanes/target txns). This pushes the recommended first implementation toward **key-hash routing with a seq-frontier** rather than per-shard, *unless* those two ordering interactions prove cheaper to satisfy under per-shard for the live Track-B workload. This is the open fork the implementation resolves; both are recorded so the choice is deliberate.

**Status of the merged code.** The Phase-1 commit-pipeline is on `main` (`4b52f16`), inert at the default depth (0/1 ⇒ serial, byte-identical) and exactly-once-correct, kept as the scaffolding the concurrent-dispatch rework extends. It must NOT be shipped as a throughput fix until concurrent dispatch lands — an operator enabling `--apply-pipeline-depth>1` today gets W idle backends and no improvement. (Roadmap item 23(c) tracks the rework; the per-shard Phase 2 and per-key Phase 3 levers are unchanged and still gated behind the now-corrected Phase 1.)
