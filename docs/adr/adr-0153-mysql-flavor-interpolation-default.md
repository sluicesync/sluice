# ADR-0153: PlanetScale/Vitess flavors default to client-side interpolation (1 RTT per statement)

## Status

**Accepted (2026-07-08).** Audit item N-15(a) (repo-audit 2026-07-08 Â§4.3), operator-approved as "flavor default, explicit DSN wins". Closes the roadmap N-15(a) entry. Companion of ADR-0150 (byte-targeted INSERT batching) and ADR-0139 (CDC statement coalescing) â€” those size the statements; this removes the hidden second round trip each statement paid.

## Context

sluice's MySQL write path never set go-sql-driver's `interpolateParams`, so every parameterized `Exec` was COM_STMT_PREPARE + COM_STMT_EXECUTE â€” two round trips, prepared fresh per call (no statement cache). On a WAN target (the defining PlanetScale/Vitess deployment shape) the hidden PREPARE is one full RTT per statement, in every regime. The original audit entry prescribed a prepared-statement cache and explicitly rejected interpolation ("the value-fidelity choice stands") â€” but the 2026-07-08 real-PlanetScale bench (`workspace/n15a-bench-2026-07-08.md`, reproduced here since workspace/ is gitignored) overturned both halves: the fidelity concern did not survive measurement, and a cache recovers the RTT only on repeated identical statement shapes â€” weakest exactly in the trickle-CDC regime where shapes churn and latency is user-visible, and still 2 RTT on every first-seen shape.

Bench (released sluice 0.99.202, local MySQL 8.0.46 source, real PlanetScale PS-10 us-east target, ~100 ms client RTT, A/B alternated; config A = binary protocol, B = `interpolateParams=true`):

The per-flush p95 delta of almost exactly one median RTT at ~1.2 statements/flush is the smoking gun: the entire gap is the hidden PREPARE. Bench fidelity was byte-exact on 9 families; the residual families were left to this chunk's pin matrix.

## Decision

At the DSN parse choke point (`parseDSNForFlavor` / `parseServerDSNForFlavor`, `internal/engines/mysql/connect.go`): if the engine's flavor is `planetscale` or `vitess` (`Flavor.usesVStream()`) and the raw DSN does **not** explicitly carry an `interpolateParams=` parameter, set `cfg.InterpolateParams = true`. The parsed cfg carries the decision, so every derived connection â€” the ADR-0104 applier lane pool (`pipelineCfg`), VStream shard-discovery and purged-GTID probe clones, snapshot pools, resnapshot re-parses â€” inherits it with no per-site logic. One INFO line per process announces the RESOLVED protocol whatever its source â€” the flavor default or an explicit `interpolateParams=true` opt-in on any flavor (vanilla's documented high-RTT lever, docs/throughput-tuning.md) â€” and every guard below keys on the resolved `cfg.InterpolateParams` state, never on the flavor, so a vanilla opt-in gets identical protections: the driver's unsafe-collation refusal, the maxAllowedPacket ErrSkipâ†’prepared fallback, and the NBE status-flag escaper all read cfg/connection state (flavor-blind by construction; the unsafe-collation refusal is additionally pinned per-flavor), and the `prepareValue` guards (NaN/Â±Inf refusal, the âˆ’0.0 wart) are unconditional across flavors and protocols. The Phase-1 fidelity matrix itself runs on vanilla MySQL containers with explicit `interpolateParams=true` DSNs â€” the encoder evidence is flavor-agnostic and covers the vanilla opt-in verbatim.

**Explicit-DSN-wins contract.** `mysql.ParseDSN` consumes `interpolateParams` into a bool, collapsing "explicitly false" with "absent", so explicitness is detected on the RAW DSN string before parsing (`dsnSetsInterpolateParams`, mirroring the driver's own after-the-last-`/` DSN anatomy). Any explicit setting â€” true or false â€” is respected verbatim. `WithDatabase` re-materializes an explicit `interpolateParams=false` on the derived DSN that `FormatDSN` would otherwise drop (it omits default-valued params).

**Vanilla MySQL stays on the binary protocol.** No evidence of need: vanilla deployments are typically LAN-RTT, where the saved round trip is negligible and interpolated text is fatter on the wire; protocol conservatism is free there, and it keeps a clean A/B escape hatch (and `interpolateParams=true` in the DSN remains a valid manual opt-in on any flavor â€” see docs/throughput-tuning.md). A prepared-statement cache for vanilla can still be layered later if a real deployment shows PREPARE cost there; the bench numbers don't justify building both.

**Zero-value safety (the v0.99.51 trap).** There is no config bool whose zero value could invert the default: the behavior derives from `Flavor` at parse time, and the zero Flavor (vanilla) means no flip â€” every non-CLI construction gets today's behavior.

## Fidelity evidence (the Bug-74 matrix, pinned before the flip)

The interpolation encoder is a second, independent value encoder dispatching on the Go driver-value family (int64 / uint64 / float64 / bool / string / []byte / time.Time / nil). Phase 1 of this chunk pinned every family Ã— {binary, interpolation} Ã— {bulk batched-INSERT, CDC coalesced multirow apply} byte-exact on real MySQL (`interpolation_fidelity_integration_test.go`): microsecond temporals, DECIMAL(20,6)/(65,30) extremes, DOUBLE extremes, VARBINARY/BLOB across all 256 octets, 4-byte UTF-8 with quotes/backslashes/newlines/NUL, BIGINT (UNSIGNED) extremes, per-column NULLs, JSON, BIT(1/8/64), SET, ENUM, GEOMETRY (SRID-prefixed WKB), the zero-date substitutes (ADR-0127 `null`/`epoch` â€” no mode ever writes a literal zero date), plus a text-vs-binary READ decode-parity leg (with interpolation on, arg-bearing SELECTs return text-protocol rows; the pre-existing, contract-permitted int64/uint64/[]byte integer-shape wobble is documented in that test).

The matrix caught two real divergences, both closed in the Phase-1 commit:

- **NaN/Â±Inf (SLUICE-E-VALUE-UNREPRESENTABLE).** Interpolation renders float64 NaN as the bare literal `NaN` â†’ Error 1054 ER_BAD_FIELD_ERROR â€” the code the Bug-F8 schema-drift self-healing deliberately retries â€” so an unguarded NaN (PG DOUBLE PRECISION legitimately carries them) spun the cold-copy reparent-retry window (~30 min, observed live) instead of failing fast. `prepareValue` now refuses NaN/Â±Inf loudly, naming the column, before the driver sees the value â€” identically on both protocols, on the bulk path AND the applier (pinned at the ApplyBatch layer, where the Bug-F8 backoff would otherwise ride the drift classification forever; the applier's defensive cold-cache/unknown-column branches route through the guard too). Known-loud residual: a trigger-CDC reader that delivers the TEXT `"NaN"` (a string, not float64) bypasses the guard by type and draws the server's own strict-mode conversion error instead â€” loud and non-retriable, just less well-named.
- **Negative zero (the `-0` literal mangle).** MySQL's parser reads the interpolated literal `-0` as unary minus on an unsigned zero â€” plain `+0` â€” silently dropping the sign bit the binary protocol preserves (stored `0` vs `-0`). `prepareValue` encodes float64 âˆ’0.0 as the STRING `'-0'`, which rides MySQL's sign-preserving stringâ†’double data conversion on BOTH protocols; the wart fires for every `ir.Float` precision (FLOAT and DOUBLE are both in the matrix â€” Â±float32-range extremes and denormals included, plus an out-of-FLOAT-range loud-refusal pin). Because the wart changes what BOTH legs bind, the differential alone is structurally blind to a server-side sign drop â€” so the pins additionally assert the ABSOLUTE stored text `"-0"` on every leg (bulk, CDC, vtgate) and compare against a raw `database/sql` control that binds âˆ’0.0 as an untouched binary-protocol parameter (the pre-ADR-0153 vanilla behavior). (The bench missed this because a `-0.0` SQL literal can't get a true âˆ’0 into the source in the first place; injecting it takes the binary protocol.)

A third-codec leg covers vtgate (`TestVStream_InterpolationResidualFamilies_ByteExact`, vstream-tagged): interpolated text transits vtgate's OWN parser/normalizer before the backing MySQL â€” an independent literal-handling layer the plain-MySQL matrix cannot vouch for â€” so the review-flagged residual families (BIT, SET, ENUM, GEOMETRY, JSON, DECIMAL(65,30) extremes, float âˆ’0, zero-date epoch) run the same differential against a real vttestserver, with the absolute negative-zero pin.

## Guards

- **NO_BACKSLASH_ESCAPES: no guard needed â€” ground-truthed correct.** go-sql-driver v1.10 switches its interpolation escaper on the SERVER-reported session status flag (`statusNoBackslashEscapes` â†’ `escapeStringQuotes`/`escapeBytesQuotes`, quote-doubling with literal backslashes), which is exactly right under NBE; the flag tracks the session-init `SET sql_mode` sluice injects. `TestInterpolation_NBE_SessionMode_ByteExact` pins a backslash/quote/NUL-heavy corpus byte-exact under an NBE session on real MySQL, with a sanity assertion that NBE semantics actually held on the wire. (This diverges from the task spec's refuse-or-WARN binary: the driver neither errors nor falls back â€” it escapes correctly â€” so a refusal would break a working combination and a WARN would be false. If that pin ever fails on a driver upgrade, the posture flips to a loud connect-time refusal.) The pre-existing SEC-1 DDL-emitter NBE disagreement WARN is orthogonal and unchanged.
- **Unsafe collations AND charsets: the default steps aside; explicit combinations stay loudly refused.** The driver refuses `interpolateParams=true` under the big5/cp932/gb2312/gbk/sjis/gb18030 collation families (trailing-byte 0x5C injection hazard) â€” but its check (v1.10.0 `dsn.go:173`) reads ONLY `cfg.Collation` and entirely ignores the separate `charset=` DSN param (its unexported `cfg.charsets`, which drive the connection's `SET NAMES`), so `...?charset=gbk` sails through the driver with interpolation on. sluice polices BOTH shapes: the FLAVOR DEFAULT skips the flip with a WARN when the collation or ANY `charset=` list entry is in the unsafe family set (`interpolationUnsafeCollations` / `interpolationUnsafeCharsets` â€” any entry disqualifies, since which SET NAMES candidate wins depends on the server at connect time), because a perf default must never turn a previously-working DSN into a connect failure. Operator-explicit `interpolateParams=true` + unsafe collation remains the driver's own loud `ParseDSN` refusal; the explicit + unsafe-CHARSET combination â€” the hole the driver does not police â€” is refused by sluice at parse (`refuseExplicitInterpolationUnsafeCharset`, wired into `parseDSN`/`parseServerDSN` so every connection path is covered), matching the collation posture. Drift posture is pinned two-sided: every copied collation entry must be refused by the driver (`TestInterpolationUnsafeCollations_SubsetOfDriver`), and a future driver-added entry we lack fails loudly at connector build, never silently.
- **Packet headroom: the existing margin absorbs the inflation; the boundary degrades gracefully.** Interpolated text inflates binary-heavy values (worst case ~2Ã— via escaping), but the ADR-0150 ~1 MiB statement byte target (and ADR-0139's equal CDC coalesce cap, `maxCoalescedStatementBytes`) sits so far under every `max_allowed_packet` (MySQL default 64 MiB, PlanetScale 16 MiB+) that even a maximally-inflating multi-MiB single row lands byte-exact â€” the estimator does NOT need to become interpolation-aware. At the true boundary the driver returns `driver.ErrSkip` when the interpolated text would exceed its `maxAllowedPacket`, and database/sql transparently falls back to the prepared path for that statement â€” no silent loss, no spurious "packet too large". Both legs pinned by `TestInterpolation_PacketHeadroom`.

## Coverage declaration (perf-parity matrix row 18)

Reached: MySQL-target **migrate bulk copy**, **sync cold-start** (all VStream-flavor cold copies funnel through the same row writers), **restore into MySQL targets** (same WriteRows), **CDC apply** incl. the ADR-0104 concurrent lanes (pipelineCfg inheritance, pinned), and the auxiliary engine connections under the flavors (schema/verify probes, migration-state, control tables, shard discovery). Not reached, deliberately: vanilla MySQL (above); the redact keyset store (no engine in scope â€” commented at `openKeysetStore`); and **every MySQL ROW-DATA READ session** (OpenRowReader, the binlog/backup snapshot row readers) â€” the read-fidelity exemption: MySQL DOES NOT round-trip FLOAT in its floatâ†’text conversion (ground-truthed 8.0.46: stored float32 8388608 renders "8388610"; DOUBLE renders shortest-round-trip), so a text-protocol chunk read would silently display-round FLOAT columns; those sessions keep the prepared/binary protocol, pinned end-to-end by `TestInterpolationFlavorDefault_RowReaderKeepsBinaryProtocol` (exact float32 back + COM_STMT_PREPAREs on the wire). The SAME ground truth exposed a PRE-EXISTING silent-precision-loss independent of this ADR â€” **FIXED in this branch**: every read page WITHOUT bind args travels the text protocol (the arg-less full-scan ReadRows on every release, plus a cursor-paged read's first unbounded page), so FLOAT columns on those pages were display-rounded on every release before this fix (stored 8388608 landed as 8388610 on the target). The fix reads single-precision FLOAT through `(col * 1E0) AS col` â€” a version-universal DOUBLE promotion (CAST AS DOUBLE is 8.0.17+ syntax and 5.7-compatible sources like Aurora MySQL 2.x are on the managed-guides roadmap; `+ 0E0` would be WRONG: IEEE âˆ’0.0 + 0.0 = +0.0 drops the sign, while multiplication preserves it) â€” in the shared read projection (`selectColumnExpr`, the same seam as the Vector-A temporal CAST â€” one fix point covers full-scan, chunked, binlog/backup snapshot readers, which all construct RowReader): the float32â†’double widening is exact and sign-preserving, and DOUBLE's text form is shortest-round-trip, so text pages are now bit-exact. DOUBLE columns need no detour (shortest-round-trip printing, pinned). Pinned by `TestRowReader_FloatFullScan_ExactRoundTrip` (failing-first raw-SQL repro of the server's rounding + the float32 torture matrix bit-exact through the REAL full-scan reader + chunked/binary cross-check + NULL legs) and the vstream test's vtgate promotion-parsing leg. Affected surface, precisely: MySQL-source FLOAT columns read on ARG-LESS pages â€” full scans and first unbounded cursor pages â€” on all prior releases; bound pages were prepared/binary and exact. The read-session interpolation exemption above is retained as defense in depth (the prepared path's fidelity never depends on the server's text formatter); revisit with RTT evidence. The wire pin (`TestInterpolationFlavorDefault_ProtocolPin_EndToEnd`) observes Com_stmt_prepare = 0 on the default path at a real server, through the engine registry (the Bug-180 layer discipline).

## Alternatives considered

- **Per-connection prepared-statement cache** (the audit's original prescription): recovers the RTT only for repeated identical shapes; weakest in the trickle regime (1â€“4-row coalesced UPDATEs, alternating kinds) where latency is user-visible; still 2 RTT on first-seen shapes; and it is new stateful machinery with its own invalidation bugs. Interpolation is one driver flag with a pinned encoder. A cache remains available as a later vanilla-side layer if evidence appears.
- **Interpolation everywhere (vanilla too):** rejected â€” no measured need at LAN RTTs, fatter wire text, and keeping the maximally-conservative default where it costs nothing preserves an A/B escape hatch.
- **A sluice-level config flag:** rejected â€” the DSN param already exists, is driver-native, and survives sluice's DSN parsing; a second knob would add a zero-value trap surface for no expressiveness.
