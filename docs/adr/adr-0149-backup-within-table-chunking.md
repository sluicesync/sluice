# ADR-0149: Backup within-table read chunking

- **Status:** Accepted
- **Date:** 2026-07-04
- **Related:** ADR-0019/0096 (migrate within-table chunk strategies — MIN/MAX divide + sampled keyset), ADR-0123 (budget-wide chunk work-stealing / the copyGate discipline), ADR-0084/0088 (backup cross-table pool, lazy PG + eager MySQL reader supply), ADR-0079 v1.1 (pinned-reader estimate cost decision), ADR-0085/0086 (backup anchor adoption + progress sidecar — unchanged), Bug 135 (partial-table re-stream posture — unchanged). Roadmap item 54 / perf-parity matrix gap #1.

## Context

The backup row sweep got its cross-table axis in ADR-0084, but each table still streamed through ONE full-scan `ReadRows` (`backup.go`, the pre-ADR `backupTable`). A corpus dominated by one huge table therefore backed up at single-connection speed while `migrate` split the same table into PK ranges (ADR-0019/0096) and `restore` fanned the writes (ADR-0112) — the top-ranked accidental asymmetry in the 2026-07-04 perf-parity inventory (gap #1: "one huge table backs up single-stream while migrate chunks it").

The pieces already existed: the chunk-eligibility classifier and boundary planners are engine-neutral pure functions over the IR (`chunk.go`), the cursor-paged bounded range read is the exact surface migrate's `copyChunk` drives (`readChunkBatch` — SQL-side inclusive upper bound via `ir.BoundedBatchedRowReader`, the ADR-0096 exactly-once contract), and the backup pool already had a snapshot-pinned reader supply (free-reader channel + importer factory). What was missing was the composition, a budget story for the two read axes, and one estimator fix.

## Decision

Reuse migrate's within-table chunk machinery inside the ADR-0084 table pool — no new planner, no forked comparator:

1. **Eligibility (all must hold; anything else = today's single-stream path with the reason logged + observable).** (a) Mode: the snapshot is shareable via the LAZY importer path — `snap.SnapshotName != ""` and the source implements `ir.SnapshotImporterOpener` (`backupWithinChunkingEligible`, the `backupParallelEligible` lazy clause). (b) Table: `canParallelChunkTable` returns a strategy (reused verbatim), the held reader implements `ir.BatchedRowReader` plus the strategy surfaces (`ir.RangeBoundsQuerier` for MIN/MAX divide; `ir.KeysetSampler` **and** `ir.BoundedBatchedRowReader` for keyset — the collation-clip requirement), and `ir.BatchedReadDisqualifier` does not veto. (c) Size: the pre-stream estimate clears `resolveBulkParallelMinRows` (same constant, same many-table adaptation as migrate).
2. **Planning** runs on the table worker's held snapshot-pinned reader, strictly pre-stream (`planBackupTableChunks`): chunk count = `clampParallelChunkCount` (the pure core extracted from migrate's `resolveParallelChunkCount` — ceil(est/threshold), floored at the parallelism, capped at max(64, read budget) so one table can fill the whole budget at the tail), boundaries via `computeChunkBoundaries` / `computeKeysetChunkBoundaries`.
3. **Reading** (`backupTableRanges` + `backupRange`): range 0 rides the worker's held reader; ranges 1..N−1 draw a token from ONE run-wide `copyParallelismGate` sized to tableParallelism × bulkParallelism (table workers draw a BASE token too — the ADR-0123 discipline, so the product ceiling holds by construction and finished tables' tokens are stolen by a big table's surplus ranges) and claim readers through the SAME `acquireBackupReader` path as cross-table peers (idle pooled reader first, importer-minted snapshot-pinned reader otherwise). Each range pages `(pk) > lower AND (pk) <= upper` via `readChunkBatch` with the Bug-68 sticky-error and ADR-0109 sibling-cancel guards.
4. **Budget split** (`resolveBackupReadParallelism`): one source probe, one reserved coordinator slot (ADR-0088 unchanged), then `resolveCopyParallelismBudget` with the axes **swapped relative to migrate**: backup satisfies the shipped CROSS-TABLE axis first (its resolved value is byte-identical to pre-ADR behaviour — a within-first split would have silently shrunk `--table-parallelism` on small-budget many-table sources), and the within axis gets whole multiples of the remainder — which for the headline single-huge-table corpus (tableP clamped to 1) is the entire remaining budget.
5. **Chunk files + manifest.** `chunkFilePath(table, idx)` naming is unchanged; `idx` is allocated from a per-table atomic shared by the table's range workers — collision-free AND gapless, but the manifest's per-entry chunk list accrues in **arrival order** (verified assumption: `restore.go partitionChunks` slices the list positionally, verify/compact/chain walk it, and nothing parses indexes out of filenames — the chunk SET is the contract). All flush/committer plumbing is ONE implementation (`backupChunkStreamer`) shared by the single-stream and range paths; every entry mutation stays under the committer mutex; `finishTable` records the atomic row-count sum. The per-chunk CEK path (`resolveChunkCEK`) was already documented concurrency-safe and is now exercised concurrently by the per-chunk-mode pin. A chunked table may carry up to N−1 extra partial-size chunk files (each range flushes its own tail) — named in the `--bulk-parallelism` help text.
6. **The 59c55e27 estimator trap.** The backup snapshot's primary reader (`postgres/backup_snapshot.go`) was pinned with NO estimator DSN, so the chunk decision's `EstimateRowCount` would have reported 0 for every table — and even with the DSN, a never-ANALYZEd source hits the reltuples −1 sentinel, which pinned readers resolve to 0 (the ADR-0079 v1.1 cost decision for sync-import readers). Backups get the same fix migrate's shared-snapshot primary got: the primary reader is minted with `estimatorDSN` + `estimatorExactCount` (exact COUNT(*) on the SAME fresh off-snapshot conn — the decision is a size estimate with no consistency requirement; bounds and row streams stay pinned), and importer-minted range readers are opted in by the backup factory through the new `ir.ExactCountEstimateOptIn` surface. Sync cold-start minting does not share that path — v1.1 unchanged, pinned by the exporter test.
7. **Resume posture UNCHANGED (Bug 135).** Partial tables re-stream from scratch (the fresh placeholder replaces the prior entry; prior chunk files are overwritten or skipped content-addressed); whole-table fully-complete reuse stays; the anchor-adoption rules (ADR-0085) are untouched — chunking never moves `EndPosition`.

Config: `sluice backup full --bulk-parallelism` (0 = auto = min(8, NumCPU) then budget-split; 1 = off) and `--bulk-parallel-min-rows`, mirroring migrate's flags. `--chunk-size`/`ChunkRows` keeps its meaning — rows per chunk FILE (the roll boundary), orthogonal to the range count.

## Rejected alternatives

- **Per-range chunk-file namespaces** (`<table>-r<k>-<i>.jsonl.gz`). Would make index allocation trivially local, but changes the on-store layout contract every downstream reader (restore, verify, compact, prune, chain) and every existing backup's resume path can see — for zero benefit over an atomic counter, since the chunk list is already a set. Rejected: keep the layout byte-compatible.
- **Per-chunk resume / revisiting Bug 135.** Range-scoped chunks are deterministic-ish (boundaries re-derive from the same data), which tempts resuming a partial table per completed range. Rejected: the boundaries re-derive from a *fresh* estimate/sample on a *new* snapshot — not guaranteed identical — and Bug 135 exists precisely because "the two runs' streams line up" was an accident that stopped holding. Partial tables re-stream from scratch; flush's content-addressed same-path SHA skip already avoids re-uploading identical bytes. Future work if ever needed: persist boundaries in the in-progress manifest and pin them across attempts — a separate ADR.
- **MySQL eager idle-reader stealing.** The FTWRL path opens N coincident readers up front; when fewer tables than readers remain, the idle ones could serve ranges of a big table. Rejected for now: the eager supply is FIXED (no importer to mint more, so a big table could never exceed leftover width), the readers' per-session snapshots coincide only because they were opened inside one FTWRL window (safe, but the pairing logic is new complexity), and the shipping demand signal is PG-shaped. Filed as an explicit gap below.

## Engine × mode coverage (perf-parity working agreement)

- **Reached:** BK × PG-source lazy-snapshot (the `SnapshotImporterOpener` path) — matrix row 2's BK cell, gap #1 closed for this cell.
- **Explicit gaps (deliberate absences, cited):** BK × MySQL vanilla (eager FTWRL — fixed reader supply; see rejected alternative above), BK × non-snapshot fallback (no consistent multi-reader view exists to chunk against), BK × SQLite (single-file source; no importer surface), BK × PlanetScale/Vitess (VStream-COPY backup path, read-side chunking is vtgate-internal — same cell as the CS absence). Each is named at runtime by the one INFO `resolveBackupWithinTable` emits.

## Implementation notes

- The dispatch is observable via the test-only `backupChunkDispatchObserver` seam (mirrors `backupDispatchObserver`), so the integration pins assert which path ran rather than inferring from chunk counts.
- The per-table INFO for "big table would have chunked but the mode can't" is approximated by the ONE run-level INFO naming the mode reason (probing per-table sizes on the eager/fallback readers just to enrich a log line would add reader round-trips mid-FTWRL); the per-table single-stream reasons are DEBUG + observer.
- Concurrency chunk: the `-race` integration gate must run before any tag that carries this (CI-only on the CGO=0 dev box).
