# ADR-0099: Cross-table VStream cold-copy concurrency (K independent COPY streams, disjoint table partition)

## Status

Accepted. Realizes the **"Bounded-parallelism (deferred sub-decision)"** ADR-0095 explicitly left as a follow-on ("K concurrent single-table VStreams, each still constant-memory… the set-min stitch is parallelism-agnostic… does not change the consistency model"). Builds on [ADR-0095](adr-0095-vstream-auto-shard-by-table-copy.md) (the per-table auto-shard COPY pump + the per-shard GTID-set-min stitch — reused verbatim per stream and across all streams), [ADR-0098](adr-0098-auto-shard-aware-vstream-resume.md) (the auto-shard-aware resume — composed per stream), [ADR-0097](adr-0097-parallel-writer-fanout-vstream-snapshot-copy.md) (the write-side fan-out — composes multiplicatively: streams × fan-out workers), [ADR-0071](adr-0071-vstream-snapshot-bounded-memory.md) (bounded-memory streaming pump — one table in flight *per stream*), [ADR-0010](adr-0010-idempotent-applier.md) (idempotent apply — absorbs the per-table overlap each stream's stitch produces), and [ADR-0007](adr-0007-position-persistence.md) (position-then-data — the global position advances only after every stream completes). Does not touch the binlog (vanilla MySQL) or Postgres cold-copy paths.

## Context

### The measured problem (validated; not re-litigated here)

[ADR-0097]'s write-side fan-out gave only ~1.4× on the VStream path in live Track-B measurement. The root cause is structural: the fan-out parallelizes the **write** side, but the VStream COPY has a **single un-splittable vtgate read stream** feeding **one dispatcher** — the write workers starve waiting on the one read stream. You cannot PK-range-chunk a VStream read (the very reason [ADR-0097] exists), so adding write workers behind one read stream hits a read-side ceiling.

The parallelism experiment that scaled **~4× near-linearly** did so by running **N independent processes**, each with its **own** VStream read stream, each on a **different** table. That is the lever: N independent read streams, not N writers behind one read stream. The read stream is the thing that actually scales, because each one is an independent vtgate gRPC session pulling its own table's COPY.

[ADR-0095] already copies tables one at a time, each as its own single-table VStream — but **sequentially**: one Recv loop alive at a time. The dominant cost on a 302 GB source is wire throughput, and a single sequential stream leaves the source/target/network N-fold under-utilized. This ADR internalizes the experiment's N-independent-streams shape into one process: open **K concurrent VStream COPY streams**, partition the in-scope table set into K disjoint groups, each stream reading + writing its own group. K independent read streams at a bounded memory cost of K × one-table.

### Why this is the read-side analogue of ADR-0097

[ADR-0097] is the **write-side** lever (N writers, one read stream, one active table). This ADR is the **read-side** lever (K read streams, K active tables). They are orthogonal and compose multiplicatively: K streams each running an ADR-0097 fan-out of degree D → K independent read streams × D writers each. The connection budget must bound K × D (§3).

## Decision

Make the VStream auto-shard COPY **run K concurrent single-table COPY sub-sequences**, each over a **disjoint** subset of the in-scope tables, on its **own** vtgate gRPC stream. Each sub-sequence is exactly the [ADR-0095] per-table auto-shard pump (constant memory, no interleave, the same liveness/reconnect/checkpoint machinery, [ADR-0098]-aware resume) restricted to its assigned tables. Every per-table snapshot from every stream flows into one shared collection; after **all K** streams finish **all** their tables, the engine computes the per-shard GTID-set **minimum** across the **union** of every stream's every per-table snapshot ([ADR-0095]'s `stitchSnapshotMin`, unchanged) — the single gapless CDC-resume position — and opens the one keyspace-wide CDC tail.

The user runs one `sluice migrate` / `sluice sync` command; K is invisible above the `ir.SnapshotStream` contract, exactly as the per-table loop already is. The orchestrator is untouched (IR-first tenet): it still drains tables one-at-a-time via per-table `ReadRows`; the K-way concurrency lives entirely inside the engine, behind the same `SnapshotStream`.

The six design questions, resolved:

### 1. Table → stream partitioning — deterministic, size-balanced when sizes are known, else round-robin

The in-scope table sequence (`copyTablesSeq`, the orchestrator's filtered schema order) is partitioned into K groups by a **deterministic pure function** `partitionTablesForStreams(tables, k) [][]string`:

- **Coverage + disjointness (the load-bearing silent-loss invariant).** Every in-scope table lands in **exactly one** group — no table in two groups (double-stream of one table — re-emits the same table into the shared `rowBuffer` from two pumps, breaking the single-writer-per-table-queue contract) and no table in zero groups (a silently un-copied table — the worst silent-loss class). Pinned hard by unit test: the multiset union of all groups equals the input exactly, and no table appears twice.
- **Determinism / stability across cold-start vs resume (the second load-bearing invariant).** The partition is a pure function of the **sorted** in-scope table list + K — same input, same assignment, every run. This is the [ADR-0098] table-order-stability concern raised to the K-stream level: on resume each stream must re-derive **the same** table→stream assignment it had on cold-start, or a table could land in a different stream than the one whose cursor names it (or be missed entirely). The function sorts the table list before assigning, so the assignment is independent of the order the caller happens to pass — pinned by a unit test that shuffles the input and asserts an identical partition.
- **Balancing.** When per-table size estimates are available (the source schema's row-count / data-length statistics, already read during schema introspection), the partition is **size-balanced**: a longest-processing-time greedy assignment (sort tables by estimated size descending, assign each to the currently-smallest group) so one stream isn't the long pole because it drew the BLOB table. When sizes are unavailable, it falls back to **round-robin over the sorted list** (deterministic, no size dependency). Either way the balancing is a *pure function of the (sorted tables, sizes, K)* — determinism is preserved because the size estimates are read once at open and passed in, not re-sampled per stream. v1 ships the round-robin floor as the always-available default; size-balancing is wired through the same function signature and engaged when estimates are present (documented; the greedy is a small, well-understood addition that does not change the coverage/disjointness/determinism invariants).

`K` is clamped to `len(tables)` — never more streams than tables (an empty group is wasteful, not wrong, but we avoid it). `K <= 1` collapses to a single group = today's sequential single-stream auto-shard, **byte-identical**.

### 2. Each stream = an ADR-0095 auto-shard sub-sequence over its subset

Each of the K streams is a `copyStream`: its own vtgate gRPC stream (its own `client.VStream(...)` call, its own gРС sub-context for cancellation), its own per-stream cursor / field cache / breadcrumb state, copying its assigned tables one at a time exactly as [ADR-0095]'s `pumpOneTableCopy` does. **One single-table COPY in flight per stream → bounded memory per stream; total = K × one-table** (bounded by `--max-buffer-bytes` per stream — the cap is per-stream, so total RAM is K × cap; documented, and the operator who sets a tight cap for a large K gets K × that). Each stream is scoped by its filter rules to **only its tables**, so vtgate never streams another stream's tables down it (no cross-stream interleave; the ADR-0071 per-stream constant-memory property holds within each stream).

The shared `rowBuffer` is keyed by unqualified table name. Because the partition is disjoint, **no two streams ever append to the same table's queue** — each table's queue has exactly one producer (its stream's pump) and one consumer (the orchestrator's `ReadRows` for that table). The existing per-table `ReadRows`↔pump backpressure (cond/byte-cap) is unchanged; what changes is that K pumps now share the mutex/cond and each backpressures on *its own* active table(s). The single `activeTable` field becomes an **active-table set** (`activeTables map[string]int` refcount) so the ADR-0071 cross-table loud-refusal (which only fires in the legacy single-stream interleave path, already gated on `copyTablesSeq == 0`) stays correctly disarmed under K streams, and the per-stream backpressure-wait is the response for every active table.

### 3. Degree + budget — a DSN knob, zero-value-safe, multiplicative with ADR-0097 fan-out

K is a source-DSN parameter `vstream_copy_table_parallelism=K`, parsed `vstreamCopyTableParallelismFromDSN` — the **same operator-knob pattern** as `vstream_copy_single_stream` and the liveness windows, all of which live entirely inside the engine (the orchestrator is never coupled to a VStream specific). It is **not** a pipeline CLI flag, because the partitioning is a property of the VStream COPY mechanism, not an engine-neutral orchestrator concern (mirroring how [ADR-0095]'s entire per-table loop is engine-internal).

**Zero-value-safe (the v0.99.51 trap).** `resolveCopyTableParallelism(n, nTables) int` maps:

- `n <= 1` → **1** (sequential single-stream auto-shard — today's exact behavior, the safe default for every constructor / test / non-DSN caller that gets the Go zero value),
- `n > 1` → `min(n, nTables, maxCopyTableParallelism)`.

There is no value of the knob that produces "0 streams = copies nothing." The default (absent param / zero value) is **1** — sequential — because the read-side concurrency is a deliberate throughput opt-in for a known-large cross-region copy, and defaulting it >1 would multiply connection pressure on every cold-start by surprise. (Contrast [ADR-0097]'s fan-out, which defaults to 4 because the write workers are cheap and the fan-out is the common win; here each stream is a full vtgate gRPC session, so the conservative default is 1.)

**Connection budget (multiplicative with ADR-0097).** During cold-start copy the target connection count is `K × D` where D is the [ADR-0097] fan-out degree (each of K streams runs its own active table through a D-way write fan-out), plus K source-side gRPC streams, plus the one CDC connection opened later. The connection-budget preflight (`resolveTargetCopyParallelism` in `coldStartOpenTargetWriters`, already run) is extended to account for `K × D` so the loud refusal still fires if the target has no slots and `--max-target-connections` is honored. We do not silently exceed the budget: K is clamped down (with a WARN naming the clamp) if `K × D` would blow the resolved target-connection budget, rather than opening connections the preflight forbade.

### 4. Snapshot → CDC handoff — the correctness crux (set-min across ALL streams' ALL tables)

This is the silent-loss surface. Each of the K streams finishes its subset at its own per-table positions `P_{i}` (table i's snapshot VGTID). The single CDC tail must resume from a position that loses no change after **any** table in **any** stream.

**The stitch: `P_start = ⋂ over (every table in every stream) P_i` — the per-shard GTID-set minimum across the UNION of all K streams' all per-table snapshots.** This is exactly [ADR-0095]'s `stitchSnapshotMin`, fed the concatenation of every stream's `perTableSnapshots`. The stitch is **parallelism-agnostic** (ADR-0095 §"Bounded-parallelism" anticipated this): it selects the per-shard set that is `Contain`-ed by every other captured set, regardless of which stream produced it.

Why this is gapless and overlap-safe — the argument the review must check hardest (and it is *identical* to ADR-0095's, just over a larger candidate set):

- **No gap (no silent loss).** `P_start ⊆` every `P_i` for every table in every stream, so CDC replays the full window `(P_start, P_i]` and everything after, for **every** table regardless of which stream copied it. No committed change after any table's snapshot can be skipped. The forbidden direction — `⋃` (the maximum) — would skip `(P_i, max]` for every lagging table; the stitch never constructs it. The K-stream case does **not** weaken this: the per-table snapshots are still each a real VStream COPY completion point, and the set-min over more of them is still `⊆` each of them. The one new subtlety — the K streams are K **independent** vtgate sessions, not one causally-ordered session, so ADR-0095's "one captured set is always a subset of the others (the earliest)" no longer holds *by causal construction* — is handled by `selectGTIDSetMin`'s existing **loud refusal** when no captured candidate is a subset of all the others: a genuinely disjoint per-shard set (which K concurrent sessions against the same shard primary should still not produce, because they read the same monotonic GTID stream, but *could* in principle) is refused loudly with the shard + divergent sets named, never guessed. The opening-VGTID-of-the-first-table-per-shard floor ([ADR-0095]'s safety net) remains available as the provably-`≤`-everything fallback.
- **Overlap absorbed.** `(P_start, P_i]` re-delivery is the same at-least-once seam every cold-start→CDC handoff already relies on (idempotent upsert, [ADR-0010]; the path already declares `CopyNeedsIdempotentWriter() = true`). K streams produce per-table-sized overlaps, the same kind ADR-0095 already produces — just sourced from K streams instead of one.

**Gapless ordering (ADR-0007).** The global CDC position is recorded onto the `SnapshotStream` (`finishCopyAutoShardConcurrent`) **only after ALL K streams have completed ALL their tables** — never after a subset. CDC therefore never starts past any table in any stream that hasn't fully copied. The happens-before edge to the orchestrator's post-bulk-copy `Position` read is the same as today: the `Position` write precedes the global `copyComplete` flip under `mu`, and the orchestrator reads `Position` only after every `ReadRows` channel has closed (which, per stream, only happens at that table's `tableCopyComplete`).

### 5. Resume — composes with ADR-0098, per stream, over the STABLE partition

On resume each stream resumes its own subset auto-shard-aware ([ADR-0098]). The correctness hinge is **partition stability**: the resume must re-derive **the same** table→stream assignment as cold-start, or the persisted cursor's in-progress table could be assigned to a different stream than the one that re-derives it — missing it or double-assigning it.

This is guaranteed by §1's deterministic partition: `partitionTablesForStreams(sort(tables), K)` is a pure function, so cold-start and resume produce identical groups given the same `(tables, K)`. The persisted cursor names exactly one in-progress table (auto-shard scopes each per-table COPY to one table); the resume:

1. re-derives the K groups deterministically,
2. finds which group contains the cursor's in-progress table (`resolveResumeAutoShard` validates the cursor names exactly one in-scope table and refuses loudly otherwise — unchanged from ADR-0098, now run against the **full** in-scope set before partitioning),
3. seeds **that** stream's auto-shard sub-sequence with the cursor (so its in-progress table resumes from its PK cursor, the tables before it in *that group* re-copy idempotently, the tables after it in *that group* copy fresh), and
4. the **other** K-1 streams re-copy their entire groups from-beginning (idempotent upsert absorbs the re-copy — the same Bug-125 overlap path; ADR-0098's re-copy-the-finished-prefix decision, applied per stream).

Because every table in every stream contributes a captured `P_i` (resumed, re-copied, or fresh), the stitch's correctness on resume is identical to a fresh cold-start. The resume's K (read from the DSN) must match the cold-start's K for the partition to be stable; a changed K is **not** silently honored — it re-derives a different partition. v1 requires the operator to resume with the same `vstream_copy_table_parallelism` (the DSN is operator-supplied and stable across a resume of the same job); a changed-K resume is documented as "restart the cold-start" rather than risked. (A future enhancement could persist K in the position token and refuse a mismatched resume loudly; v1 keeps the token shape unchanged and documents the operator contract — consistent with ADR-0098's "no new token field" decision.)

### 6. Error / lifecycle — any stream's failure aborts loudly, clean K-way cancel, no leaks

- **Any stream's failure aborts the whole copy LOUDLY.** The first stream to error records the terminal error (`failCopy`, first-error-wins) and cancels the shared copy context; the other K-1 streams observe the cancel, unwind their Recv loops, and exit. The global `copyComplete` flips with `err != nil`, so every blocked `ReadRows` consumer wakes and surfaces the error (Bug 68 loud-failure gate, unchanged). **No partial silent success, no global position advance** — `finishCopyAutoShardConcurrent` only records the stitched position when `err == nil` after all K streams joined.
- **Clean ctx-cancel across K streams.** A shared `errgroup`-style join: the K pump goroutines are spawned under one `sync.WaitGroup`; the driver waits for all K, then stitches. On cancel (CloseFn cancels the stream context), each pump's Recv unblocks (its gRPC sub-context is a child of the shared one), each exits, the WaitGroup drains, the driver returns. No leaked goroutines (every pump is joined), no leaked gRPC streams (each stream's sub-context cancel tears its stream down; the shared conn is closed once in `close`), no leaked connections. **v1 reuses ONE gRPC `*grpc.ClientConn`** across the K VStream streams (gRPC multiplexes concurrent streams over one HTTP/2 connection) — so K streams cost one TCP/TLS connection to vtgate, K logical streams. Pinned by a goroutine-count-delta + no-leak `-race` test.

## Alternatives considered

- **Raise ADR-0097 fan-out degree higher.** Measured to give only ~1.4× — the write workers starve behind the one read stream. Does not address the read-side ceiling. **Rejected** (the very finding that motivates this ADR).
- **K streams each on its own gRPC connection (not multiplexed).** Costs K TCP/TLS connections to vtgate for no throughput gain over HTTP/2 stream multiplexing (the read bottleneck is per-stream vtgate-side COPY work, not the shared transport). **Rejected** in favor of one conn / K streams; revisitable if a real workload shows transport-level contention.
- **Round-robin partition only (no size-balancing).** Simpler, fully deterministic, but lets one stream draw all the large tables and become the long pole — defeating the parallelism on a skewed schema (the BLOB-table case). **Kept as the floor** (always available, the no-estimates default) but size-balanced when estimates exist, because the motivating 302 GB source has exactly that skew.
- **Partition by hashing the table name (stateless, no sort).** Deterministic without sorting, but not size-aware and risks an empty group / lopsided split for small K. The sorted-greedy is barely more code and is both deterministic and balanced. **Rejected.**
- **Persist K (and the partition) in the position token; refuse a mismatched-K resume loudly.** Strictly safer for the changed-K resume edge, but adds a token field (ADR-0098 deliberately added none) and the v1 operator contract (resume with the same DSN) covers the real case. **Deferred** as a clean follow-on; v1 documents the contract.
- **A pipeline-level CLI flag instead of a DSN param.** Would couple the engine-neutral orchestrator to a VStream-specific concurrency knob (the partition is a VStream COPY property). [ADR-0095]'s entire per-table loop is engine-internal behind the `SnapshotStream` for exactly this reason. **Rejected** in favor of the DSN knob (the established VStream-tuning surface).

## Consequences

- **PS/Vitess cold-copy throughput rises ~K-fold on the read-bound path** (the experiment's near-linear K-stream scaling), closing the gap ADR-0097's write-side fan-out alone left at ~1.4×. Composes multiplicatively with the fan-out (K read streams × D writers). Bounded by source/target/network capacity and the `K × D` connection budget, not by sluice.
- **Memory is K × one-table** (bounded by `--max-buffer-bytes` per stream). The operator who sets K trades RAM for throughput linearly and predictably; documented.
- **The snapshot→CDC seam is at-least-once across K streams** (per-table overlaps re-applied idempotently). Identical correctness to ADR-0095's single-stream seam, just sourced from K streams; the stitch is the same set-min, parallelism-agnostic.
- **One new silent-loss-class surface, three guards.** (a) The partition covers every table exactly once — unit-pinned (coverage + disjointness). (b) The global position advances only after all K streams complete — structural (the join-then-stitch ordering). (c) The partition is stable cold-start vs resume — pure-function determinism, unit-pinned (shuffle-invariant). All three are the review-critical invariants.
- **Concurrency chunk → `-race`-before-tag.** This adds K concurrent Recv loops sharing one mutex/cond/rowBuffer, a K-way join, and K-way cancel. Per the project rule, the integration **`-race`** gate MUST pass **before** any tag is cut (push-first, tag-after, or `scripts/race-integration.ps1`). CGO is off on the dev box, so `-race` is CI-only here.
- **K=1 is byte-identical to today** (single group = today's sequential single-stream auto-shard). The zero value and the absent DSN param both resolve to K=1, so every non-opt-in caller is unchanged.

## Testing

- **Partition coverage + disjointness (silent-loss class, load-bearing):** `partitionTablesForStreams` over a known table set × K ∈ {1, 2, 3, len, len+1} — the multiset union of all groups equals the input exactly, no table on two groups, no empty input table dropped. K > len(tables) clamps to len groups (no empty group).
- **Partition determinism / stability (silent-loss class, load-bearing):** the same `(tables, K)` produces an identical partition across calls, and a **shuffled** input produces the **same** partition (the sort-before-assign invariant) — this is the cold-start-vs-resume stability pin. Size-balanced and round-robin variants both pinned deterministic.
- **Global stitch = set-min across ALL streams' ALL tables:** drive K concurrent fake streams (the `fakeVitessClient` pattern, one scripted stream per table per group + one CDC handoff), each completing its tables at distinct per-table GTIDs; assert the captured handoff Position is the per-shard set-MIN across the **union** (never the max — the silent-loss guard), and the CDC handoff opens one keyspace-wide stream from it.
- **Global position only after ALL streams complete:** a fake where one stream lags; assert `stream.Position` is unset until the lagging stream's last table completes, and the global `copyComplete` does not flip early.
- **Resume composes per stream over the stable partition (silent-loss class):** a resume with K > 1 where the persisted cursor names a table in a specific group — assert that group's stream seeds from the cursor (in-progress table resumes past its PK, no row-0 restart), the other groups re-copy fresh, every table contributes a `P_i`, and the stitch is non-empty. Re-derives the same partition as cold-start (asserted by the determinism pin).
- **Any stream error aborts loudly, no position advance:** a forced error in one of K streams makes the copy fail (`Err() != nil`), the other streams cancel, and `stream.Position` never advances.
- **ctx-cancel leaks nothing:** cancelling mid-copy with K streams running leaks no goroutines (goroutine-count delta == 0 after) and no gRPC streams / connections, and does not report success.
- **Zero-value-safe K:** a zero-value snapshot stream / absent DSN param resolves to K=1 (sequential, byte-identical to today); K=1 takes the single-group path; neither hangs nor copies zero tables.
- **Integration (`vstream` / `vitesscluster` tag):** cold-copy a multi-table keyspace with K > 1 → every table lands, target `COUNT(*)` + content checksum == source per table (no gap/dup), clean CDC handoff from the stitched position; and an interrupt+resume with K > 1 → still complete + exactly-once (composes with ADR-0098), no ADR-0071 buffer-cap refusal on any stream.

## Silent-loss surfaces for value-fidelity review

Three invariants, each a silent-loss class if broken, called out explicitly:

1. **Every in-scope table is copied by exactly one stream** (coverage + disjointness of the partition). A table in zero streams is silently never copied; a table in two streams double-produces into one shared queue. Guard: `partitionTablesForStreams` coverage/disjointness unit pin.
2. **The global CDC position advances only after ALL K streams complete ALL their tables.** A position recorded after a subset would let CDC start past an un-copied table → silent loss. Guard: structural join-then-stitch ordering + the lagging-stream unit pin.
3. **The partition is stable (identical) across cold-start and resume.** A drifting partition could re-assign the cursor's in-progress table to the wrong stream → missed or double-copied. Guard: pure-function determinism (sort-before-assign) + shuffle-invariance unit pin; operator contract to resume with the same K.
