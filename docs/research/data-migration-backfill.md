# Research: in-place data migration / column backfill (the expand-contract "migrate" step)

Date: 2026-07-14 · Trigger: operator idea — after a schema change shipped via a PlanetScale **deploy request** (e.g. add a new column), users routinely need a **data migration**: backfill/transform data from an old column into the new one. PlanetScale has no native tool for this middle step; Prisma documents it as the expand-contract pattern. · Method: the Prisma data-migration guide + the PlanetScale deploy-request / online-DDL model, crossed with sluice's existing chunked + resumable + PlanetScale-aware machinery. · Status: research + phased recommendation; roadmap item 62 tracks it. Nothing implemented.

## Headline conclusions

1. **The gap is real and sharply shaped.** Expand-contract is **expand** (add the new column) → **migrate** (backfill/transform data into it) → **contract** (drop the old column). PlanetScale gives you expand and contract via deploy requests; the **migrate** step in the middle has no native tool, and it is the hard one. Prisma's documented answer — a one-off `findMany` + `update` loop inside a single transaction — is fine for a toy table and a **scale liability** everywhere else: one big transaction on PlanetScale/Vitess hits the exact synchronous-transaction-time wall a direct `ALTER` hits (errno 3024, proven in ADR-0148), and on any engine it holds locks and bloats the undo/rollback segment. The migrate step *wants* to be batched, resumable, and online — which is sluice's wheelhouse.
2. **sluice already owns the hard machinery; only the surface is missing.** There is **no standalone backfill command** today, but every part exists: the **ADR-0096 keyset chunker** + the resumable-cursor store (ADR-0019/ADR-0111) that already drive bounded, restartable table reads; the **PlanetScale-aware throttle / grow-gate** posture (ADR-0110/ADR-0115); and a **sync-time backfill primitive** — `--BackfillAddedColumn` (ADR-0058 §1c) already issues a bounded PK-cursor `SELECT (pk, new_col)` and emits synthetic UPDATE events to populate a forwarded ADD COLUMN *during a live sync*. A standalone backfill is that primitive's **one-shot, same-database, expression-driven sibling** — reusing the same chunker, cursor, and throttle.
3. **MVP = `sluice backfill`: a same-DB, keyset-chunked, resumable, online-safe UPDATE** that sets a column from a native SQL expression over existing columns, guarded by a `--where` predicate for idempotency. Zero control-plane coupling — it owns the *migrate* step while the operator does expand/contract via PlanetScale deploy requests.
4. **Native per-engine SQL for the expression, not the IR translator.** A backfill runs inside **one** database, so there is no cross-dialect translation to do — the `--set` expression is emitted verbatim in the target engine's own SQL (the same posture `--expr-override` takes for the escape-hatch cases the translator can't cover). A `--dry-run` shows the exact generated `UPDATE` before anything runs.
5. **Orchestrating the full expand→backfill→contract via the PlanetScale API is the stretch, and deliberately separate.** Driving add-column (deploy request) → backfill → drop-column (deploy request) end-to-end couples sluice to the PS **control plane** (a service token + the dev-branch/deploy-request model + the safe-migrations prerequisite — exactly the extra surface that kept ADR-0148's deploy-request index path opt-in and deferred). High value, but a bigger PS-specific project; it belongs in its own phase, not the MVP.
6. **Non-goals stay explicit.** Not a general `sluice exec <sql>` runner; not a cross-engine transform (backfill is same-DB — cross-engine value movement is what `migrate`/`sync` already do); not auto-detecting which columns "need" backfilling (the operator declares `--set`/`--where` explicitly, per the "explicit, never silently auto-handled" tenet); not subsuming the sync-time `--BackfillAddedColumn` (they are siblings for two different moments).

## Design facts that shape the design (verified)

- **The PlanetScale synchronous-transaction wall is why batching is mandatory, not optional.** ADR-0148 ground-truthed that a direct large `ALTER` on PlanetScale runs *synchronously* and hits **errno 3024** (query exceeded the statement/transaction time limit) — which is *why* PS index builds must go through the async deploy-request workflow. A single large `UPDATE … SET new = f(old)` is the same shape of statement and hits the same wall. A backfill that isn't chunked simply does not complete on a large PlanetScale table.
- **`--BackfillAddedColumn` (ADR-0058 §1c) is the proven kernel.** Its help/mechanism: *"a bounded PK-cursor `SELECT (pk, new_col)` against the source [that] emits synthetic UPDATE events to populate the new column … backfill cost is proportional to the table's row count … operators must opt in knowingly."* That is precisely a batched, cursor-driven, opt-in backfill — the standalone command generalizes it from "populate a forwarded column during CDC from the source's values" to "set a column in place from an operator SQL expression."
- **Idempotency is a `--where` guard + a persisted cursor, not bookkeeping.** With `--where 'new_col IS NULL'` (or any predicate that is false once a row is done), a re-run resumes from the cursor and skips completed rows; a re-run of a fully-completed backfill matches zero rows and is a **no-op**. This is the same "resume is safe because the target predicate self-describes doneness" property the CDC apply idempotency relies on.
- **Online safety = bounded batch × optional throttle.** A keyset-ranged `UPDATE … WHERE <pk-range>` of N rows holds a lock only for that batch; sizing N keeps each statement well under the PS txn-time limit and short enough not to block writers. An optional inter-batch pause reuses the AIMD/grow-gate posture already built for CDC apply and cold-copy on managed targets.
- **Verify is a natural pass.** "Did the migrate step finish?" = count rows still matching the `--where` guard; zero means every row is backfilled. This mirrors sluice's existing count/sample `verify` discipline and closes the expand→**migrate**→contract loop before the operator ships the contract deploy request.
- **Prior art is thin, which is the opportunity.** Prisma (and most ORMs) document the app-script approach and stop; gh-ost/pt-osc are *schema*-change tools, not data-transform tools; PlanetScale's own docs point at the deploy-request for schema and leave the data migration to the user. Nobody ships a batched, resumable, online **data-migration** primitive as a first-class product feature — the same "either the opportunity or the warning" read the flat-file research reached, here leaning opportunity because sluice already has the machinery.

## Recommended phases (roadmap item 62)

- **Phase 1 (M) — standalone `sluice backfill`.** Same-DB, single `--driver`/`--dsn`, keyset-chunked, resumable, online-safe `UPDATE`. `--set 'col = <native SQL expr over existing columns>'` (repeatable), `--where '<predicate>'` (the idempotency/scope guard), `--batch-size`, `--dry-run` (print the generated statement + an affected-row estimate), progress + ETA (reuse the copy-phase ETA). Reuse the ADR-0096 keyset chunker + the resume-cursor store verbatim. Pins per the loud-failure discipline: idempotent re-run (completed ⇒ no-op), resume-after-kill lands the exact remaining set, bounded per-batch lock, and a PlanetScale batch that stays under the txn-time wall. Engines: MySQL + Postgres, with the MySQL path covering PlanetScale/Vitess at a batch size tuned below errno-3024.
- **Phase 2 (S) — backfill verify + preview.** A post-pass (or `--verify`) that counts rows still matching `--where` and reports 0 = complete; the `--dry-run` from phase 1 doubles as the pre-flight preview. Closes the expand→migrate→contract loop with an explicit "safe to drop the old column now" signal.
- **Phase 3 (M/L, demand-gated) — PlanetScale expand-contract orchestration.** Drive the whole pattern: open a dev branch, deploy-request the ADD COLUMN, run the phase-1 backfill against the promoted data, then deploy-request the DROP COLUMN — via a thin PS API client + service token (the ADR-0148 pattern, no `planetscale-go` SDK). Couples to the control plane (extra credential surface, dev-branch/deploy-request model, safe-migrations prerequisite), so strictly opt-in and separate from the MVP. This is where sluice becomes an **expand-contract orchestrator**, not just the migrate step — and it shares the credential/deploy-request plumbing with ADR-0148, so the two PlanetScale-control-plane features want a common client.
- **Non-goals (explicit):** a general `sluice exec <sql>`; cross-engine backfill (that's `migrate`/`sync`); auto-inferring which columns need a backfill; and any silent/automatic contract (dropping the old column is always the operator's explicit, verify-gated step).
