# v0.69.x real-user-readiness campaign — close-out

## What this was

An autonomous battle-test → fix → release → re-battle-test loop, launched to answer: *how close is sluice to letting real users try it?* Driven by a real-world adversarial corpus (the 329-table `interesting_create_table_examples_with_unique_constraints` set, MySQL→PG) plus per-shape PG-source fixtures, run against testcontainers `mysql:8.0` / `postgres:16` (vanilla) + a local validation rig.

## Arc

`v0.69.0` (329-corpus pass-3/4 batch — #8–#23, 8 issues) → `v0.69.1` (Bug 68 PG→MySQL multi-dim-array silent total-row-loss + a whole reader-stream silent-swallow class) → `v0.69.2` (Bug 69 unconstrained PG `numeric`) → `v0.69.3` (Bug 70/71/72 PG→PG nullable/multi-dim array COPY, `timetz`, wide `varchar`) → `v0.69.4` (Bug 74 + 73/75/76). Five consecutive Option-C/Option-B-clean releases; each fix Phase-A-pinned + independently reviewed.

Bugs 68–76 were almost all **PG→PG / PG→MySQL** — directions the MySQL→PG corpus structurally cannot exercise; they were found by per-shape PG-source fixtures, one latent peeled per release as deeper/rarer shapes were exercised (each often masked by a prior fix's early-abort).

## The one self-introduced regression (the load-bearing lesson)

v0.69.3's Bug 70 fix introduced **Bug 74**, a CRITICAL silent-loss regression: PG→PG `numeric[][]` (and `uuid`/`inet`/`cidr`/`time` multi-dim) silently flattened to 1-D. The Bug 70 pin and the independent review both covered `int[][]`/`text[][]` but not the `numeric`/string-leaf pgx encode path (pgx's array codec dispatches on the *target element OID*; an identical sluice code path can encode differently per target type). Caught by the next cycle's terminal battle-test *before any "real users go" signal*; v0.69.3 got a public correction banner (now pointing at v0.69.4); forward-fixed in v0.69.4; review gap closed by maintainer-personal Docker re-verification.

→ Codified as a permanent discipline in `CLAUDE.md` ("Pin the class, not the representative" — every family-dispatched encoder/codec change must be pinned per type *family* × {scalar/1-D, multi-dim, NULL-element}).

## Residual known limitations (all LOUD, documented, no corruption — not blockers)

`varchar(N)[]`/`char(N)[]` PG→PG element-length DDL-emit (`22023`); unbounded `varbit` → COPY-time `22001` vs preflight; composite / `interval` / `money` / enum-array / multi-dim `bit[]` cross-engine refusals. All fail loudly with zero data loss; documented in `docs/type-mapping.md`.

## The 329-corpus question (asked repeatedly — answer here for future runs)

~320/329 migrate clean MySQL→PG; ~9 are genuine untranslatable-construct schemas **correctly loud-refused before any DDL** (`regexp_like`, `str_to_date`, `substring_index`, `unhex`, `CAST AS UNSIGNED`, spatial `POINT`, gen-col-ref-gen-col — the gate only scans DDL-embedded DEFAULT/GENERATED/CHECK bodies, not data/queries). That split has been **byte-identical to the v0.68.2 baseline every release** (no narrowing/widening). The loud-failure (names construct + table + remedy at preview *and* preflight) is the by-design awareness mechanism; these constructs are uncommon in real app DDL (the corpus over-represents them by design).
