# Prep: translation policy edges

Roadmap reference: [docs/dev/roadmap.md §7](../roadmap.md). Related: [docs/type-mapping.md](../../type-mapping.md), [docs/value-types.md](../../value-types.md).

## Goal

Close the cross-engine bug surface that hasn't been covered by the existing integration tests. Each of these is a small policy decision baked into the engine pair; bundled because they share a pattern (a type or feature that exists in one engine and needs an explicit policy on the other) and because shipping them together produces a coherent "v1 cross-engine compatibility" deliverable.

The roadmap names five items. After auditing the current code state, the work splits into two prongs:

1. **Verification of policies already implemented.** Three of the five items are already wired through the engines but lack cross-engine integration coverage. The fix is to extend the cross-engine integration tests to exercise each.
2. **Implementation of the remaining gaps.** Two items have real implementation work — a `now()` → `CURRENT_TIMESTAMP` default-value translator (PG → MySQL direction) and a post-bulk-copy identity-sequence-sync phase on the Postgres side.

## Files to add / touch

For the verification prong (items 1, 2, 3):

- `internal/pipeline/migrate_cross_integration_test.go` — extend the existing MySQL→PG seed schema to include UNSIGNED INT, an ENUM column, and a JSON column. Assert the IR shape on the PG target after migration: `Decimal{Precision: 20, Scale: 0}` for UNSIGNED, `Enum` with the right values, `JSON{Binary: true}`.
- `internal/pipeline/migrate_pg_to_mysql_integration_test.go` — extend with the reverse direction where it makes sense (PG numeric+enum back to MySQL is a separate test scope, but JSONB → MySQL JSON is a clean addition).

For the implementation prong:

**Default-value translation (item 4):**

- `internal/engines/mysql/ddl_emit.go` — extend `emitDefault` (or add a new helper called from `emitColumnDef`) with a small lookup table that translates known PG default expressions to their MySQL equivalents:
  ```
  "now()"             → "CURRENT_TIMESTAMP"
  "current_timestamp" → "CURRENT_TIMESTAMP"  // already passes; defensive
  ```
  Anything not in the lookup table passes through verbatim — that's the documented policy ("loud failure beats silent corruption").
- `internal/engines/mysql/ddl_emit_test.go` — table-driven unit test covering `now()` translation and the passthrough path.

**Identity sequence sync (item 5):**

- `internal/engines/postgres/schema_writer.go` — new `SyncIdentitySequences(ctx, schema)` method. For each `IDENTITY` column in the schema, execute:
  ```sql
  SELECT setval(
      pg_get_serial_sequence('schema.table', 'id'),
      COALESCE((SELECT MAX(id) FROM schema.table), 0) + 1,
      false  -- next nextval returns this value
  );
  ```
  Iterate per-table, per-identity-column. The `+ 1` plus `is_called=false` semantics means the next `nextval` returns `MAX(id) + 1` cleanly. `COALESCE` covers empty tables.
- `internal/ir/interfaces.go` — add `SyncIdentitySequences` to the `SchemaWriter` interface. MySQL's implementation is a no-op (MySQL's AUTO_INCREMENT counter is automatically updated by direct INSERTs of explicit values, so no post-copy sync is needed).
- `internal/pipeline/migrate.go` (`runBulkCopy`) — add a new phase between bulk-copy and CreateIndexes:
  ```
  Phase 3.5: SyncIdentitySequences(ctx, schema)
  ```
  Naming the phase explicitly in the orchestrator keeps it visible alongside the existing 1/2/3 schema-apply phases.
- `internal/engines/postgres/schema_writer_test.go` — unit test for the SQL builder. Integration: the existing PG→PG and MySQL→PG cross-engine tests already insert explicit IDs that go through GENERATED BY DEFAULT AS IDENTITY; add an assertion at the end that runs `SELECT nextval(...)` on the target and confirms the next ID would not collide with bulk-copied rows.

## Open questions for the user

1. **`SyncIdentitySequences` on the IR `SchemaWriter` interface vs. a separate `IdentitySync` interface.** Adding it to `SchemaWriter` forces every writer implementation to provide it (MySQL would be a no-op stub). A separate optional interface (`type IdentitySyncer interface { SyncIdentitySequences(...) error }`) keeps `SchemaWriter` lean. *Recommendation:* add to `SchemaWriter` — same pattern as the existing three-phase methods (`CreateTablesWithoutConstraints`, `CreateIndexes`, `CreateConstraints`); MySQL's stub is a one-line return-nil that documents why no-op is correct. Confirm?
2. **Phase 3.5 placement.** After bulk-copy but before indexes/constraints (so any post-sync errors don't block index creation), or at the very end? *Recommendation:* between bulk-copy and indexes — symmetric with how the three existing phases are ordered around bulk-copy. Confirm?
3. **`now()` translation scope.** Just `now()` ↔ `CURRENT_TIMESTAMP`, or a broader lookup (`current_date`, `current_time`, `localtimestamp`)? *Recommendation:* just `now()` for v1. Other PG-specific defaults are rarer in the wild and adding them speculatively grows surface without coverage. Add when a real test fails. Confirm?
4. **Cross-engine ENUM with characters needing escaping.** MySQL ENUM values can contain single quotes (`'foo''bar'`); the PG writer's `quoteSQLString` already handles that. *Recommendation:* test it — add a value with a single quote to the cross-engine ENUM seed. Catches escape-handling regressions cheaply. Confirm?
5. **Identity sequence sync on tables with composite PKs that include an identity column.** `pg_get_serial_sequence` works per-column; iterate over every identity column in the table, not just the first. *Recommendation:* iterate. Composite PK with an identity column is rare but legal; doing it right is one extra loop. Confirm?

## Anticipated rough edges

- **Schema reader's `data_type` for unsigned MySQL → reading back from PG**. After MySQL → PG migration, the PG side has `NUMERIC(20,0)`. Reading the PG schema back gives `ir.Decimal{Precision: 20, Scale: 0}` — *not* `ir.Integer{Width: 64, Unsigned: true}`. That's correct: the PG side has no concept of unsigned, so the IR shape after read-back reflects PG's representation. The cross-engine test should assert `ir.Decimal{20, 0}`, not the original MySQL shape.
- **JSON value round-trip across engines**. MySQL JSON values arrive as `[]byte` (the IR contract). PG's writer accepts `[]byte` via pgx and stores it as `JSONB`. The bytes round-trip but PG re-formats them on read (key order, whitespace). The integration test should use a JSON value that round-trips deterministically (`{"a":1}` is fine; `{"a":1, "b":2}` may have key-reordering surprises).
- **MySQL ENUM with leading/trailing whitespace in values.** PG ENUM type values are exact strings; MySQL trims trailing spaces in some configurations. The cross-engine test should pick values without whitespace ambiguity.
- **`pg_get_serial_sequence` returns NULL for non-identity columns.** The setval query needs a guard against this — only run for columns where `ir.Integer.AutoIncrement` is true on the IR side.
- **Empty table after bulk-copy.** The setval expression `MAX(id) + 1` is undefined when MAX is NULL; `COALESCE(MAX(id), 0) + 1` covers it. Test with an empty source table.
- **MySQL's AUTO_INCREMENT on cross-engine PG → MySQL.** When the source is PG identity → target MySQL AUTO_INCREMENT, the MySQL side's auto-increment counter advances naturally as explicit values are inserted (MySQL InnoDB auto-bumps internally). No sync needed on the MySQL side; the no-op `SyncIdentitySequences` documents this.
