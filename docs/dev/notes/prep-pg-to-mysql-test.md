# Prep: Postgres → MySQL integration test

Roadmap reference: [docs/dev/roadmap.md §1](../roadmap.md).

## Goal

A new integration test mirroring `internal/pipeline/migrate_cross_integration_test.go` (which covers MySQL→PG), exercising the *opposite* direction: read schema and rows from Postgres, write them to MySQL, assert the target shape and data.

## Files to touch

New file (only one):

- `internal/pipeline/migrate_pg_to_mysql_integration_test.go` — `//go:build integration`, package `pipeline`.

No new helpers needed. The pipeline package already has both `startMySQL` and `startPostgres` (each returns a source/target DSN pair on a single container plus a cleanup func), `applyMySQLDDL` and `applyPGDDL` (multi-statement DDL apply against the right driver), `closeIf`, `findTable`, `findColumn`, `targetTableNames`, `readAll`. ## Skeleton (mirror the existing cross-engine test)

```go
func TestMigrate_PostgresToMySQL(t *testing.T) {
    // PG source DSN of a fresh PG container
    pgSource, _, pgCleanup := startPostgres(t)
    defer pgCleanup()
    // MySQL target DSN of a fresh MySQL container
    _, mysqlTarget, mysqlCleanup := startMySQL(t)
    defer mysqlCleanup()

    const seedDDL = `
        CREATE TABLE users (
            id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
            email      VARCHAR(255) NOT NULL,
            active     BOOLEAN      NOT NULL DEFAULT true,
            created_at TIMESTAMP(0) NOT NULL DEFAULT CURRENT_TIMESTAMP,
            CONSTRAINT users_email_unique UNIQUE (email)
        );
        CREATE TABLE posts (
            id      BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
            user_id BIGINT NOT NULL,
            body    TEXT   NOT NULL,
            CONSTRAINT posts_user_id_fk FOREIGN KEY (user_id)
                REFERENCES users (id) ON DELETE CASCADE
        );
        CREATE INDEX posts_user_id_idx ON posts (user_id);
        INSERT INTO users (email, active) VALUES
            ('alice@example.com', true), ('bob@example.com', false);
        INSERT INTO posts (user_id, body) VALUES
            (1,'first post'),(1,'second post'),(2,'a post by bob');
    `
    applyPGDDL(t, pgSource, seedDDL)

    // mig := Migrator{Source: pgEng, Target: mysqlEng, ...}.Run(ctx)
    // sr := mysqlEng.OpenSchemaReader(ctx, mysqlTarget); read it back; assert
}
```

Conservative seed deliberately matches the MySQL→PG seed in shape so the assertion code can be near-identical. Spicy types (`JSONB`, native arrays, custom enum types) are deferred — the spine first.

## Anticipated bugs / risks

These are the type-translation paths the test will exercise that haven't been validated yet. Treat each as "if this fails, here's the likely diagnosis":

1. **PG `BOOLEAN` → MySQL `TINYINT(1)`.** Should round-trip cleanly because the IR carries `ir.Boolean` and the MySQL writer already emits `TINYINT(1)` for it. Verify by asserting `users[0]["active"]` is `true` (Go bool) on the MySQL side.
2. **PG identity column → MySQL AUTO_INCREMENT.** PG `GENERATED BY DEFAULT AS IDENTITY` should land as `ir.Integer{Width:64, AutoIncrement:true}`, and the MySQL writer already emits `AUTO_INCREMENT`. If the PG SchemaReader doesn't set `AutoIncrement: true` for identity columns, this will surface as a missing AUTO_INCREMENT on the MySQL side.
3. **PG `TEXT` → MySQL `TEXT`.** PG TEXT has no length; MySQL TEXT is sized (TINYTEXT/TEXT/MEDIUMTEXT/LONGTEXT). The IR uses `TextSize` buckets; the question is whether the PG reader maps PG TEXT to `TextLong` or `TextRegular` — `TextLong` is safer (4 GB) and matches the practical PG semantics.
4. **PG `VARCHAR(255)` → MySQL `VARCHAR(255)`.** Should be straightforward. `ir.Varchar{Length: 255}`.
5. **FK ON DELETE CASCADE survives both readers and both writers.** The MySQL side already passed this test in the MySQL→PG direction; the PG SchemaReader populates `ReferencedSchema = "public"` (its actual schema) and the MySQL writer ignores `ReferencedSchema`. Should be fine.
6. **`CURRENT_TIMESTAMP` default.** PG's reader will produce `DefaultExpression{Expr: "CURRENT_TIMESTAMP"}` or `now()`. The MySQL writer needs to translate the latter to `CURRENT_TIMESTAMP` if it appears. *This is the most likely place to find a bug.* If failure surfaces as "Invalid default value for `created_at`", the fix is a small lookup table in the default-value emitter.
7. **PG `BIGINT` (signed) → MySQL `BIGINT`.** Should be clean. UNSIGNED isn't on the PG side so no policy question.

## Open questions for the user

None should be blocking — the conservative seed avoids the policy edges. If a bug surfaces in #6 (CURRENT_TIMESTAMP), the fix is well-scoped. If anything spicy comes up, fall back to documenting it in the test as a known-broken case rather than fixing in the same chunk.
