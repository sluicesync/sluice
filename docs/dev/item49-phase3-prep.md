## (b) Capture-payload trimming — MODERATE, self-contained, tackle first
**Why:** the capture trigger (`sqlite-trigger/setup.go` ~L328 `CREATE TRIGGER … AFTER INSERT/UPDATE/DELETE`, image built by `captureImageExpr` ~L339) writes a full `(typeof, text/hex)` json_object of EVERY column on EVERY change — and on UPDATE it stores BOTH before+after images. Write amplification; on D1 it's billable rows-written/storage.
**What (first cut — trim the redundant image):** analyze what the applier actually consumes. `sqlite-trigger/cdc_reader.go` (`decodeCell`, the `json_object('t',typeof,'v',value)` parse ~L424) builds the `ir.Change`; the `ChangeApplier` applies it idempotently keyed on PK. If the applier needs only (PK + AFTER-image) for INSERT/UPDATE and (PK only) for DELETE, then: drop the DELETE full-before-image down to the PK columns, and drop the UPDATE before-image entirely (keep PK+after). That's the cleanest first cut. A harder follow-up is changed-columns-only on UPDATE (SQLite triggers can't cheaply diff columns — needs `UPDATE OF col` triggers or per-column CASE).
**Gotchas:** confirm the applier doesn't rely on the before-image for anything (conflict detection? no — it's idempotent upsert). Exactly-once + `(typeof,text/hex)` fidelity for kept columns must hold. Value-fidelity-review. Changing the trigger DDL is a `trigger setup` schema change → bump the setup fingerprint / document a re-setup.

## (a) Schema-change forwarding — BIG, needs its own ADR
**Why:** SQLite/D1 have NO DDL triggers, so a source `ALTER TABLE` is never captured by the data-capture triggers. Today the captured-column **fingerprint guard** (ADR-0135) detects the drift and **REFUSES LOUDLY** (schema-drift refusal) rather than forwarding it. Phase 3(a) forwards it instead.
**What:** (1) detect drift — already done (fingerprint mismatch, in `cdc_reader.go`); (2) diff the current source schema (`sqlite_master` / `PRAGMA table_xinfo`, via the `sqlite`/`d1` reader) against the captured schema → the DDL delta; (3) apply the delta to the TARGET via the existing schema-forward path (see item #24 / ADR-0103 mid-sync index-DDL forwarding + `internal/pipeline/migrate_schema_forward_helpers*.go` — the PG/MySQL CDC schema-forward is the model); (4) re-setup the capture triggers on the source (drop+recreate for the new column set) and update the fingerprint.
**Gotchas / why it's ADR-worthy:** SQLite `ADD COLUMN` is easy (append); DROP/RENAME/MODIFY COLUMN use SQLite's limited ALTER (often a full table-rebuild) — decide which shapes forward vs loud-refuse. The model is the PG/MySQL CDC schema-forward (item #24, DDL-triggered), but here drift is detected by fingerprint-poll (no DDL event), so the *trigger* is different. This is the trigger-CDC analog of #24. Design the drift→diff→forward→re-setup flow + supported-DDL matrix in a new ADR before coding.

## (c) Replica-aware poll — MODERATE, D1-specific, NICHE (lowest priority)
**Why:** the `d1-trigger` poll uses D1's DEFAULT primary (strongly-consistent) — NOT Sessions/read-replica routing — because commit-order = id-order holds at the write-serialized primary but wobbles against a lagging replica (ADR-0136 §4). A replica poll risks out-of-order / missed-recent-commit reads.
**What:** an opt-in replica-aware poll using D1's **Sessions API** (bookmark-based read-your-writes for monotonic reads) + a **safety-lag** predicate (poll `id <= MAX(id) - margin` so a lagging replica can't miss recent commits — the D1 analog of pgtrigger's bigserial safety-lag). Lives in `cdc_executor.go` `d1Executor` + `sqlite/d1_*.go` (D1Conn). Only matters for a high-read continuous D1 sync offloading reads to a replica — lowest value of the three.
