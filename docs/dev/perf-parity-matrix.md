# Performance-technique parity matrix

The living answer to "did this optimization reach every engine and mode it should have?" — created 2026-07-04 after the operator hit repeated cases of an optimization landing in one engine or mode but not its siblings (sync vs migrate, reader vs backup). Ground-truthed at `12feb24a` (v0.99.175); every cell cites code or names the absence. Updated 2026-07-04 by three chunks (perf-parity-checker verified faithful at `20668ed7`): replay-path hardening (gaps #2/#4 closed + the chain-tail identity-sequence correctness fix), migrate quick wins (gap #6 closed; new rows 15 shared-snapshot + 16 analyze-after), and the gap-#3 cold-start defaults chunk (binlog leg resolved; VStream leg reclassified deliberate). Updated again 2026-07-04 by the ADR-0149 chunk: gap #1 (backup within-table read chunking) closed for the PG lazy-snapshot cell, residual engine cells reclassified deliberate. Updated again 2026-07-04 by the ADR-0150 chunk: row 3's MySQL INSERT cells moved from the fixed 500-row cap to byte-targeted ~1 MiB statements (10k-row safety ceiling + 60k placeholder bound), matching the CDC applier's ADR-0139 statement size. Updated 2026-07-08 by the audit-N-15b chunk: new row 17 (target reparent/storage-grow ride-out) declares the ADR-0108/0110/0114 coverage per mode — the MySQL overlapped index builds were the one cell without the ride-out and now carry it on both paths; row 5 + the deliberate-absences entry corrected for the v0.99.200 serial overlap (the old "PS declines overlap" wording lagged the code). Updated 2026-07-08 by the audit-N-15a chunk (ADR-0153): new row 18 (MySQL statement protocol) — the PlanetScale/Vitess flavors default to client-side interpolation (1 RTT/statement; −33% bulk copy, −26% CDC burst drain at ~100 ms RTT on real PS), vanilla stays binary (deliberate), explicit DSN wins. Updated 2026-07-09 by the VStream-COPY FLOAT display-rounding fix: not a new technique but a correctness re-read with a throughput COST — the VStream **CS** (sync cold-start) and **BK** (backup) cells now carry ONE extra source read pass over each repairable single-precision FLOAT table (the `(col*1E0)` exact re-read that repairs the rowstreamer's 6-sig-digit rounding), bounded to the PK + FLOAT columns and cursor-paginated (O(1) mem) on CS / a per-table PK→float map on BK CAPPED at `--float-reread-max-rows` (default 2M, serial sweep → one table at a time; over-cap falls back loudly to rounded/refuse — never unbounded, ADR-0071); `--no-float-exact-reread` removes the pass (rounded, consistent). Documented in throughput-tuning; row 18's VStream FLOAT note updated below. Updated 2026-07-08 by the audit V-1 + P′-1 chunk: new row 19 (batched catalog existence probes) — the MySQL schema phases' three N+1 detect/verify sites (VerifyIndexes, per-table index detect-then-skip, per-FK probe) collapse to one chunked `IN (…)` read apiece; and row 6's CS cell — the VStream snapshot drain (the engines' last unbuffered `ir.Row` channel) now carries the house 64 buffer + 16-row batched pops (was one row per mutex acquisition + broadcast), its former deliberate-absence entry retired by the audit. Updated 2026-07-10 by the audit PERF-P1 + ARCH-F3 chunk: the VStream-COPY FLOAT-repair WRITE side (the sync cold-start **CS** cell's post-COPY PK-keyed UPDATE) moved from one `UPDATE … SET floats WHERE pk` per row (O(rows) round-trips — hours-to-days on a WAN PlanetScale target) to a batched UPDATE-against-a-VALUES/UNION join, 500 rows/statement (O(rows/batch) round-trips, ~500× fewer statements) for BOTH mysql and postgres targets; the repaired values + the set of rows updated are byte-identical (same `prepareApplierValue` shaping — Postgres casts each first-row VALUES column to its target storage type so the coercion matches the per-row assignment). The two engine writers' shared skeleton is hoisted to `internal/engines/internal/floatrepair` (ARCH-F3). Reaches the sync-cold-start FLOAT-repair write cell for BOTH engines; the BK (backup) FLOAT-patch path is in-memory and unaffected. Row 18's VStream CS note updated below.

**Maintenance protocol.** This matrix is hand-maintained and therefore rots — the same failure mode as the CI shard list before `check-shard-coverage.sh`. Two defenses: (1) any chunk that adds or extends a performance technique MUST update its row in the same PR (the CLAUDE.md working agreement), and (2) the `perf-parity-checker` agent re-derives the matrix from code on demand and reports drift — run it after perf chunks land and before releases. A cell that is deliberately absent must say so and cite the rationale (ADR/bug/comment); an uncited ABSENT is presumed accidental and belongs on the gap list below.

## Deliberate absences (cited — do not "fix" without revisiting the rationale)

VStream cold-copy default K=1 (gap-3 survey: K is NOT persisted in the position token, so a default flip silently re-derives a DIFFERENT table→stream partition for any durable-watermark job resumed across the upgrade boundary — missed/double-copied in-progress table, the ADR-0099 §5 silent-loss class; plus the set-min-stitch divergence-refusal class and the `--max-buffer-bytes`/K per-stream split a surprise K>1 would spring on every cold start. The sequential path INFO-names `--vstream-copy-table-parallelism`) · trigger-flavor serial cold-start (single-conn snapshot/anchor consistency argument; INFO added in pgtrigger + sqlite-trigger/D1 openers; parallelizing is the gap-3 residual L item) · olap off chunked reads (Bug 132) · `--upfront-indexes` sync omission (`streamer_coldstart_parallel.go:304`) · PS/Vitess CONCURRENT index-overlap decline (ADR-0080; the serial build-as-copied overlap ships since v0.99.200 — only concurrent ALTERs against vtgate stay declined) · MySQL no pgx.Batch pipelining (coalescing ADR-0139/0140 substitutes) · backup partial-table resume-from-scratch (Bug 135) · sync cold-start import readers decline the exact-count never-ANALYZEd estimate fallback (`ir.ExactCountEstimateOptIn` never applied in `postgres/snapshot_importer.go` minting — the ADR-0079 v1.1 cost decision, reaffirmed ADR-0149 §6 and pinned by `TestExportSnapshot_EstimateRowCount_NeverAnalyzed`; consequence: a fresh never-ANALYZEd PG source cold-starts single-stream where migrate/backup chunk) · backup within-table chunking limited to the LAZY-importer snapshot path (ADR-0149: MySQL-eager FTWRL readers are a fixed supply with no importer to mint range readers — the idle-reader-stealing alternative is a rejected-for-now in the ADR; the non-snapshot fallback has no consistent multi-reader view; SQLite has no importer surface; PlanetScale/Vitess backup rides VStream-COPY where read-side chunking is vtgate-internal — one INFO names the engaged/not-engaged mode per run) · object-store O(tables) manifest rewrites (named wart) · sqlite/D1 not apply targets · D1 within-table chunking deferred (`d1_rows.go:54`) · raw-copy PG→PG-only gates (ADR-0078 "widen on evidence").
