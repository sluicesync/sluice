### 1b. Resilient, resumable VStream cold-start COPY ([ADR-0072](../adr/adr-0072-resumable-coldstart-copy.md)) — *SHIPPED v0.99.4–v0.99.5*

**Why.** A large-table PlanetScale cold-start that hits a transient network fault mid-COPY (`Unavailable: connector reset by peer`) currently has to restart the **entire** copy from row 0 — and for a multi-GB table over a flaky link, the re-copy may fault again before finishing, never converging. Steady-state CDC tailing already resumes from the last GTID and PG→PG copy-pool has AIMD backoff; the VStream cold-start COPY phase is the one long-running path with no resume.

**What.** Three interlocking gaps (full trace in ADR-0072):
- **Gap 1 — gRPC transient classification (SHIPPED).** `classifyReaderError` now honors native gRPC status codes (`Unavailable`/`Aborted`/`Unknown`/`ResourceExhausted` → retriable) instead of only MySQL-`1105`/raw-text shapes, so a VStream `Recv` reset triggers `runWithRetry` instead of failing terminally. `reader_errors.go` + `TestClassifyReaderError_GRPCStatusCodes`.
- **Gap 1.5 — idempotency prerequisite.** `runWithRetry` clears `ResetTargetData` after the first attempt (`streamer.go:884`), so a retry re-copies into a target holding the partial rows → non-retriable `1062` unless the COPY writer is idempotent. **Ties to the Bug 125 idempotent-COPY-writer fix; the two must ship together** or cold-start retry is inert.
- **Gap 2 — resumable COPY cursor (the design).** `shardGtid` drops Vitess's per-table `TablePKs` (the COPY-resume cursor); the position is persisted only at `COPY_COMPLETED`. Carry `TablePKs` in the position (Phase A), checkpoint it on a bounded cadence during COPY (Phase B), and reconnect-and-resume in the snapshot pump (Phase C).

**Gotchas.** Position format gains an additive field — pin that old tokens still decode to "COPY from beginning". Checkpoint cadence trades fault-loss-window vs. control-table write amplification (bounded N-rows/T-seconds, not per-row). Concurrency-adjacent (pump goroutine + in-place reconnect) → **`-race`-before-tag**. Validate jointly with the Bug 125 idempotency fix and ground-truth the resume row-count on a real Vitess target.

### 3. Cross-table copy concurrency for many-table schemas (the pgcopydb many-table gap) — *(a) cross-table pool SHIPPED (ADR-0076); (b) adaptive threshold SHIPPED*

**Why.** A controlled PG → PG benchmark vs pgcopydb (the tool sluice's bulk-copy was modeled on; secondary session, 2026-06-07, draft `docs/comparison-pgcopydb.md` pending review) surfaced a real out-of-box gap on **many-table** schemas: `sluice migrate` has **no cross-table concurrency**. The per-table loop in `internal/pipeline/migrate.go` copies tables **serially**, and `--bulk-parallelism` only splits *within* a table (ADR-0019 PK-range chunking). pgcopydb's `--table-jobs` (default 4) copies multiple tables at once. On 30 tables × 50k rows (each below the 80k within-table-split threshold, so each is both single-streamed *and* serial), sluice's default was **16.1 s vs pgcopydb's 6.1 s (~2.6× slower)** — cores sit idle between tables. On a single large table sluice's default auto-split actually *wins* (5.05 s vs 6.14 s); the gap is specifically the many-medium-table shape. Correctness was never in question — every run was zero-loss (aggregate-checksum verified). This is a throughput/UX gap, not a silent-loss class, so it is demand/competitive-gated, not correctness-gating.

**What.** Two candidate directions (the benchmark draft names both; pick after a design pass):
- **(a) Cross-table copy concurrency — SHIPPED (ADR-0076).** A bounded `errgroup` worker pool over the per-table copy loop (`--table-parallelism`, default `0`=auto→4 à la pgcopydb's `--table-jobs`; `1` disables), composed *with* the existing within-table `--bulk-parallelism`. The two axes' connection budget is split at the SINGLE existing chokepoint (`resolveCopyParallelismBudget`): within is satisfied first, the table axis gets `budget/withinP`, so the product `tableP × withinP <= CopyBudget` holds **by construction** — no global shared gate (the rejected option ii). `--max-target-connections` bounds the product. Resume-under-concurrency extends the chunk axis's clone-under-lock discipline to every table-level state write (peer tables write distinct keys of the same map). The `migrate` path only; the sync cold-start path stays serial by design. Measured ~2.76× speedup (5.88s→2.13s) on the 30×50k benchmark shape, closing the ~2.6× pgcopydb gap; every run zero-loss. Pinned by unit tests (`resolveCopyParallelismBudget` product-bound matrix incl. budget=1 and `--max-target-connections`; table-pool peak-concurrency + all-complete) + 3 PG integration tests (many-table speedup, resume-under-concurrency mid-run kill, combined-budget on a small `max_connections` target).
- **(b) Adaptive `--bulk-parallel-min-rows` — SHIPPED.** `--bulk-parallel-min-rows` is now a `0 = auto` sentinel (was a fixed `80000` CLI default): an explicit value is honoured verbatim, but the auto path scales the within-table-split threshold DOWN as the table count rises (`default/tableCount`, floored at `10000`; `internal/pipeline/chunk.go` `resolveBulkParallelMinRows`). A single/few-table schema keeps the full default (the single-large-table auto-split win is unchanged); a many-medium-table schema now engages within-table parallelism instead of copying each medium table serially + single-streamed. Closes most of the gap with no new concurrency surface (tables still copied serially, so ≤ one table's connections open at once → no budget-math change). The benchmark measured 6.97 s vs pgcopydb 6.1 s (~14 %) with the threshold lowered by hand; this makes that automatic. Pinned by `TestResolveBulkParallelMinRows` (explicit-honoured / monotonic / floored / single-table-unchanged). Leaves the residual serial-table scheduling overhead, which (a) removes.

**Gotchas.** (1) **Resumability + checkpoint state** — the per-table `table_progress` cursor and the parallel-chunk `Chunks` state must stay correct under cross-table concurrency (peer tables writing the state row concurrently; the chunk path already guards its slice with a mutex + `cloneStateForWrite`, extend that discipline to the table axis). (2) **Connection budget** — cross-table × within-table parallelism multiplies open connections; the ADR-0043 budget preflight + the 53300 AIMD backoff gate must govern the *combined* count, not each axis independently, or a wide schema exhausts the target's connection slots. (3) **Snapshot consistency** — ADR-0019 already notes `migrate` uses per-connection snapshots (no shared snapshot for the cold migrate); more concurrent readers widens that window — fine for a quiesced source, but document it (the `sync start` cold-start path pins all readers to one exported snapshot via `ir.SnapshotImporter` and is unaffected). (4) **`-race`-before-tag** — this is a concurrency chunk.

**Operator demand check.** Competitive-gated: the gap only bites a *many-table* cold migrate at default settings, and it's tunable today (lower `--bulk-parallel-min-rows`). Promote when a real operator hits a slow wide-schema migrate, or alongside publishing the pgcopydb comparison (so the doc can say "known, tracked, tunable today"). Pairs naturally with finishing the `docs/comparison-pgcopydb.md` review (the draft's stated owner-review task).

### 3b. PG→PG copy throughput: index-build overlap + identity passthrough — *(a) index-build overlap SHIPPED (ADR-0077); (b) identity passthrough SHIPPED (ADR-0078)*

**Why.** The 110 GB / 43-table at-scale comparison (`docs/comparison-pgcopydb.md`, "At scale" section) showed pgcopydb **~1.75× faster end-to-end** (895 s vs 1564 s) on a realistic mixed corpus — *after* item 3 closed the cross-table gap. The remaining difference is two **structural** things (NOT parallelism — the run is disk-bound at ~100–122 MB/s; tuning workers/splitting did nothing or hurt). Both are concrete, independently-shippable optimizations.

**What.** Two separate chunks (each likely an ADR + a `-race`/value-fidelity pass):

- **(a) Overlap index builds with the copy — the broad win (every engine pair) — SHIPPED ([ADR-0077](../adr/adr-0077-overlap-index-builds-with-bulk-copy.md)).** `migrate` no longer runs a full bulk-copy phase *then* a separate deferred index phase (a sequential ~457 s tail at 110 GB — 29% of total — that pgcopydb hides by building each table's indexes as its data lands). On a PG target it now builds each table's secondary indexes as soon as its copy completes, concurrently with the still-copying tables, via the optional `ir.IncrementalIndexBuilder` surface: the copy pool's per-table success callback (`onTableCopied`) forwards each table onto a buffered channel that a separate index pool drains, both under one errgroup (copy error cancels index pool and vice versa). **Gotchas handled:** (1) the combined connection budget is split ONCE at the single chokepoint — `splitCopyAndIndexBudget` reserves `clamp(0.25×CopyBudget,1,8)` for the index pool and hands the rest to the copy axes, with the invariant `indexBudget + copyBudget' <= CopyBudget` (the index pool sizes from the reserved slice, NOT a self-probe, since copy connections are now open simultaneously); (2) constraints/FKs + identity-sync stay AFTER the combined phase; (3) resumability — `ir.TableProgress.IndexesBuilt` (additive on the wire) drives "copied-not-indexed → re-feed to index pool / fully-indexed → skip", with `CREATE INDEX IF NOT EXISTS` guarding a crash mid-build; (4) `-race`-before-tag (concurrency chunk extending ADR-0076's pool — the `-race` Integration gate is CI-only). PG-only (the index-build-parallelism machinery is PG-only); MySQL falls back to the pre-existing post-copy whole-schema `CreateIndexes`. Pinned by unit tests (`splitCopyAndIndexBudget` invariant matrix; overlap-orchestrator fed-once/channel-closed/error-cancellation) + PG integration tests (overlap-actually-happens via `min(indexBuildStart) < max(copyComplete)` observability seam; zero-loss + all indexes via pg_indexes; resume short-circuits already-indexed tables; combined budget on a small max_connections target; MySQL fallback).

- **(b) PG→PG identity passthrough — closes the per-stream rate gap — SHIPPED ([ADR-0078](../adr/adr-0078-pg-pg-identity-passthrough-raw-copy.md)).** The IR copy decodes every row into the typed IR and re-encodes via `pgx.CopyFrom` — the per-stream gap the at-scale comparison measured (99.5 vs 122.6 MB/s, ~23%). A same-engine fast lane now byte-pipes the raw COPY stream (`COPY (SELECT <readable cols> [WHERE pk range]) TO STDOUT` → `io.Pipe` → `COPY tbl (<non-generated cols>) FROM STDIN`) via pgx's raw `pgconn` `CopyTo`/`CopyFrom`, escaping `database/sql` through the same `Conn.Raw` path `writeViaCopy` uses; composes with the chunk machinery (per-chunk byte-pipe) and the whole-table single-stream path. Engine-neutral OPTIONAL surfaces (`ir.RawCopyExporter`/`RawCopyImporter`/`RawCopyVersionProber`/`RawCopyChunk`/`RawCopyFormat`); MySQL doesn't implement them so a MySQL pair never takes the lane. **The GATE (the correctness-critical surface, one auditable pure predicate `rawCopyGate`):** same-engine (`Source.Name()==Target.Name()`, never naming "postgres") + no `--redact` + no `--type-override`/`--expr-override` + no shard-injection + per-table `identityProjection` (excludes extension/verbatim/bit/geometry — OID/wire-format-sensitive — in v1); any transform present → IR fallback (a miss would silently bypass a transformation, a silent-loss class). The CRUX sub-invariant: exporter projects an explicit SELECT of the source-readable columns (generated EXCLUDED), never a bare `COPY tbl TO STDOUT`; importer's column list comes from the SAME helper so they line up by construction. Cold-start + `migrate`-only (slotted in the ADR-0043 fast-loader branch; resume + sync stay on the IR path). Format: `--raw-copy-format=text|binary|auto`, default text (cross-major safe, pgcopydb's default); binary opt-in on a matched-server-major probe of both endpoints (mismatch → text loudly). Pinned by unit tests (`rawCopyGate` negative matrix per transform + all-clear positive; `identityProjection` per type family; `negotiateRawCopyFormat` downgrade; PG statement-builder projection-exclusion) + PG integration tests (positive zero-loss text+binary with raw-lane-taken seam; chunked; generated-column-recomputed; resume-falls-back; one negative-fallback per transform asserting raw NOT taken AND transform applied). `-race` concurrency chunk (local `io.Pipe`+2 goroutines per chunk inside the existing pools) → the `-race` Integration gate is CI-only. **Value-fidelity follow-up (shipped):** both raw sessions pin `client_encoding=UTF8` (`rawCopyForceUTF8`) before the COPY — the byte-pipe encodes under the source session's `client_encoding` and decodes under the target's, so an asymmetric DSN (`LATIN1` one side only) would silently corrupt non-ASCII text; forcing UTF8 on both makes the stream self-consistent by construction (the one place the raw lane asserts session state rather than passing bytes untouched).

**Operator demand check.** Competitive-gated (the pgcopydb-comparison story), operator-chosen 2026-06-08 to do next. (a) is the higher-value/broader; (b) is the headline per-stream-gap closer for the PG→PG audience. Reproduce the benchmark via `benchmarks/pgcopydb/`. **At-scale result (110 GB, measured 2026-06-09, `docs/comparison-pgcopydb.md`):** combined (a)+(b) closed the gap to pgcopydb from 1.75× → 1.42× (1564→1269 s), all zero-loss. Honest split: disk-bound, so (a) overlap was −60 s (overlapped index builds contend for the saturated disk) and (b) passthrough was the bigger lever at −235 s. Both are automatic/capability-detected (no flag), migrate-cold-start-only.

### 3c. Extend index-build overlap to MySQL targets — *SHIPPED v0.99.30, [ADR-0080](../adr/adr-0080-mysql-index-build-overlap.md) (`988ca79`); combined-`ALTER` follow-up SHIPPED (`591da65`)*

Re-land note: the first land regressed (its own overlap test hit `Error 1089` building a SPATIAL index) and was reverted; root cause was a PRE-EXISTING emit bug — `emitIndexColumnList` rendered the column prefix `(N)` on SPATIAL/FULLTEXT indexes (the reader surfaces a SUB_PART on the geometry column), which MySQL rejects. Fixed by suppressing the prefix for those kinds (also fixes the serial `CreateIndexes` path). Re-landed via PR with the `-race` gate green BEFORE main.

**Measured (`benchmarks/mysql/`):** −13 % median total wall (−21 % best case), MySQL→MySQL 30×1.5 M, 4 indexes/table, 10.7 GiB, disk-bound — a clear repeatable win (the serial post-copy index tail ~1432–1500 s, *larger than the copy*, folds into the copy window). Unlike ADR-0077's PG −60 s regression. The bench surfaced the within-table MDL serialization (per-`ALTER` metadata lock serializes a table's N indexes), which motivated the **combined-`ALTER` follow-up** (`591da65`): build a table's combinable BTREE/UNIQUE indexes in one `ALTER … ADD INDEX a, ADD INDEX b` (one InnoDB scan, one MDL), FULLTEXT/SPATIAL each separate (Error 1795 / no-LOCK=NONE). **Measured incremental: −18.1 % median** (combined median 1783 s vs per-index overlap 2177 s; per-pair −15 %/−21 %; all zero-loss + all-indexes) — a further win on top of the −13 % overlap. See the ADR-0080 amendment.

**Why.** Index-build overlap (ADR-0077, item 3b(a)) is engine-NEUTRAL in concept but PG-only in implementation: it engages only when the target writer implements `ir.IncrementalIndexBuilder`, which today only the PG writer does. MySQL-target migrates therefore still run the pre-ADR-0077 order — full bulk-copy phase, *then* a separate whole-schema `CreateIndexes` tail. A MySQL target would get the same tail-collapse the PG target got, for free on every MySQL-*target* migrate (MySQL→MySQL and PG→MySQL).

**What.** Implement `ir.IncrementalIndexBuilder` (+ the `TableIndexedNotifier` / completion-channel surface ADR-0077 defined) on the MySQL `SchemaWriter`: build a table's secondary indexes via `ALTER TABLE … ADD INDEX` (or multiple in one ALTER) as soon as its copy lands, concurrently with the still-copying tables, under the same combined connection budget (`splitCopyAndIndexBudget`). Constraints/FKs stay a final phase. PKs are already created with the table. The orchestrator change is zero — it already routes to the overlapped phase whenever the surface is present; this is purely the engine-side implementation + its budget participation.

**Gotchas.** (1) MySQL/InnoDB online-DDL `ADD INDEX` has different cost/locking characteristics than PG's `CREATE INDEX` (inplace vs copy algorithm, metadata locks, the row-log) — measure at scale before claiming a win; the PG number doesn't transfer. (2) PlanetScale/Vitess targets route DDL through their own online-DDL/Safe-Migrations machinery, so the overlap may be a no-op or need to defer to the platform — gate on flavor. (3) Concurrent `ALTER`s against the same instance compete for `innodb_buffer_pool` + I/O exactly like the PG disk-contention finding — the at-scale win may again be modest on a saturated disk and larger on fast storage. (4) `-race` chunk (extends the cross-table pool). Its own at-scale `benchmarks/pgcopydb/`-style MySQL measurement before the doc claims a number.

### 3d. Fast cold-start for the `sync` path — the pgcopydb `--follow` equivalent at full speed — *v1 SHIPPED v0.99.29; v1.1 (within-table chunking) SHIPPED v0.99.30 (`509fdeb`), [ADR-0079](../adr/adr-0079-fast-cold-start-for-sync-path.md) + v1.1-CORRECTION addendum*

v1.1 re-land note: the first v1.1 land regressed broadly (relaxing pinned-reader `CountRows` let the during-copy ETA probe race the row-stream on the pinned conn → `bad connection` → 0 rows) and was reverted. Re-designed: `CountRows` stays returning 0 for pinned readers (ETA probe safe by construction); a new `ir.RowCountEstimator` serves the pre-stream chunk decision off a FRESH non-pinned conn. value-fidelity-reviewer SHIP; the exact regression tests + a within-table-chunking-engaged pin pass locally; re-landed via PR with `-race` green on the combined state BEFORE main.

**Why.** This is the migrate-vs-sync gap a user hit: today the three cold-start speedups — cross-table pool (`--table-parallelism`, ADR-0076), index-build overlap (ADR-0077), and PG→PG raw passthrough (ADR-0078) — are **`migrate`-only**. `sync start`'s own initial cold-start calls the serial `runBulkCopyWithOpts` (`streamer.go:2811`, `streamer_multidb.go:587`) "serial by design," so the **copy-then-continuously-follow** workflow (pgcopydb's `--follow`; sluice's `sync start` is the one-command equivalent) does the initial copy on the SLOW path. There is also no clean `migrate` (fast) → `sync start` (CDC, no re-copy) handoff: `sync start` re-runs its own cold-start. So a user who wants *both* a fast initial copy *and* continuous CDC currently can't get the fast copy. pgcopydb gives fast-copy + follow together; sluice should too.

**What — two candidate shapes (pick after a design pass):**
- **(A) Bring the fast machinery to the sync cold-start.** Route `sync start`'s cold-start through the same parallel-pool + overlap + (for same-engine no-transform) raw-passthrough path `migrate` uses. The reason it's "serial by design" is the resumable-checkpoint coupling (the durable mid-COPY cursor that lets an interrupted cold-start warm-resume, ADR-0072) — so the work is making the fast paths checkpoint-compatible (the cross-table pool already threads `state.TableProgress`; the raw byte-pipe is the hard one — it has no per-row checkpoint granularity, so it would degrade to IR on the resumable path and engage only on a fresh/`--restart-from-scratch` cold-start, exactly as the migrate resume path already does).
- **(B) A `migrate` → `sync` position handoff.** `migrate` takes a consistent snapshot; if it recorded that snapshot's CDC start position (binlog GTID / PG LSN at snapshot time) into a manifest, `sync start --position-from-manifest`-style could begin CDC from there WITHOUT re-copying — reusing the existing chain-terminal handoff plumbing (`--position-from-manifest` already does this for backup chains). This keeps `migrate` and `sync` as separate fast primitives and bridges them, rather than fattening the sync cold-start.

**Gotchas.** (A) is the better UX (one command, like pgcopydb `--follow`) but couples the fast paths to the checkpoint machinery; (B) is lower-risk (reuses backup-chain handoff) but is a two-command workflow and needs `migrate` to capture+persist the consistent CDC start position (it takes the snapshot but may not persist a resumable CDC position today — verify). Either way: the snapshot/CDC boundary must stay gap-free (the FTWRL / exported-snapshot boundary discipline from the v0.99.16 silent-loss fix applies). `-race` chunk. Design-pass (Plan agent) before building; own ADR.

### 3e. Parallel writer fan-out on the VStream/CDC snapshot cold-start copy (the PS-MySQL throughput gap) — *ADR-0097 written; implementation in PR (do NOT tag until `-race` green)*

**Why.** Item 3d brought the FAST parallel cold-start to the `sync` path, but only for sources whose snapshot can be PK-range-chunked on the READ side (exported-snapshot PG, raw passthrough). The **VStream/PlanetScale-MySQL** cold-start cannot: vtgate streams the snapshot down one logical channel (no arbitrary range-`SELECT`), and PlanetScale blocks `LOAD DATA LOCAL INFILE`, so the writer falls back to a SINGLE cross-region-RTT-bound batched-`INSERT` connection (~13 GB/h). A live experiment proved N independent `sync` processes scaled near-linearly (N=3 → ~128 GB/h), beating PG single-stream COPY at N≥3–4.

**What.** The WRITE-side analogue of ADR-0019's read-side chunking: one reader goroutine PK-hash-partitions the single incoming snapshot row stream to N per-worker channels; N writer workers each run the existing idempotent batched-INSERT core on their own pinned connection. Engine-neutral pipeline routing + a new opt-in `ir.ParallelIdempotentCopyWriter` capability (MySQL implements; PG stays serial). See [ADR-0097](../adr/adr-0097-parallel-writer-fanout-vstream-snapshot-copy.md).

**Gotchas.** Exactly-once routing (no drop/dup; every row to one worker) is silent-loss-class. No position advances until every worker is durably flushed (ADR-0007 handoff). The ADR-0072 mid-COPY durable-progress checkpoint is DISABLED on fan-out tables (a flat flushed-row count is not order-equivalent to the enqueue frontier across workers → a mid-COPY breadcrumb could checkpoint past an un-flushed early row, silent-loss-on-resume); the whole-table join is the sole durability guarantee for a fanned-out table. PK-hash (not round-robin) so Bug-125 COPY re-emissions of a PK serialize on one worker. Zero-value-safe degree int (`0`→default; the v0.99.51 trap). Resume composes via the ADR-0098 auto-shard-aware resume (the fan-out runs against the one active table at a time, fresh or resumed). `-race` chunk — the integration `-race` gate is the merge gate and MUST be green before any tag.

### 3f. Auto-shard-aware VStream cold-copy resume (bounded-memory resume of a large multi-table keyspace) — *ADR-0098 written; implementation in PR (do NOT tag until `-race` green)*

**Why.** ADR-0095 (item 3, v0.99.62) made the VStream cold-copy auto-shard by table (one single-table COPY at a time, bounded memory) — but only on a FRESH cold-start (`start == nil`). A `--resume` (`OpenSnapshotStreamFromPosition`, a position carrying the ADR-0072 `TablePKs` cursor) fell back to the LEGACY single keyspace-wide interleaved stream → ADR-0071 buffer-cap **crash-loop**. Found live (BUG-CATALOG Bug 156): a resumed 28-table / 302 GB keyspace copy crash-looped (`table "binary_blobs" would buffer … exceeding --max-buffer-bytes … while "audit_trail" is being copied`); loud but FATAL-for-resume — the interleaved path can never make progress on a large keyspace, so resume of exactly the case auto-shard exists for was broken. ADR-0095 called this "a v1 limitation, not a correctness gap" — the live finding proves it is not benign.

**What.** Drop the `start == nil` gate (`autoShard := len(tables) > 1 && !singleStream`); drive the SAME per-table pump on resume. The persisted cursor names exactly the one in-progress table (`resolveResumeAutoShard`); re-copy the tables before it idempotently (Bug-125 upsert absorbs the overlap), resume that table from its cursor (`reopenForTableSeeded`, no row-0 restart), copy the tables after it fresh, and reuse the ADR-0095 per-shard GTID-set-min stitch for the CDC handoff. See [ADR-0098](../adr/adr-0098-auto-shard-aware-vstream-resume.md).

**Gotchas.** Gapless exactly-once across the resume seam is silent-loss-class: every in-scope table must contribute a captured per-table snapshot so the set-min stitch is the cold-start stitch (re-copy the finished prefix rather than skip it — skipping needs a persisted GTID floor + interval-min arithmetic over GTID sets, the Bug-74 risk class). CDC is recorded only at the LAST table's completion → never advances past an incompletely-copied table (ADR-0007). Bounded memory because exactly one single-table COPY is ever in flight. State-shape: NO new token field (completion re-derived from the cursor + table order). Back-compat: a released-code single-in-progress-table cursor round-trips; a legacy multi-table / out-of-scope cursor is REFUSED loudly (never silent re-copy/skip). `-race` chunk — the integration `-race` gate is the merge gate and MUST be green before any tag.

### 11.original (historical context, retained for traceability)

**Why.** Postgres' extensibility — PostGIS, pgvector, pg_trgm, hstore, citext, ltree, pgcrypto, uuid-ossp, etc. — is a major reason operators choose PG specifically. Today sluice's IR doesn't represent extension types, so PG-source columns of those types either get dropped (silent — not OK per the loud-failure tenet) or surface a loud refusal at schema-read time. **For PG → PG syncs where both sides have the same extensions installed, those columns should "just work"** — pass through with native fidelity rather than being treated as hostile. Cross-engine targets (PG → MySQL) keep the loud-failure default; explicit operator-supplied translations stay the escape hatch (parallel to ADR-0016's expression translator catalog).

**What.** Allowlist-based opt-in passthrough mode. New CLI flag `--enable-pg-extension EXT` (repeatable). When set:
- PG schema reader recognizes columns of those extension types and emits them into a new IR variant (`ir.ExtensionType{Extension, Name, ...}`).
- PG schema writer detects same-engine target + matching extension installed + recognized extension type → emits the column verbatim.
- CDC reader / row reader pass through the binary representation as-is (no value-shaping).
- Pre-flight refuses cleanly if target doesn't have the extension installed (`SELECT 1 FROM pg_extension WHERE extname = $1`).
- Cross-engine targets (MySQL): existing loud-error path preserved; operator can supply explicit `--type-override` to cast to a fallback type.

**Three tiers based on what fidelity needs.** Tier classification informs which extensions are cheap to support vs. need real engineering:
- **Tier 1: Type-only.** Extension defines new column types; values are opaque bytes/text from sluice's POV. Examples: hstore, citext, ltree, cube, intarray. ~50-100 LOC per extension.
- **Tier 2: Type + indexes.** Type plus index access methods (GIN, GiST, BRIN-via-extension) operators rely on. Examples: PostGIS (gist), pgvector (ivfflat / hnsw), pg_trgm (gin). ~150-300 LOC per extension; need index-method awareness in schema reader so round-trip preserves indexes.
- **Tier 3: Type + functions in defaults / generated columns.** Extension-defined functions appear in column defaults or generated expressions. Examples: uuid-ossp's `uuid_generate_v4()`, pgcrypto's `digest()`. Adds expression-translator catalog entries for PG-to-PG passthrough; loud failure on cross-engine unless explicitly translated. Per-extension policy.

**Gotchas.**
- Extension version skew (PostGIS 3.4 source → 3.0 target) could surface subtle behavior gaps. v1 only checks extension presence, not version; document the limitation; operator-supplied version-pinning could come later if real-world drift causes pain.
- Per-extension allowlist is more conservative than auto-detect ("we have the same extensions, pass them through automatically") but also higher operator burden. Worth considering an `--auto-pg-extensions` opt-in escape hatch in v2 once the allowlist is battle-tested.
- The PostGIS PG-to-PG passthrough overlaps with the "GEOMETRY/SPATIAL support" entry (which today addresses cross-engine PG ↔ MySQL geometry). Decision: this entry covers the PG-to-PG path for PostGIS as part of the broader extension-passthrough mechanism; the GEOMETRY/SPATIAL entry retains its cross-engine + VStream focus. Bug 26/27 stay parented under GEOMETRY/SPATIAL.

**Operator demand check.** Strong indirect signal — the PG ecosystem has well-documented extension-heavy adoption. pgvector specifically has had massive AI/ML adoption since 2023. PG-to-PG syncs where extensions are blocked are a real pain point. v1 ships the allowlist + 3-5 most-deployed Tier 1+2 extensions; further extensions follow demand.

**v1 shortlist (pinned by item 12's research doc, [`docs/research/pg-extensions-deployment-frequency.md`](../research/pg-extensions-deployment-frequency.md))** — implementation order:

1. **pgvector** (Tier 2) — leads; establishes the Tier-2 index-method-passthrough machinery PostGIS will reuse. Strongest demand trajectory (AI/ML).
2. **pg_trgm** (Tier 2 lite — operator classes only, no new column type) — validates the index path on something simpler than full pgvector.
3. **hstore** (Tier 1) — first Tier 1 ship; type-only opaque-text validation.
4. **citext** (Tier 1) — second Tier 1, even simpler (text + collation); pair with hstore in one PR if the IR shape is clean.
5. **PostGIS** (Tier 2) — last in v1; coordinates with the "GEOMETRY/SPATIAL support" entry above (the cross-engine PostGIS path stays parented there; this entry covers the PG-to-PG passthrough as the broader extension mechanism).

**Surprises documented in item 13's doc**: uuid-ossp + pgcrypto are universal across all four sources but are Tier 3 (function-in-defaults expression-translator work) — strong v2 candidates after the v1 Tier 1+2 machinery is in place. ps-extensions.io's #1 (`pg_search`, 116 votes) is single-vendor (paradedb) and not on Neon — poor v1 fit despite vote count. hstore + citext have clean cross-engine paths to MySQL JSON / VARCHAR-collated respectively (worth ADR-0016 default-translator entries when ready).

### 13. View support Phase 2/3 — materialized-view refresh + cross-engine view-body translation

**Phase 2 shipped v0.36.0.** `sluice matview refresh --target=DSN [--matview NAME] [--concurrently] [--target-schema=NAME]` — one-shot subcommand that drives `REFRESH MATERIALIZED VIEW [CONCURRENTLY]` against PG matviews on the target. PostgreSQL-only (refuses on MySQL targets — MySQL has no matview concept). Operator drives cadence from their own scheduler (cron / k8s CronJob / Airflow); sluice deliberately does NOT own a refresh loop because (1) cadence is operator-policy, and (2) integrating with operator scheduling means alerting, backoff, and observability for the cron itself are already in place. Concurrent refresh is pre-checked for the required unique index on the matview; missing-index matviews are skipped with a clear operator-actionable reason. Missing-matview filters surface as loud-failure errors. Per-matview timing is captured in both text and JSON output for metrics scraping.

**Phase 3 — cross-engine view-body translation.** Phase 1 emits the source-dialect SELECT verbatim on cross-engine pairs; non-portable definitions surface as a target-side parse rejection at apply time (loud-failure tenet). The right Phase 3 path is to extend ADR-0016's expression translator to a SELECT grammar — a strict subset (column refs, function calls, JOIN/WHERE/GROUP BY) that covers the high-frequency patterns. `--view-override TABLE.VIEW=DEFINITION` is the always-works escape hatch for cases the translator can't handle. Deferred until operator demand surfaces a specific cross-engine view that hits the loud-failure at apply time.

**Gotchas.**
- Phase 2 MVP intentionally skips dependency-ordered refresh. Operators with nested matviews (matview A reads from matview B) should either pass `--matview` repeatedly with the right ordering, or invoke the subcommand twice (idempotent). A dedicated dependency-ordered refresh is a Phase 2.1 follow-on if real workloads surface the need; pg_depend has the data, just not wired up.
- Phase 3 SELECT translation is open-ended — a real SQL parser dependency would simplify but adds a heavy library; the existing hand-coded ADR-0016 approach stays preferable until rule count exceeds ~30 entries.
- Phase 1 deliberately punted: view definitions with explicit column lists `CREATE VIEW name (a, b, c) AS ...`, MySQL `CREATE ALGORITHM=UNDEFINED VIEW`, PG `WITH (security_invoker=true)`, and PG RULE-based pseudo-views. Each has different demand signals — pick what to surface based on real-world operator reports.

### 15. PII redaction during logical replication and migration (GitHub #24)

**Why.** Common operator ask in the replication space: *"I want a copy of this production database, but without the PII."* Typical destinations: staging/dev environments seeded from production schema + realistic-shape data without exposing real users; analytics warehouses that aggregate but never identify individuals; cross-region / cross-tenancy moves where compliance (GDPR / CCPA / HIPAA) requires PII to stay in the source jurisdiction; vendor handoffs where a third-party data processor needs the schema + the data shape but not the PII. Today operators reach for separate tools (Tonic, Privacy Dynamics, Debezium SMTs, custom ETL); sluice could absorb the use case as a first-class feature since the IR-first row pipeline already passes every value through a typed transform stage. See [GitHub issue #24](https://github.com/sluicesync/sluice/issues/24) for the full motivation + comparable-products analysis.

**What — four-phase sequence:**

- **15a. Phase 1 — simple strategies + framework. SHIPPED v0.53.0** (plus Phase 1.5 closure v0.54.0–v0.55.0: CDC apply-path redaction, YAML `redactions:` config, schema-preview annotation, backup-stream redaction; Bug 58 schema-namespace fix v0.54.1, Bug 59 kong-comma fix v0.56.1). `--redact TABLE.COLUMN=STRATEGY[:options]` with `null` / `static` / `hash:sha256` / `hash:hmac-sha256` / `truncate`; `internal/redact` package; audit log line. See "Recently landed: PII redaction track".

- **15b. Phase 2 — format-preserving + tokenize. SHIPPED v0.56.0 → v0.61.0.** Phase 2.a generic `mask:inner`/`mask:outer` + Luhn helper (v0.56.0); Phase 2.b country/format mask presets — ssn/pan/pan-relaxed/email (v0.57.0), ca-sin/uk-nin/iban/uuid (v0.58.0), Bug 60 `mask:uuid` preflight (v0.58.1); Phase 2.c replay-stable `randomize:*` — int/email/us-phone/uuid (v0.59.0, ADR-0039), ssn/pan/ca-sin/uk-nin/iban (v0.60.0); Phase 3 dictionary strategies `randomize:dict` / `tokenize:dict` + YAML `dictionaries:` (v0.61.0, ADR-0040). See "Recently landed: PII redaction track".

- **15c. Phase 3 — JSON-path redaction. PENDING (no release target).** `jsonpath` strategy with child redactions per JSON path: `paths: [$.payment_method: tokenize, $.last_visit: truncate_to_day]`. Requires an in-IR JSON walker that can rewrite nested values without re-encoding the whole document. Most invasive but smallest fraction of real-world asks (most PII is in dedicated columns, not nested JSON). Genuinely not yet shipped — v0.53.0's CHANGELOG explicitly defers it and no later release lands it. Demand-gated.

- **15d. Phase 4 — deterministic-tokenize keyset persistence. LANDED (v0.63.0; ADR-0041 Accepted 2026-05-15).** Persists the HMAC keyset across multiple sluice streams pulling from the same source so user `alice@example.com` becomes the same surrogate on streams to staging-1 AND staging-2. Shipped: `Keyset` type + loader, `--keyset-source=file:|env:|db:`, `key:` rule option, `sluice_keysets` DDL on PG + MySQL, startup audit-log line, preflight refusal. **Two deviations from the ADR draft (recorded verbatim in ADR-0041's "Decisions / deviations" section):** (D1) startup-snapshot only — the keyset is resolved once at process start and is immutable for the run; rotation takes effect on next restart only; no file-watch / db-poll hot-reload (deferred to Phase 4.5). (D2) clean break — the Phase 1 `--redact-key-source` flag and the hardcoded v0.61.0 `tokenize:dict` key were deleted with no back-compat shim; `hash:hmac-sha256` and `tokenize:dict` now REQUIRE `--keyset-source` (loud preflight refusal otherwise). Out of v1 scope: `sluice keyset rotate`/`list` CLI (manual SQL/YAML), KMS/Vault adapters, encryption-at-rest of the `bytes` column.

**Key invariants the design enforces (from the issue body):**

1. **Determinism per stream-id.** Two different sluice sync streams over the same source produce *consistent* redactions for the same input — same hash key, same tokenize seed. Otherwise CDC apply would produce row-version drift.
2. **Schema preview shows redactions.** `sluice schema preview` and `sluice schema diff` annotate redacted columns (`-- REDACTED via tokenize:email_format`) so operators see what they're agreeing to.
3. **Verify modes downgrade gracefully.** `sluice verify --depth=count` is unaffected (row counts unchanged). `sluice verify --depth=sample` automatically excludes redacted columns from the row hash (otherwise it'd always fail by design).
4. **Generated-column behavior unchanged.** Redactions apply at the reader's IR-emit step, BEFORE the target's `GENERATED ALWAYS AS` recomputes. So a generated column on the target derives from the redacted inputs naturally.
5. **Refuse to start without acknowledgment when no redactions declared.** Optional `--require-redactions` flag for the safety-conscious operator who wants the pipeline to refuse if they forgot to configure them.
6. **Audit logging.** Single INFO line at stream start summarizing how many columns are redacted (`columns_redacted=12 strategies=[...]`).

**Where redaction sits in the pipeline.** Bulk-copy reader → IR rows → redact → bulk-copy writer. CDC reader → decoded events → IR row → redact → applier. Migrate uses the same path. The redaction layer is a single typed function composed with the existing translation step; adds no new I/O.

**What it explicitly does NOT try to be (per the issue body).**

- Not a PII discovery scanner (column-name regex / heuristics). Operators declare what to redact; sluice applies it. Discovery is a separate problem.
- Not encryption at rest. Sluice runs in the operator's environment; the redacted output is plaintext on the destination. (Encryption during transport is TLS; encryption at rest is destination-side.)
- Not reversible. Hash + tokenize are one-way by design. Operators needing reversibility should add their own reversible-encryption middleware outside sluice.

**Gotchas.**

- **Determinism across stream restarts**: the HMAC keyset (for tokenize / hash strategies) must persist somewhere durable so a stream restart produces the same surrogate for the same input. Phase 1 can punt this by deriving the keyset from `--stream-id` + a static salt (stable across restarts of the same stream); Phase 4 lands the proper keyset persistence.
- **Backup chain integration** (cross-reference with item 14): redactions declared in YAML need to flow through `backup full` / `backup stream run` paths so the backup chain itself is PII-clean. Phase 1 lands sync-path redaction; backup-path follow-on is a small extension since both paths share the IR pipeline.
- **Cross-engine + redaction**: same-column redactions must work whether the target is same-engine or cross-engine. The redaction layer operates on IR values, which are engine-neutral; should compose cleanly with existing cross-engine translation.

**Sizing.** Phase 1 (the simple strategies + framework) is ~500-800 LOC + tests — IR-side `Redact()` function, YAML schema entries, four strategy implementations, CLI flag, `schema preview` annotation. Phase 2 adds ~400-600 LOC (format-preserving requires format-detection utility code). Phase 3 is ~600-900 LOC (in-IR JSON walker; non-trivial). Phase 4 is its own ADR + ~400-600 LOC (key-management + persistence).

**Sequencing.** Behind the backup-chain track (chunk 14 phases 1-3 close before phase 4 starts). Doesn't compete with #18 (AIMD) or #23 (silent-stall fix) — different surfaces, different operators benefit. Probably the most operator-facing new feature in the v0.55.0+ roadmap window.

### 18. postgres-trigger CDC runtime performance — batched-apply latency, lighter capture payload, drain throughput — **(a) + (b) SHIPPED (v0.94.0); (c) demand-gated**

**Source.** The sluice-vs-Bucardo head-to-head benchmark (2026-05-29; the v0.89.0 regression cycle) plus the follow-up latency measurement that *re-attributed* its headline finding. Both are trigger-based PG capture replicators; on identical local workloads (1M-row initial copy + 110k-change CDC stream) sluice's `postgres-trigger` engine won initial copy (~208k vs ~77k rows/s, 8-way parallel COPY), setup (single static binary vs Bucardo's plperl+DBI control DB + Perl daemon), clean teardown (0 residue vs Bucardo's leftover `bucardo` schema + source triggers), and the structural differentiator — cross-engine to MySQL/PlanetScale, which Bucardo fundamentally cannot do. All enhancements — no correctness bug (both tools verified byte-identical).

**Why (corrected — the latency premise was measurement-disproven).** The original framing blamed the fixed-interval poll and proposed a NOTIFY-kick as the headline win, citing single-change latency of ~5.9s vs Bucardo's ~0.95s. A direct measurement disproved that premise: the **default per-change apply path is already ~0.88s** (≈ Bucardo's ~0.95s), so the poll interval is *not* the dominant latency term. The ~5.9s was an artifact of running `--apply-batch-size=auto` and traced to two bugs in the *batched* applier, not the poller:
1. **5s idle-flush on partial batches.** A trailing partial batch waited up to the 5s idle-flush grace for the next change before committing — on a single sparse change that is ~5s of pure latency, the dominant component of the 5.9s.
2. **AIMD latency-includes-wait-for-work collapse.** The batch-latency clock started *before* the pre-tx wait loop, so a sparse/bursty stream's first batch reported a latency dominated by time spent blocked waiting for the first change (observed ~31s). The AIMD controller (ADR-0052) read that as "apply is catastrophically slow" and collapsed batch size 1000→1 (12 consecutive decreases, 0 increases), throttling drain throughput ~2x.

So NOTIFY-kick is **demoted** — the poll isn't the bottleneck. Closing the real gap is the two batched-apply fixes (done) plus a lighter capture payload for write-heavy sources.

**What (three sub-items, re-ordered by the corrected attribution).**
1. **[SHIPPED v0.94.0] Short idle-flush grace + AIMD apply-only timing.** (a) `defaultIdleFlushPeriod` reduced 5s → 100ms in *both* engines' batched appliers, so a drained/sparse stream's partial batch flushes within ~100ms instead of 5s. (b) the batch-latency clock now starts *after* the pre-tx wait loop. Both engines pinned with unit tests; CI green incl `-race`. Single-change latency drops from ~5.9s toward ~1s; drain throughput recovers ~2x from the un-collapsed controller.
2. **[SHIPPED v0.94.0 — ADR-0068] Lighter capture payload.** `sluice trigger setup --capture-payload=full|changed|minimal` (default `full`, all three selectable for head-to-head benchmarking). Trigger-only change; reader/applier are payload-shape-agnostic. `changed` trims the UPDATE `after` to PK+changed-columns (keeps full `before`); `minimal` also trims `before` to the OLD PK (→ PK-`WHERE`, reaches toward ~2x source-write amplification). A silent-loss bug caught in review (minimal's UPDATE-before was the NEW PK → PK-changing UPDATEs would diverge) was fixed pre-merge by using the OLD-PK projection + adding PK-update to the integration matrix.
3. **Optional `--poll-interval` config + configurable idle grace.** `--poll-interval` **SHIPPED in v0.91.0** (`f063dc8`, `cmd/sluice/cli.go:677`; overrides the CDC reader's poll cadence for poll-based engines — today postgres-trigger; push-based engines ignore it). The remaining half — making the (now-100ms) idle-flush grace operator-tunable — stays **demand-gated**: no operator has surfaced a workload that needs it. NOTIFY-kick stays demoted — poll latency isn't the bottleneck the measurement found.

**Gotchas / open questions.**
- The 100ms idle grace rides producer/scheduler jitter within a burst while staying negligible against the ~1s poll interval. If a future poll-interval config drops the interval well below ~1s, re-check that the grace still doesn't truncate in-flight bursts (it shouldn't — full batches immediate-repoll — but verify).
- ADR-0068 covers the payload-mode design. `changed`/`minimal` KEEP self-containment + point-in-time fidelity (changed values are in the log); only `minimal` changes apply semantics (full-before divergence `WHERE` → PK-`WHERE`), acceptable for one-way CDC. Default stays `full`.
- The ~10.8x figure is a 50k-UPDATE microbench; the ADR's head-to-head measurement plan confirms the reduction on a realistic mix before any default flip.
- The batched-apply fixes touch the shared CDC apply path (both engines, concurrency-adjacent: goroutines + channels + the idle timer), so the `-race` + integration CI gate is the real validation per the concurrency-chunk rule.

**Sizing.** (1) is done (~small diff: one const per engine + the latency-clock move + pins). (2) lighter payload (ADR-0068) is trigger-only — per-mode trigger-function bodies + a `--capture-payload` flag + tests + the head-to-head measurement (no reader/applier change, since they're payload-shape-agnostic), so smaller than first estimated. (3) is a small config surface, partly free.

**Sequencing.** Demand-shaped; (1) landed because the measurement re-attribution made it a cheap correctness-adjacent win. (2) pairs with the postgres-trigger Phase 2 work and is gated on a write-heavy-source operator flagging it. Not a correctness gate — sluice already wins setup/portability/cross-engine; this closes the runtime-CDC gap the Bucardo comparison surfaced, minus the NOTIFY-kick that the measurement showed wasn't needed.

### 19. Vitess VStream throttle resilience — stress-characterize event suppression + evaluate a lock-free OLAP-dump fallback (operator-flagged 2026-06-13)

**Why.** sluice's Vitess/PlanetScale path is **VStream-only** for BOTH the backup snapshot (`backup_snapshot.go` → `usesVStream()` → `openBackupSnapshotVStream`, VStream COPY across shards) and live CDC. Vitess's **lag-based tablet throttler** can suppress VStream events: the operator-flagged failure mode is that even at **0 replication lag**, the **heartbeat lag** value can exceed the throttler threshold (stale/slow heartbeat writes, heartbeat-interval vs threshold config) → the throttler engages → VStream COPY/streaming is throttled or paused → a sluice user perceives a sync stall / backup hang with no obvious cause. sluice today only handles this **reactively** — `applier_errors.go` / `reader_errors.go` classify the throttler's `ResourceExhausted` as a transient-retry — so a *sustained* throttle becomes a sustained, silent-ish stall, not a loud diagnosis. Meanwhile `pscale database dump` sidesteps VStream entirely (session `set workload=olap` + `@replica`/`@rdonly` routing → a streaming SELECT, NOT throttler-gated), and **sluice already has an `workload=olap` full-scan RowReader** for vtgate (`engine.go` `olapFullScan`) — it's just not wired to the Vitess *backup snapshot* path. So a lock-free OLAP dump is a plausible throttle-resilient *backup-data* tactic worth measuring.

**What.** Two parts; (a) is the primary ask (characterize), (b) is the conditional mitigation.

- **(a) VStream throttle stress-test (characterization first).** Build a repeatable stress harness that drives the conditions under which VStream stops emitting and observe how sluice behaves (silent stall? retry-loop? loud?). Conditions to sweep: throttler engaged via **heartbeat lag > threshold at 0 replication lag**, sustained high write rate, low `--throttle_threshold`, replica-routing under load. Run it first against the **local Vitess** harness (the `vstream` / `vitesscluster` build-tag suites, vttestserver + the multi-process cluster — the `vitess-cluster-validator` agent), then against a **real PlanetScale** branch (the `psverify` harness — relate to item 10's Path A/C and its credential gating). Output: a documented matrix of "VStream stalls when …" + sluice's observed behavior, and whether the current transient-retry classification is adequate or whether sluice should **detect throttle-state and surface it loudly** (so operators see "source throttler engaged" instead of an unexplained lag).
- **(b) Lock-free OLAP-dump fallback for the Vitess BACKUP snapshot (conditional on (a) showing it's worth it).** Offer a `workload=olap`, replica-routed, parallel streaming-SELECT dump (pscale-dump shape; reuse the existing `olapFullScan` reader) as an alternative to VStream COPY for the snapshot/backup-data leg, so a throttled cluster can still be backed up. **Key constraint (do not skip):** v0.44.0 deliberately moved the PlanetScale backup *to* VStream COPY precisely because a pinned-conn/SELECT path "produces a binlog-shape position the VStream continuous-sync path can't decode, breaking incremental + stream-run chain-resume entirely" — so an OLAP-SELECT dump cannot anchor the shard-GTID position that incremental chaining needs. It would therefore be a **full-only / one-shot** fallback (no chainable incremental), or require a separate position-capture story. Live CDC has **no** SELECT alternative (VStream is the only change-event source), so the mitigation there is throttle-state surfacing + tuning, not a dump swap.
- **(c) Vitess backup/restore perf fair-fight (the comparison-doc gap).** `docs/comparison-backup.md` has the vanilla-MySQL fair-fight but **no Vitess numbers**. Measure `sluice backup full`/`restore` (VStream COPY) on a Vitess source against `mysqldump --single-transaction` (via vtgate), `mydumper`/`myloader`, and **[aquarapid/go-mydumper](https://github.com/aquarapid/go-mydumper)** — the Vitess-native parallel dumper the [Vitess docs recommend for exporting data](https://vitess.io/docs/24.0/user-guides/configuration-basic/exporting-data/) (and the lineage `planetscale/cli`'s own internal dumper derives from). This directly informs (b): if VStream COPY is competitive AND throttle-resilient, the OLAP fallback isn't worth building; if go-mydumper's lock-free OLAP path is markedly faster or survives throttle where VStream COPY stalls, (b) earns its keep. Add a "Vitess-side fair fight" section to `comparison-backup.md` with the measured table.

**Sequencing of the test work:** stage it — (1) local `vitesscluster` docker harness (primary+replica + the chaos helpers in `cdc_vstream_cluster_chaos_helpers_test.go` to induce lag/reparent; the `vitess-cluster-validator` agent drives it) to PROVE the throttle repro + watch VStream suppression cheaply; (2) a Vultr VM (more cores/RAM than the local Rancher) for sustained-load stress + the (c) perf benchmark on meaningful data volume; (3) a real PlanetScale branch (`psverify`) for vendor realism (TLS / vtgate-proxy / real throttler tuning). Relate to item 10's Path A/C + credential gating.

**Gotchas.** Throttler config knobs (`--throttle_threshold`, heartbeat interval/lease, `--throttle_check_as_check_self`); the GTID-position constraint above; PlanetScale credentials are operator-run-only (not in CI); and the OLAP-dump path must NOT silently lose the chain-resume capability operators get from the VStream-COPY backup — if (b) ships it must be an explicit, loud, full-only mode. **Sequencing:** demand/curiosity-flagged, not a correctness gate today (the retry classification keeps it from being data loss — it's a *latency/availability* perception issue). Do (a) first; (b) only if the characterization shows real-world throttle stalls that the OLAP path would meaningfully relieve.

**FINDINGS (2026-06-13, local `vitesscluster` v24.0.1 + Vultr `vhf-8c-32gb`; full log + writeups in sluice-testing `workspace/vitess-vultr-run/`).** Characterized; all sub-points confirmed:
- **Silent stall is REAL + ROOT-CAUSED — and the root cause is a Vitess `vtgate` gap, not sluice-only.** The tablet vstreamer DOES signal throttle in-band (`VEvent.throttled`/`throttled_reason`, binlogdata.proto fields 24/25, set on its heartbeats by `injectHeartbeat()`), but `vtgate`'s `vstream_manager.go` **filters out tablet heartbeats** (`// Remove all heartbeat events for now`) and **synthesizes its own throttle-blind heartbeats** → an external gRPC consumer never sees `throttled`. So a mid-stream throttle reaches sluice as content-free heartbeats + zero ROW events + no gRPC error: the `ResourceExhausted` retry classifier never fires, the 45s progress watchdog stays re-armed by heartbeats, the pump parks in `grpc Recv()` → unbounded lag, zero diagnostic (converges on unthrottle; no data loss). Vitess-team writeup + proposed fix (vtgate propagate `throttled` onto its synthesized heartbeat — the external-consumer analog of #16575/#16577's vplayer fix) at `workspace/vitess-vultr-run/vitess-vstream-throttle-signal-gap.md`. The "heartbeat-lag-at-0-repl-lag" knob is SELF-HEALING for active streams (on-demand heartbeat leasing); a sustained throttle needs a real app-rule / sub-floor threshold / real replica lag.
- **(a) sluice observability fixes — LAYERED (the out-of-band SQL signal is real but PARTIAL, verified 2026-06-13):** (1) the at-stream-OPEN throttle fails loud (30s Phase-1 liveness, heartbeats don't re-arm Phase 1) but the message MIS-DIAGNOSES it as "no PRIMARY tablet / set vstream_tablet_type=primary" — name the throttler as a candidate cause. (2) mid-stream is ALREADY resilient (Phase-2 watchdog re-arms on heartbeats → stays alive + catches up on unthrottle, never errors) — it's just SILENT; the fix is a WARN, not a behavior change. (3) **Out-of-band detection over vtgate's MySQL data plane WORKS but is incomplete:** `SHOW VITESS_THROTTLED_APPS` / `SHOW VITESS_THROTTLER STATUS` route through vtgate's MySQL port to the PRIMARY (default routing works; `@replica` → VT09010/VT09007), so sluice CAN poll them on a primary-routed connection (the backup-COPY path already connects to primary). BUT they only reflect an **explicit per-app deny** (`SHOW VITESS_THROTTLED_APPS` lists `vstreamer`/`rowstreamer`/`vreplication` with expiry+ratio); they do **NOT** reflect a **threshold/lag-metric** throttle — `SHOW VITESS_THROTTLER STATUS` returns static config only (enabled/threshold/lag-query, no live metric, no "currently throttling" bool), and the threshold case shows only the built-in `always-throttled-app` sentinel. The live metric-vs-threshold decision is only on the gRPC/control-plane `CheckThrottler` (NOT reachable over sluice's MySQL conn). So: poll `SHOW VITESS_THROTTLED_APPS` for the streamer app names → catches explicit throttles (real signal, do it); for the common **threshold/lag** throttle, fall back to the heuristic ("heartbeats but no data > window → WARN: stream idle or throttled, check source throttler"). The clean, complete in-band signal (`VEvent.throttled`) is still GATED on the vtgate fix (vtgate strips it today).
- **(c) perf fair-fight (9.16 GB / 33.5M rows via vtgate):** sluice = 115s dump / **1.1 GB** (gzip) / 332s restore, zero-loss. `mysqldump --single-transaction` and apt `mydumper` BOTH FAIL outright against vtgate (row-count cap / no `workload=olap` / pre-Vitess); `go-mydumper` (aquarapid, olap) dumps fine (113s) but go-myloader panics on MySQL 8.0 (tool bug). → **sluice is the most usable end-to-end Vitess backup of the four.**
- **(b) verdict: WORTH BUILDING for RESILIENCE not speed.** Decisive: with `rowstreamer` + `vstreamer` both APP_DENIED (the exact throttle that stalls sluice's VStream COPY), `go-mydumper`'s `workload=olap` dump produced a byte-identical full dump at full speed — OLAP SELECT is NOT throttler-gated. A lock-free OLAP-dump path would back up a *throttled* cluster VStream COPY can't (full-only/explicit/loud per the GTID constraint). Sequence after (a).
- **Not yet done:** real-PlanetScale (`psverify`) leg (credentials operator-only); the replica-lag throttle vector (only the app-rule vector exercised).

**(d) Real-world trigger guide (operator-flagged 2026-06-13, IN PROGRESS).** The characterization so far used an *artificial* throttle (explicit `--throttle-app` deny / sub-floor threshold) to prove sluice's RESPONSE. The open question is the **Vitess-side cause**: which real workloads drive the lag-based throttler to engage (and thus stall an external VStream)? The mechanism: the throttler periodically compares a **metric** (default = replication lag, measured via the `_vt.heartbeat` table: primary writes a heartbeat row, replica reads it, lag = now − heartbeat_ts) against a **threshold** (default ~5s); over-threshold ⇒ it denies `CheckThrottler` for apps (`vstreamer`/`rowstreamer`/`vreplication`/`online-ddl`), which back off. So anything that grows real replication lag — or makes the lag metric read high — trips it. Candidate real-world triggers to QUANTIFY (intensity/threshold at which each trips + the resulting VStream delay), building a guide:
  1. **Write-heavy primary** — replica can't apply binlog fast enough ⇒ lag ⇒ throttle (the classic; worsened by large/long txns serializing on the replica).
  2. **VReplication activity from a user migration** — `OnlineDDL` (vreplication copy phase, itself throttled), `MoveTables`/`Reshard`/`Materialize` competing for primary+replica I/O AND self-throttled by the same throttler — a heavy migration throttles unrelated external VStream consumers.
  3. **Replica resource saturation** — heavy `@replica`/OLAP reads or CPU/IO contention slows binlog apply ⇒ lag.
  4. **vtgate / connection load** — and any **custom-metric** throttle (newer Vitess can throttle on MySQL `threads_running` / history-list-length), so raw concurrency can trip it directly without classic lag.
  5. **Heartbeat-staleness artifact** — stale `_vt.heartbeat` reads high lag at ~0 real lag (self-heals for active streams via on-demand leasing; edge configs — heartbeat disabled / unleased / sub-floor threshold — can still trip).
  **Deliverable:** a guide (workload → does-it-throttle, at what intensity, mechanism, mitigations) as general operator knowledge — run realistic load (sysbench write/read, an OnlineDDL/MoveTables migration, vtgate load) against a default-config cluster, measure throttler engagement (`CheckThrottler` / lag metric) and the VStream delay it produces. Distinct from the artificial-repro work above; this is "what actually causes it in production."

  **RESULT (2026-06-13, default-config v24.0.1 cluster, full guide at sluice-testing `workspace/vitess-trigger-guide/vitess-trigger-guide.md`):** the default lag throttler does **NOT** trip on write volume against a healthy cluster — it trips when the **replica genuinely falls behind**. Two confirmed real causes, three confirmed negatives:
  - **#1 (CONFIRMED, strongest — the operator's prime suspect): a concurrent VReplication user migration.** A routine `OnlineDDL ddl-strategy=vitess` ALTER on a 1.06M-row table drove shard lag 0→**251s** and held `vstreamer THRESHOLD_EXCEEDED` for the whole ~4-min copy phase, **totally stalling an UNRELATED external sluice stream** (target frozen while source advanced). The migration's vreplication copy moves the *shared shard-lag metric*, which gates the unrelated external vstreamer. Migration size = throttle duration (a 530k-row ALTER only oscillated around the threshold).
  - **#2: an under-resourced / contended replica** (CPU/IO-starved) → lag 5s→64s→251s, target frozen 60s+.
  - **NEGATIVE — write-heavy primary alone does NOT trip it:** sysbench write_only at 795–904 tps + ~840 MB bulk binlog kept peak lag ~1.0s; vttablet caps tx concurrency (~900 tps) before the replica falls behind on a healthy box.
  - **NEGATIVE — vtgate/connection load does NOT trip the default (lag) metric:** but a v24 **custom `threads_running` throttle** IS configurable and *can* gate on raw concurrency with zero lag (vttablet pooling keeps it low, so it needs a low threshold to bite).
  - **NEGATIVE — heartbeat-staleness self-heals:** idle lag drifts to 25–60s (stale `_vt.heartbeat`) but collapses to <1s the instant a streamer leases on-demand heartbeats.
  - **Cross-cutting: no data loss ever** (lag collapses + target catches up within ~one poll of unthrottle; src==tgt after every scenario) — pure latency/availability. **sluice was completely silent through every real throttle** (only its 10s heartbeat lines), confirming the 19(a) silent-stall now driven by a *real* cause — which is exactly what the 19(a) WARN (PR #211) surfaces. **Operator takeaway: the prime real-world trigger is a co-tenant Vitess migration (OnlineDDL/MoveTables/Reshard) on the same keyspace, not your own write rate.** TODO: fold this guide into `docs/vitess-vstream-troubleshooting.md` (the doc #211 adds) once #211 merges.

### 21. Concurrent cold-copy: work-stealing / dynamic rebalancing of the disjoint table partition (close the K-wide tail) — *throughput optimization; not a correctness gate*

**Why.** The cross-table concurrent cold-copy (VStream ADR-0099/0100 `vstream_copy_table_parallelism=K`; native-MySQL ADR-0101/0102 `copy_table_parallelism=N`) partitions the in-scope tables into K (or N) **disjoint groups once at cold-start** via `partitionTablesForStreams`, and each read→write pipeline drains **its** group **serially**. The partition is **static — no work-stealing/rebalancing** — so the effective width tapers below K at the tail: as the lighter groups finish, their pipelines go **idle** while the heavier groups grind their last large table. Observed live 2026-06-18 on Track B (K=4 → only 2 read streams active at 22/27 tables copied). The partition balances by **row-count estimate**, but actual copy time is driven by **bytes + row width** — a group that draws the blob/wide-row tables (binary_blobs, large-payload tables) runs disproportionately long, so the tail is bounded by the single slowest group rather than by total/K. (The D-way write fan-out still runs per active table, so the *write* side isn't fully serial at the tail — but the *read* side, the per-table cap, is.)

**What.** Add work-stealing so a pipeline that drains its group picks up more work instead of idling. Two tiers: **(a) table-level stealing** — an idle pipeline claims an unstarted table from a still-busy group (cheap; closes most of the gap when ≥K-ish tables remain). **(b) intra-table stealing** — split the in-progress *large* table into PK-range chunks an idle pipeline can take (the stronger form; bounds the tail by chunk, not by whole-table). (b) needs PK-range chunking on the snapshot read path: the `migrate` path already has keyset chunking (ADR-0096), but the `sync`/VStream COPY path streams one logical channel per table and is **not** arbitrarily range-splittable today — that's the core constraint to solve (or restrict (b) to the native-MySQL `SELECT`-range path first, where it composes with the existing chunker). A claim/lease protocol over the shared table set replaces the up-front static partition; both engines share `partitionTablesForStreams`, so the rebalancing lives at that layer + the ADR-0100 consumer.

**Gotchas.** Must preserve the exactly-once + gapless invariants the static partition gives for free: **every table (and every chunk) copied by exactly one pipeline** (the claim/lease must be atomic — no double-claim, no dropped table), and the snapshot→CDC seam stitch stays correct (VStream per-shard set-min across all streams; native single FTWRL position — neither depends on *which* pipeline copied a table, so stealing is seam-safe as long as coverage stays complete). Keep the byte budget bounded: VStream's per-stream `--max-buffer-bytes / K` cap and the `K × D` connection budget must not balloon as work moves. Resume composes with the ADR-0098 auto-shard-aware resume (the claim state must be re-derivable or persisted). Likely a new ADR. **Not a correctness fix** — the static partition is correct and fast for the bulk of the copy; this reclaims the tail and makes K robust to row-width skew. Sequence after the large-scale program's throughput-arc settles.

### 22. Progress ETA: label `total_rows` as an estimate (avoid "rows > total" confusion) — *cosmetic UX hardening; ETA safety already handled*

**Why.** The bulk-copy progress ticker's `total_rows` is ALWAYS a statistics ESTIMATE — MySQL `information_schema.TABLE_ROWS`, PG `pg_class.reltuples` (`row_reader_range.go` `CountRows` on both engines; the native concurrent reader uses the same TABLE_ROWS estimate via the side metadata pool, item 21a follow-up). Do NOT fudge the number (bumping `total` up to `rows` would misrepresent it). Repo-wide progress-UX touch (`internal/pipeline/progress.go`), pinned by a ticker unit test (rows > total → estimated flag set, ETA stays `-1`).

**Gotchas.** Keep it honest — the value stays the real estimate; only the LABEL changes. Don't introduce a divide-by-zero or a negative when rows == total (the existing `total > rows` guard already covers it; preserve it). Low priority — purely cosmetic; the safety-critical ETA behavior is already correct.

### 23. Cross-region CDC apply to PlanetScale-MySQL: per-shard progress stall + apply-throughput under a high-write multi-shard Vitess source — *B-1 per-shard WARN + pre-seed SHIPPED (Unreleased, -race green); B-2 auto-reopen CONTRAINDICATED (load-forced capture) → real fix is 23(c) apply throughput*

**Why (live finding, 2026-06-18, Track B re-test on v0.99.73).** A Vitess→PS-MySQL `sync` cold-copy completed and entered CDC cleanly (the auto-resnapshot/retention fix held — item-21/ADR-0093 path validated), but in steady-state CDC the apply does NOT keep up with a continuously-writing 2-shard source, and the lag is **per-shard asymmetric**: shard `80-` applied at a healthy clip (lag ~37K and roughly stable), while shard `-80`'s applied GTID position **froze** (e.g. `3,598,251` unchanged across many minutes) with its lag **GROWING** (303K → 349K and climbing) — even though the source keeps committing to `-80` and the stream logs no error (only heartbeats + AIMD `p95≈10s` batch-shrink). So one shard's apply effectively stalls while the other proceeds — observable via the per-shard source-vs-applied GTID gap (`sluice sync status` token vs tablet `gtid_executed`). NOT silent (the lag is visible; CDC never claims caught-up), but a stalled shard on a "healthy" stream is a serious shape.

**What (investigate first — three-phase; do NOT assume the cause).** Candidate hypotheses to disprove with ground truth: **(a)** the `-80` vstreamer is throttle-suppressed at the source (the documented Vitess lag-throttler shard-stall, roadmap item 19 — an INFRA condition, not a sluice bug; confirm via `SHOW VITESS_THROTTLED_APPS` / the tablet throttler + whether `-80` events are arriving at all); **(b)** the keyspace-wide CDC apply orders/serializes across shards such that one shard starves the other (a sluice apply-scheduling issue — the K-stream COPY is concurrent but the CDC tail is a single keyspace-wide stream whose per-shard apply progress can diverge); **(c)** cross-region commit latency (`p95≈10s`, AIMD shrinking toward batch=1) caps apply throughput below the source write rate generally, and `-80` just lost the scheduler race. Phase A = instrument/observe which shard's events reach the applier and whether `-80` commits ever land; only then fix the confirmed cause.

**Then (if it is throughput, not a stall-bug).** The cold-copy got W×D write fan-out (ADR-0097/0100/0102), but the **CDC apply path has no equivalent throughput lever** for a cross-region RTT-bound target — and unlike cold-copy it must preserve apply order, so any parallelism needs per-key/per-table ordering guarantees (composing with ADR-0010 idempotent apply) or per-shard apply pipelines. Design-gated on the Phase-A finding. **Gotcha:** whatever the cause, the per-shard lag must stay observable (it is — the gap is the signal) and exactly-once/ordering must not be weakened to chase throughput.

**PHASE A FINDINGS (2026-06-18, observed on the live Track-B v0.99.73 run).** Ground truth gathered; the leading hypothesis is now (b) refined to a **per-shard buffer-starvation wedge under a slow apply**, NOT throttle: (1) **throttle REJECTED** — both shards' replicas `Seconds_Behind_Source=0`, `THROTTLED_APPS` shows only Vitess's built-in placeholder; (2) **source IS delivering `-80`** — the `-80` primary's `VStreamerEventsStreamed` advanced at a rate comparable to `80-`; (3) the stuck stream had ONE `stream starting`, **zero** reopens/errors — it looked healthy (heartbeats, `80-` applying) while `-80`'s applied GTID was frozen ~1h with growing lag — a **silent per-shard wedge** detectable ONLY via the per-shard source-vs-applied gap; (4) **a warm-resume (fresh vstream open from the persisted vgtid) RECOVERED `-80`** — it resumed applying immediately — so the wedge is a **recoverable stream-level stall, not a permanent structural bug** (the restart also destroyed the wedged-state evidence, so dissecting the exact mechanism needs a re-wedge captured live at `--log-level=debug`); (5) **decisive contrast** — Track B2 (SAME Vitess source, **PG** target, fast ADR-0092 pipelined apply) keeps BOTH shards caught up, while Track B (MySQL target, latency-saturated cross-region apply, AIMD churning toward batch=1) wedged `-80`. That correlation points at: under a slow apply the bounded shared vstream buffer (ADR-0071 `--max-buffer-bytes`) fills with the **dominant** shard's events (here `80-`, the write-skew winner) and **starves the minority shard (`-80`)** until its sub-stream wedges; a fast apply drains the buffer so neither starves. **Fix directions to evaluate:** per-shard buffer fairness (don't let one shard monopolize the shared cap), and/or a **per-shard stall detector → auto-reopen** (reuse the ADR-0094 reshard-reopen machinery: if a shard's position stalls while the source advances and peers progress, reopen the stream — which provably recovers it). Reproduce-with-debug to confirm received-but-not-applied vs not-received before the fix.

**B-1 — per-shard progress WARN: ✅ SHIPPED (Unreleased).** The observability-only first increment of the per-shard stall detector. A `shardProgressWatchdog` (`internal/engines/mysql/cdc_vstream_shard_progress.go`, structured exactly like the whole-stream `vstreamLiveness`: single watchdog goroutine owns the timer/state, the dispatch path feeds it a non-blocking buffered per-VGTID shard-advancement diff) emits a single rate-limited WARN — naming the wedged shard + keyspace — when one shard's emitted GTID component has not advanced for a window WHILE a PEER shard HAS advanced within that same window (and Phase-2 serving is proven). That asymmetry (S frozen while a peer progresses) is precisely the silent per-shard wedge the merged-channel whole-stream watchdog cannot see. The WARN names BOTH candidate causes — a genuine per-shard source stall (throttle / large-tx / tablet issue → check `SHOW VITESS_THROTTLED_APPS` on that shard's primary) OR a normal `MinimizeSkew` catch-up hold (vtgate holding the ahead shard back while a behind peer drains) — because the reader cannot distinguish them without a source probe; it is WARN-only (never setErr / cancel / alter resilience) so a benign skew-hold WARN is acceptable signal. Knob: `vstream_shard_stall_warn_timeout` (Go duration, default 60s; `0`/negative disables this WARN only; malformed ⇒ loud error; stripped from the MySQL session per the Bug-126 class). Pinned by unit tests (scan logic deterministic via the injectable timer + clock seam: asymmetric-stall-fires-once-latched, latch-clears-on-advance, global-idle-does-not-fire, before-serving-proven-does-not-fire, disabled, nil-safe, the per-VGTID advance diff, the DSN parse + strip, and the message contract). **CONCURRENCY chunk** — the `-race` integration gate must pass on CI before tagging.

**B-1 follow-up — pre-seed the known shard set: ✅ SHIPPED (Unreleased, `-race` GREEN on main).** The original B-1 watchdog seeded its per-shard clock only on a shard's FIRST observed advance, so a shard wedged from the very first CDC event (never delivers an advancing VGTID) was never tracked and `scan` never considered it — exactly the original live Track-B silence (the frozen shard had never advanced post-handoff). Fix threads the resolved shard layout (`r.shards`) into the watchdog and pre-seeds every known shard at start time. Pinned by `TestShardProgressWatchdog_EndToEnd_FromStartFrozenShardWarns` (a shard NEVER observed advancing still WARNs once stale vs a fresh peer). `-race` Integration gate GREEN on `main`.

**B-2 — auto-reopen: ❌ CONTRAINDICATED for the dominant variant (load-forced capture, 2026-06-19).** A `--log-level=debug` capture under load-forcing (7 parallel writers, source ~700/s vs cross-region apply ~50–60/s) reproduced the wedge and settled the open questions — but REFUTED the auto-reopen plan. Ground truth: **(1) the freeze is DELIVERY-side, not apply-side** — the instrumented reader log shows the held shard's VGTID FROZEN (`advanced=[<other shard>]` on every event; the held shard never in the advanced set), so vtgate is NOT delivering it; the applier is correctly applying everything it receives. **(2) The dominant wedge is a `MinimizeSkew` hold under an apply-deficit:** when apply < source, one shard accumulates a large backlog; `MinimizeSkew` delivers the MORE-behind shard (to keep the merged stream commit-time ordered) and HOLDS the other, and because apply never catches up the hold is effectively permanent (observed frozen 12+ min, lag growing, and it does NOT self-resolve even after the causing load is removed — the behind shard must fully drain its backlog first, ~hours). **(3) A reopen does NOT clear this variant** — a clean warm-resume was tested live against the frozen state and the held shard stayed frozen across the reopen (vtgate re-formed the identical skew-based hold from the persisted positions). So promoting the B-1 WARN into an auto-reopen would THRASH uselessly here (reopen → re-wedge → reopen …). The real lever for this (the common cross-region case) is **apply throughput** — see item 23(c) / per-shard apply concurrency: keep apply ≥ source so no large asymmetric backlog forms and `MinimizeSkew` never holds a shard indefinitely. **(4) A SEPARATE, rarer variant exists** — the original v0.99.73/v0.99.75 wedge (one shard CAUGHT UP, the other frozen-and-growing) DID recover on reopen; that is a distinct genuine stuck-stream not controlled-reproduced here, and the only case where a (bounded, peer-caught-up-gated) reopen would help. **Revised plan:** keep the B-1 WARN as the detector (correct for both variants); do NOT build blanket auto-reopen; pursue **item 23(c) apply throughput** as the real fix for the dominant variant; revisit a narrowly-gated bounded-reopen ONLY for the genuine stuck-stream variant if it recurs and is controlled-reproduced. The B-1 WARN message already names both candidate causes, which is the right operator-facing signal in the meantime.

**B-2 RE-CONFIRMED LIVE on v0.99.81 (2026-06-20, three-phase Phase-A re-run).** A fresh credential-safe source+target probe on the live Track-B `lst-b-v75` run independently re-validated the B-2 ground truth and showed the throughput fix (23(c)) **self-healing** the wedge: (1) **source healthy on BOTH shards** — `-80` (tablet 80a, `77ee51e9`) and `80-` (tablet c0a, `77ee8962`) each advancing ~20 GTID/s, `SHOW VITESS_THROTTLED_APPS` EMPTY on both primaries (throttle re-refuted); (2) **the hold is delivery-side MinimizeSkew** — sluice's persisted `source_position` mirrors the received VGTID, and over a 30s window shard `-80` advanced +3,681 (~123 GTID/s) while shard `80-` advanced **0** (held); the held shard is the relatively-AHEAD one (`80-` resumed less-behind), exactly the MinimizeSkew "deliver the behind shard, hold the ahead shard" signature; (3) **structural confirmation it is not a sluice scheduling bug** — the ADR-0104 key-hash apply lanes route by `(table, PK)` and are shard-AGNOSTIC, so sluice cannot preferentially starve a shard it actually receives; a frozen persisted per-shard gtid ⟺ vtgate froze that shard's VGTID component; (4) **self-healing under `--apply-concurrency=4`** — apply (~123 GTID/s on the draining shard) now FAR exceeds source (~20/s/shard), so the behind shard `-80` drained +420K GTID since the run's warm-resume and will let MinimizeSkew release `80-` once it reaches parity. This closes the three-phase RCA: **no sluice bug; expected MinimizeSkew-under-apply-deficit behavior, already mitigated by the shipped 23(c) apply-throughput lever.** One OPTIONAL optimization surfaced (logged as **item 27**): relax `MinimizeSkew` on the concurrent-apply path — cross-shard changes are independent (range-sharding ⇒ a given `(table, PK)` lives in exactly ONE shard ⇒ no cross-shard same-key dependency, and `StopOnReshard=true` already prevents the only overlap window), so commit-time cross-shard ordering is unnecessary for sluice's correctness; dropping the hold would let both shards stream concurrently and remove the catch-up-latency penalty on the held (ahead) shard during a backlog drain. It is a CRITICAL ordering/position change so it is design-gated, not a quick toggle.

### 24. Secondary-index / index-only DDL is NOT forwarded to the target mid-sync (ALL CDC sources) — *confirmed gap (live, 2026-06-18); designed in [ADR-0103](../adr/adr-0103-forward-index-ddl-during-cdc.md); demand-gated; loud-failure-safe (no silent loss)*

**Why (live finding, 2026-06-18, Track B/B2).** Added a secondary index to a live source table via Vitess online DDL (`ALTER TABLE wide_payload ADD INDEX idx_lst_ts (ts)`, `ddl_strategy=vitess`); the migration cut over on both shards (index present on the source), but the index appeared on NEITHER target — not the PS-MySQL target (`lst-mysql-b-v2`) nor the PS-PG target (`lst-pg-b2`). Data kept flowing correctly; only the index did not cross. **No silent loss** — DATA is unaffected; this is a schema-fidelity gap, not a value-fidelity one.

**Root cause (code-confirmed — GENERAL across all three CDC sources, NOT VStream-specific).** sluice HAS a `CREATE INDEX` / `DROP INDEX` forwarding path (`ShapeKindCreateIndex` / `ShapeKindDropIndex` → `applyShapeForward` → `ir.ShapeDeltaApplier`), but it is driven by an `ir.SchemaSnapshot` whose IR carries index metadata — and **no CDC reader populates secondary indexes into that IR**, because the CDC wire protocols are all column-oriented: (1) **VStream** — `projectVStreamFields` (`field_to_ir.go`) returns `&ir.Table{Schema, Name, Columns}`, no `Indexes`, not even the PK, since vtgate `FIELD` events are columns-only; (2) **binlog (vanilla MySQL)** — the reader DOES re-introspect on a DDL event, but its cached `tableSchema` is **columns + PK only** by deliberate design (`cdc_reader.go`: *"Indexes (other than the PK) and foreign keys are not consulted on a row event so we don't pay to load them"*); (3) **PG logical replication** — pgoutput Relation messages carry no secondary-index metadata, and `CDCSchemaSnapshotNormalizer` explicitly sets `out.Indexes = nil` (`postgres/cdc_normalize.go`; pinned by a test asserting *"pgoutput carries no secondary index metadata"*). So on EVERY source an index-only DDL leaves the projected signature unchanged ⇒ no `SchemaSnapshot` delta ⇒ `ClassifyShape` never sees the index shape ⇒ nothing forwarded. The `ShapeKindCreateIndex`/`DropIndex` arms are reachable only via the **migrate-path** full-introspection diff, never the live CDC forward. **Confirmed live 2026-06-18** for the VStream→PS-MySQL and VStream→PS-PG paths; the binlog/PG-source conclusions are code-confirmed (the wire protocols cannot carry it without re-introspection). Data is unaffected on all paths.

**What (when demanded) — detection is the hard, source-specific half; apply is mostly built.** The cross-engine index *translation* + creation already exists (cold-copy creates source indexes on the target via deferred index creation), so the gap is **detecting** the index delta and **applying it non-blockingly**.

- **Detection (per source):** (a) **Vitess/PlanetScale** — poll `SHOW VITESS_MIGRATIONS` (authoritative: it records the completed online-DDL statement + a completion signal that the VStream event stream does NOT cleanly surface, because an online DDL cuts over via `_vt_` shadow-table rename, not a literal `ALTER` on the logical table — so re-introspecting on the post-cutover `FIELD` re-emit is the alternative in-band hook); (b) **vanilla MySQL** — a *direct* `ALTER … ADD INDEX` IS a literal DDL query event in the binlog, so extend the existing on-DDL re-introspection to ALSO load secondary indexes into the snapshot IR (gh-ost/pt-osc shadow-table DDLs need the same VITESS-style cutover-re-introspect handling); (c) **PostgreSQL** — hardest, because logical replication does not stream DDL at all; needs an event-trigger-based DDL capture (the pgtrigger engine already captures DDL via event triggers — reuse that) or a periodic index re-introspection.
- **Apply (per target — use each target's NON-BLOCKING index primitive, since an index build on a large table must not lock):** (a) **PlanetScale MySQL** — issue a PlanetScale **deploy request** carrying the schema change and let PS's native online-DDL machinery apply it non-blocking with its review/revert rails (the operator-proposed approach; index ordering relative to the CDC data stream does NOT matter for correctness, which makes the async deploy a clean fit). Caveat: this couples sluice to the PS **control-plane** API (org/db/branch + a service token — a different credential surface than the target MySQL/VStream data-plane DSN) and to PS's dev-branch→deploy model, so it is best as an **opt-in** target capability (or a REPORT — "detected index X; here is the deploy request to apply" — consistent with sluice's "index lifecycle is explicit, not silently auto-handled" tenet) rather than a silent auto-apply; (b) **self-hosted MySQL** — `ALTER … ALGORITHM=INPLACE, LOCK=NONE` or a gh-ost-style helper; (c) **PostgreSQL** — `CREATE INDEX CONCURRENTLY` (non-blocking, no table lock; but it cannot run inside a txn and can leave an INVALID index on failure that needs cleanup — handle that).

**Today's operator workaround:** add the index on the target out-of-band (for PS-MySQL, via a deploy request you issue yourself; for PG, `CREATE INDEX CONCURRENTLY`). This gap is specific to indexes added AFTER cold-copy during continuous sync; cold-copy itself carries source indexes to the target.

### 25. Pre-existing target table whose schema DIFFERS from the source is silently accepted (`CREATE TABLE IF NOT EXISTS` skips) — target-schema-drift preflight — *demand-gated; mostly loud, but a relaxed-`sql_mode` MySQL target makes it a latent SILENT-coercion vector*

**Why (Bug 157 Phase-A finding, 2026-06-18).** A cold-start's create-tables step uses `CREATE TABLE IF NOT EXISTS` (`internal/engines/mysql/ddl_emit.go` ~915; PG path equivalent). If an **empty** target table already exists with a schema that differs from what sluice would emit (a stale/reused target, a hand-created table, a prior run's leftover DDL), the `IF NOT EXISTS` makes creation a NO-OP and sluice copies into the DRIFTED schema. This was the real cause of Bug 157's MySQL-target Error 1264: `lst-mysql-b` carried a stale **signed** `bigint` column from earlier drop/recopy cycles, so the otherwise-correct `bigint unsigned` cold-copy aborted mid-write. (The Bug-9 populated-target gate catches a target with *data*, but an *empty-but-drifted* table passes it.) **Class:** for an over-range integer it is LOUD (the write aborts) — but a narrower drifted column under MySQL relaxed `sql_mode` (e.g. target `VARCHAR(10)` vs source `VARCHAR(255)`) is the Vector-B SILENT-truncation class, so this is a latent silent-loss vector, not purely UX.

**What.** A preflight that, before copying into a pre-existing target table, compares the existing target column set/types against what sluice would emit and REFUSES LOUDLY (or WARNs under an explicit opt-in) on a material mismatch — naming the drifted columns + the remediation (`--reset-target-data` for a clean re-create, or fix the target DDL). This is engine-general (all types, not just `bigint unsigned`) and target-side. **Gotchas:** must not false-fire on benign/representational differences sluice itself introduces by design (e.g. the deliberate `bigint unsigned`→PG `bigint` mapping, wide-varchar→TEXT down-map, unconstrained-numeric widening — compare against sluice's OWN emitted type, not the source type); should compose with the existing cold-start gate ordering (after stale-backend reap, around the Bug-9 populated-target check). Demand-gated: surfaced by the large-scale program's reused-target churn, not a normal fresh-target migration.

### 26. PG-side concurrent CDC apply (the ADR-0104 key-hash-lane analog for the ADR-0092 Postgres path) — *operator-prioritized 2026-06-20; designed in [ADR-0105](../adr/adr-0105-postgres-concurrent-cdc-apply.md); cross-region PG apply throughput*

**Why (live finding, 2026-06-20, Track B vs B2).** The cross-region CDC apply wedge (item 23) is now closed for **MySQL** targets via ADR-0104 `--apply-concurrency` (key-hash lanes, ~4×, live-validated). It remains OPEN for **Postgres** targets: Track B2 (same Vitess source → PS-PG) uses the ADR-0092 within-transaction pipelined apply (single AIMD controller, ceiling 1000) and on a contended cross-region link was measured applying ~6.5 rows/s — ~10× slower than Track B's concurrent MySQL apply (~66 rows/s) under the same source. ADR-0092 overlaps only the commit RTT; it does not parallelize across keys, so a high-RTT PG target is apply-bound with no concurrency lever. The asymmetry is structural, not a bug — but it makes PG the slower cross-region target.

**What.** Bring the ADR-0104 concurrent key-hash apply to the Postgres `ChangeApplier`: merged change stream → W in-order lanes by `hash(qualified table, ordered PK values)` (same-key → same lane → in-order; cross-key concurrent), each lane committing on its own backend, resume position advanced only to a source-tx boundary durable across all lanes (the seq-frontier). It must compose with — not replace — ADR-0092 (a single lane W=1 ≡ the current pipelined path) and preserve ADR-0007 position-then-commit, ADR-0010 idempotent UPSERT, and the ADR-0089 keyless guard, exactly as the MySQL path does.

**Architecture decision (the hard call).** Most of ADR-0104's machinery is engine-neutral: the lane router, the `checkpointFrontier` (markCommitted / recordTxBoundary / checkpointPosition / waitForFrontier), the seq-monotone guard, the per-lane AIMD wiring, and the in-lane tx-killer/error recovery. Today it lives in `internal/engines/mysql/change_applier_concurrent.go`. Two options: **(A) EXTRACT** the engine-neutral lane+frontier core into a shared package (e.g. `internal/cdcapply`) that both engines drive via a small per-engine "apply one batch on this backend" seam, then wire the PG applier to it; **(B) REPLICATE** the pattern in `internal/engines/postgres`. (A) is the IR-first, DRY choice and avoids two divergent exactly-once implementations (the Bug-74 class risk of a fix landing in one engine only), at the cost of a careful extraction that must not perturb the live-validated MySQL path; (B) is faster but doubles the critical exactly-once surface. **Lean (A)** — but the extraction is itself a CRITICAL refactor (the MySQL path is GA and live-load-validated), so it needs the full `-race` integration gate + a serial-vs-W byte-identical differential on BOTH engines before tag, and the MySQL behavior must stay byte-identical across the extraction (pin it first, refactor under the pin).

**Gotchas.** PG has no PlanetScale tx-killer (Error 1105) but DOES have serialization failures / deadlocks (SQLSTATE 40001/40P01) — the in-lane retry/recovery seam must map to those for PG, not assume the MySQL error shape. The PG path's "dedicated backend per lane" must respect the connection budget (the ADR-0092 conn-budget fix, v0.99.47). Keyless tables (ADR-0089 at-least-once WARN) route to a single lane (no PK to hash) exactly as MySQL does. **CONCURRENCY chunk** — `-race` integration gate before tag, on both engines.

### 27. Relax `MinimizeSkew` on the multi-shard VStream concurrent-apply path (remove the artificial per-shard catch-up cap) — *optional throughput optimization surfaced by the item-23 v0.99.81 RCA; CRITICAL ordering, design-gated*

**Why (item-23 Phase-A RCA, 2026-06-20).** The VStream request sets `MinimizeSkew=true` (`cdc_vstream.go:894`) so vtgate holds the ahead shard back to keep the merged stream commit-time ordered. Under an apply-deficit this is the benign-but-throughput-capping per-shard hold item 23 documents: the ahead shard's delivery is frozen until the behind shard fully drains (~hours for a large backlog), even when apply > source. Live-confirmed on v0.99.81: shard `80-` delivery frozen for many minutes while `-80` drained at ~123 GTID/s.

**What.** Make `MinimizeSkew` opt-out (or default-off) **specifically when the concurrent key-hash apply is engaged**, so both shards stream concurrently and the held shard isn't delayed during a backlog drain. **Why it is safe for sluice's correctness:** range-sharding means a given `(table, PK)` lives in exactly ONE shard, so there is NO cross-shard same-key dependency; the key-hash lanes already serialize same-key changes within a shard, and `StopOnReshard=true` closes the only window (a reshard) where a key could transiently exist in two shards. So cross-shard commit-time ordering buys sluice nothing — the per-shard positions are tracked independently and the seq-frontier persists only fully-durable boundaries regardless of inter-shard skew. **Gotchas / why design-gated:** this touches CDC ordering + the position token, the CRITICAL exactly-once surface — must audit every consumer that might assume commit-time merge order (the position orderer, the snapshot→CDC stitch, the per-shard watchdog's "peer advanced" signal), confirm resume-safety with widely-skewed per-shard positions, and validate with a live A/B (MinimizeSkew on vs off) measuring both-shard drain + a `-race` integration pass. Lower priority than item 26 because item 23(c)'s throughput fix already prevents the pathology from forming in steady state; this only improves catch-up latency for an already-formed backlog.

### 28. ✅ FIXED (2026-06-20, v0.99.87 + v0.99.88): native-MySQL CDC supports `binlog_transaction_compression=ON` — `TRANSACTION_PAYLOAD_EVENT` unpacked + applied, resume-safe for large transactions — *was HIGH silent non-application; v0.99.87 added support, v0.99.88 fixed a CRITICAL large-payload resume silent-loss in that support*

**v0.99.88 follow-up (commit `efa4575`) — CRITICAL large-payload resume silent-loss fix.** v0.99.87 stamped every inner payload event with the payload's END LogPos. Correct for a single-batch (small) transaction, but a transaction larger than `--apply-batch-size` commits in several target batches mid-payload, each persisting a position PAST the whole payload — so a mid-payload interruption + warm-resume silently dropped the un-applied rows. Caught by a large multi-row compressed-transaction resume test (the "test the size class, not the representative" discipline; the single-row v0.99.87 pin couldn't trip it). Fix: inner events have no independent on-wire positions, so the payload is atomic for resume — every inner row/table-map/BEGIN event anchors at the payload START (a mid-payload interrupt re-reads + idempotently re-applies the whole payload, ADR-0010), and only the final inner event (the commit) advances to the payload END. Live-validated (20000-row compressed INSERT killed mid-apply → resume converges, src md5 == dst md5) + pinned (`TestCDCReader_BinlogTransactionCompression_LargePayloadPositionAnchoring`, fail-without/pass-with). v0.99.87 warned.

**FIX (v0.99.87, commit `c9d9cbe`).** `dispatch` (`internal/engines/mysql/cdc_reader.go`) now has a `case *replication.TransactionPayloadEvent:` that iterates the decoded inner events (go-mysql v1.15.0 `Events []*BinlogEvent`) through the same routing, stamping each inner header with the OUTER payload event's `LogPos` (the transaction boundary) so the persisted file/pos resume point is payload-aligned (resume starts at the NEXT payload, never the mid-payload misalignment that fails with "no corresponding table map event"); GTID mode ignores LogPos so the stamp is a no-op there. `isRowRelevantEvent` recognizes the payload (no-events watchdog suppression). Live-validated (Docker MySQL 8.0, compression ON): compressed INSERT/UPDATE/DELETE apply exactly-once (src md5 == dst md5) in steady state + warm-resume from a compressed-tx position drains the backlog. Pinned by `TestCDCReader_BinlogTransactionCompression_AppliesAndResumes` (reader-level, steady-state + warm-resume), verified to FAIL without the fix ("got 0 row changes; want 3") and pass with it. **Note (investigated 2026-06-20, NOT a real issue).** During the repro, UPDATEs to a `TIMESTAMP(6)`/`BLOB`-bearing table appeared to silently no-op (`buildUpdateSQL` uses a full-row `WHERE` over all non-generated columns, so any column whose before-image ≠ the target's stored value matches 0 rows — tolerated per the Bug-6 comment). A focused CLEAN repro disproved a generic class: cold-copy round-trips `TIMESTAMP(6)`, `BLOB`, `DOUBLE`, `DECIMAL`, and `NULL` byte-for-byte, so the full-row WHERE matches and UPDATE/DELETE apply correctly (src md5 == dst md5). The earlier misses were an ARTIFACT of the messy multi-run repro (repeated `--reset-target-data` + partial applies left the target's `ts`/`blob` out of sync with the source before-image). The full-row WHERE is robust as long as cold-copy is value-faithful (guarded by the value-fidelity pins); JSON — the one historically-fragile type — is already handled via `CAST(? AS JSON)`. No action needed.

**(historical) CONFIRMED (2026-06-20, clean local repro): native-MySQL CDC did NOT support `binlog_transaction_compression=ON`** — *HIGH silent non-application; loud-failure-discipline violation.*

**ROOT CAUSE (confirmed by clean local repro + code-reading, 2026-06-20).** sluice's binlog event dispatch (`internal/engines/mysql/cdc_reader.go` `dispatch`, ~line 601) has cases for `RotateEvent`/`GTIDEvent`/`TableMapEvent`/`RowsEvent`/`QueryEvent`/`XIDEvent` but **NO case for `*replication.TransactionPayloadEvent`** (MySQL 8.0.20+ `binlog_transaction_compression=ON`, event type `0x1f`/31 — the ZSTD-compressed container that wraps a whole transaction's TABLE_MAP + ROWS + XID). A compressed transaction therefore falls through to the `default` (ignored): its inner events are never unpacked, **nothing is applied, the resume position never advances, and there is NO error** — only a soft `"no binlog row events received during startup grace period"` WARN while heartbeats keep the stream alive. `isRowRelevantEvent` (~line 586) likewise omits the payload type. **Reframes the original Track-D symptom:** this is NOT a resume edge — sluice has zero compression support; the Track-D `"no corresponding table map event"` was a variant of the same gap surfacing under its (ENOSPC-confounded) conditions.

**Clean repro (local Docker, 2026-06-20).** MySQL 8.0 source with `--binlog-transaction-compression=ON --gtid-mode=OFF` (file/pos), native MySQL→MySQL sync: cold-start copies the 500-row seed fine and enters CDC at `binlog.000003:4212`; then 5 compressed transactions (UPDATE/INSERT/DELETE) are generated → **dst stays at 500, id-10 stays `seed-10`, persisted position stays frozen at `4212`, zero applied, no error.** `SHOW BINLOG EVENTS` confirms the txns are present as `Transaction_payload` events. Warm-resume behaves identically (frozen at 4212). The reader is effectively stuck at the first payload event from the moment a compressed transaction appears.

**Severity: HIGH, silent non-application.** Not silent *data loss* (source intact; target merely frozen at the cold-start snapshot, uncorrupted), but a CDC tool that silently replicates NOTHING while looking healthy is the loud-failure-discipline's core sin. `binlog_transaction_compression` is a real, documented MySQL 8.0.20+ feature (default OFF, commonly enabled for WAN replication / disk savings), so a real operator config silently breaks continuous sync.

**Fix.** (1) **Immediate (loud preflight refusal):** detect `@@binlog_transaction_compression=ON` at CDC stream start and REFUSE LOUDLY (clear message: sluice does not yet support binlog transaction compression; set `binlog_transaction_compression=OFF` on the source — affects only newly-written binlogs — or wait for full support). Converts silent-non-application → loud failure; small, safe. (2) **Real fix (full support):** add `case *replication.TransactionPayloadEvent:` to `dispatch` that iterates `e.Events` (go-mysql v1.15.0 decodes the inner `[]*BinlogEvent`) and processes each inner TableMap/Rows/Xid as if uncompressed. **CRITICAL gotcha:** the inner events carry synthetic/relative `LogPos`; the persisted resume position MUST use the OUTER payload event's `LogPos` (the transaction boundary), never an inner event's — getting this wrong is exactly the "no corresponding table map event" resume failure. Needs: file/pos AND GTID-mode coverage, exactly-once + warm-resume-mid-compressed-stream integration pins (compression ON), and the `-race` gate. **Concurrency/CDC-path chunk.**

### 29. ✅ FIXED (2026-06-21, v0.99.89): serial CDC applier persisted a mid-transaction checkpoint → native-MySQL warm-resume crash-loop — *HIGH; loud-failure (crash, not silent loss) but an unrecoverable-without-resnapshot stall; this was item 28's TRUE root cause, NOT compression*

**ROOT CAUSE (confirmed live on the large-scale program, 2026-06-21).** The shared serial batched-apply loop (`internal/appliershared/batch_loop.go`) persisted the LAST ROW's binlog position (`lastPos.Token`) on EVERY flush — row-cap, byte-cap, idle-grace, keyless guard, channel-close — not just on a source-transaction boundary (`TxCommit`). When a source transaction has more rows than the effective batch size — which happens whenever the AIMD controller collapses the batch to 1 under a slow target (e.g. a cross-region PlanetScale target during initial backlog drain) — the transaction splits across batches and a mid-batch flush persists a position pointing INTO the transaction, at a `ROWS` event. MySQL file/pos resume cannot start mid-transaction: go-mysql seeks to the byte offset, reads a `ROWS` event whose `TABLE_MAP` was earlier in the same tx, and dies with `"no corresponding table map event"` — an unrecoverable warm-resume crash-loop. Confirmed via `mysqlbinlog` (the persisted `binlog.000021:13244716` pointed at an `Update_rows: table id 507`) + the original CDC entry started fine at an earlier in-process position (`173733`) and only the COLD reconnect from the persisted mid-tx checkpoint crashed. The concurrent `laneapply` frontier was already immune (it advances the position only to a fully-durable source-tx boundary — "must NOT persist a partial point"); the serial loop lacked the same guard. **Reframes item 28:** the Track-D `"no corresponding table map event"` was this checkpoint bug, not binlog compression (the failing event was `0x1f` = UPDATE_ROWS on a clean, compression-OFF source).

**FIX (v0.99.89, commit `f22476e`).** `BatchConfig.CheckpointOnlyAtTxBoundary` (MySQL=true; Postgres unchanged — PG LSN resume tolerates a mid-tx restart point and the walsender resends whole transactions, so its slot ack keeps advancing per flush). Under the flag a non-boundary flush commits its DATA but SKIPS the position write, so the persisted resume position advances only to a source-tx / DDL boundary; a mid-tx interrupt re-reads the whole in-flight transaction from the previous boundary and idempotently re-applies it (ADR-0010 — at-least-once for the interrupted tx, as the keyless guard and concurrent frontier already are). The trailing `TxCommit` of a tx whose rows committed in earlier batches lands on an empty batch (the pre-tx wait loop) and is checkpointed via `writeBoundaryOnly` (a dedicated position-only tx) — the only place the boundary can advance under batch-size 1. Pinned at the shared-loop seam (`TestRunBatchLoop_CheckpointOnlyAtTxBoundary_NeverPersistsMidTxPosition` + the PG-unchanged guard + the atomic-boundary case); composes with the existing reader-level resume tests that prove go-mysql resumes cleanly from a tx-boundary position. `-race` integration gate before tag (CDC/exactly-once chunk).

**Related (Track-D slow-apply, investigated 2026-06-21, NOT a bug).** The accompanying `p95≈10s` AIMD floor was a TRANSIENT effect of the initial backlog drain running large (1000-row) batches that each took ~10s, polluting the AIMD p95 window and collapsing batch size to 1; the target is fast (TCP RTT ~10–14ms, ~94ms/commit incl. mysql-client startup). With the checkpoint bug fixed — or `--apply-concurrency` (immune) — apply keeps pace.

### 30. Non-Metal PlanetScale target: storage auto-grow resilience + low-tier throughput comparison (operator-flagged 2026-06-21) — *resilience test; Track D continuation*

**Why.** Every PlanetScale target tested so far has been **Metal** (fixed, pre-provisioned storage sized well above the test data — no in-flight resizing). **Non-Metal** PlanetScale databases start with a small volume (~10 GB) and **auto-grow** behind the scenes (10 GB → 39 GB → larger) as data lands. The operator has observed that an in-progress import can **stall during an automatic storage-resize operation**. sluice must be **resilient** to that: treat the resize-induced stall/error as transient (reconnect / warm-resume / retry) and **continue automatically** once the volume has scaled up — never die, never require manual intervention, never silently lose or duplicate. This is the inverse environment of the Metal runs and exercises a class of transient platform stall the Metal tests structurally cannot.

**What.** As a Track-D continuation, provision a **non-Metal** PlanetScale MySQL database and run the Track-D vanilla-MySQL→PS cold-copy + CDC against it, sized so the ~26 GB corpus crosses at least the 10 GB→39 GB (and ideally a later) auto-grow boundary mid-copy. Observe: does the bulk-copy / CDC apply stall during a resize? If so, does sluice's error classification treat it as retriable (reconnect + warm-resume, no data loss, idempotent re-apply) and resume automatically once storage scales? Capture the resize events (timing) against the apply timeline. Compare end-to-end throughput vs the Metal-320 baseline.

**Size options (operator-offered).** (a) **`PS-320`** — apples-to-apples with the Metal-320 runs for the cleanest throughput comparison (but fewer/late resize events given larger initial headroom). (b) **`PS-80`** — mid-range; a balance of resize events + reasonable throughput. (c) **`PS-10`** — smallest/lowest-resourced; the MOST auto-grow events (10 GB corpus crossing many steps to 26 GB) AND the most resource contention → the strongest resilience signal + a low-tier throughput data point. *Recommendation: run **PS-10** (or PS-80) for the resilience observation — maximize resize events — and optionally a **PS-320** pass for the clean throughput-vs-Metal number.*

**Gotchas.** (1) The resize stall may surface as a connection hang, a write/commit timeout, or a specific PlanetScale error code — confirm sluice's applier-error classifier (ADR-0038) routes it to retriable, not fatal; this is the same resilience family as the v0.99.84 bounded-verify-timeout and the #54 tx-killer handling. (2) Verify warm-resume after a resize-induced reconnect re-reads from a clean tx boundary (now guaranteed by item 29) and converges exactly (src checksum == dst). (3) Non-Metal storage reclaim is not instant — account for it in teardown/space accounting. (4) Loud-failure-safe regardless: a resize stall must never be mistaken for completion.

### 31. Fast-by-default CDC apply: make `--apply-concurrency` default to an adaptive value instead of serial (operator-flagged 2026-06-21) — *out-of-the-box throughput; design-gated (ADR), default-behavior change*

**Why.** Most of sluice's speed axes are ALREADY adaptive-default-on and connection-budget-bounded: `migrate`/`backup`/`restore --table-parallelism` (auto: 4), FAST cold-start `--bulk-parallelism` (min(8,NumCPU)) + `--table-parallelism` (auto: 4), and VStream snapshot `--copy-fanout-degree` (auto: 4). The ONE major lever still defaulting to serial is **`--apply-concurrency`** (steady-state CDC key-hash apply, ADR-0104/0105): `ApplyConcurrency` defaults to `0` (= serial) unless the operator opts into `W>1` (`internal/pipeline/streamer.go:~508`). So CDC catch-up/steady-state throughput is the only part of sluice that is NOT fast out-of-the-box. It shipped a v0.99.77 PREVIEW → GA in v0.99.80, but the default never flipped.

**Why it's safe to flip NOW (the gate the tenets require is met).** The concurrent apply path is GA + hardened: per-lane AIMD self-throttle (auto-backs-off on a weak/slow target), in-lane PlanetScale-tx-killer re-chunk recovery (v0.99.80), the lane-local committable-size read cap (v0.99.81), exactly-once cross-lane frontier on BOTH MySQL (ADR-0104) and Postgres (ADR-0105), Bug 158 (PG concurrent first-boundary silent loss) fixed in v0.99.83, and v0.99.89 hardened even the serial default path. The item-30 PS-10 run is live evidence for whether budget-bounded concurrency stays safe + self-throttles on a 1/8-vCPU instance (the worst case for a concurrent-by-default decision).

**What.** Make `--apply-concurrency` resolve `0 → auto:N` (a conservative, connection-budget-bounded default, e.g. `min(4, budget-derived)`), keeping `=1` as the explicit serial opt-out — mirroring how `--table-parallelism` already does `0 = auto:4 / 1 = disable`. **Critical design detail:** resolve the default at the STREAMER/applier construction level, NOT only in the CLI — exactly as the sibling `AutoTune` field does ("the default at the streamer level is also true so any programmatic caller that doesn't set it gets the opted-in shape"). Resolving only in the CLI re-triggers the zero-value-safe-default trap (the v0.99.51 / Bug-158 family): every test, broker, and programmatic Streamer would silently get serial.

**Gotchas.** (1) MySQL targets have no connection-slot probe (`--max-target-connections` is "inert against engines without a connection-slot model (MySQL target)"), so for PlanetScale-MySQL the auto cap must be a sane fixed ceiling, not a probe. (2) Keyless tables: at-least-once amplification under concurrency is already guarded (per-row keyless flush) — re-verify under the new default. (3) Ordering/exactly-once already handled by the frontier; the differential (serial == W>1, byte-identical) pin must run under the new default. (4) Ships as its OWN release with a full regression cycle — it changes default behavior for EVERY user. (5) Decide whether `--copy-fanout-degree`/`--table-parallelism` auto:4 values should also scale with detected target capacity rather than the fixed 4 (same ADR).

### 32. PlanetScale Prometheus metrics integration — consume target CPU / memory / storage (+ lag, connections) for proactive adaptivity + operator observability (operator-flagged 2026-06-21) — *PlanetScale-specific; optional control-plane capability; design-gated (ADR)*

**Why.** sluice's adaptivity today is **reactive**: the AIMD controller sees only per-batch commit latency, and the tx-killer / storage-resize / throttle handling all respond *after* the target pushes back. PlanetScale already exports a rich per-branch Prometheus metrics surface that would let sluice see the target's resource state **directly and act proactively**. Operator's stated priority: **CPU / memory usage + storage usage** first (other metrics secondary).

**Exposure (confirmed from the docs).** Per-org endpoint `https://api.planetscale.com/v1/organizations/${ORG}/metrics`, scraped via Prometheus **HTTP Service Discovery** (per-branch targets, ~10-min refresh); auth is a **service token** (`${TOKEN_ID}:${TOKEN}`) granted the **`read_metrics_endpoints`** permission — a control-plane credential distinct from the data-plane DSN (same opt-in posture as ADR-0103's deploy-request token).

**Key metrics.** CPU `planetscale_pods_cpu_util_percentages`, memory `planetscale_pods_mem_util_percentages` (both engines); storage — Vitess `planetscale_vttablet_volume_{available,capacity}_bytes` + `planetscale_vttablet_table_storage_all_bytes`, Postgres `planetscale_volume_{available,capacity}_bytes`; plus lag (`planetscale_{mysql,postgres}_replica_lag_seconds`, `planetscale_workflow_vreplication_lag`, `planetscale_postgres_wal_size_bytes`) and connections (`planetscale_edge_active_connections`, `planetscale_vttablet_connection_pool_active`, PG `planetscale_postgres_connection_state` / `planetscale_postgres_settings_max_connections`).

**What (three uses, in priority order).** An OPTIONAL PlanetScale capability that polls the metrics endpoint OFF the hot path (~15–30 s), filters to the sync's target branch/keyspace, and feeds:
- **(a) Proactive back-off** — when target **CPU% / memory%** saturate, reduce apply concurrency / batch size *before* a tx-kill or stall (a forward-looking complement to the reactive AIMD; directly feeds item 31's adaptive concurrency and item 19's Vitess-throttle resilience). The connection metrics also give a real budget number for the item-31 MySQL "no connection-slot probe" gap.
- **(b) Storage-resize awareness** — read `volume_available_bytes` / `volume_capacity_bytes` to **anticipate** the non-Metal auto-grow (item 30) and emit a clear "target storage resizing" signal instead of a confusing timeout; pairs with the item-30 resilience work (detect → explain → ride through).
- **(c) Operator observability** — surface target CPU/mem/storage/lag alongside sluice's own progress (`sluice diagnose` / logs / an optional dashboard panel) so the operator sees *why* apply is slow without leaving the tool.

**Gotchas.** (1) PlanetScale-SPECIFIC → must be a clean **optional capability** behind a capability declaration, with **zero PS leakage into the engine-neutral core** (IR-first + contain-Postgres/PS-complexity tenets); wired only when the operator supplies `--planetscale-org` + the metrics service token. (2) **Advisory only** — metrics are a proactive *hint*; the frontier (exactly-once), AIMD, and tx-killer machinery stay authoritative, and a metrics outage/staleness must never stall or corrupt the sync (degrade to today's reactive behavior). (3) The documented set has **no throttler-state or tx-kill metric**, so item 19 / the Vitess throttle signal still needs its data-plane source (`SHOW VITESS_THROTTLED_APPS`); metrics complement, not replace. (4) HTTP-SD is per-org; sluice would query the org endpoint and select the branch series (or hit the discovered per-branch target). (5) **Sibling sub-item (the inverse direction):** sluice could ALSO *export* its OWN Prometheus `/metrics` (progress %, lag, rows/s, per-lane AIMD batch size, tx-killer/resize counts) so operators watch sluice in the SAME Grafana/Datadog stack as their PlanetScale metrics — a smaller, engine-neutral, broadly-useful observability win that doesn't need any PS credential. Worth scoping alongside (a)–(c).

### Performance frontier (2026-06-12 profiling round — where the remaining time actually goes)

Two evidence rounds on v0.99.41-era main; profiles under `workspace/pprof-migrate-v09941/`
(gitignored), experiment logs in sluice-testing `workspace/perfB/`:

- **PG→PG migrate (raw-copy lane) is wire-bound, not compute-bound.** pprof on
  the 136 GB rig: sluice on-CPU ~6% of wall; the ADR-0078 passthrough does zero
  per-row work (0 flat samples in sluice functions; no steady-state allocs; no
  lock contention; `--table-parallelism=1` has the identical CPU shape). The
  only sluice-reducible item is the reader→writer `io.Pipe` memmove at <0.5% of
  wall. **Consequence: the positional-`ir.Row` rewrite (repo-audit P3) is NOT
  justified by this lane** — if it pays anywhere it's the cross-engine typed
  lane (TOAST/wide-row corpus differential is the test that would decide).
- **MySQL-target migrate sits at the InnoDB server floor.** sluice's LOAD DATA
  path (already 8-way parallel within-table via ADR-0019 chunking) matches a
  hand-run `LOAD DATA LOCAL INFILE` to rig noise (~95k vs ~98k rows/s on the
  1M-row anchor); InnoDB single-table clustered-PK ingest does not parallelize
  (1×/4×/8× concurrent loaders all ≈100k) and is ingest-bound, not commit-bound
  (`innodb_flush_log_at_trx_commit=2` ≈ +5%). **No Go-side or parallelism lever
  moves this number; myloader faces the same wall.** Per-direction tracking +
  the full numbers: sluice-testing `THROUGHPUT.md` (#34).
- **10k-tiny-tables (decides #56, measured 2026-06-12 — sluice-testing
  `workspace/perfC/VERDICTS.md`): sluice ≈13 ms/table vs pgcopydb ≈128 ms/table
  on the same corpus — sluice is ~10× AHEAD on this shape**, so the tiny-table
  batch lane is demand-gated, not perf-gated: do not build without a real user
  hitting it. The future lever, should demand appear, is precise: ~80% of wall
  is the serial CREATE TABLE phase (one network round-trip per table,
  `internal/engines/postgres/schema_writer.go:288-296`, flat across
  `--table-parallelism` which only governs bulk_copy) — a batch lane must
  batch DDL statements per round-trip, not COPY.
- **Wide-row typed lane (decides positional-`ir.Row`, audit P3 — same round):
  do NOT schedule the XL rewrite.** On a 750k × ~4.2 KB/row MySQL→PG copy
  (~200 MB/s, 8-way), the typed lane is 22.5% on-CPU and driver-dominated:
  pgx binary COPY-encode 42%, read/scan ~30%, sluice-owned typed-lane ~15% of
  which the `ir.Row` map cost (`copy_source.go:68` per-column lookup + value
  boxing) is ~8-9%. A perfect positional rewrite addresses single-digit CPU —
  far below the XL bar. Re-open only on separately-measured GC pressure at
  high concurrency.
- Remaining measured frontier worth a future chunk: backup-path zstd (32% of
  backup CPU post-codec — level sweep / parallel chunk compression).

### Connection-resilience + index-build tuning arc (2026-06-02, unreleased)

Surfaced while building/load-testing the sluice-powered Heroku→PlanetScale migrator (`sluicesync/sluice-heroku-migrator`, a private fork of `planetscale/heroku-migrator` with the replication engine swapped from Bucardo to sluice's postgres-trigger engine). Five chunks merged to `main`:

- **connection-resilience Phase 1** (`internal/engines/postgres/connect_appname.go`, `connection_budget.go`; `internal/pipeline/connection_budget.go`) — every PG connection now carries `application_name=sluice/<role>/<id>` (role-before-id so the prefix+role survive PG's 63-byte truncation); a target connection-budget preflight auto-caps parallel COPY to fit the slot budget and **refuses loudly** when `copy_budget < 1` (`--max-target-connections` override); PG connections adopt the `netkeepalive` dialer. Validated live (the `application_name` labels + a clean cross-region migration that previously failed on slot exhaustion).
- **connection-resilience Phase 2** (`internal/engines/postgres/stale_backend.go`) — cold-start preflight **detects** stale `sluice/%`-labeled, own-role, lock-holding/idle-in-tx backends and **reaps** them only with `--reap-stale-backends`. Terminate scoping is a single greppable constant; `pg_terminate_backend` sits in the SELECT projection (guaranteed scope-before-kill). The orphan-lockout failure mode (a SIGKILL'd COPY half-open, holding a target lock + a slot) is what this addresses. See [`docs/dev/notes/orphaned-backend-resilience.md`](notes/orphaned-backend-resilience.md).
- **connection-resilience Phase 2b** (`internal/pipeline/copy_backoff.go`, `copy_parallelism_gate.go`; `internal/engines/postgres/connection_slot.go`) — AIMD backoff on mid-copy `53300`/superuser-reserved-slots: multiplicative-decrease parallelism + retry the chunk rather than abort, with a bounded loud give-up. Classifier is narrowly scoped to `53300` (never masks a real error); retry is double-copy-safe because `53300` fails at connection-establishment, before any COPY write.
- **index-build tuning Phase A** (`internal/engines/postgres/index_build_tuning.go`; `CreateIndexes` refactor) — the dedicated index-build session SETs a `maintenance_work_mem` autotuned from `shared_buffers` (the clean RAM proxy; `RAM ≈ shared_buffers × 7`), pushing past the provider's ~4%-of-RAM steady-state default during the idle build phase (`--index-build-mem` override). - **index-build tuning Phase B** — concurrent index builds via a bounded worker pool; N clamped by the connection budget, memory budget (**aggregate `N × per-build-mem ≤ budget`** — the OOM guard), and `--index-build-parallelism`. The note's tier data shows parallelism barely helps below PS-640 (`max_worker_processes` flat at 4), so auto-N stays conservative.

### GEOMETRY / SPATIAL — PostGIS-aware translation (v0.28.0)

- **PG → MySQL geometry round-trip with SRID preserved (Bug 26 closed).** PG `geometry(POINT, 4326)` lands as MySQL `POINT NOT NULL SRID 4326` instead of dropping to SRID 0; ST_SRID(loc) on the target returns the source SRID. The MySQL DDL emit grew a `SRID <n>` clause on geometry columns when the IR carries a non-zero SRID; the IR's `ir.Geometry.SRID` field threads from the PG schema reader's `geometry_columns` lookup through to the MySQL writer.
- **VStream POINT bytes prefix stripping (Bug 27 closed).** VStream's `query.Type_GEOMETRY` cell decoder now strips the 4-byte little-endian SRID prefix that MySQL's GIS storage format prepends to the OGC WKB body. Downstream consumers see raw WKB matching the IR contract for `ir.Geometry` values, identical to the vanilla `database/sql` driver path. Fix is fixture-tested at the unit-test layer; operator-run end-to-end verification via `psverify` (PlanetScale credentials) follows the existing PS test pattern.
- **Cross-engine geometry no longer refuses.** `pipeline.checkCrossEngineSupportable` previously refused PG → MySQL `ir.Geometry` blanket; with SRID round-trip closed, the conservatism is no longer load-bearing. Refusal narrows to `ir.ExtensionType` (ADR-0032 pgvector / hstore / etc.) which has no portable MySQL equivalent. Chain-restore + sync-from-backup paths now ship geometry through cross-engine without operator intervention.
- **`postgis/postgis:16-3.4` integration image.** New `integration postgis` build-tag layer for the cross-engine PG ↔ MySQL geometry round-trip tests. Image is ~500 MB (heavier than `postgres:16` and `pgvector/pgvector:0.7.4-pg16`) so the postgis-tagged tests run as a separate go-test invocation in the CI Integration job rather than expanding the default `integration` tag's footprint.
- **PostGIS-absent target stays loud-failure.** PG SchemaWriter and RowWriter detect PostGIS at engine open time via `pg_extension`; without the extension installed, ir.Geometry columns refuse with `GEOMETRY requires PostGIS; install with CREATE EXTENSION postgis;` rather than silently downgrading to `bytea`. See [ADR-0035](adr/adr-0035-postgis-geometry-spatial-support.md).

### Multi-source aggregation Phase 1 + 2 (`--target-schema` + stream-id collision detection)

- **`--target-schema NAME` flag** on `migrate`, `sync start`, `schema preview`, `schema diff`. PG-only — flat-namespace engines (MySQL) refuse with a clear "use a different --target DSN database" message. When set, every emitted CREATE TABLE / ALTER TABLE / CREATE INDEX / CREATE TYPE prefixes the identifier with the schema name; the schema is auto-created via `CREATE SCHEMA IF NOT EXISTS`. Type-name derivation (PG enums) namespaces through the schema so two sources both having `accounts.status` enums coexist without collision. Implements Shape B (microservices → analytics warehouse) of the [proto-design](design/multi-source-aggregation.md). See [ADR-0031](adr/adr-0031-multi-source-aggregation-target-schema.md).
- **Stream-id collision detection.** `sluice_cdc_state` grew a `source_dsn_fingerprint TEXT NULL` column (additive, idempotent migration via `ADD COLUMN IF NOT EXISTS`). Streamer fingerprint check at startup refuses if a different source DSN tries to reuse an existing stream-id. Catches operator-typo / wrong-source cases loudly before any data moves. Fingerprint is truncated SHA-256 of host+port+database — credential rotation doesn't trip false-positives.
- **New IR surfaces.** `ir.SchemaSetter`, `ir.SourceFingerprintRecorder`, `ir.StreamStatus.SourceDSNFingerprint`. PG implements via SetSchema on SchemaReader / SchemaWriter / RowReader / RowWriter / ChangeApplier; the applier carries a separate `controlSchema` field so per-target `sluice_cdc_state` stays in the DSN's default schema while user-data INSERT/UPDATE/DELETE land in the per-source schema. MySQL deliberately doesn't implement (the validate-time refusal is the load-bearing gate).
- **Operator UX.** `sluice sync status` renders cross-source streams from the same control table; the existing query already lists every row by stream-id. Each per-source stream stays a separate `sluice sync start` invocation — N-process aggregation rather than single-process multi-source (failure isolation, resource isolation, k8s/systemd lifecycle).

### Mid-stream live add-table — strict zero-loss correctness (v0.32.0)

- **v0.24.0's documented best-effort gap closed (ADR-0036).** A four-round Phase A diagnose campaign (A.1 ruled out long-txn-straddle / snapshot-LSN-race / broker-re-emit; A.2 v0.29.0 falsified reframed M3 via per-row source-LSN instrumentation; A.3 v0.31.x confirmed applier-side drop 10/10) localized the drop to `internal/engines/postgres/change_applier.go::dispatch` returning `nil` (silently dropping) when `colTypesFor` hit `errUnknownTable` because the target table didn't exist yet. Fix (v0.32.0, ~150 LOC): `AddTable.Run` step 3a creates the target table BEFORE `ALTER PUBLICATION ADD TABLE`; `runBulkCopyForAddTable` routes the copy through `ir.IdempotentRowWriter`. Engine-symmetric (MySQL's ADR-0034 filter-flip didn't manifest the race but inherits the cleaner shape). `TestAddTable_LiveMode_PG_UnderLoad` tightened to a strict zero-loss assertion; the Phase A.3 capture probe stays as a permanent proof-of-falsification artifact. Forward options B (dual-slot) / C (operator quiesce) shed for this surface but retained for unrelated edge cases.

### Mid-stream add-table — MySQL Phase 2 (live add via filter-flip)

- **`sluice schema add-table TABLE --stream-id ID --no-drain`** — now also works against a MySQL source. Same CLI surface as the PG path; the orchestrator dispatches by source-engine capability and routes binlog sources through the new filter-flip mechanism. Operator-facing UX is identical (one flag, no operator restart). See [`adr-0034-mysql-phase-2-live-add-table.md`](adr/adr-0034-mysql-phase-2-live-add-table.md).
- **New surfaces.** Pipeline-side optional interfaces `liveAddedTablesWriter` / `liveAddedTablesReader` (target applier records / streamer reads the filter-flip column on `sluice_cdc_state`). MySQL `ChangeApplier.RecordLiveAddedTable` writes the new table into the comma-separated `live_added_tables` column; `ChangeApplier.ReadLiveAddedTables` is what the streamer's poll consults. Streamer side: `liveAddedFilter` (atomic-pointer-to-immutable-map) + a poll goroutine paired with the existing stop-signal poll. The dispatch filter merges base + live-added with OR semantics — additive grants without touching the operator's existing `--include-table` / `--exclude-table`.
- **Best-effort caveat (parallel to PG Phase 2).** Events on the new table arriving at the streamer's dispatch BEFORE its poll observes the new column value are dropped (poll cadence 5s default). Operators with high write rates on the new table at the moment of live-add use the drained flow or quiesce briefly. Same shape as PG Phase 2's in-flight gap (ADR-0030 § "what could go wrong"); the strict-zero-loss roadmap entry now covers both engines.

### Mid-stream add-table — Phase 2 (live add, no drain) — PG

- **`sluice schema add-table TABLE --stream-id ID --no-drain`** — runs add-table against an actively-streaming sync without first running `sync stop --wait`. Strategy C variant (c) from the proto-ADR: single slot, publication-add-then-snapshot ordering. v0.27.0 extended the same CLI surface to MySQL via filter-flip (entry above + ADR-0034). See [`adr-0030-mid-stream-live-add-table.md`](adr/adr-0030-mid-stream-live-add-table.md).
- **New surfaces.** Pipeline-side optional interfaces `slotPositionReader` / `snapshotLSNExtractor` / `lsnComparer` so engines opt into the live-mode invariant check structurally. `Postgres.Engine.ReadSlotPosition` reads `confirmed_flush_lsn` from `pg_replication_slots`; `Postgres.Engine.ExtractSnapshotLSN` and `Postgres.Engine.CompareLSN` close the loop without leaking the engine's position envelope into the orchestrator.
- **Operator safeguards.** Live mode refuses on engine pairs that expose neither `publicationAdder` (PG path) nor `liveAddedTablesWriter` (MySQL path) — clear error directing at the drained flow. Captures the active stream's slot `confirmed_flush_lsn` before publication-add; verifies snapshot-LSN ≥ slot-LSN after the snapshot opens — refuses loudly if the invariant fails (would silently drop events on the new table; ADR-0030 explains why standard ordering can't trip this in practice but the check pins the invariant against future regressions).

### Mid-stream add-table (Phase 1 MVP)

- **`sluice schema add-table TABLE --stream-id ID`** — brings a new source table into an active CDC stream's scope without a destructive `--reset-target-data` cycle. Drained-stream workflow only (Phase 1): operator runs `sync stop --wait`, then `schema add-table`, then `sync start --resume`. Live add-table without the drain shipped in Phase 2 (entry above). Implements the design in `docs/dev/design/mid-stream-add-table.md`.
- **New surfaces.** `pipeline.AddTable` orchestrator (mirrors `Migrator` shape); pipeline-side optional interfaces `publicationAdder` / `snapshotSlotOpener` / `slotDropper` so engines opt in structurally. `Postgres.Engine.AddPublicationTables` issues `ALTER PUBLICATION ... ADD TABLE` (additive; existing scope untouched; idempotent on partial-add re-run). MySQL participates with no engine surface change — binlog already covers every table.
- **Operator safeguards.** Refuses if no row exists for the supplied `--stream-id` (catches typos / wrong-target). Refuses if `stop_requested_at` is set (catches "forgot to drain"). Refuses if the target table already has rows (`TableEmptyChecker` preflight, same shape as cold-start). Refuses if the named table doesn't exist on the source (catches "ran add-table before CREATE TABLE landed"). All refusals surface clear messages with recovery steps. Typed-confirmation prompt mirroring `--reset-target-data`'s friction tier; `--yes` bypasses.
- **Persisted position is intentionally NOT updated.** The stream's existing `sluice_cdc_state` position is still the right resume point for the other tables; the applier's idempotent upsert handles the [persisted_LSN, snapshot_LSN] overlap on the new table on resume.

### Backup chain retention — chunks 14a–14e complete (v0.47.0 → Task #16)

The local-FS-chain operator pain that GitHub #20 was filed for (unbounded inode growth + linear-in-chain restore time) is closed end-to-end:

- **14a. `chain.json` catalog at chain root (v0.47.0).** Single file listing `[backup_id, parent_backup_id, created_at, end_position, bytes, file_path]` per manifest in chain order. Restore / verify read it once instead of `ListObjects` / `readdir`-ing `manifests/`. Atomic write-temp + rename on every rollover. Backwards-compat: missing-file fall-back to directory scan, so pre-v0.47.0 chains keep working without migration. Unblocks 14b/14c/14d/14e — they all need O(1) chain-content lookup.
- **14b Phase 1 — rotation-EXIT thresholds (v0.51.0, SUPERSEDED).** `--exit-after-age=DUR` / `--exit-after-chain-length=N` on `backup stream run`: commit + exit cleanly when the threshold trips; chain.json gets `RotatedAt` + `RotationReason` markers. Phase 2 supersedes; the Phase 1 flags now error with a clear migration message (clean break per the zero-users tenet).
- **14b Phase 2 — `--retain-rotate-at=DUR` in-process inline rotation (v0.67.0; ADR-0046; hardened v0.67.1 / Bug 66).** When the threshold trips, the *same goroutine* opens a new snapshot stream, bulk-copies source state into a sibling chain root, writes the `SucceededBy` auto-stitch pointer, and resumes streaming — no operator wrapper. The snapshot/CDC overlap during rotation (the hard part) is the bounded-segment lineage model in ADR-0046, refined by **ADR-0067 (v0.88.0)** which made rotation born-contiguous (the `(P_N, S]` overlap), so rotated chains are compactable (closes Bug 95).
- **14c. `sluice backup prune --from-dir CHAIN [--keep-incrementals N | --keep-duration DUR]` (v0.50.0).** One-shot cleanup for existing chains. Drops oldest-prefix incrementals; first kept incremental's manifest gets re-stitched (`ParentBackupID` → full's ID, `StartPosition` → full's `EndPosition`) so chain restore's parent-link walk + StartPosition validation pass on the re-stitched chain. Event windows in dropped incrementals are LOST from the restore range (operator opts in explicitly). Refuses on `chain.json` absent / both flags / neither flag / structural-break catalogs.
- **14d. `sluice backup compact --from-dir CHAIN --merge-window DUR` (v0.77.0; Task #15).** Merges K consecutive incrementals into a single combined incremental with the same end-position but smaller file/inode footprint. v1 is the naive concat — read all change events in order, write to a single new `.jsonl.gz` chunk under a new `_changes/<merged-ts>/`, delete sources, tombstone merged manifests in `chain.json`. The new manifest gets the last source's end-position. Restore-side is no-change: merged manifests look like larger incrementals to the existing chain iterator.
- **14e. Smart compaction — same-row event collapsing (Task #16; ADR-0064).** Within the merge window, collapse same-PK event chains into the net result per the [ADR-0064](adr/adr-0064-backup-smart-compaction.md) policy table: INSERT+UPDATE(s) → INSERT with final values; UPDATE(s) → one UPDATE; INSERT+DELETE → nothing; UPDATE(s)+DELETE → just DELETE; DELETE+INSERT (row reused) → both verbatim. TRUNCATE/DDL are barriers; tables without a declared PK fall through to naive concat and are listed in the compaction report. Opt-in via `--smart-compaction` (default off in v1 for the cautious-ship reason; a default flip is a one-line change after field data confirms the safety profile). Refuses LOUDLY on corrupt PK with a `--smart-compaction-off` recovery hint. Encrypted chains refuse loudly in v1 (no cross-keyset re-encrypt yet). Observed reductions on update-heavy synthetic workloads: 50–95% event drop + matching byte savings.

The only forward-looking piece is the `--smart-compaction` default flip — gated on operator field data; pickup is one ADR-update + a `--no-smart-compaction` opt-out flag.

### Logical backups Phase 5 (cross-engine chain restore)

- **Cross-engine `sluice restore --from=<chain-url> --target-driver=<engine>`** — was supported for full-only chains since v0.16.x; v0.21.0 extends it to chains with incrementals. Lifts the loud refusal at `chain_restore.go:99` (`"cross-engine chain restore is a Phase 5+ topic"`). Schema deltas in incremental manifests now route through `internal/translate.RetargetForEngine` before invoking `ir.SchemaDeltaApplier.AlterAddColumn` on the target. PG-source `ADD COLUMN UUID` lands as MySQL `CHAR(36)`; INET → VARCHAR(45); Array → JSON. Existing `RetargetForEngine` rules are reused verbatim — no new translation surface.
- **Cross-engine `sluice sync from-backup run --target-driver=<engine>`** — same delta-translation pass on each tick's incremental in `pipeline.SyncFromBackup.applySchemaDeltas`. Detects cross-engine at startup and logs `INFO broker: cross-engine chain — chain's EndPosition not written to sluice_cdc_state; use --at-chain-id for cross-engine resumption assertions`.
- **Change-event value translation reuses live-CDC machinery.** Cross-engine row payloads in change chunks land at the engine appliers' existing live-CDC value-translation path: each applier looks up its own *target* column types and routes every value through `prepareValue` for target-shape preparation. - **Loud refusal for unsupportable types.** PG-source PostGIS `Geometry` columns refuse cross-engine restore to MySQL with the offending table + column named and an `--exclude-table` recovery hint. Pre-flighted at chain start so the operator gets a clear failure before any work happens. Same refusal pattern as full cross-engine restore, extended to cover incremental schema deltas (a delta that introduces a PostGIS column refuses with the incremental's BackupID named).
- **Integration coverage on PG and MySQL.** `chain_restore_cross_integration_test.go` covers acceptance criteria 1-4 (PG → MySQL chain, MySQL → PG chain, broker schema-evolution mid-stream, PostGIS refusal); same-engine paths regression-clean via the existing `incremental_pg_integration_test.go` / `incremental_mysql_integration_test.go` running unchanged.

### Logical backups Phase 4.5 (backup-as-broker / `sync from-backup`)

- **`sluice sync from-backup run` / `sluice sync from-backup stop`** — new CLI surface implementing `docs/dev/design/logical-backups-phase-4-5.md`. The consumer-side companion to Phase 4's `backup stream`: a long-running broker that polls a backup chain at the configured `--poll-interval` cadence (default `30s`) and replays new incrementals into a target via the existing `ChangeApplier.ApplyBatch` path. Decouples source from target via the chain as the message log — no direct source-target connectivity required.
- **`pipeline.SyncFromBackup` orchestrator** — opens the target's `ChangeApplier` ONCE for the broker's lifetime; each tick lists manifests, filters to those NOT yet applied (via the persisted `last_applied_backup_id` in `sluice_cdc_state`), and replays each in chain order — schema deltas first (via `ir.SchemaDeltaApplier.AlterAddColumn` from Phase 3.2), then change chunks through the engine's batched applier.
- **Replay state via existing `sluice_cdc_state`** — no schema change. New position-shape sentinel: `position_engine = "backup-broker"`, `position_token = '{"chain_url":"...","last_applied_backup_id":"<id>"}'`. ADR-0007 transactional position-and-data atomicity makes broker crashes mid-replay safe to re-apply (ADR-0010 idempotent applier). Reuses the v0.19.1 heartbeat read-modify-write helper to preserve concurrent stop_requested_at across heartbeat writes.

Cross-machine: an operator on machine B can stop a broker on machine A without process access. New exported helper `pipeline.RequestSyncFromBackupStop`. Two override flags: `--reset-target-data` runs an inline `ChainRestore` to land the full + every incremental, then transitions to live polling. `--at-chain-id=<BACKUP-ID>` is the operator's assertion that the target is already at that chain ID (typical workflow: manual `sluice restore --from=<chain-url>`); broker writes a fresh `sluice_cdc_state` row and transitions to live polling.
- **Integration coverage on PG and MySQL** — `broker_pg_integration_test.go` covers acceptance criteria 1, 3, 4, 5, 6, 7 (happy-path catch-up, schema evolution, cooperative stop, restart resumes, cold-start refusal, cold-start with --reset-target-data); `broker_mysql_integration_test.go` covers MySQL happy-path (criterion 2); `broker_fanout_integration_test.go` covers criterion 9 (1 source + 2 brokers reading the same chain → both targets converge with source slot count unchanged).

### Logical backups Phase 4 (continuous-incremental long-running stream)

- **`sluice backup stream run` / `sluice backup stream stop`** — new CLI surface implementing `docs/dev/design/logical-backups-phase-4.md`. The stream is a single long-running process that produces rolling incrementals at a configured cadence; no per-incremental cron orchestration. Fits k8s "always-on protection" deployments naturally and pairs with continuous CDC + chain-restore for full DR coverage.
- **Rollover policy.** Hybrid time-bound + size-bound + change-count-bound, first-fired wins (mirrors single-shot incremental's existing shape). `--rollover-window=DURATION` (default `5m`), `--rollover-max-changes=N` (default `100k`), `--rollover-max-bytes=BYTES` (default `64Mi`). Window extends to the next `TxCommit` so the chain doesn't end mid-tx (same as Phase 3.1). Empty rollovers skipped by default; `--rollover-include-empty` opts in for heartbeat-shape monitoring.
- **`pipeline.BackupStream` orchestrator** — opens the engine's CDC pump ONCE for the lifetime of the stream and reuses it across rollovers. Each rollover writes a NEW manifest at `manifests/incr-<unix-millis>-<seq>.json` (mirrors Phase 3.1 + Bug 35's per-Run namespace). The chain-walker (Phase 3.2) handles stream-written chains transparently — restore + verify don't care that the chain came from `backup stream` vs `backup incremental`.
- **`stream_state.json`** — small mutable file at `manifests/stream_state.json` carrying `{pid, host, started_at, last_rollover_at, stop_requested_at}`. Updated on every successful rollover. Concurrent-writer protection: refuses to start a second stream when the file shows a recent (`< 2 × rollover-window`) `last_rollover_at` from a different (pid, host); `--force` bypasses with a WARN. Operator-actionable error message names the conflict and the override.
- **Cross-machine stop** — `sluice backup stream stop --target=<url>` writes `stop_requested_at` to the state file; the running stream polls between rollovers and exits cleanly when set. Mirrors the `sync stop` pattern (ADR-0025) but via the backup destination as the rendezvous point — both sides agree on the destination, not on the host. New exported helper `pipeline.RequestStreamStop`.
- **Signal handling** — SIGINT/SIGTERM via the existing `kongContext` notifier propagates as ctx.Done through the rollover loop; mid-rollover cancel surfaces as a clean nil exit (the rollover's chunks may be partially-written but the manifest never finalises; on restart the stream picks up at the previous rollover's EndPosition).
- **Rollover hooks** — `--rollover-hook=<cmd>` runs after each rollover commits with env vars `SLUICE_ROLLOVER_MANIFEST_PATH`, `SLUICE_ROLLOVER_PARENT_BACKUP_ID`, `SLUICE_ROLLOVER_BACKUP_ID`, `SLUICE_ROLLOVER_CHANGES`, `SLUICE_ROLLOVER_BYTES`, `SLUICE_ROLLOVER_ELAPSED_MS`. 30 s timeout. Hook errors WARN-log but don't fail the stream — the rollover already committed. Examples in docs: push to Prometheus pushgateway / send Slack notification / write to monitoring datastore.
- **Integration coverage on PG and MySQL** — `stream_pg_integration_test.go` covers acceptance criteria 1, 4, 5, 6, 7, 8 (long-running stream produces rolling incrementals, chain restore round-trip, ctx-cancel clean exit, cross-machine stop request, concurrent-writer refusal); MySQL counterpart covers the binlog flavour. Bounded windows (2 s) and small ceilings (10 changes) keep test runtime under a minute per scenario.

- **`ir.SchemaWriter.CreateViews`** new interface method; both engines implement. New Phase 6 in the simple-mode orchestrator (after constraints) emits `CREATE OR REPLACE VIEW` (regular) and `CREATE MATERIALIZED VIEW ... WITH DATA` (matviews; PG-only). View-to-view dependency ordering uses single-pass-with-up-to-2-retries policy — no SQL parser, surfaces clear error if budget exhausted.
- **CLI flags** `--include-view PATTERN`, `--exclude-view PATTERN`, `--skip-views` on `migrate` / `sync start` / `schema preview` / `schema diff`. `ViewFilter` mirrors `TableFilter` shape; filtered independently from tables.
- **Schema diff for views** — `ir.SchemaDiff.ViewsMissing` / `ViewsExtra` / `ViewsMismatched` populated by `DiffSchemas`; text + JSON renderers updated. Definition comparison is trim-and-equal; cross-engine drift is high-noise by design (PG canonicalises view bodies via `pg_get_viewdef`); the renderer hedges with a low-confidence comment.
- **Phase 1 limitations documented** — cross-engine view-body translation (PG → MySQL or vice-versa) emits the source dialect verbatim and relies on the loud-failure tenet. View definitions with explicit column lists (`CREATE VIEW v(a,b,c) AS ...`), MySQL `CREATE ALGORITHM=UNDEFINED VIEW`, PG `WITH (security_invoker=true)`, and PG RULE-based pseudo-views are out of scope for Phase 1.
