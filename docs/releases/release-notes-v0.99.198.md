# sluice v0.99.198

**New: `--skip-foreign-keys` (opt-in, on `migrate` and `sync start`) skips creating foreign-key constraints on the target while keeping each skipped FK's referencing columns indexed — so an FK-bearing source can transition onto a target with limited/no FK support (Vitess/PlanetScale sharded keyspaces, or FKs managed out-of-band) WITHOUT first stripping FKs from the source, and joins stay fast because sluice synthesizes a backing index over each skipped FK's referencing column tuple (only when no existing target index already covers those columns as a left-prefix, so it never creates a redundant index). Drop-in: purely additive and off by default — unset, every migrate and sync is byte-identical to before.**

## Added

- **`--skip-foreign-keys` on `migrate` and `sync start`: create no FK constraints on the target, but ensure each skipped FK's referencing column tuple stays indexed (new opt-in capability; off by default, no data-path change).** The primary use case is a target with limited or no FK support — Vitess / PlanetScale sharded keyspaces, where cross-shard FKs don't work and FKs are opt-in per database, or any target where FKs are managed out-of-band — so an existing database that HAS foreign keys can be migrated or synced onto it without first stripping the FKs from the source. To preserve join performance, for every FK sluice would have created it synthesizes a plain non-unique backing index over the FK's referencing columns (in FK-column order), but ONLY when no existing target index already covers those columns as a left-prefix: a primary key or secondary index whose leading columns already match the tuple is left alone, so no redundant index is ever created. This matters most on a MySQL target, where MySQL auto-creates an FK's backing index ONLY when the FK itself is created — so a naive skip would leave the referencing column unindexed and slow every join through it; synthesizing the index is what makes the skip safe there. The transform runs once on the finalized schema, before any DDL phase, and applies to both the migrate foreign-key phase and the sync cold-start schema-apply path (and it suppresses the multi-database deferred cross-database FK pass, since there are no FKs to create). It is mutually exclusive with `--allow-degraded-fks` (opposite intents — one skips FK creation entirely, the other creates FKs and tolerates dirty source rows — so setting both is a loud refusal). The change is never silent: each skipped FK is reported on its own log line (naming the table, the referencing columns, and the synthesized backing index or the existing index that already covered them) plus a summary count of FKs skipped, indexes synthesized, and columns already covered. Composite FKs index the full tuple in FK-column order; partial (WHERE) indexes and expression/functional indexes are correctly NOT treated as covers (they don't guarantee the columns are fully indexed for joins), while a MySQL prefix-LENGTH index IS a cover; and re-running is idempotent because the synthesized indexes ride the existing `CREATE INDEX IF NOT EXISTS` / detect-then-skip path, so a `--resume` never double-creates them. Unset, behavior is byte-identical to before — FK constraints are created as always. Pinned by a left-prefix-cover unit matrix (exact match / PK left-prefix / composite-column order / too-narrow / partial / expression / MySQL prefix-length), a same-columns de-dup pin, a mutual-exclusion refusal pin, an over-long-name hash-fallback pin, and a cross-engine integration test (PG→MySQL: the target has zero FKs, an already-indexed FK column is not duplicated, an un-indexed FK column gets a synthesized index, and all rows copy).

## Compatibility

**No breaking changes; drop-in. Purely additive and opt-in — no change to any default behavior, format, or data path.** With `--skip-foreign-keys` unset (the default), every migrate and sync — including the FK phase and the cold-start schema-apply — is byte-identical to before, on every engine. The flag only changes behavior when explicitly set, and only for the FK-creation step: FKs are not created and each referencing column tuple is kept indexed instead. It is engine-agnostic (any source→target combination) and mutually exclusive with `--allow-degraded-fks` (setting both is refused loudly at validate). Two documented scope boundaries: (1) the fleet `sync run` (`syncs.yaml`) surface intentionally does not carry the flag yet — it follows the `--skip-views` precedent and is easy to add on demand; and (2) a table added mid-stream via live add-table is not covered — only the `migrate` path and the `sync` cold-start create FKs (steady-state CDC apply never does), so those are the only two surfaces the flag shapes.

## Who needs this — action required

- **Anyone migrating or syncing a foreign-key'd source into Vitess/PlanetScale (or any target where FKs must be off but the columns should stay indexed): pass `--skip-foreign-keys` to transition the schema without stripping FKs from the source first.** sluice will skip the FK constraints on the target and synthesize a backing index for any referencing column tuple not already covered, so joins stay fast — watch the per-FK log lines and the summary count to see exactly which FKs were skipped and which indexes were created. This is especially important on a MySQL target, where the FK's backing index otherwise only exists alongside the FK. If you drive syncs from a fleet `syncs.yaml`, note the flag is not yet wired into `sync run` — use single-target `sync start` for now.
- **Everyone else: no action.** The flag is opt-in and off by default; not setting it leaves every migrate and sync exactly as it was. There is nothing to re-run or re-verify.
