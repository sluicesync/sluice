# sluice v0.99.70

**New: native (non-Vitess) MySQL cold-copy can now copy multiple tables to the target CONCURRENTLY — closing the serial-copy bottleneck that pinned vanilla-MySQL → PlanetScale/MySQL migrations at the serial single-table rate (ADR-0101).** Opt-in and drop-in: it engages on a new source-DSN knob `copy_table_parallelism=N`; absent / `0` / `1` (or a single-table scope) resolves to N=1 and runs the existing serial single-snapshot cold-copy byte-identically, so existing syncs are unchanged. This is the native-MySQL analogue of the VStream cross-table concurrency (ADR-0099/0100) — but native MySQL has no exported/shareable snapshot (Postgres's lever) and no VStream (Vitess's), so the consistent multi-table snapshot uses the mydumper / `mysqldump --single-transaction --master-data` pattern, generalized from sluice's existing one-reader binlog snapshot to N readers under one held lock. **Honest scope: the v1 native concurrent copy is W × 1** — W tables copied concurrently, one plain-INSERT writer per table — which lifts the serial-consumer ceiling (throughput rises ~N× the serial single-table rate) but does **not** by itself reach the Vitess W × D ceiling (~26 MB/s on cross-region PlanetScale-MySQL); each table is still a single cross-region INSERT stream. Folding the per-table write fan-out in (true W × D) on the native path is a documented follow-up.

## Features

- **Native-MySQL concurrent multi-table cold-copy — N consistent-snapshot readers under one FLUSH TABLES WITH READ LOCK (ADR-0101).** Live Track-D measurement showed a vanilla MySQL → PS-MySQL cold-copy held at the serial single-table rate because the native binlog cold-start copied tables ONE AT A TIME — the same serial-consumer bottleneck ADR-0100 diagnosed for the VStream path, but with no native knob to lift it (`vstream_copy_table_parallelism` is VStream-source-only). This release opens **N reader connections, each running `REPEATABLE READ` + `START TRANSACTION WITH CONSISTENT SNAPSHOT`, all under ONE briefly-held FLUSH TABLES WITH READ LOCK**, records ONE binlog position, then `UNLOCK TABLES` — so all N readers share a single consistent cut and the snapshot→CDC seam is that single recorded position (no per-reader stitch, simpler and more obviously gapless than the VStream set-min seam). The in-scope tables partition into N disjoint groups via the same deterministic `partitionTablesForStreams` ADR-0099 already unit-pins, surfaced through the existing `ir.ConcurrentCopyPartitioner` capability, so the existing ADR-0100 pipeline consumer (`runConcurrentTableCopy`) drives W = N concurrent read→write pipelines verbatim — each pipeline copying its group's tables from its own pinned-snapshot connection. The lock is held only across opening the N transactions and reading the position (millisecond-scale, no data read under the lock), and the only new core is the FTWRL-coordinated N-snapshot opener plus a multi-snapshot RowReader that routes each table's `ReadRows` to its group's connection. The target PROCESSLIST should now show up to W = N tables receiving rows concurrently instead of one at a time; the win is the removal of the serial-consumer ceiling (~N× the serial single-table rate), bounded by source/target/network capacity and the N × D connection budget. No headline native MB/s figure is quoted — the live re-measurement runs after this ships; the mechanism is the multiplicative cross-table concurrency, not a measured number. Pinned and value-fidelity reviewed on four silent-loss-class invariants: (a) all N readers pin the SAME cut as the recorded position (the FTWRL is held across all N `START TRANSACTION` calls AND the position read, then released — the open/record/unlock ordering is pinned); (b) every in-scope table is read by exactly one reader (the disjoint partition IS the table→connection map; a table owned by no group is refused loudly, never silently skipped); (c) the CDC position equals the one recorded position and is read only AFTER the W-way errgroup joins (structural — no per-pipeline position write); (d) N = 1 is byte-identical to the existing serial single-snapshot path. ctx-cancel mid-copy cancels the W-way errgroup and `Close` commits + closes all N pinned connections with no leaked goroutines or connections (the FTWRL is already released by then, held only during open); an open-failure partway through unwinds every already-opened connection AND releases the lock before returning. The concurrency-chunk `-race` Integration gate is the merge gate (CGO is off locally).

## Compatibility

- **No breaking changes; drop-in upgrade.** Native concurrent cold-copy engages only on the new source-DSN knob `copy_table_parallelism=N`. Absent the knob (or `0` / `1`, or a single-table scope) the cold-copy takes the existing serial single-snapshot table loop exactly as v0.99.69 did — no FTWRL-N-open, no errgroup, the byte-identical pre-ADR-0101 path (the engine surfaces no concurrent groups at N=1, so the pipeline never enters the concurrent branch). The knob is zero-value-safe by design (`resolveCopyTableParallelism` maps `n ≤ 1 → 1`, reusing ADR-0099's resolver — the v0.99.51 default-inversion trap avoided by defaulting to serial, never a surprise multiplier on every cold-start). There is no new CLI flag and no change to the persisted handoff/position token shape (the snapshot→CDC anchor is the same single recorded binlog position the serial path already uses).
- **`copy_table_parallelism` is distinct from the VStream knob and is stripped before the MySQL session.** It deliberately omits the `vstream_` prefix because it governs the native binlog cold-copy, not VStream (the VStream path keeps its own `vstream_copy_table_parallelism`). It is parsed entirely inside the mysql engine and is added to the DSN-param strip allowlist (Bug-126 class) so it never reaches a MySQL session as `SET copy_table_parallelism=…` (which MySQL would reject as an unknown system variable).
- **FTWRL-required, with a documented serial fallback — never a silent inconsistent snapshot.** The N-snapshot opener needs `FLUSH TABLES WITH READ LOCK` (the RELOAD / FLUSH_TABLES privilege), which is blocked on some managed MySQL (RDS, Aurora). If FTWRL fails AND N > 1 was requested, sluice does NOT proceed with N independent snapshots (which could capture an inconsistent multi-table view) — it FALLS BACK to the serial single-snapshot path with a loud WARN naming the reason, concurrency disabled, consistency preserved. N is clamped to `[1, min(len(tables), 32)]` so a typo can't open a thousand connections.
- **Connection budget: N is NOT folded into the target-connection preflight in v1 — operator responsibility.** During the concurrent copy the source sees N pinned reader connections (plus the one CDC connection opened at handoff); the target sees W = N write streams. N is a source-DSN knob parsed inside the mysql engine, invisible to the pipeline-layer connection-budget preflight, so v1 cannot clamp it — sluice emits a WARN at copy start naming N and stating that `N ≤ source max_connections` and the target write demand are the operator's responsibility (no auto-clamp claimed that isn't implemented). A future enhancement folds N into the preflight.
- **Scope: the native (binlog-flavor) MySQL cold-copy only.** The VStream / Vitess / PlanetScale-source cold-copy (its own `vstream_copy_table_parallelism`), the PostgreSQL cold-copy, the ADR-0079 fast shareable-snapshot path, and the `sluice migrate` path (its own within-table chunking, ADR-0096) are all untouched. A single-table scope is already single-stream and unaffected.

## Who needs this — action required

- **Anyone running `sluice sync` from a vanilla / native (self-managed) MySQL source into PlanetScale/MySQL over a large multi-table database who wants a faster cold-copy:** set `copy_table_parallelism=4` (or higher) in the source DSN, provided the source user has the RELOAD privilege (FTWRL) and you mind the connection budget (keep `N ≤ source max_connections` and account for N concurrent target write streams — v1 does not auto-clamp N; the WARN at copy start names N and restates the contract). This lifts the serial single-table cold-copy bottleneck to ~N× by copying N tables at once. Note the honest v1 scope: each table is still a single plain-INSERT writer (W × 1), so this removes the serial-consumer ceiling but does not by itself reach the Vitess cross-region W × D ceiling — the per-table write fan-out on the native path is a documented follow-up. On managed MySQL without RELOAD (RDS, Aurora) the knob safely degrades to the serial path with a loud WARN. Resume the job with the same N you started it with (the partition is a pure function of the sorted table set and N).
- **No re-verification of prior cold-copies is required.** This is a throughput optimization to how fast the native cold-copy runs, not a fix to a path that produced wrong or missing data. The serial single-snapshot native cold-copy on prior versions was correct and gapless; completed copies are unaffected and need no re-check, and N=1 (the default) reproduces that serial path byte-for-byte.
