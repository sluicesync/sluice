# sluice v0.99.189

**Fixed — two agent-usability / robustness fixes surfaced by a live validation of the Tier-1 agent skills (the skills-validation follow-up to v0.99.186–188; lower severity — CLI-contract robustness, not data corruption): `sluice slot drop` without `--yes` now refuses loudly instead of silently no-op'ing on a non-TTY, and `backup full`'s suggested next command now points at the correct verify flag. Drop-in: no data-path, backup-format, or default change; a non-interactive `slot drop` that previously prompted-then-aborted now exits 3 with a clear code, so any script relying on that prompt must pass `--yes`.**

## Fixed

- **`sluice slot drop` without `--yes` now refuses loudly (exit 3, `SLUICE-E-CONFIRMATION-REQUIRED`) instead of prompting — which on any non-TTY silently aborted a destructive command (agent-contract fidelity; low severity, no data loss).** AGENTS.md guarantees every sluice command is non-interactive ("no prompts, ever"), but `slot drop` without `--yes` printed an interactive `Drop replication slot "…"? [y/N]` confirmation via `confirmDestructive`. On a non-TTY — any agent, CI job, or script — the read hit EOF, was interpreted as "no", printed `aborted`, and returned exit 0: a destructive command silently no-op'ing, exactly the failure the "refuse loudly" tenet exists to prevent (the caller believed the slot was dropped when it was not). `slot drop` without `--yes` now returns a `ClassRefusal` coded error — the new `SLUICE-E-CONFIRMATION-REQUIRED` code (exit 3) — naming the slot with the hint `pass --yes (or -y) to confirm`, and the refusal fires fail-fast *before* any source connection is opened. `--yes`/`-y` clears the gate and proceeds exactly as before; `--if-exists` and `--force` are unchanged. `confirmDestructive` is retained (the `trigger` teardown path still uses it). Pinned by two unit tests: `slot drop` without `--yes` returns the exit-3 coded refusal without reading stdin (no database needed), and with `--yes` it clears the gate and reaches the engine-resolution path.
- **`backup full`'s `next_steps` hint now points at `--from-dir`, the flag `backup verify` actually accepts.** The success envelope suggested `sluice backup verify --output-dir <BACKUP_DIR>`, but `backup verify`'s flag is `--from-dir` (`--output-dir` is a usage error there) — so the binary's own suggested next command failed for anyone who copy-pasted it. The hint now emits `--from-dir`; the same wrong string was corrected in the envelope test that pinned it.

## Changed

- **Three Tier-1 agent skills hardened from the same validation pass (docs/skills only — no shipped-binary change).** `fidelity-verify` now states that cross-engine `verify` is count-only and that a clean count is **not** a value-level fidelity guarantee (closing a false-FAITHFUL reading); `sluice-error-triage` now handles the generic-`code`-with-the-real-cause-in-`message` shape, adds a wrong-hint warning and a codeless-failure branch, and lists the full connect-error codes; `backup-chain-operator` corrects `incremental` (no `--format`), `prune` (no `--encrypt`), and the `slot drop` invocation (positional name, `--yes` for non-interactive confirm rather than an interactive prompt). These change agent guidance only; the sluice binary behaves identically.

## Compatibility

**No breaking changes to any data path; drop-in.** No CLI flag semantics beyond `slot drop`, no default, and no backup format-version change; all migrate / sync / backup / restore data paths are byte-identical, and every engine and direction is unaffected. The one observable behavior change is intentional and confined to `slot drop`: an invocation without `--yes` that previously printed an interactive `[y/N]` prompt (and, on a non-TTY, silently aborted with exit 0) now exits 3 with `SLUICE-E-CONFIRMATION-REQUIRED` and does not read stdin. `slot drop --yes` (or `-y`) is byte-for-byte the same as before, and `--if-exists`/`--force` are unchanged. Any script that relied on the interactive prompt to confirm a drop must now pass `--yes` explicitly — which is the correct non-interactive contract. The `backup full` hint change is purely the text of a suggested next command; no automated path parsed it.

## Who needs this — action required

- **Anyone driving `sluice slot drop` non-interactively (agents, CI, scripts) without `--yes`.** Before v0.99.189 the command **silently did nothing** (prompt → EOF → `aborted` → exit 0) on a non-TTY; the slot was **not** dropped despite a success exit. Add `--yes` (or `-y`) to those invocations. On v0.99.189 the same command without `--yes` now fails loudly (exit 3), so a wrapper that treated exit 0 as "dropped" will start surfacing the missing flag instead of masking it. If you have leftover replication slots you believed a prior automated `slot drop` had removed, re-check with `sluice slot list` and drop them with `--yes`.
- **Anyone who followed `backup full`'s suggested `backup verify --output-dir …` next command.** That command errored on the wrong flag; use `--from-dir` (what v0.99.189 now suggests). No backup artifact was affected — only the suggested verify invocation failed.
- **Everyone else: no action.** This is not a data-fidelity fix. Interactive terminal users of `slot drop` are unaffected in practice (they always confirmed), migrate / sync / backup / restore results are unchanged, and no re-verification of migrated data is warranted by this release.
