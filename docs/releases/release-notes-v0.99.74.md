# sluice v0.99.74

**New: the native-MySQL concurrent cold-copy now WORK-STEALS tables across its W readers (no more idle readers waiting on one skewed partition), gains a per-table ETA on that path, and a multi-shard Vitess/PlanetScale source gets a loud per-shard progress WARN that surfaces a single wedged shard the whole-stream liveness watchdog cannot see.** All three are throughput/observability improvements; none changes the data a sync moves. Drop-in upgrade — the one new DSN knob (`vstream_shard_stall_warn_timeout`) defaults to a safe 60s, the work-stealing/ETA changes add no knob, and existing syncs behave exactly as on v0.99.73.

## Features

- **Table-level work-stealing for the native-MySQL concurrent cold-copy — the W readers now pull from a shared cursor instead of each draining a fixed group, closing the K-wide tail (roadmap 21a).** ADR-0099/0101's cross-table concurrent cold-copy partitions the in-scope tables into K disjoint groups ONCE at cold-start and drains each group serially per pipeline, with no rebalancing — so the effective width tapers below K at the tail as the lighter groups finish and idle while a heavier group grinds its last large table (observed live on Track B: K=4 collapsing to 2 active read streams at 22 of 27 tables). This release adds work-stealing for the engine where it is clean: the native-MySQL multi-snapshot reader's N connections are each a consistent-snapshot transaction from the SAME FLUSH TABLES WITH READ LOCK cut (ADR-0101), so ANY connection can read ANY table identically. A new engine-neutral `ir.WorkStealingCopyReader` capability (`ConcurrentReaderCount` + `ReadRowsOn`) is implemented by the native concurrent reader; when present, the pipeline flattens the static partition into one ordered work list and runs W = min(N, tables) pipelines that PULL the next table from a shared atomic cursor, each reading its claimed table on its OWN pinned connection — keeping the copy W-wide all the way to the last table. The D-way per-table write fan-out (ADR-0097/0102) composes per claimed table unchanged, so total concurrency stays W × D. Scoped to native MySQL on purpose: the VStream reader does NOT implement the surface (its K streams are each `Match`-scoped to their group at the source, so a stealing consumer would have no rows for an out-of-group table) and keeps the static partition — VStream work-stealing needs source-stream restructuring and is deferred (roadmap item 21). Exactly-once is preserved by the atomic claim (each table claimed by exactly one pipeline; full coverage), one in-flight read per connection (the reader index is owned by one pipeline), and the native single-FTWRL position being independent of which connection read a table, so the snapshot→CDC seam is unaffected. Pinned by `TestRunConcurrentTableCopy_WorkStealing_ExactlyOnce` (full coverage with the WS path taken and no static `ReadRows`) and `_CrossesGroupBoundary` (a deterministic proof that a pipeline pulls tables from BOTH original static groups — something the static drain can never do); as a concurrency chunk the `-race` Integration gate is the merge gate (CGO is off locally).

- **Per-table ETA for the native concurrent cold-copy, via a side metadata pool.** The native concurrent reader's progress ticker had no per-table total because its inner snapshot readers no-op `CountRows` — a metadata query on a connection actively streaming a table would be a concurrent query on a single `*sql.Conn`, which deadlocks. The native concurrent reader now implements `ir.RowCounter` by running the cheap `information_schema.TABLE_ROWS` ESTIMATE (the SAME estimate the serial RowReader uses, not an exact `COUNT(*)` scan) on a FRESH connection drawn from the retained source metadata pool: catalog metadata needs no snapshot and never touches the N pinned snapshot connections, so it is collision-free, and that pool has no `MaxOpenConns` cap so the count never starves behind the pinned conns. The work-stealing adapter forwards `CountRows` so the per-table ETA works on the work-stealing path too, and a reader without `ir.RowCounter` degrades gracefully to no-ETA. This also gives the STATIC native concurrent path (no work-stealing) an ETA it never had. Honest about what the number is: `total_rows` here is an ESTIMATE from `information_schema`, not an exact count — it can drift from the true row count and the progress bar may briefly read past 100% on a fast-growing table (labeling it explicitly as an estimate is roadmap item 22, a follow-up). Pinned by `TestPinnedRowReader_ForwardsCountRows` (forwards the estimate; `(0, nil)` fallback for a non-`RowCounter` reader) plus compile-time `ir.RowCounter` assertions on both the concurrent reader and the work-stealing adapter.

- **Per-shard VStream progress WARN — a loud, rate-limited heads-up when one shard of a multi-shard Vitess/PlanetScale source wedges while a peer shard keeps advancing (roadmap item 23, increment B-1).** On a multi-shard source the VStream reader merges every shard into ONE channel, so any shard's events (plus the ~5s heartbeats) keep the whole-stream Phase-2 liveness timer and the soft idle-WARN re-armed even while another shard's emitted GTID component is frozen — i.e. the existing single-channel liveness watchdog structurally cannot see a per-shard wedge. Live testing hit exactly this: shard `-80` frozen for hours while `80-` advanced, no stream error, only heartbeats. This release adds a per-shard progress watchdog that emits ONE rate-limited WARN naming the wedged shard when its GTID component has frozen while a peer shard keeps advancing and the overall stream stays alive. The WARN names BOTH candidate causes so the operator knows what to check — a genuine per-shard source stall (a throttle, a large transaction, or a tablet issue; check `SHOW VITESS_THROTTLED_APPS`) OR a normal Vitess `MinimizeSkew` catch-up hold — and states explicitly that the stream stays connected. The watchdog mirrors the existing whole-stream liveness watchdog's race discipline exactly: a single goroutine owns the timer and per-shard state, fed a non-blocking buffered per-VGTID shard-advancement diff computed where the current VGTID is updated. Controlled by a new source-DSN knob `vstream_shard_stall_warn_timeout` (Go duration, default 60s; `0`/negative disables this WARN only; malformed value is a loud error), which is added to the `vstream_` DSN-param strip allowlist (Bug-126 class) so it never leaks as a `SET` on the MySQL session. As a concurrency chunk (new goroutine/timer/shared state) the `-race` Integration gate is the merge gate.

## Compatibility

- **No breaking changes; drop-in upgrade.** The work-stealing and per-table-ETA changes add NO new knob and are inert unless the native concurrent cold-copy is already engaged (`copy_table_parallelism` / W > 1); with W = 1 the cold-copy is the serial single-snapshot path exactly as v0.99.73. The per-shard WARN is OBSERVABILITY-ONLY — it never sets an error, cancels, or alters resilience, so it cannot change the outcome of any sync — and its `vstream_shard_stall_warn_timeout` knob defaults to a safe 60s with `0` disabling it. No flags, config, position/state formats, or on-the-wire behavior change.

- **Scope: the native (binlog-flavor) MySQL concurrent cold-copy (work-stealing + ETA) and the multi-shard VStream CDC observability (per-shard WARN).** Work-stealing is deliberately NOT applied to the VStream/Vitess/PlanetScale-source cold-copy (its `Match`-scoped streams keep the static partition; VStream work-stealing is deferred to roadmap item 21). The PostgreSQL cold-copy, the ADR-0079 fast shareable-snapshot path, and the `sluice migrate` path are untouched. The per-shard WARN affects only multi-shard Vitess/PlanetScale CDC streams; single-shard and non-Vitess sources never arm it.

## Who needs this — action required

- **Running `sluice sync` from a vanilla / native MySQL source into PlanetScale/MySQL with the concurrent cold-copy (`copy_table_parallelism` > 1) over a table set with skewed sizes:** just upgrade — work-stealing engages automatically with no new knob and keeps all W readers busy to the tail instead of letting them idle behind one heavy partition, and you also get a per-table ETA on that path for the first time (an `information_schema` estimate; see the honesty note above). No re-verification of prior copies is required.

- **Running `sluice sync` from a MULTI-SHARD Vitess/PlanetScale source and want early visibility into a stalled shard:** upgrade and optionally tune `vstream_shard_stall_warn_timeout` (default 60s; set higher to reduce noise during normal `MinimizeSkew` holds, or `0` to disable the WARN). When the WARN fires, check the named shard for a source-side throttle/large-transaction/tablet issue (`SHOW VITESS_THROTTLED_APPS`) or confirm it is a benign catch-up hold. This increment is observability-only: it never fails or cancels the stream, and automatic recovery/reopen on a wedged shard is a future release (roadmap item 23 increment B-2).

- **No re-verification of prior copies, and no action for PostgreSQL or single-shard users.** Every change here is a throughput optimization or an observability signal; none produced or could produce wrong or missing data, and completed copies are unaffected.
