# sluice v0.99.35

> ⚠️ **KNOWN ISSUE (Bug 135, found by the post-release battle-test; FIXED in
> **v0.99.36** — upgrade): do NOT `--resume` a crashed/killed PARALLEL
> `backup full` on this version.** Resuming a backup that was interrupted
> while running with the new parallel sweep (the v0.99.35 default) can
> produce a backup containing duplicate and missing rows while exiting 0
> — detectable at restore time only on PK'd tables, and not at all on
> tables without unique constraints. Fresh (uninterrupted) parallel
> backups are correct, serial backups (`--table-parallelism=1`) resume
> correctly, and v0.99.34 is unaffected. **Workaround:** if a parallel
> backup is interrupted, discard the partial output and re-run fresh
> instead of resuming — or run backups with `--table-parallelism=1`
> until v0.99.36.

**The backup surface, hardened and made fast — `backup full --chain-slot` provisions a zero-gap incremental chain in one flag (ADR-0083); two silent-loss chain bugs, a phantom-schema-delta bug, and a stop-restart walsender stall are closed; and cross-table parallelism (ADR-0084) lands on both `backup full` and `restore` with measured numbers: backup 2367 s → 881 s (2.7×) and a projected ~3 h restore done in 2810 s (≥3.8×) on a 133 GB / 43-table corpus. Drop-in from v0.99.34 with one default change: backups and restores now process 4 tables at a time by default — `--table-parallelism=1` restores the old serial behavior.**

## Features

- **`backup full --chain-slot` — one-flag incremental-chain provisioning (Postgres, ADR-0083).** The persistent chain slot (named by `--slot-name`) is created *as* the snapshot anchor — its consistent point IS the manifest's recorded EndPosition — and the pgoutput publication is ensured *before* the anchor, so `backup incremental` chains with zero gap and zero manual setup. Previously the operator had to create both objects by hand *before* the full, and pgoutput's historic-catalog rule means a late-created publication can never decode the chain's first window (observed live in benchmarking). The slot is kept only when the backup completes — a failed run drops it so retries start clean — and an already-existing slot is refused loudly. Engines without a slot concept (MySQL) log a loud no-op. Pinned by unit tests (commit-only-on-success, fallback refusals) and an end-to-end zero-setup chain on real PG with checksums.
- **`--table-parallelism` on `backup full` and `restore` (ADR-0084, default `auto` = 4).** Cross-table fan-out for both halves of the backup lifecycle — see Performance for the mechanism and measured numbers.

## Fixed

- **Silent-loss class closed: an incremental against a late-created or foreign-advanced slot now refuses loudly instead of silently gapping the chain.** A replication slot created (or advanced by another consumer) *after* the parent backup cannot serve the WAL in between — PostgreSQL silently fast-forwards `START_REPLICATION` to the slot's `confirmed_flush_lsn`, so `backup incremental` used to SUCCEED, exit 0, while the chain silently missed every write in the gap. A new chain-resume preflight refuses before opening CDC, naming the exact positions and the recovery (`backup full --chain-slot`); the slot-missing refusal also now says the slot may never have existed (the old message blamed WAL pruning). Affected releases: every release with backup chains, v0.17.0 through v0.99.34 — the condition requires a mis-provisioned or foreign-consumed slot, exactly the setup mistake `--chain-slot` now makes impossible. Pinned by late-created-slot and missing-slot integration tests on real PG.
- **Silent-loss class closed: backup chains no longer ack streamed-but-uncommitted positions.** `backup incremental` / `backup stream` have no applier, so the CDC reader's keepalive fell back to acking the *streamed* LSN — events parsed by the pump but discarded at window close could advance `confirmed_flush_lsn` past the recorded EndPosition, silently gapping the next link (timing-dependent: a keepalive must fire in the teardown window). The chain consumers now hold the slot ack at the stream start and release it only to durably committed window ends; on `backup stream` this also bounds source WAL retention to ~one rollover window. Affected releases: v0.17.0 through v0.99.34 (the streamed-LSN keepalive fallback predates chains; chain consumers carried the exposure from the day they shipped). Pinned by ack clamp/ratchet unit tests.
- **DDL-free incrementals no longer record phantom `alter_table` schema deltas.** Schema readers drained indexes and foreign keys through Go map iteration (randomized order), so two reads of the *same* schema could be structurally unequal — a recorded parent manifest vs the end-of-window catalog read then diffed as spurious `alter_table` entries (observed: 6 phantom deltas on a DDL-free incremental), and `ComputeSchemaHash` fingerprints diverged for identical schemas. Both engines' readers now drain in sorted (table, name) order; the incremental schema diff compares indexes as a name-keyed set — so chains rooted in pre-fix manifests stop false-alarming too — and the schema hash fingerprints a canonical view (non-semantic collections name-sorted; table and column order stay semantic). No data was ever lost — the deltas were noise, not corruption. Affects all prior releases with backup chains. Pinned by tests reproducing the exact benchmark rotation (3-index shuffle → zero deltas) plus the inverse guards (real change → delta; rename → delta).
- **Stop-then-restart no longer stalls on a lingering walsender.** Closing a replication connection with an already-cancelled context skipped the protocol's graceful `Terminate` message, so the server-side walsender kept the slot marked active until `wal_sender_timeout` (60 s default) reaped it — longer than the v0.99.34 slot-active retry budget, producing `slot is active for PID N` refusals on immediate restarts. Every replication-conn close (pump teardown, error paths, snapshot cold-start paths) now runs under a fresh bounded context so `Terminate` is always sent and the slot releases in milliseconds; the v0.99.34 retry stays as defense in depth. This was always a loud failure — no data was ever lost or skipped. Affects all prior versions with PG slot-based CDC (pre-v0.99.34 it surfaced as an immediate 55006 exit; on v0.99.34 as a burned retry budget then the same loud exit).

## Performance

- **`backup full` reads tables in parallel on Postgres sources (`--table-parallelism`, ADR-0084).** The full-backup row sweep fans out across a bounded worker pool (default 4 tables at once, pgcopydb `--table-jobs` parity), every reader pinned to the SAME exported snapshot via `SET TRANSACTION SNAPSHOT` — cross-table consistency is identical to the serial sweep. MySQL backups stay serial (per-session snapshot, not shareable — an INFO log names the reason), as does the non-snapshot fallback. Manifest table order and resume semantics are unchanged-by-construction: entries are pre-staged in schema order, and a crashed run leaves at most `--table-parallelism` tables with partial chunk lists, which the existing resume path already handles. **Measured on the 133 GB / 43-table benchmark corpus: 2367 s → 881 s (2.7×) with defaults**, closing the gap vs `pg_dump -j8` from 10.2× to 3.2× (`docs/comparison-backup.md`).
- **`restore` bulk-applies tables in parallel on EVERY target (`--table-parallelism`, ADR-0084 restore side).** The restore chunk-apply phase fans out across a bounded writer pool (default 4 tables at once), one dedicated row-writer connection per concurrent table — engine-generic, since parallel writers need no shared snapshot (PG and MySQL targets alike), bounded by the target's connection budget. Restore wall time is the operator's recovery-time objective. Per-table chunk ordering, per-chunk SHA-256 verification, and chain restores' ordered incremental replay are unchanged; chain restores parallelize each segment full's bulk-apply via the same flag. **Measured: a projected ~3 h serial restore completes in 2810 s (≥3.8×) with defaults**, closing the gap vs `pg_restore -j8` from ~11.5× to 3.1×.
- **The incremental headline these fixes protect:** once a chain is provisioned, capturing a 3.6M-row-event delta costs **104 s / 1.5 GB**, vs `pg_dump`'s only comparator — a 232 s + 16 GB full re-dump, every cycle, forever.

## Compatibility

- **Default change: `backup full` and `restore` now run cross-table parallel by default** (`--table-parallelism` auto = 4) — this includes chain restores and the broker's `SyncFromBackup` cold start, which parallelize each segment full's bulk-apply through the same path. Output and consistency are identical to serial; concurrency against the source/target rises (up to 4 reader connections on backup, 4 writer connections on restore). `--table-parallelism=1` restores the exact old serial behavior.
- **`OpenBackupSnapshot` now takes `ir.BackupSnapshotOptions`** — an internal engine-interface change; affects only out-of-tree engine implementations and embedders, not CLI users.
- **`ComputeSchemaHash` values change** for schemas whose index/FK collections weren't already name-sorted. The stored hash is write-only in production code — nothing verifies against old hashes, so no verification breaks; new manifests simply record the canonical fingerprint.
- **Resume manifests now pre-stage all tables:** a crashed backup's manifest lists every table up front, untouched ones marked `Partial=true` with zero chunks (previously tables appeared only as they completed). The existing resume path handles both shapes.
- No other flag, default, or invocation changes. MySQL backup reads, the non-snapshot fallback, and all migrate/sync paths are untouched by the parallelism work.

## Who needs this — action required

- **Anyone running backup chains (`backup incremental` / `backup stream`)** — two silent-loss classes are closed. If you ever (re)created or let another consumer touch the chain slot between links, a pre-v0.99.35 incremental could have exited 0 with the gap's writes missing. Action: upgrade, then **verify your existing chain end-to-end** (`restore --verify-only`, or restore to a scratch target and compare counts); if verification fails or you're unsure of the slot's history, re-anchor with `backup full --chain-slot` and start a fresh chain. New chains provisioned via `--chain-slot` are gap-free by construction.
- **Anyone backing up or restoring large multi-table databases** — the 2.7× backup / ≥3.8× restore wins come with the default; nothing to configure. If your source/target is connection-constrained, set `--table-parallelism=1` (old behavior) or a lower value.
- **PG sync operators who stop and restart streams** — immediate restarts no longer burn the slot-active retry budget against a doomed walsender; remove any sleep wrappers added to space out restarts. No data was ever affected — the stall was always a loud exit.
- Everyone else: just upgrade; drop-in from v0.99.34.
