# sluice v0.99.248

**New: `sluice expand-contract` ‚Äî the full expand‚Üímigrate‚Üícontract pattern against a PlanetScale database as one opt-in command (ADR-0162, roadmap item 62 Phase 3), live-validated end-to-end against a real PlanetScale database before release.** A purely additive drop-in upgrade: a new command, a new internal control-plane client, and three new coded errors ‚Äî no change to any existing command, flag, or default. The live validation caught three real defects before this shipped; all three are fixed in this release and none ever appeared in a published version.

## Added

- **`sluice expand-contract` ‚Äî the whole online-schema-change pattern, one command (ADR-0162).** After ADR-0159 shipped the middle step as `sluice backfill`, the expand and contract steps on PlanetScale still meant hand-driving the deploy-request workflow twice ‚Äî dev branch, DDL on the branch, deploy request, deploy, finalize ‚Äî around the backfill. This command drives all of it: **preflight** (service token, org/database/branch, safe migrations, table + walkable PK) ‚Üí **expand** (dev branch + verbatim `--expand-ddl` + deploy request, deployed and finalized) ‚Üí **migrate** (the ADR-0159 `Backfiller` reused whole, never forked) ‚Üí **verify** (the whole-table remaining-count gate on `--where`, which is *required* here ‚Äî it is what authorizes the destructive step) ‚Üí **contract** (a second dev branch + `--contract-ddl` + deploy request, hard-gated behind `--yes`) ‚Üí **cleanup** (the run's dev branches deleted best-effort, even on failure; `--keep-branches` opts out). Without `--contract-ddl` or without `--yes`, the run stops after verify **as a success** and prints the exact resume command ‚Äî there is no interactive prompt (sluice commands are non-interactive by contract), and `--dry-run`/`--yes` are mutually exclusive so a plan can never confirm a drop. The command is PlanetScale-specific by construction: the data-plane engine is fixed to the `planetscale` flavor (no `--driver` to mis-set), and safe migrations disabled on the production branch is a loud coded refusal (`SLUICE-E-PS-SAFE-MIGRATIONS-DISABLED`, naming the toggle and the `pscale` command) ‚Äî sluice never flips that toggle itself, because the enable/disable propagation lag makes wrap-around toggling unsafe (ADR-0148 finding #7). The live end-to-end run completed the full pattern ‚Äî expand deploy request, 32,768-row backfill in 7 chunks, verify remaining=0, contract deploy request, cleanup ‚Äî in 145 seconds on a PS-10; a second full run at 131,072 rows (27 chunks, ~66 s backfill leg) confirmed the keyset walk, bounded per-chunk UPDATEs, and the single-row verify count all stay clear of PlanetScale's per-query result-set and DML limits at >100 K rows.

- **Honest v1 resumability: deterministic branch names + refuse-on-leftover + `--resume-from`, no persisted state machine.** Dev branches are named deterministically from a hash of the table + DDL, so a crashed run's branch is found *by name* on re-run and **refused** with guidance (inspect the deploy request; `--resume-from` past the leg if its DDL deployed, else delete the branch) ‚Äî guessing whether a leftover branch is reusable would be the silent path. `--resume-from expand|migrate|contract` is the explicit continue point; the migrate leg is natively resumable via the ADR-0159 cursor, and verify is deliberately *not* a resume point ‚Äî it always runs before contract, even under `--resume-from contract`. The deploy-request poller classifies only *terminal* states by name (success, failure, `no_changes`, closed); an unknown intermediate state waits out `--deploy-timeout` rather than failing a healthy deploy, and failures/timeouts surface as the coded runtime `SLUICE-E-PS-DEPLOY-REQUEST-FAILED` carrying the leg, DR number, observed state, and URL. `--dry-run` prints the full plan ‚Äî branch names, DDLs, the DR flow, the rendered per-chunk backfill statement, the gates, the cleanup posture ‚Äî with **zero PlanetScale API calls** (pinned by a call counter); it opens the data-plane DSN read-only for schema and statement rendering only.

- **A thin shared PlanetScale control-plane client (`internal/planetscale/api`), no SDK.** Verbs-only HTTP ‚Äî service-token auth via the pscale CLI's env convention (`PLANETSCALE_SERVICE_TOKEN_ID`/`PLANETSCALE_SERVICE_TOKEN`), PlanetScale error-envelope decoding, modest 429 backoff, and a no-URL/no-token-in-errors guarantee ‚Äî with branch and deploy-request verbs. Workflow logic (polling, branch lifecycle ordering) lives in the orchestrator, not the client, so future deploy-request features compose the same verbs. The telemetry provider's authenticated service-discovery leg was refactored onto it (one client for both PS control-plane features); its behavior is unchanged and its credential-leak pins now cover the shared client.

## Fixed

Three defects in the new command, all caught by the first live validation run against a real PlanetScale database *before* this release ‚Äî the unit fakes could not have caught any of them, and **no published release ever carried them** (they existed only between the two commits of this release on `main`). Full write-up in ADR-0162 "Live-validation findings".

- **`deployable` lives inside the nested `deployment` object of GET `/deploy-requests/{number}`, not at the top level.** The first cut read a top-level field the real response does not have, so the deploy wait never fired and **every real run would have timed out at `--deploy-timeout`** with misleading requires-review guidance. The hand-written test fixtures had encoded the same wrong shape ‚Äî self-consistent fixtures that marshal/unmarshal the client's own struct can never catch this class. Fixed: `DeployRequest.CanDeploy()` reads both locations, the api package pins a sanitized **verbatim capture** of the real API response, and the orchestrator fake serves the real nested shape.

- **Safe migrations blocks the backfill's own control-table bootstrap.** Safe migrations ‚Äî this command's prerequisite ‚Äî refuses *every* direct DDL statement on the production branch (Error 1105), including the ADR-0159 state store's `CREATE TABLE IF NOT EXISTS` (Vitess refuses the statement, table-exists or not), so the migrate leg could never bootstrap its cursor tables. Fixed twice over: the MySQL `EnsureControlTable` is now detect-then-create (zero DDL statements when the tables already exist ‚Äî behavior-preserving everywhere else), and the expand leg stages the control tables **on the dev branch** so they ship to production inside the expand deploy request, the governed channel PlanetScale mandates.

- **A new dev branch's schema base can lag production ‚Äî and a deploy request from a stale base silently reverts newer schema.** Observed live: a branch created 14 minutes after a deploy still lacked the deployed column, and a deploy request from it diffed as *dropping that column from production* (empirically deployed as cleanup ‚Äî the phantom revert is real); on the contract leg that would have dropped the freshly backfilled expand column. The lag is intermittent and its mechanism undocumented, so the guard treats freshness as unknowable a priori: every dev branch's schema is now compared to production's via the API before any DDL, a stale base self-heals once (delete branch ‚Üí on-demand backup of production ‚Üí recreate ‚Üí recheck, narrated ‚Äî the backup's duration scales with database size), and a still-stale base refuses with the new coded `SLUICE-E-PS-BRANCH-STALE-BASE`. The fresh path costs two schema GETs.

## Compatibility

- **Purely additive; no breaking changes.** One new command and three new coded errors (`SLUICE-E-PS-SAFE-MIGRATIONS-DISABLED`, `SLUICE-E-PS-DEPLOY-REQUEST-FAILED`, `SLUICE-E-PS-BRANCH-STALE-BASE`); no flag, default, or behavior change to `migrate`, `sync`, `backup`, `restore`, `backfill`, or any other existing command. The MySQL migrate-state `EnsureControlTable` change is behavior-preserving ‚Äî it detects before creating, issuing zero DDL when the tables already exist and the same `CREATE TABLE IF NOT EXISTS` when they don't; as a side benefit, a standalone `sluice backfill` against a safe-migrations-enabled PlanetScale production branch whose control tables already exist no longer trips Error 1105 on the (previously unconditional) bootstrap DDL. The telemetry provider's refactor onto the shared client changes no behavior. One posture note: `expand-contract` is the first sluice command needing a control-plane credential (the PlanetScale service token) on top of a DSN ‚Äî accepted as inherent to the deploy-request workflow, and scoped to this one opt-in command.

## Who needs this

No action required for anyone ‚Äî the three fixes above never shipped in a published release, so there is nothing to re-verify or re-run. Pick this release up if you run schema changes on PlanetScale: `expand-contract` turns the two hand-driven deploy requests around a backfill into one command with the destructive half hard-gated behind a clean verify plus `--yes`. Prerequisites to know before your first run: safe migrations must be enabled on the production branch (sluice refuses loudly and names the toggle rather than flipping it), and the service token needs branch + deploy-request access. Start with `--dry-run` ‚Äî it prints the entire plan without a single control-plane call.
