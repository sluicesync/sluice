# sluice v0.99.212

**New (opt-in Security) — KMS-backed manifest signing now covers GCP KMS and Azure Key Vault, completing the three-cloud provider matrix (ADR-0154 Phase 3b). Same as AWS in v0.99.210: the signing private key stays in the cloud HSM, and verification is pure local crypto (a DR/CI host verifies with only a public key). Both providers ride the SAME `kms` scheme and composite `kms/<algorithm>` token, so a GCP- or Azure-signed chain verifies IDENTICALLY to an AWS one — the provider isn't recorded in the on-disk format at all. Fully opt-in and drop-in: nothing changes unless you pass `--sign-key kms://gcp/…` or `kms://azure/…`, and every existing AWS / Ed25519 / HMAC / pre-v6 signed backup verifies and restores unchanged (no canonicalization bump).**

## Security

- **GCP KMS + Azure Key Vault manifest signing (ADR-0154 Phase 3b; opt-in via `--sign-key kms://gcp/…` / `kms://azure/…`).** Phase 3a (v0.99.210) shipped AWS KMS signing; 3b adds the two remaining major clouds behind the same `kms` scheme family. It was signer-adapter-only exactly as the ADR predicted — the pure verifier already covered ECDSA (DER) / RSA-PSS / Ed25519, and nothing in the canonical-serialization, verify-policy, freshness, scheme-binding, or compact/prune machinery changed. Provider selection is not in the on-disk format: a GCP- or Azure-signed `kms/ecdsa-p256` chain verifies bit-for-bit identically to an AWS one via the same composite scheme token, which is folded into the signed bytes so scheme/algorithm confusion still fails closed across all three providers.
- **GCP (`--sign-key kms://gcp/<versioned-cryptoKeyVersion-resource>`).** AsymmetricSign over ECDSA P-256/P-384, RSA-PSS, or **Ed25519** — the one primitive AWS KMS lacks (GCP signs the whole message, no external pre-digest). GCP returns ECDSA signatures as DER, like AWS, so there is no signature-encoding conversion. The GCP-specific correctness detail is the **CRC32C wire-integrity handshake**: every AsymmetricSign request carries the digest/data CRC32C, and the adapter refuses loudly unless the server echoes it as verified AND the returned signature's own CRC32C matches — both directions, so a signature corrupted in transit is caught, never emitted. GCP signs a specific key VERSION, so a bare crypto-key resource is refused up front with a clear message. Pinned by `TestGCPKMSSigner_RoundTrip` (P-256/P-384/RSA-PSS/Ed25519), `_ECDSA_IsDER_NotRawConcat`, and the three CRC-mismatch refusals.
- **Azure (`--sign-key kms://azure/<key-vault-key-url>`).** ECDSA P-256/P-384/P-521 or RSA-PSS, normalizing Azure's two real format divergences so the single-form pure verifier stays untouched. **(1)** Azure returns ECDSA signatures as raw r‖s (IEEE-P1363), not DER — the adapter converts r‖s→ASN.1 DER before returning; the conversion is pinned across P-256/P-384/**P-521** (whose odd 66-byte coordinate half is the specific trap, per the Bug-74 family-matrix discipline), with an explicit "the raw form does NOT verify, the DER-converted form DOES" assertion so a DER-vs-raw confusion is caught, not silently accepted. **(2)** Azure exports its public key as a JWK, rebuilt into a stdlib key (RSA exponent range-guarded). Sign/Verify are on `*azkeys.Client` directly in the vendored SDK (v1.5.0) — no separate CryptographyClient. Azure has no Ed25519. Pinned by `TestAzureKMSSigner_RoundTrip_AllCurvesAndRSA`, `_ECDSA_RawConcat_ConvertedToDER`, `_JWKPublicKeyRoundTrip`, and the wrong-length/wrong-key/unsupported-type refusals.
- **Verification unchanged from Phase 3a — pure local crypto, anchored on the OPERATOR's key.** `--verify-key` accepts a local SPKI PEM (offline DR, verified locally against the recorded algorithm) OR the `kms://gcp|azure/<key-ref>` reference (online public-key fetch of the operator-NAMED key). No provider Sign call on the verify path; the manifest's advisory `key_ref` is never trusted, so a store adversary rewriting it cannot redirect trust.

## Compatibility

**No breaking changes; fully opt-in; every existing signed backup verifies and restores unchanged.** Three things to know:

1. **Opt-in.** Nothing engages unless you pass a `kms://gcp/…` or `kms://azure/…` key. Migrate, sync, verify, plaintext/encrypted backup/restore, and every AWS / Ed25519 / HMAC signing path are byte-identical. New CLI surface: the `kms://` `--sign-key`/`--verify-key` forms now accept `gcp` and `azure` providers alongside `aws`.
2. **No canonicalization bump — all prior chains are byte-unchanged.** GCP/Azure signatures ride canon v3 via the composite `kms/<algorithm>` token exactly as AWS does; `ManifestCanonVersion` stays v3 and the `hmac-kek`/`ed25519`/`aws-kms` signed-byte goldens are frozen. A v0.99.208 HMAC chain, a v0.99.209 Ed25519 chain, and a v0.99.210 AWS-KMS chain all verify and restore exactly as before.
3. **A chain keeps its scheme + provider across extension and maintenance.** You cannot mix providers or schemes within one chain (extending or compacting a GCP-signed chain needs the same GCP key). Signing remains independent of the encryption keystore — the signing key may differ from the `--kms-key-arn` (or GCP/Azure) encryption key.

## Who needs this — action required

- **Operators on GCP who want HSM-custody signing keys: point `--sign-key kms://gcp/<.../cryptoKeyVersions/N>` at an ASYMMETRIC_SIGN key version (EC P-256/P-384, RSA-PSS, or Ed25519) on `backup full`/`incremental`, and verify with `--verify-key <exported-PEM | kms://gcp/<...version>>`.** GCP is the only provider offering Ed25519 signing. Distribute only the public key (exported PEM) to your verify/restore/CI hosts for offline verification.
- **Operators on Azure who want HSM-custody signing keys: point `--sign-key kms://azure/<vault-key-url>` at a Key Vault key with the sign operation enabled (EC P-256/P-384/P-521 or RSA), and verify with `--verify-key <exported-PEM | kms://azure/<key-url>>`.** Managed HSM vault URLs work the same way.
- **The keyless-signing maintenance cron (ADR §7-Q4) now works on all three clouds.** Run compact/prune/incremental with the provider `kms://` key; the cron re-signs the restructured chain via the provider's IAM/RBAC-granted Sign and never holds raw key material.
- **Operators already using AWS KMS, Ed25519, or HMAC signing: nothing to do.** Your existing signed backups verify and restore unchanged; chains continue to extend with the same scheme/provider. No re-signing, no migration.
- **Everyone else: nothing to do — purely additive, opt-in.** No existing backup, restore, migrate, sync, or verify behavior changes; no data is retroactively affected; unsigned and pre-v6 backups restore exactly as before. There is no silent-loss fix here to re-verify.

## Note on validation

Real-cloud GCP/Azure validation is deferred (the N-9 pattern — no first-party local emulator exists for GCP KMS or Azure Key Vault signing). Both adapters are pinned against faithful in-process fakes that return each provider's NATIVE wire format (GCP's CRC32C integrity fields + DER signatures; Azure's raw r‖s + JWK public keys), so the format conversions and integrity checks are genuinely exercised in CI. A live-cloud pin is a flagged follow-up alongside the AWS `kmsverify` localstack leg.
