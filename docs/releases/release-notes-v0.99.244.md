# sluice v0.99.244

**New: `sluice backfill` — the expand-contract "migrate" step as a first-class command: online-safe, resumable, keyset-chunked in-place data backfills for MySQL (including PlanetScale/Vitess) and Postgres.** Purely additive — a drop-in upgrade with no behavior change to any existing command.

## Added

- **`sluice backfill` — batched, resumable in-place data migration (ADR-0159, roadmap item 62 Phase 1).** After a schema change ships (on PlanetScale, via a deploy request), the middle step of expand→migrate→contract is a data migration that fills or transforms values into the new column — and no vendor or ORM ships that step as a batched, resumable, online primitive (the common one-shot `UPDATE`/ORM-loop shape hits PlanetScale/Vitess's synchronous-transaction-time wall, errno 3024, and holds long locks on any engine). The new command is that primitive: `sluice backfill --driver <mysql|planetscale|vitess|postgres> --dsn <dsn> --table <t> --set 'col = <expr>' [--set ...] [--where '<predicate>'] [--batch-size N] [--dry-run] [--restart]`. It is single-endpoint — the backfill reads and updates one database, no source/target pair.

- **Bounded keyset-chunked UPDATEs — online-safe by construction.** The engine walks the table's primary key and issues exactly one `UPDATE ... WHERE (pk...) > (after...) AND (pk...) <= (upper...) AND (<where>)` per chunk, clipped to at most `--batch-size` PK-walk rows (default: sluice's bulk-copy default, 5000). Both bounds are row-comparison predicates on the PK tuple, compared by the server in the column's native collation — the same exactly-once contract the chunked bulk copy pins (ADR-0096), so the boundary walk and the chunk UPDATE agree on one total order and no boundary-straddling row is skipped or double-covered. Short statements mean short locks, and on PlanetScale/Vitess each statement stays far under the errno-3024 statement-time wall. Phase 1 is deliberately sequential (one cursor, predictable load on a live table). Tables without a usable key — no PK, or a non-orderable PK type (JSON/array/geometry) — are refused loudly with `SLUICE-E-BACKFILL-NO-PRIMARY-KEY`; there is deliberately no force flag, because an unbounded whole-table UPDATE is exactly the statement shape the command exists to avoid.

- **Resumable and idempotent — safe to kill and re-run unattended.** The cursor persists in the same database via the existing `sluice_migrate_state` control tables, keyed `backfill:<table>:<12-hex hash of set+where>`: a killed run resumes where it left off, a re-run of an already-completed spec is a reported no-op (exit 0, zero rows touched), a different `--set`/`--where` spec starts fresh, and `--restart` discards the stored cursor and walks the table again from the start (`--batch-size` is excluded from the spec hash, so retuning it mid-run doesn't orphan the cursor). The cursor is written only after each chunk commits, so a crash in the gap re-executes at most one chunk on resume — which is why the `--where` predicate should be **self-describing** (e.g. `new_col IS NULL`): a guard that describes doneness makes the bounded replay a no-op, the same idempotency property sluice's CDC apply relies on. A replay without such a guard can double-apply that one chunk; this is documented, and the resume log line names the replayed cursor.

- **Native verbatim SQL, previewed with `--dry-run`.** A backfill runs inside one database, so there is no cross-dialect translation to do: the `--set 'col = expr'` right-hand side and the `--where` predicate are emitted verbatim in the engine's own dialect (the `--expr-override` posture — engine-specific functions just work). `--set` is repeatable for multi-column backfills and is split at the first `=`, so expressions containing `=` (CASE arms) pass through intact; a `--set` column that doesn't exist on the table is refused up front with `SLUICE-E-BACKFILL-UNKNOWN-COLUMN` (the message lists the table's actual columns) rather than failing per-chunk — or worse, a typo silently creating the impression the backfill ran. `--dry-run` prints the exact engine-rendered per-chunk UPDATE plus a remaining-row estimate and exits without writing anything (rows or control table); the estimate query doubles as the `--where` preflight, so an unparsable predicate fails before any UPDATE runs.

- **Engines: MySQL (all flavors) and Postgres; three new coded refusals.** MySQL — including the `planetscale` and `vitess` flavors, which ride the same bounded-UPDATE path — and Postgres implement the new optional engine surface; SQLite/D1 are refused loudly with `SLUICE-E-BACKFILL-UNSUPPORTED-ENGINE` (a single-file/edge database doesn't need the online-safety machinery — run the transform directly). All three new codes (`SLUICE-E-BACKFILL-NO-PRIMARY-KEY`, `SLUICE-E-BACKFILL-UNSUPPORTED-ENGINE`, `SLUICE-E-BACKFILL-UNKNOWN-COLUMN`) are refusal-class (exit 3) and documented in `docs/operator/error-codes.md`.

## Compatibility

- **Purely additive.** A new command, a new optional engine interface, and three new error codes — no flag, default, or behavior change to `migrate`, `sync`, `backup`, `restore`, or any other existing command; the orchestrator and reader/writer surfaces are untouched. One observability note: the `sluice_migrate_state` header table now also carries namespaced `backfill:<table>:<hash>` ids (with a `backfill` running phase) alongside migrate ids — inspection tooling that assumed only migrate ids will see them; terminal states reuse the existing phase strings.

## Who needs this

No action required for anyone — this release contains no fixes, so there is nothing to re-verify or re-run. Pick it up if you do expand-contract schema changes and need the data-migration middle step as a safe, unattended command: PlanetScale/Vitess users blocked by the errno-3024 wall on one-shot backfill UPDATEs, and anyone on MySQL or Postgres who wants a column backfill that runs in bounded batches, survives being killed, and no-ops on re-run instead of holding a whole-table lock.
