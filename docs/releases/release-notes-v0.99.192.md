# sluice v0.99.192

**Fixed — pointing the vanilla `mysql` driver at a PlanetScale host now fails loudly *up front* instead of obscurely mid-run. A `*.connect.psdb.cloud` / `*.private-connect.psdb.cloud` endpoint needs the `planetscale` driver, not `mysql`; a new connection-free preflight now refuses before any work begins with the coded error `SLUICE-E-DRIVER-HOST-MISMATCH` (exit 3) and the hint `pass --<role>-driver planetscale`. Drop-in: no data-path or format change, and no effect on non-PlanetScale hosts or on the `planetscale`/`vitess` flavors — purely an earlier, clearer failure for a misconfiguration.**

## Fixed

- **`--source-driver mysql` / `--target-driver mysql` aimed at a PlanetScale endpoint is now refused before any connection, naming the host and the driver flag to fix, instead of failing obscurely partway through a run (UX/foot-gun fix; no data effect; the obscure mid-run failure was present in every prior release).** A PlanetScale host (`*.connect.psdb.cloud` / `*.private-connect.psdb.cloud`) can only be driven by the `planetscale` driver: the vanilla `mysql` flavor uses binlog CDC and `LOAD DATA` cold-copy, both of which Vitess/PlanetScale block, so `mysql` against a PlanetScale host previously connected and then failed deep in the run with a confusing low-level error. Because the hostname is already known from the DSN, sluice can now catch the mismatch from the string alone: a new optional engine surface (`ir.DSNValidator`, modeled on `ir.ShardDiscoverer` and implemented by the mysql engine) is consulted for the source **and** the target at the very top of both migrate and sync — before any reader or writer opens — and the vanilla flavor refuses a PlanetScale host with a role-agnostic message naming the host and the reason (binlog CDC + `LOAD DATA` are Vitess-blocked). The orchestrator wraps that refusal in a coded error (`SLUICE-E-DRIVER-HOST-MISMATCH`, a refusal → exit 3), prefixes the role (`source` / `target`, source checked first), and supplies the exact hint `pass --source-driver planetscale` / `pass --target-driver planetscale`. The `planetscale` and `vitess` flavors — the correct drivers for that host — are a no-op, and any engine that doesn't implement the surface is silently skipped. Pinned by a unit matrix over the mysql `ValidateDSN` (both PSDB suffixes × all three flavors, plus no false positives on normal / garbage / Unix-socket / empty DSNs) and over the pipeline preflight (coded refusal, role prefix, source-before-target ordering, and a no-op when no engine implements the surface).

## Compatibility

**No breaking changes; drop-in.** No CLI flag semantics change, no default that affects an existing correct invocation, no data-path change, and no backup format-version change — every migrate / sync / backup / restore data path is byte-identical. The only new behavior is a preflight refusal, and it fires exactly on the one misconfiguration it names: the vanilla `mysql` driver pointed at a PlanetScale endpoint, which never worked (it failed mid-run before). Non-PlanetScale hosts are entirely unaffected, and the `planetscale` / `vitess` flavors — the correct drivers for a PlanetScale host — are a no-op in the check, so any invocation that succeeds today still succeeds and behaves identically. A run that used to fail obscurely partway through now fails immediately with an actionable, coded message instead.

## Who needs this — action required

- **PlanetScale users who might reach for `--source-driver mysql` / `--target-driver mysql`.** You now get an up-front, coded refusal (`SLUICE-E-DRIVER-HOST-MISMATCH`, exit 3) naming the host with the hint to pass `--source-driver planetscale` / `--target-driver planetscale`, instead of a confusing mid-run failure. No data was ever silently mis-migrated by the old behavior — the vanilla `mysql` driver against a PlanetScale host always failed loudly (just late and obscurely) — so there is nothing to re-verify; simply re-run with the `planetscale` driver.
- **Everyone else: no action.** Non-PlanetScale hosts and the `planetscale` / `vitess` flavors are unaffected; no migrated data needs re-verification.
