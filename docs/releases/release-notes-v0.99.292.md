# sluice v0.99.292

**Four new preflight gates — led by a `binlog_format=ROW` refusal that closes a fully silent MySQL/MariaDB CDC empty-stream mode (ground-truth-verified: a STATEMENT-format source cold-copies clean, then silently loses every subsequent change while sync stays green) — plus four loud, zero-loss fixes from the v0.99.291 regression cycle and roadmap residuals.** Drop-in upgrade, no breaking changes for correctly-configured sources. If you run binlog CDC from a MySQL or MariaDB source, check `binlog_format` now — see "Who needs this" below: on every earlier release a non-ROW source froze the target at the snapshot with no error at all.

## Features

**`binlog_format=ROW` upfront gate — binlog CDC now refuses a STATEMENT/MIXED-format MySQL or MariaDB source at every CDC start, with the new coded `SLUICE-E-CDC-BINLOG-FORMAT-NOT-ROW` (roadmap item 68e).** The premise was verified on a real `mysql:8.0` source before gating: with `binlog_format=STATEMENT` the cold copy lands, then every live INSERT/UPDATE/DELETE is silently lost — DML arrives as QueryEvents the dispatcher treats as generic DDL, the target freezes at the copy snapshot, the persisted position never advances, and both the stream and shutdown exit clean. The worst silent-loss shape, so the gate ships: `@@GLOBAL.binlog_format` is read at the same three chokepoints as the Bug-193 row-image preflight (StreamChanges — covering sync cold-start handoff, warm resume, and `backup incremental` — plus both snapshot openers, so a cold start refuses before the bulk copy), and anything but ROW refuses with the value and the remedy (`SET GLOBAL binlog_format=ROW` / provider console). MIXED is refused as the same class — deterministic writes are statement-logged, and MIXED is MariaDB's platform default, so an un-tuned MariaDB source hits this out of the box. VStream sources never take this path (vtgate owns the row-event contract) and bulk-only runs (`migrate`, `backup` full) are not gated. Documented residual: a SUPER session-level STATEMENT override slips the global read. Pinned by a value-family unit matrix plus a real-server integration matrix (mysql STATEMENT, mysql MIXED, mariadb STATEMENT), each asserting the coded refusal fires with the target untouched.

**Replication-headroom preflight — a slot-creating Postgres CDC cold start refuses up front, coded, when the source has no room for one more replication slot or WAL sender (roadmap item 68d).** Previously a full `max_replication_slots` ceiling failed mid-cold-start with the raw `all replication slots are in use` (53400), after the schema read, with no inventory and no pointer at the tooling that frees leftovers. The new preflight probes `max_replication_slots` vs `pg_replication_slots` and `max_wal_senders` vs `pg_stat_replication` before creating the slot, refusing with the new `SLUICE-E-CDC-REPLICATION-HEADROOM` naming the usage, the ceiling, each existing slot (active/inactive), and the remedies — `sluice slot list`, `sync decommission` for finished streams, `slot drop` for abandoned leftovers, or raising the setting (with per-stream publications making one-slot-per-stream the staged-wave pattern, a full default ceiling usually means leftover slots from finished waves). Cold start only: warm resume and the interrupted-COPY resume reuse their existing slot and never probe (integration-pinned on a server AT the ceiling). Managed-PG permission variance degrades gracefully — a failed census WARNs and continues, so the refusal only fires on a successful probe; `postgres-trigger` (slot-less) and MySQL sources are excluded by the capability gate.

**Large-object stranding advisory — a Postgres source holding `pg_largeobject` blobs now WARNs at migrate/sync preflight that the blobs are not copied (roadmap item 68c).** Large-object contents live outside every user table; sluice copies the referencing `oid`/`lo` column values faithfully as plain integers, and until now the stranded blobs left no signal at all. The new census probes `pg_largeobject_metadata` plus the in-scope `oid`/`lo`-typed columns: suspects in scope get a WARN naming every `table.column`, stating the target's ids will point at nothing, and pointing at the new `docs/type-mapping.md` "Postgres large objects" section (carry recipes: `lo_export`/`pg_dump --blobs`, or convert to inline `bytea` via `lo_get` first); large objects with no in-scope suspect column get a single quieter WARN. Advisory only, never a refusal, and a failed probe skips silently — the census adds no failure mode.

## Fixed

**The ADR-0166 pre-create shape gate now guards the SYNC cold-start leg — an empty-but-drifted pre-existing target table refuses up front with `SLUICE-E-TARGET-TABLE-SHAPE-MISMATCH` instead of cold-copying into the drifted schema (roadmap item 25 residual; present on the sync leg since sync shipped — the migrate leg was closed in v0.99.258).** The Streamer's cold start called `CreateTablesWithoutConstraints` (IF NOT EXISTS) bare, so a drifted empty table passed the Bug-9 populated-target check and the copy ran into the wrong schema — silent value coercion under a relaxed-`sql_mode` MySQL target, and on a strict target a mid-copy `Unknown column` error riding the transient-retry wall (reproduced pre-fix: the sync wedged for the full 10-minute test budget with no refusal). The comparison core is now shared with migrate's gate: matching shape → INFO skip exactly like migrate, drifted → the coded refusal phrased for the sync leg, uncomputable/no-mapping → WARN and proceed as before. The skip branches keep their contracts: `--reset-target-data` and `--restart-from-scratch` drop tables first, `--schema-already-applied` is the operator's promise, the interrupted-COPY resume mirrors migrate's `--resume` carve-out, and warm resume never creates tables. Filed, not implied: the multi-database (ADR-0074) sync cold-start path still creates tables ungated.

**Multi-table filtered sync on a native MySQL source no longer refuses at cold start (Bug 201 — affected v0.99.276–v0.99.291, since filtered sync shipped; loud, zero-loss).** The ADR-0101 concurrent cold-copy reader — the default whenever a native-MySQL stream covers 2+ tables — never implemented `ir.RowFilterSetter`, so `sync start --where` refused at cold start with the misleading `source engine "mysql" does not support row-level filtering` (the serial `--copy-table-parallelism=1` path worked, which was the verified workaround). The concurrent reader now stamps the per-table predicate map onto every pinned-snapshot inner reader, so each concurrent table leg ANDs its predicate server-side into both the keyless full scan and the keyed/chunked paged reads — identical semantics to the serial reader — and an ADR-0111 mid-copy re-snapshot recovery re-stamps the map onto its fresh readers, so a recovery can never silently resume a filtered table unfiltered. The refusal gate survives for readers that genuinely lack the setter (SQLite/D1/flat-file). Pinned RED-before-GREEN by a real-MySQL integration test (3 tables, parallelism 2, distinct predicates, out-of-predicate rows asserted absent, CDC leg still filtering after handoff).

**The apply-retry budget no longer carries over across ridden-out target outages (Bug 202 — affected v0.99.290–v0.99.291, since target-outage ride-out first worked; loud, zero-loss).** The consecutive-failure counter's only reset path required a successful position read from the TARGET — but the target is also the position store, so when it was down again at the next failure the read failed, progress applied between outages was never credited, and the second target outage of a stream's lifetime exited `apply retry budget exhausted` on its first failures despite hours of verified progress in between. The retry loop now keeps an in-memory progress ledger with the D0-4 discipline (a failed read is never evidence in either direction): a committed-progress sentinel polls the persisted position at a bounded 5s cadence while each attempt is flowing, and any successful anchor-bearing observation showing the token advanced since the previous counted failure proves durable commits happened (position writes ride the batch tx) and grants a fresh budget. The loud-failure floor is pinned both ways: a genuinely stuck batch and a never-reachable target both still exhaust in exactly the configured attempts.

**A retry re-establish landing in a restarting Postgres's `57P03 database system is starting up` window now rides the retry budget instead of exiting terminal (Bug 203 — affected v0.99.288–v0.99.291 at the applier-open re-establish site; loud, zero-loss).** The connect-phase classifier delegated to the shared network-shape matcher only (whose corpus deliberately excludes PG lifecycle wordings) and had no structured SQLSTATE check at all — while the applier classifier and the trigger-CDC poll classifier both classified the same SQLSTATE retriable. The set (57P01/57P02/57P03 + class 08) is now single-homed as `nettransient.IsConnectionAvailabilitySQLState`, matched structurally via the driver error's `SQLState()` method; `postgres.IsReadTransientSQLState` delegates to it and a delegation-parity pin fails CI if either home drifts. Auth, invalid-catalog, and every other SQLSTATE stay terminal, pinned per cell. The MySQL sibling was ground-truthed rather than assumed: a restarting MySQL presents no structured handshake errno (the whole window surfaces as transport shapes the corpus already classifies), so no MySQL leg was added and the asymmetry is documented.

## Compatibility

No breaking changes for correctly-configured setups; drop-in upgrade, no flag changes. Three refusals and one WARN are new, all deliberate: a STATEMENT/MIXED `binlog_format` source now refuses at every binlog CDC start — such a source was never actually syncing (the stream ran green while losing every change), so this converts an existing silent failure into a loud one; a full replication-slot/WAL-sender ceiling on a PG source now refuses before the slot create (only on a successful census — permission-limited managed-PG roles degrade to a WARN); and a sync cold start into an empty-but-drifted pre-existing target table now refuses instead of copying into the wrong schema (`--schema-already-applied`, `--reset-target-data`, and resume paths are deliberately ungated; a matching pre-existing table is now INFO-skipped exactly like migrate). One refusal is removed: multi-table filtered sync on native MySQL now works instead of refusing, so the `--copy-table-parallelism=1` workaround is no longer needed. VStream flavors and bulk-only runs are unaffected by the binlog-format gate; `migrate` is unaffected by the retry/budget fixes.

## Who needs this — action required

- **Anyone running binlog CDC (`sync`, `backup incremental`) from a MySQL or MariaDB source — check your `binlog_format` NOW** (`SELECT @@GLOBAL.binlog_format`). If it is not ROW, on every release before v0.99.292 the stream silently froze at the cold-copy snapshot: every change since then was lost with sync green and exit 0. MIXED counts — it is MariaDB's platform default. Remedy: set `binlog_format=ROW` (or the provider-console equivalent), then `--restart-from-scratch` and re-verify (`sluice verify`); v0.99.292 refuses such a source up front so this cannot recur. If your format is already ROW, nothing was lost and no action is needed.
- **Anyone who cold-started a sync into pre-existing (empty) target tables on a relaxed-`sql_mode` MySQL target** — if the target tables' shapes could have drifted from the source, re-verify those tables (`sluice verify`): pre-292 a drifted empty table was copied into with silent value coercion. Targets created by sluice itself, strict-mode targets (which failed loudly), and `--schema-already-applied` runs where you controlled the shape are not exposed.
- **Multi-table filtered native-MySQL sync operators** — the `--copy-table-parallelism=1` workaround for the cold-start refusal is no longer needed; drop it to get the concurrent cold copy back.
- **Anyone whose long-lived stream exited `apply retry budget exhausted` early in a second target outage on v0.99.290–v0.99.291** — that was Bug 202; upgrade and restart the stream (warm resume — the exit was loud and lost nothing).
- Everyone else: no action needed beyond upgrading.
