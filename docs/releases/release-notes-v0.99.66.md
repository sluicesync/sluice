# sluice v0.99.66

**HIGH fix — resuming a `sluice sync` cold-copy of a large multi-table Vitess/PlanetScale keyspace no longer crash-loops on the bounded-memory cap; `--resume` now drives the same per-table auto-shard path as a fresh cold-start (ADR-0098).** Drop-in: nothing changes in your invocation, and a resume of a keyspace that auto-sharded on cold-start (v0.99.62+) is now memory-equivalent to that cold-start instead of falling back to the legacy interleaved stream. No data was ever silently lost by this bug — the old resume failed loudly (a buffer-cap error / restart crash-loop), it never completed with missing rows.

## Fixed

- **Auto-shard-aware VStream cold-copy resume (Bug 156, HIGH).** ADR-0095 (v0.99.62) made the VStream/PlanetScale cold-copy auto-shard by table — open, drain, and complete one single-table VStream COPY at a time, bounded memory, no interleave — but it gated that path on a *fresh* cold-start (`start == nil`). A `--resume` (a position carrying an ADR-0072 per-shard `TablePKs` cursor) fell back to the legacy single keyspace-wide interleaved stream, which buffers every not-yet-active table and hits the ADR-0071 bounded-memory cap; on a large multi-table keyspace the second table inevitably exceeds any host-fittable `--max-buffer-bytes` while the first drains, so the resume failed with `table "X" would buffer … exceeding the --max-buffer-bytes cap …` and, under a restart-on-failure supervisor, crash-looped without ever making progress. The resume of exactly the large-keyspace case auto-shard exists for was broken. This release drops the `start == nil` gate (`autoShard := len(tables) > 1 && !singleStream`) so a resume drives the *same* per-table pump as cold-start: the persisted cursor names the single in-progress table `t_k`, and in unchanged schema order the pump re-copies the already-finished tables `[0..k)` from-beginning (the Bug-125 idempotent upsert absorbs the overlap — no double-apply), resumes `t_k` from its persisted cursor (no row-0 restart), and copies the remaining tables `(k..n)` fresh. Because every table — re-copied, resumed, or fresh — contributes a captured per-table snapshot, the ADR-0095 per-shard GTID-set-minimum stitch for the CDC handoff is the cold-start stitch verbatim: the resume is gapless (CDC is recorded only after the last table completes, so it never advances past an incompletely-copied table — ADR-0007) and bounded-memory (exactly one single-table COPY is ever in flight, so the ADR-0071 interleave refusal is structurally unreachable on resume). Pinned by unit tests for `resolveResumeAutoShard` (single in-scope cursor → table; multi-table / out-of-scope / empty → loud refuse) and the resume routing (engages the auto-shard pump, seeds the in-progress table from the cursor, re-copies the prefix, copies the tail fresh — including the in-progress-at-index-0 edge), plus a `vstream`-tagged integration regression test that cold-starts an auto-shard copy where Σ(tables) far exceeds a tiny `--max-buffer-bytes`, interrupts mid-copy after ≥1 table completes, resumes under the same tiny cap, and asserts no buffer-cap error on any table, the in-progress table resuming strictly past its cursor with its full tail, full per-table delivery, and a non-empty stitched handoff position.
- **No new position-token field; a legacy/ambiguous cursor is refused loudly.** Completion is re-derived from the persisted cursor plus the known table order rather than carried in new state — the cursor names exactly the one in-progress table, so everything before it in the sequence is finished and everything after is unstarted. The skip-the-re-copy optimization (persist a completed-tables GTID floor and start the prefix past it) was deliberately rejected for v1: it needs hand-rolled GTID-set interval arithmetic to stay gapless — the Bug-74 subtle-codec risk class — whereas re-copying the finished prefix is provably correct by reduction to the already-reviewed cold-start path. A position written by released auto-shard code (v0.99.62+) round-trips unchanged; a legacy single-stream / multi-table cursor, or a cursor naming a table no longer in scope, is refused with a named, actionable error (resume on the legacy `vstream_copy_single_stream=true` path, or restart the cold-start) rather than silently re-copying the whole keyspace or skipping a table.

## Compatibility

- **No breaking changes; drop-in upgrade.** Nothing changes in your `migrate`/`sync` invocation, and the persisted handoff/position token shape is unchanged (no new field). A resume of a keyspace that auto-sharded on cold-start now stays on the auto-shard path instead of falling back to the legacy interleave.
- **Scope: the `sluice sync` / VStream (Vitess/PlanetScale-source) cold-copy resume path only.** Cold-start itself was already correct (auto-shard worked from v0.99.62); only the resume path was affected. A single-table scope is already constant-memory single-stream, so this changes nothing there. Vanilla MySQL binlog and PostgreSQL cold-copy paths are untouched.
- **Opt-out unchanged.** `vstream_copy_single_stream=true` in the source DSN still restores the legacy single keyspace-wide interleaved COPY (and its ADR-0071 multi-table loud-refusal floor) for both cold-start and resume; that is also the path a refused legacy/ambiguous cursor points the operator to.

## Who needs this — action required

- **Anyone running `sluice sync` from a Vitess/PlanetScale source over a large multi-table keyspace who relies on resume after an interruption:** upgrade to v0.99.66. Before this release a `--resume` of such a keyspace could fail with an ADR-0071 buffer-cap error and crash-loop under a restart supervisor, never making progress; now the resume completes at bounded memory, resuming the in-progress table from its cursor and re-copying the already-finished tables idempotently. If you previously worked around this by restarting the whole copy or by raising `--max-buffer-bytes`, neither is needed anymore.
- **No re-verification of prior syncs is required, and no completed migration is at risk.** This bug never produced wrong or missing data — the old resume path failed loudly (a buffer-cap error / zero-progress crash-loop), it never exited 0 with a gap. Cold-start was always correct, and any sync that actually completed on a prior version is unaffected and needs no re-check.
