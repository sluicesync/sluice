## What was happening

`ClassifyShape`'s `diffAlteredColumn` compares `preCol.Type` vs `postCol.Type` via `reflect.DeepEqual`. The cold-start seed comes from PG's `SchemaReader` (rich projection â€” reads `information_schema` for `attnotnull`, `is_identity`, collations, etc.), but the CDC `SchemaSnapshot` comes from pgoutput's `RelationMessage` (poor projection â€” just `name`, `OID`, `typmod`, key-flag). Same column, two different IR struct shapes â†’ `DeepEqual=false` â†’ false `altered-col=true` on every existing column. Combined with a legitimate `added=1` on the new column, that's a "multi-shape combo" refusal via DP-E.

Phase A instrumentation in `diffAlteredColumn` against a local PG repro captured the exact smoking-gun field on the catalogued repro schema:

```
col=id  pre_type=ir.Integer{Width:64, Unsigned:false, AutoIncrement:true}
        post_type=ir.Integer{Width:64, Unsigned:false, AutoIncrement:false}
        type_differs=true
```

PG's `SchemaReader` sets `Integer.AutoIncrement=true` for `BIGINT GENERATED BY DEFAULT AS IDENTITY`; pgoutput's OID-to-type mapping can't distinguish IDENTITY from a plain `BIGINT`, so the CDC projection has `AutoIncrement=false`. Same class of asymmetry applies to `Varchar`/`Char`/`Text.Collation` (PG reads explicit column collations from `pg_attribute.attcollation`) and `Decimal.Unconstrained` (PG distinguishes bare `numeric` from `numeric(p,s)`).

## Fixed

- **`fix(adr-0054): Bug 84 â€” PG cold-start seed CDC-projection mismatch`.** New optional engine interface `ir.CDCSchemaSnapshotNormalizer` with method `NormalizeForCDCComparison(*Table) *Table`. PG implements it (`internal/engines/postgres/cdc_normalize.go`) and zeros the five known-asymmetric fields documented above. MySQL deliberately does NOT implement it â€” its `TableMapEvent` decoder re-reads `information_schema` on schema-change boundaries, so the CDC projection already matches the SchemaReader's output. `synthesizeColdStartSeedSnapshots` now normalizes each seed table via the engine's normalizer (when implemented) before storing it in the intercept's cache. Subsequent `(pre, post)` comparisons on unchanged columns see byte-identical projections â€” no more false `altered-col=true`.
