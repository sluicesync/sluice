# sluice v0.99.144

**The SQLite / Cloudflare D1 source now carries the schema features it used to silently drop — generated columns land as real GENERATED columns, CHECK constraints are enforced on the target, and partial/expression indexes keep their predicate/expression. Plus a latent-bug fix in the DDL emitters so an unknown-dialect expression is never silently mistranslated. Purely additive for SQLite/D1 imports; byte-identical for every existing Postgres/MySQL path.**

## Features

**SQLite/D1 source carries generated columns, CHECK constraints, and partial/expression indexes (ADR-0133).** The `sqlite` (file/`.sql`-dump) and `d1` (query-API) readers previously read columns/PK/FK/plain indexes but silently omitted three schema features the IR already models — and the Postgres and MySQL engines already read and emit. SQLite was simply the one reader not wired into them; this wires it in. Generated columns are detected via `PRAGMA table_xinfo` (the `hidden` flag: 2 = VIRTUAL, 3 = STORED) with their generation expression parsed from the `CREATE TABLE` SQL, so on the target the column lands as a real GENERATED column that re-derives its value (excluded from the bulk-copy INSERT, matching the existing convention) rather than as a static copy. CHECK constraints — table-level and column-level, named and anonymous — are parsed from the `CREATE TABLE` SQL by a paren/string/identifier/comment-aware tokenizer (it matches `CHECK` only as a standalone keyword, so a `checkout` column or a string default containing the word CHECK never false-positives) and are enforced on the target. Partial indexes carry their `WHERE` predicate — closing a Bug-19a-class silent uniqueness-widening for a UNIQUE partial index on a Postgres target — and expression indexes carry each indexed expression; an expression-index column list that can't be cleanly parsed is WARN-skipped rather than carrying a guessed set. The SQLite/D1 engine now declares `SupportsCheckConstraint` and `SupportsGeneratedColumns` true.

**Verbatim carry with loud target rejection, never a silent guess.** SQLite is a third dialect, and sluice's cross-dialect expression translator is mysql↔postgres only. Rather than guess, a carried SQLite expression is emitted VERBATIM (tagged dialect `"sqlite"`): portable constructs (`a + b`, `length(x)`, `x || y`, comparisons) work on the target, while non-portable ones (`strftime(...)`, SQLite-only functions) are rejected LOUDLY at target DDL time naming the rejected function — never a silent drop or a silent mistranslation. Each table/index that carries such a body emits a one-time WARN naming it, so an operator can verify the one residual edge (a portable-looking operator with SQLite's loose-typing semantics). A real SQLite→canonical expression translator for that verbatim tail is the tracked next increment (roadmap item 49).

## Fixed

**Writer dialect-guard — an unknown-dialect schema expression is no longer silently mistranslated (latent bug).** The Postgres and MySQL DDL emitters dispatch on an IR expression's dialect tag (generated-column body, CHECK, partial-index predicate, expression-index column, and column default): same-dialect/untagged emits verbatim, otherwise the cross-dialect translator runs. That translator is mysql↔postgres only, and the dispatch's `else` branch was hardcoded to "the one other known engine" — so any expression tagged with a third dialect (now `"sqlite"`, or any future engine) was silently fed through the wrong translator (`translateExprForPG` expects MySQL input; `translateExprForMySQL` expects PG input). The concrete hazard: SQLite's double-quoted-string misfeature `DEFAULT "draft"` would be rewritten into the backtick *identifier* `` `draft` `` on a MySQL target — a silently-wrong default. Every dispatch site on both engines now translates ONLY when the source dialect is the specific other engine it can translate from (Postgres iff `"mysql"`, MySQL iff `"postgres"`); every other value — `"sqlite"`, `""`, any unknown — emits verbatim and surfaces a non-portable construct as a loud target rejection.

## Compatibility

Additive for SQLite/D1 imports and byte-identical for every existing Postgres/MySQL path. The dialect-guard change is provably a no-op for current producers: the only non-self, non-empty dialect any reader emits today is the one-other-engine, for which "translate unless self/empty" and "translate only from the known other" select the same branch; only a `"sqlite"`/unknown tag flips from mistranslate to verbatim. Shipped under an independent value-fidelity review (which caught and got fixed the DEFAULT-expression and CHECK-refuse dispatch sites that were initially left unguarded) plus reader-extraction unit tests, writer-guard matrix tests on both engines, and cross-engine integration tests (generated columns re-derive on PG/MySQL, a CHECK is enforced on the target, a partial-index predicate is present on PG, and a non-portable generated column is loud-rejected).

## Who needs this

Anyone migrating a SQLite or Cloudflare D1 database that uses generated/computed columns, CHECK constraints, or partial/expression indexes — these now carry to the Postgres/MySQL target instead of being silently dropped. Plain Postgres↔MySQL users are unaffected (the dialect-guard fix is a behavior-preserving latent-bug fix for them).
