# sluice v0.99.193

**New: `--control-keyspace` — `sync` now works against a *sharded* PlanetScale/Vitess target by routing sluice's three internal CDC control tables to an unsharded sidecar keyspace, auto-detected when unset. Previously a sharded target rejected those vindex-less tables with `VT09001: … does not have a primary vindex` and aborted the sync *before any rows copied*, so a sharded→sharded region move was impossible. Drop-in: unsharded targets and non-Vitess engines (Postgres/SQLite) are byte-identical to before — the feature engages only for a sharded MySQL/PlanetScale/Vitess target, and on one you usually need not pass the flag at all.**

## Features

- **`--control-keyspace <name>` routes sluice's CDC control tables (`sluice_cdc_state`, `sluice_cdc_schema_history`, `sluice_shard_consolidation_lease`) to a separate *unsharded* keyspace, unblocking sync into a sharded PlanetScale/Vitess target.** Vitess requires every table in a sharded keyspace to carry a primary vindex; sluice's three internal control tables have none, so a sync against a sharded target keyspace previously died at `CREATE TABLE` with `VT09001: … does not have a primary vindex` before a single row copied. The flag keyspace-qualifies every control-table statement (backtick-quoted `` `ks`.`sluice_cdc_state` ``, so PlanetScale's hyphenated default-keyspace names are accepted) and scopes the `information_schema` lookups to that keyspace, so the vindex-less tables live in an unsharded keyspace that accepts them while the data keeps flowing into the sharded one. Threaded through `sync start` / `sync stop` / `sync status` / `sync health` and `schema add-table`. Pinned by the pure `selectControlKeyspace` decision matrix and unit + real-2-shard-PlanetScale integration coverage.
- **On a sharded target with the flag unset, sluice auto-detects the default unsharded sidecar keyspace — zero manual setup.** It enumerates the vtgate's keyspaces (one `SHOW VITESS_SHARDS`, classifying each keyspace as unsharded when it reports the single `-` shard) and, when the data keyspace is sharded, picks the sole unsharded keyspace that isn't the data keyspace. Exactly one candidate is used automatically; **zero or several candidates is a loud refusal that names them** and asks you to pass `--control-keyspace`. You pass the flag only to override the auto-pick or to disambiguate. A shard-discovery failure (endpoint isn't really Vitess, transient error) is not fatal — it falls back to the unchanged bare-name behavior, and a genuinely sharded target then still fails loudly at `CREATE TABLE` rather than silently. Pinned by the decision-matrix unit tests plus a live auto-detect on a real 2-shard PlanetScale target that found the hyphenated default keyspace with no manual setup.

## Compatibility

**No breaking changes; drop-in.** No CLI flag semantics change for existing invocations, no default that affects a correct invocation today, no data-path change, and no backup format-version change. The whole feature engages **only** for a sharded MySQL/PlanetScale/Vitess target: an unsharded MySQL/PlanetScale/Vitess target and every non-Vitess engine (Postgres, SQLite/D1) keep their control tables in the data keyspace on a single-keyspace transaction and are **byte-identical to before** — pinned by a unit test on the unset path. One semantics note to state honestly: on a sharded target the CDC position write still rides the same transaction as the data write, but that transaction is now a **cross-keyspace, best-effort (non-2PC) commit**. This is acceptable because a sharded apply is already cross-shard and non-atomic, and keyed idempotent apply makes a torn resume a safe no-op — live-proven: a hard crash mid-CDC warm-resumed from the sidecar-persisted position with zero duplicates, zero gaps, and source md5 == target md5. **Known limitation:** the fleet paths (`sync run`, `sync status --all`) do **not** yet carry `--control-keyspace` — that needs a per-entry `syncs.yaml` config field and is deferred; the single-target `sync` commands are covered.

## Who needs this — action required

- **Operators moving or replicating *into* a sharded PlanetScale/Vitess keyspace** (e.g. a region move that preserves sharding on the target). On a single-target `sync start` you usually need to do nothing — sluice auto-detects the unsharded sidecar keyspace; pass `--control-keyspace <name>` only if auto-detect refuses because zero or several unsharded keyspaces exist (the refusal names them). Use fleet `sync run` / `sync status --all` for such a target is not yet supported (known limitation above); use the single-target `sync` commands.
- **Everyone else: no action.** Unsharded MySQL/PlanetScale/Vitess targets and all Postgres/SQLite/D1 work are unchanged and byte-identical; nothing to re-verify, nothing to re-run.
