# sluice v0.99.191

**Fixed — a genuinely sharded PlanetScale/Vitess *source* now cold-copies all rows from all shards. On a sharded source keyspace the VStream snapshot cold-copy failed closed with `FailedPrecondition … shard provided in VGTID, -, not found` and copied *zero* rows — a loud failure, no silent loss — even when the operator passed `--allow-cross-shard-merge`. Shard auto-discovery is now the default, so a sharded source works with no extra configuration. Drop-in: sharded PlanetScale/Vitess sources only; unsharded PlanetScale (the overwhelming majority) is behaviorally unchanged. No data-fidelity or format change.**

## Fixed

- **Sharded PlanetScale/Vitess *source* cold-copy now discovers the real shard layout and copies every row from every shard, instead of failing closed with zero rows (loud failure — no silent loss; pre-existing since VStream multi-shard support landed in v0.1.0).** On a genuinely sharded source keyspace the VStream snapshot cold-copy (`sync start`'s snapshot→CDC handoff) built its VGTID from whatever `resolveVStreamShards` returned, and with no explicit `vstream_shards` and auto-discovery off that list silently defaulted to `["-"]` — the *unsharded* convention — so the copy asked vtgate for a keyspace-wide `{Shard:"-"}` VGTID that a sharded keyspace has no such shard for. vtgate rejected it (`FailedPrecondition "… shard provided in VGTID, -, not found …"`) and the cold-copy delivered *zero* rows. Because the cross-shard-collision preflight discovers the real shards independently (via the engine's `DiscoverShards`), it correctly saw `[-80,80-]` while the copy assumed `["-"]` — guard and copy disagreed — so even passing `--allow-cross-shard-merge` (which clears the collision guard) did not help: the copy still built the wrong keyspace-wide VGTID and copied nothing. Fixed by making shard auto-discovery the **default**: whenever `vstream_shards` isn't pinned, sluice reads the real layout via a plain, parameterless `SHOW VITESS_SHARDS` — which returns a single `-` for an unsharded keyspace (so that path is byte-identical to the old `["-"]` default) and every shard for a sharded one — and discovery returning zero shards is now a loud error, never a silent `["-"]` fallback that would resurrect the wrong-layout bug. Discovery is deferred to stream-open (moved out of reader construction) so building a reader stays connection-free; an explicit `vstream_shards=…` still pins/overrides and skips the query, and `vstream_auto_discover_shards=true` is retained for back-compat (it now names the default). Complements v0.99.190, which fixed the shard-discovery query itself (removing the `LIKE ?` bind parameter that vtgate rejected) for unsharded databases. Pinned by the re-worked `resolveVStreamShards` unit matrix (the no-explicit default now routes to discovery — proven by the discovery-wrapped error, impossible if it still short-circuited to `["-"]`) and, authoritatively, by the new build-tagged integration test `TestVitessReshard_ShardedColdCopy_AllShardsCopied`, which reshards a keyspace 1→2 and cold-copies all 200 rows from both shards of a real 2-shard `vitess/lite` cluster with no `vstream_shards` set (FailedPrecondition gone). The `-race` Integration matrix — including the vstream and engines-mysql suites — was green on the tagged commit before tag.

## Compatibility

**No breaking changes; drop-in.** No CLI flag semantics change, no default that affects an existing invocation, no data-path change, no backup format-version change; every migrate / sync / backup / restore data path is byte-identical and every non-VStream engine and direction is unaffected. The only behavior change is corrective and confined to VStream sources: shard resolution now auto-discovers by default instead of assuming `["-"]`. On an **unsharded** PlanetScale/Vitess source (the overwhelming majority — one keyspace per database) discovery returns the same single `-`, so behavior is unchanged; on a **sharded** source a migrate/sync that previously failed closed with zero rows now copies all shards. An explicit `vstream_shards=…` still pins the layout exactly as before, and `vstream_auto_discover_shards=true` remains accepted (it now names the default). Non-PlanetScale sources (vanilla MySQL, Postgres, SQLite/D1) are entirely unaffected — the changed path is VStream-only.

## Who needs this — action required

- **Anyone migrating or syncing from a *sharded* PlanetScale/Vitess keyspace** (necessarily with `--allow-cross-shard-merge` or `--inject-shard-column`, which the cross-shard-collision guard requires for a sharded source). On v0.99.190 and earlier such a run failed closed — `FailedPrecondition … shard provided in VGTID, -, not found`, zero rows copied — so there is no silently mis-migrated data to re-verify: the copy either loud-failed or, on this release, succeeds. Action: re-run the migrate/sync on v0.99.191; it now discovers all shards and copies every row. Keep `vstream_shards` unset to let discovery run, or pin it only if you must override the layout.
- **Everyone else: no action.** Unsharded PlanetScale/Vitess sources are behaviorally unchanged (discovery returns the same single `-`), and non-PlanetScale engines and directions are unaffected. No migrated data needs re-verification.
