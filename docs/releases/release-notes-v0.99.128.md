# sluice v0.99.128

**New: intra-table PK-range work-stealing for the native-MySQL concurrent cold-copy (ADR-0119, roadmap 21b) — idle reader connections now steal a CHUNK of the last big table, so the copy stays N-wide all the way to the tail instead of tapering to one whole table. A throughput optimization, not a correctness change; the prior path was already correct. Fully drop-in over v0.99.127 — on by default for the native-MySQL concurrent cold-copy, inert everywhere else, with a new `--no-intra-table-stealing` opt-out.**

## Features

**Intra-table PK-range work-stealing on the native-MySQL concurrent cold-copy (ADR-0119, roadmap item 21 tier b).** Tier (a) work-stealing (v0.99.74) lets an idle copy pipeline claim a whole *unstarted* table off a shared cursor, but the cold-copy tail was still bounded by the *last single large table*: once fewer tables remained than reader connections, pipelines went idle while one ground out the last big (often blob/wide-row) table. Tier (b) makes the stealable unit a **(table, PK-range) chunk** instead of a whole table. A large, chunk-eligible table — one with an orderable single or composite primary key, above the within-table row threshold — is split into PK-range chunks via the **same** boundary machinery the `migrate` path already pins (ADR-0019 integer MIN/MAX/divide, ADR-0096 keyset sampling), with the upper-bound clip pushed into SQL in the column's native collation (the Bug-74 contract), so the chunks tile the table's rows with no gap and no overlap. Idle readers then steal a *chunk* of the last big table, keeping the copy N-wide down to the tail and making the achieved width robust to byte/row-width skew across tables. Boundary sampling runs on the side meta pool (a HINT query, never a pinned snapshot connection), so it does not consume a snapshot reader slot. It is scoped to the native-MySQL consistent-N-snapshot cold-copy path — the only reader where every connection shares one FTWRL cut and can therefore read any PK range consistently. Pinned by a work-item construction matrix (integer/keyset/composite tiling; ineligible/small/no-PK/non-orderable tables fall back to a whole-table item; M-chunk clamp; opt-out; missing-table loud), per-work-item cursor-key disambiguation tests, an in-memory chunked end-to-end exactly-once test, an engine integration family matrix (int/varchar/decimal/datetime/composite) with mid-chunk drop-recovery, and pipeline integration tests doing native-MySQL→{MySQL,PG} over a skewed corpus with exactly-once checksums, a chunking-engaged observer, and the opt-out path — all under the CI `-race` integration gate.

**Exactly-once is preserved.** Each (table, chunk) carries its own resume cursor key (`table#chunkIndex`) so concurrent chunks of one table can never alias on the shared cursor map, and those keys compose with the ADR-0111 re-snapshot crash-recovery path unchanged; the CDC handoff anchor is untouched (ADR-0111 §3 intact). Whole-table callers thread `(nil, nil, table.Name)` through the same generalized resumable read path and remain byte-identical to v0.99.127.

## Fixed

No bug fixes in this release. It is a throughput feature plus dev-tooling and dependency housekeeping.

## Compatibility

No breaking changes; fully drop-in over v0.99.127. The new behavior is **on by default only for the native-MySQL concurrent cold-copy** — a self-managed (non-Vitess) MySQL source running with `--copy-table-parallelism > 1`. It is **inert (a no-op) on PostgreSQL and VStream (Vitess/PlanetScale) sources and on a serial cold-copy** (`--copy-table-parallelism=1`): those paths copy every table as a single whole-table work item exactly as before. The new opt-out flag `--no-intra-table-stealing` reverts the native-MySQL concurrent path to the prior tier-(a) whole-table-stealing behavior; it is correctly *opt-out*-named, so the Go zero value leaves stealing **on** and no non-CLI construction can silently invert it (the v0.99.51 zero-value trap). No default for any existing flag moves, no read/write/CDC value path changes, and chunk coverage is gap-free and overlap-free whether stealing is on or off. All engines and directions otherwise behave identically to v0.99.127. (Also in this release, non-runtime: a CDC memory-soak harness and findings note under `benchmarks/cdc/`, and routine dependency bumps — testcontainers postgres/mysql modules, google.golang.org/api, the aws-sdk-go-v2 S3 module, and actions/checkout — none of which affect the shipped binary's behavior.)

## Who needs this — action required

Nobody must act, and nothing needs re-verifying — this is a throughput optimization with no correctness change, so no prior release is implicated and no data path changed. One group benefits on upgrade: operators running a `sync start` cold-copy from a **self-managed MySQL source with `--copy-table-parallelism > 1`** whose corpus has a few very large (blob/wide-row) tables — those previously tapered to a single pipeline on the last big table and now keep the copy N-wide to the tail, shortening cold-copy wall-time with no configuration change. Anyone who prefers the prior tier-(a) whole-table-only stealing can set `--no-intra-table-stealing`. Operators on PostgreSQL or VStream (Vitess/PlanetScale) sources, or running a serial cold-copy, see no change at all. Upgrading is optional and safe.
