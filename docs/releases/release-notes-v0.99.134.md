# sluice v0.99.134

**New: budget-wide intra-table PK-range work-stealing for the `migrate` / PostgreSQL fast-cold-start parallel copy (ADR-0123, roadmap 21b for the migrate/PG path) — a single large table now expands to fill the WHOLE copy budget at the tail instead of staying pinned at `--bulk-parallelism` while peer-table connection slots idle. A scheduling-only throughput optimization, not a correctness change; the prior path was already correct and stays exactly-once. Fully drop-in over v0.99.133.**

## Features

**Budget-wide intra-table work-stealing on the `migrate` / PostgreSQL fast-cold-start parallel copy (ADR-0123, roadmap item 21 tier (b) for the migrate/PG path).** v0.99.128 brought intra-table work-stealing to the *native-MySQL* concurrent cold-copy; this brings the equivalent to the `migrate` orchestrator and the PostgreSQL fast cold-start. A skewed-corpus PG→PG `migrate` (one 10M-row table plus several small ones) run with `--table-parallelism 4 --bulk-parallelism 4` (a budget of 16 connections) was measured draining the small tables in ~1.5s and then copying the big table pinned at 4 readers (= `--bulk-parallelism`) for its whole ~40s while **12 of the 16 connection slots sat idle** — because the big table split into exactly `--bulk-parallelism` chunks behind a per-table fixed-width gate that could not expand into the budget freed when the peer tables finished.

The fix replaces the two independent caps — the ADR-0076 cross-table pool cap × the per-table within-table gate — with **one shared connection-budget gate**: a single counting semaphore sized to the same product ceiling (`table-parallelism × bulk-parallelism`) that every base table connection *and* every within-table PK-range chunk draws one token from. The budget is now redistributed at runtime: when a peer table finishes and releases its base token, an in-progress large table's surplus chunks — blocked on that same gate — steal it, keeping the copy budget-wide all the way down to the tail. Large tables split into a finer, deterministic chunk count (`clamp(ceil(estRows / within-threshold), bulk-parallelism, max(64, budget))`, mirroring the native-MySQL ADR-0119 clamp) — enough chunks to fill the whole budget — computed by the *same* `computeChunkBoundaries` / `computeKeysetChunkBoundaries` functions already pinned exactly-once across the PK-family matrix (the Bug-74 collation clip), so the finer split inherits their no-gap/no-overlap tiling. On the measured corpus the big table now expands from the pinned-at-4 width to the full budget (peak 16/16) at the tail.

**Exactly-once and resume are preserved — the change is scheduling-only.** Every per-chunk invariant is reused verbatim: the exactly-once half-open `(lower, upper]` tiling with the collation-correct SQL upper clip (ADR-0096), per-chunk resume (ADR-0019), the ADR-0043 fast-loader, the ADR-0078 raw-copy passthrough, the ADR-0109 source-read retry, the ADR-0110 grow-gate, the ADR-0076 cross-table state map, and the ADR-0077 index-build overlap. Chunk boundaries are persisted on the first attempt and reloaded unchanged on resume, so a resumed run re-derives the identical chunk set. The budget is redistributed, never inflated — total concurrently-open copy connections never exceed the product ceiling (the new single gate is a tighter, true single chokepoint than the prior product-of-two-caps; chunk 0 reuses the table's base connection and takes no extra token, so it can always make progress and the gate is deadlock-free). The snapshot→CDC seam is unaffected — the PostgreSQL cold-start's exported snapshot and the resume position are established once, independent of which worker reads which chunk.

## Fixed

No data-path bug fixes in this release. One CI-tooling fix rides along: `scripts/vet-tags.sh` (and its PowerShell mirror) now strips GOOS/GOARCH build constraints (`windows`, `!windows`, `linux`, `amd64`, …) from its `//go:build` discovery — those are toolchain-selected, not `-tags`, and `go vet ./...` already covers them per-platform. The v0.99.133 hot-reload introduced a platform-gated `//go:build !windows` file, a negation that tripped the tags-vet job's conjunction-only parser; the v0.99.133 shipped binaries were unaffected (the failure was a build-tag type-check the default build skips), and this restores the Lint gate to green. No shipped-binary change.

## Compatibility

No breaking changes; fully drop-in over v0.99.133. This is a scheduling optimization with **no correctness change** — the prior path was already correct and chunk coverage is gap-free and overlap-free either way. It applies to `migrate` in all four directions and to the PostgreSQL `sync` fast cold-start (which reuses the same orchestrator). The VStream (Vitess/PlanetScale) COPY-path tier (b) remains roadmap-open and is **blocked** by design: each stream is Match-scoped to its group at the source, so a single channel is not arbitrarily range-splittable — that needs new per-stream Match-scoping primitives, out of scope here. No default for any existing flag moves, and no read/write/CDC value path changes.

## Who needs this — action required

Nobody must act, and nothing needs re-verifying — this is a throughput optimization with no correctness change, so no prior release is implicated and no data path changed. One group benefits on upgrade: operators running a `migrate` or PostgreSQL `sync` cold-copy with `--table-parallelism > 1` over a corpus that has a few very large tables alongside many small ones — those previously tapered to `--bulk-parallelism` readers on the last big table and now keep the copy budget-wide to the tail, shortening cold-copy wall-time with no configuration change. Everyone else sees no change at all. Upgrading is optional and safe.
