# sluice v0.99.127

**CLI ergonomics release — flag-naming consistency, fully 100% backward-compatible (ADR-0118). Every existing command line and script keeps working byte-for-byte; the only new visible behavior is two advisory WARNs and two new opt-in `sync start` flags, with no change whatsoever to any data, copy, or CDC path. Pure drop-in over v0.99.126.**

## Changed

**Retry knobs are now mutually accepted across `backup stream run` and `sync start` (additive aliases, no deprecation).** A CLI audit found the same retriable-failure backoff concept spelled two ways: `sync start` exposes `--apply-retry-attempts` / `--apply-retry-backoff-base` / `--apply-retry-backoff-cap` while the backup-stream rollover path exposes `--retry-attempts` / `--retry-backoff-base` / `--retry-backoff-cap`. The two are conceptually the same loop with identical defaults (8 attempts / 100ms base / 30s cap), so this release cross-adds each spelling as a kong alias on the other command: `backup stream run` now also accepts the `--apply-retry-*` spellings and `sync start` now also accepts the `--retry-*` spellings. Each command's primary `--help` name is unchanged and no spelling is being retired, so there is no deprecation warning and no script edit — muscle memory from one command simply works on the other. Pinned by alias-binding unit tests asserting each alias sets the same underlying field as the canonical name.

**`cutover` gains the cleaner canonical `--sequence-margin`, with `--cutover-sequence-margin` kept working as a deprecated alias (one-time WARN on the old spelling).** `--cutover-sequence-margin` was the only flag in the whole CLI that redundantly prefixed its own command name; under the `cutover` subcommand the `cutover-` prefix is pure noise. The flag is now named `--sequence-margin`, with `cutover-sequence-margin` retained as a kong alias so every existing command line and script keeps working unchanged. Passing the old spelling now emits a one-time deprecation WARN ("`--cutover-sequence-margin` is deprecated; use `--sequence-margin`. The old name still works and will be removed in a future release."), mirroring the established ADR-0091 `--forward-schema-add-column` posture exactly. The WARN fires only when the operator actually passed the deprecated spelling — detection reads kong's per-flag set-tracking of which name was used, never the resolved value, so a default-equal explicit value is still detected and an unset flag never warns.

**Two VStream / native-MySQL cold-copy READ-axis knobs are promoted from DSN-only to first-class `sync start` flags.** The write axis of cold-copy concurrency was already a CLI flag (`--copy-fanout-degree`, ADR-0097), but the matching read axes were buried in the source DSN query-string and undiscoverable via `--help`: `vstream_copy_table_parallelism` (Vitess / PlanetScale source) and `copy_table_parallelism` (self-managed MySQL source). This release adds `--vstream-copy-table-parallelism` and `--copy-table-parallelism` as first-class `sync start` flags sitting beside their write-axis sibling. The existing DSN params keep working verbatim — precedence is explicit CLI flag > DSN param > engine default (1 = serial) — and the new flags default to `0` (unset → fall back to the DSN param), so an unset flag never silently overrides a DSN value; only an explicitly-set value wins. Pinned by a resolver precedence test (CLI > DSN > default, including the CLI-absent/DSN-present fallback and the both-absent serial default).

**The FAST-cold-start parallelism flags now lead their `--help` with applicability, and a previously-silent no-op is now a loud WARN.** `--bulk-parallelism` / `--table-parallelism` / `--bulk-parallel-min-rows` are spelled identically across `migrate`, `sync start`, `backup full`, and `restore` but carry related-but-distinct per-command meanings; their `--help` now front-loads the per-command / per-source applicability so the first thing a reader sees is where the flag actually applies. More importantly, on a `sync start` against a MySQL or Vitess/PlanetScale source these flags are inert (the cold-start is serial there; the ADR-0079 FAST cold-start they govern is PG-source-only), and previously they were accepted, parsed, and silently did nothing. `sync start` now emits a one-time WARN when one of them is explicitly set against a MySQL/VStream source, pointing the operator at `--copy-fanout-degree` plus the new read-axis flags above — turning a silent no-op into a loud one, the loud-failure tenet applied to a UX hazard. As with the deprecation WARN, "explicitly set" is read from kong's set-tracking of the literal command-line token, not the resolved value, so the WARN never fires for an unset flag holding its default.

## Compatibility

No breaking changes; fully drop-in and 100% backward-compatible. Every existing command line and script keeps working byte-for-byte: the retry-flag aliases and the new read-axis flags only *add* accepted spellings or *add* a flag that defers to the existing DSN param, and `--cutover-sequence-margin` keeps working as an alias of the new canonical name. No flag is renamed in a way that breaks an existing invocation, no default moves, and no resolved config value changes — a golden no-behavior-change pin asserts a representative command line for each touched command resolves identically before and after. There is zero change to any read, write, copy, or CDC path, and all engines and directions are unaffected. The only new visible behavior is the two advisory WARNs (the deprecated `--cutover-sequence-margin` spelling; an inert parallelism flag explicitly set on a MySQL/VStream `sync start`) and the two new opt-in `sync start` flags. The DSN forms `vstream_copy_table_parallelism` / `copy_table_parallelism` continue to work verbatim.

## Who needs this — action required

Nobody must act. This is a CLI ergonomics release with no behavior change to any data path, so there is nothing to re-verify, re-run, or reconfigure, and no prior release is implicated — nothing was broken and nothing was silently lost. Two groups will simply find the tool a little easier to use after upgrading: operators tuning VStream (Vitess/PlanetScale) or native-MySQL cold-copy who previously had to set the read-axis concurrency via DSN query params can now use the discoverable `--vstream-copy-table-parallelism` / `--copy-table-parallelism` flags on `sync start` (the DSN form still works, so there is no migration to do); and anyone who had passed `--bulk-parallelism` / `--table-parallelism` / `--bulk-parallel-min-rows` to a MySQL/VStream `sync start` expecting it to speed up the cold-copy will now get a loud WARN explaining it is inert there and pointing at `--copy-fanout-degree` and the new read-axis flags — surfacing a mis-tuning that was previously silent. Anyone scripting `cutover --cutover-sequence-margin` will see a one-time deprecation WARN and can switch to `--sequence-margin` at their leisure; the old spelling keeps working. Upgrading is optional and safe.
