# sluice v0.99.206

**Honest disclosure of a newly ground-truthed silent-precision finding on the PlanetScale/Vitess VStream COPY path, plus one loud-over-silent CLI exit-code change and a test-gate + flake-hardening internal sweep. This release ships NO code fix for the FLOAT finding — it discloses it (see Known issues) so PlanetScale/Vitess operators who cold-copy single-precision FLOAT columns can re-verify precision-sensitive data until the fix lands in a following release. The one behavior change: a declined `schema add-table` confirmation now exits 1 instead of printing "aborted" and exiting 0, unifying it with the `--reset-target-data` declined contract. Otherwise drop-in for existing invocations; the single deliberate exit-code change is listed under Compatibility.**

## Fixed

- **A declined `schema add-table` confirmation now exits 1 via the wave-4b declined sentinel instead of printing "aborted" and exiting 0 (the silent-success-on-decline shape since `schema add-table`'s confirm gate landed; through v0.99.205).** This was the one destructive-confirm decline still outside the wave-4b abort contract the `--reset-target-data` sites got in v0.99.205: typing anything but the table name printed a bare "aborted" line and returned nil, so a wrapping script that keyed on exit 0 read a declined, no-op run as success. The decline now returns `errConfirmDeclined` — a non-zero exit carrying the sentinel's own operator-facing text — matching the other destructive-confirm sites. `schema add-table` has no `--format` flag, so text is the whole surface (no JSON-envelope work needed). Pinned by `TestSchemaAddTableConfirm_CommandPath` through the real `Run()`: a decline yields the sentinel, and typing the table name passes the gate and proceeds to the downstream target-applier open (proving the answer was consumed), extending the wave-4b command-path matrix.

## Known issues

- **NEW — VStream COPY silently display-rounds single-precision FLOAT columns to 6 significant digits on PlanetScale/Vitess sources (MED-HIGH, silent-precision class; found this release ground-truthing the ADR-0153 G4 residual on real vttestserver; NO fix in this release — a fix is in progress for a following release).** A VStream cold-start COPY — the `sync` cold-start snapshot and `backup full` on the VStream path (every consumer of `openVStreamSnapshotStream`) — lands `FLOAT`/`REAL` (single-precision) values display-rounded to 6 significant digits, exit 0. The mechanism: vttablet's rowstreamer issues a bare-column `SELECT` over MySQL's text protocol, so the wire bytes are mysqld's own 6-sig-digit FLOAT rendering — the exact bug class ADR-0153's `(col * 1E0)` DOUBLE-promotion projection fixed on the SQL reader, but this SELECT is built inside vttablet and is unreachable from the client, so sluice cannot inject the projection. Ground truth on vttestserver mysql80 (2026-07-09): 7 of 11 torture values land as a DIFFERENT float32 — `8388608` → `8388610`, `-123456.789` → `-123457`, float32-max → a different float32. **Not affected:** `DOUBLE`/double-precision columns (exact on both legs), the CDC/binlog leg (vttablet re-encodes the binlog float32 bits shortest-round-trip — float32-exact, pinned), float `-0.0` sign (exact on both legs, pinned), and every non-PlanetScale/Vitess source (vanilla MySQL, Postgres, SQLite/D1 sources do not use the VStream COPY path). **Affected releases:** the VStream cold-start COPY has carried this since the VStream snapshot path landed in **v0.1.0** (`sync` cold start), and the `backup full`-on-VStream leg since **v0.44.0** — through **v0.99.205**. It is now pinned in `TestVStream_FloatCarrierParity` (`cdc_vstream_float_carrier_integration_test.go`): the CDC float32-exactness and DOUBLE/`-0.0` assertions run green today, and the COPY float32-exactness assertions sit behind a documented `t.Skipf` naming the roadmap open-bugs entry — that skip is the fix's ready failing pin, un-skipped when the fix lands. Fix-shape candidates under evaluation (the projection cannot be injected client-side): a loud per-FLOAT-column WARN/refuse at VStream COPY start steering to a binlog-first cold start or `--type-override`, a post-COPY SQL re-read of FLOAT columns through the fixed reader projection, or an upstream Vitess rowstreamer fix. Until the fix ships, treat every PlanetScale/Vitess-source cold copy of a `FLOAT` column as lossy past 6 significant digits.

## Compatibility

**No CLI-flag changes; drop-in for existing invocations, with one deliberate loud-over-silent behavior change:** a declined `schema add-table` confirmation now exits 1 (was 0). A script that treated exit 0 as "table added" on a declined `schema add-table` must update — the change is in the loud direction (nothing that previously succeeded now fails; a declined run was always a no-op, it just lied about its exit code). No engines, directions, or data paths change behavior. The VStream FLOAT finding is a disclosure, not a behavior change — the path behaves exactly as it did in prior releases; this release only documents it.

## Who needs this — action required

- **PlanetScale/Vitess-source operators who have run a `sync` cold-start or a `backup full` that copied `FLOAT`/`REAL` (single-precision) columns on any release from v0.1.0 (cold-start) / v0.44.0 (backup) through v0.99.205: re-verify precision-sensitive FLOAT data on the target.** The copied values are display-rounded to 6 significant digits and may differ from the source past that precision (exit was 0, so nothing flagged it). `DOUBLE` columns, the CDC/binlog steady-state path, and all non-PlanetScale/Vitess sources are unaffected — you only need to check single-precision FLOAT columns that went through a VStream cold-start COPY. There is no fix to upgrade to yet; upgrading to v0.99.206 does not change the path. If precision matters and you cannot wait for the fix, a binlog-first cold start or copying the affected columns as `DOUBLE`/`--type-override` avoids the vttablet text-protocol rounding.
- **Automation authors around `schema add-table`: a declined confirmation now exits 1 — update any wrapper that checked for exit 0 on a decline.**
- **Everyone else: drop-in upgrade, no action.**

## Internal

- **The weekly Vitess-version-matrix cluster suite now runs under `-race`, closing the last real-cluster leg outside the race gate.** The `vitesscluster` version-matrix legs — the concurrency-heaviest real-cluster surface (OnlineDDL cutover, PrimaryOnly) — were the only real-cluster suite still outside `-race`; the chaos/reshard legs in `extended-suites.yml` have carried it since the 2026-07-03 repo audit. The `go test` line gains `-race` and the timeouts move ~1.5× (inner 40m → 60m, job 45 → 70) to absorb the detector overhead, mirroring the chaos-leg precedent, with the inner timeout kept below the job timeout so Go's diagnostic timeout-panic fires before the hard job kill.
- **`TestStreamer_PostgresToPostgres_BatchedApply` now pins the coalescing mechanism directly instead of a timing-sensitive commit-delta constant (no product change).** The test asserted a dest-side `pg_stat_database` commit delta at or under a constant (52) — a timing artifact that flaked once on a loaded `-race` runner (200 rows landed in 55 commits). It now installs a pipeline test seam (`batchApplyObserverForTest`, an `ir.BatchObserver` on the serial ADR-0092 batch loop when the AIMD controller doesn't take the observer seat) and asserts the mechanism: every source row flowed through an observed coalesced flush, and mean rows/flush is at least 10 (per-change apply yields exactly 1 — an order of magnitude under the floor; a healthy run is 2-3 flushes of ~100). No timing-dependent constant remains; verified against real Postgres at 201 rows in 3 flushes.
- **Coming — ADR-0154 (signed backup manifests) was Accepted this release as a design/docs item; Phase 1 implementation is in progress and unreleased.** It closes the ADR-0152 honest residual (whole-manifest rollback and change-list tail truncation by a store-write adversary) via optional detached manifest+lineage signing under a new FormatVersion 6, framed as the second-order defense behind the first-order manifest-is-data-never-instructions and path-sanitisation invariants. Nothing about signing ships in this release.
