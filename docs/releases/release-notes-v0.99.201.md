# sluice v0.99.201

**HIGH fixes — two silent-failure classes closed, plus six more fixes from the 2026-07-08 audit cross-check. (1) The `postgres-trigger` CDC engine's safety-lag hold-back and snapshot→CDC anchor compared a 32-bit epoch-less `xid` against a 64-bit `xid8`, so once a Postgres cluster's lifetime transaction count crosses 2^32 (epoch ≥ 1 — routine on long-lived busy clusters, the managed-Postgres audience this engine targets) both predicates silently stopped working and change events from overlapping transactions could be silently skipped (exit 0, no warning; latent on v0.85.0 through v0.99.200, fires only on epoch-bumped clusters). Fixed by comparing the change-log's existing 64-bit `txid` column on both sides — no schema change, existing installs upgrade in place — and live-validated on an epoch-bumped PG 16. (2) The native MySQL binlog CDC replication stream ALWAYS rode plaintext — a `tls=true` DSN encrypted every query connection while every replicated row went cleartext with no warning (all releases; VStream/PlanetScale flavors unaffected); the stream now inherits the DSN's TLS posture with loud transport warnings. Also: a fleet `schema-changes` typo no longer silently enables DDL forwarding, an encrypted backup chain can no longer be silently extended with plaintext when the root manifest is unreadable, Argon2id KDF params from untrusted manifests are bounds-checked, webhook/Slack/telemetry secret URLs no longer leak into WARN logs, `--include-table`/`--exclude-table` no longer fails on an excluded owner's standalone sequence, and underscore-keyed config fields are now reachable from `SLUICE_*` env vars. Drop-in for virtually all setups — no flag or data-path changes; see Compatibility for five loud-but-deliberate behavior changes.**

## Fixed

- **The `postgres-trigger` engine's safety-lag hold-back and snapshot→CDC handoff anchor now compare in the 64-bit xid8 domain, closing a latent silent-data-gap on Postgres clusters whose XID epoch is ≥ 1 (HIGH; silent loss class — exit 0, zero WARN; hold-back affected v0.85.0 through v0.99.200, the anchor v0.86.1 through v0.99.200; fires only once a cluster's lifetime txid count crosses 2^32).** Both queries compared the change-log row's system `xmin` — a 32-bit, epoch-less `xid` — against `pg_snapshot_xmin(pg_current_snapshot())` — a 64-bit, epoch-carrying `xid8` — and the in-code comment treated the cast as a JSON-precision detail, not a cross-domain comparison. At epoch 0 the two domains coincide and everything worked (which is why every test and every young cluster was fine), but once lifetime txids exceed 2^32 the snapshot side is > 2^32 while row `xmin` stays below it: the poll's hold-back predicate becomes ALWAYS true, so the watermark advances past an in-flight transaction's already-allocated change-log ids and that transaction's changes are silently skipped when it commits (the exact overlap-commit gap the predicate exists to prevent); and the anchor's `>=` arm NEVER matches, so `COALESCE` degenerates to `MAX(id)` — an anchor too high — resurfacing the Bug-94 cold-start gap. Live-confirmed on a `pg_resetwal -e 5` epoch-bumped PG 16: the old predicate emitted a committed row while an older transaction with a lower change-log id was still open, and the old anchor returned `MAX(id)`. The fix compares the change-log's existing `txid` column — `pg_current_xact_id()::text::bigint`, recorded by the capture trigger at insert time — on the left of both queries, putting xid8 on BOTH sides, which is what ADR-0066 §2's txid rationale intended all along. `txid` has been NOT NULL and correctly populated since the engine's first release, so existing installs need no `ALTER`, there are no legacy rows to special-case, and behavior at epoch 0 is unchanged (verified live). Also in this commit: setup's version preflight now refuses PG < 13 instead of PG < 9.4 — `pg_current_xact_id()` and `pg_current_snapshot()` are PG 13+ and have been required since the engine's first release, so a PG 12 source previously passed setup and then failed EVERY write to every triggered table at first DML; the refusal is now loud and early. One honestly-documented residual (pre-existing, epoch-INDEPENDENT, deliberately not bundled here): an in-flight transaction's already-allocated change-log row is invisible to the anchor query under MVCC, so the `MIN(id)-1` arm cannot anchor below it when its id is lower than every visible not-yet-settled row; closing it needs a `pg_snapshot_xip` wait-for-settle at handoff and is filed in ADR-0066's implementation notes. Pinned by SQL-shape unit tests on both queries plus a new epoch-bump integration test (boots PG 16, stops it cleanly, `pg_resetwal -e 5` on the data volume, restarts, gates on snapshot xmin > 2^32, then pins hold-back-under-overlap and the anchor landing below the unsettled row) — mutation-verified to fail against the reverted `xmin`-based queries in the exact predicted silent-loss shapes.
- **A typo'd `schema-changes:` value in the fleet `syncs.yaml` now refuses loudly at config load instead of silently enabling source-DDL forwarding (silent config inversion; present since the fleet sync command center landed in v0.99.132, through v0.99.200).** The sole consumer treats every non-`refuse` string as "forward" (case-insensitive compare), and the fleet path plumbed the YAML value with only an empty-default and no enum check — kong's enum protected `sync start --schema-changes` only — so `schema-changes: refused`, `off`, or `no` passed the v0.99.194 unknown-key guard and the sync forwarded source DDL against explicit operator intent, silently. A new `validateSchemaChanges` refuses anything outside {`forward`, `refuse`} (case-insensitive) at spec load, naming the sync, the offending value, and the allowed set. A same-class sweep of the fleet spec's string fields confirmed `schema-changes` was the only unvalidated one (zero-date, notify-smtp, drivers, and apply-batch-size all had load- or build-time validation). Pinned through the REAL fleet-config load path on a YAML fixture — the Bug-180 "pin through the CLI layer" lesson — including the omitted-key default.
- **`--include-table` / `--exclude-table` no longer deterministically fails with 42P01 when a standalone sequence's owner table is excluded from the subset (loud failure, no data risk; present since standalone sequences became first-class schema objects in v0.99.175, through v0.99.200).** `ApplyTableFilter` pruned `schema.Tables` only; `schema.Sequences` passed through with `OwnedByTable` intact, so the Postgres writer's `bindSequenceOwners` emitted `ALTER SEQUENCE … OWNED BY <excluded_table>.<col>` against a table the filter left uncreated — a deterministic 42P01 after the subset's tables had already been created, blocking the shipped copy-table-subset guide's path for a common source shape (any re-optioned serial whose owner falls outside the subset). The fix lives in the engine-neutral filter, so every caller — migrate, sync cold-start, backup, restore, diff, preview, verify, multidb — inherits it: a sequence whose owner fails the filter is dropped from the schema with a WARN naming the sequence and its excluded owner. Dropping the WHOLE sequence rather than just the ownership is deliberate — `OWNED BY` ties the sequence's lifecycle to the excluded column, so carrying it unowned would silently change its semantics; unowned standalone sequences always pass through. Pinned by unit tests over both filter directions plus a PG→PG integration test that reproduced the pre-fix 42P01 with the fix reverted and now migrates the subset, drops the owned sequence, and still carries the unowned one.
- **`SLUICE_*` environment variables can now set underscore-keyed config fields, which were previously SILENTLY ignored (silent config drop; the overlay bug is present since the config layer landed in v0.1.0, each affected field from the release that added it, through v0.99.200).** The env overlay replaced EVERY underscore with a dot, so `SLUICE_KEYSET_SOURCE` resolved to `keyset.source` — which matches no `koanf:"keyset_source"` tag — and was silently dropped; likewise `include_tables`, `exclude_tables`, `namespace_map`, and `expression_mappings`, and `keyset_source` (a secrets pointer) is exactly the kind of value operators put in env. The docs' one worked example (`SLUICE_EXTENSIONS_ALLOW`) was the only shape that worked. The mapping is now built by reflection over the Config struct's koanf tags (new fields are automatically addressable): flat keys keep their underscores, nested keys map to dotted paths, and map-typed keys take the variable's tail as the map key (`SLUICE_NAMESPACE_MAP_APP` → `namespace_map.app`); block-structured keys (mappings/redactions/dictionaries) stay YAML-only. A `SLUICE_*` variable resolving to no known key is now skipped with a loud WARN naming it and the valid keys — WARN-and-skip rather than the hard error the audit suggested, deliberately, because documented legitimate `SLUICE_`-prefixed names exist outside the config file (kong bindings like `SLUICE_SOURCE`/`SLUICE_TARGET`/`SLUICE_NOTIFY_*`, exempted from the WARN, and operator-chosen secret holders like `env:SLUICE_BACKUP_PASS`) whose names cannot be enumerated. Also fixed in the same pass: comma-separated env values now actually split into slice fields (explicit string-to-slice hook, mirroring the fleet loader — koanf's default left `"a,b"` as a one-element slice). Pinned by config unit tests over the reflection mapping, the WARN-and-skip path, the kong exemptions, and the comma split.

## Security

- **The native MySQL binlog CDC replication stream now inherits the source DSN's TLS posture instead of ALWAYS riding plaintext (silent TLS downgrade; present on the native `mysql` flavor's binlog CDC path in every release through v0.99.200; the VStream/PlanetScale/Vitess flavors were always TLS-by-default and are untouched).** The `BinlogSyncerConfig` was built with no `TLSConfig` at all, so a `tls=true` DSN encrypted every `database/sql` query connection while the replication stream — carrying every replicated row — went cleartext with no warning (a loud failure occurred only if the server enforced `require_secure_transport`). The fix clones go-sql-driver's own DSN-resolved TLS config onto the binlog syncer, so the stream gets exactly the transport the query connections negotiate — including a REGISTERED custom `tls=` config's CA pool and client certs. The mapping: absent/`tls=false` → plaintext as before, plus a once-per-stream-open WARN that replicated rows ride unencrypted; `tls=true` → verifying TLS (ServerName from the DSN host, MinVersion floor TLS 1.2, matching the VStream path); `tls=skip-verify` → encrypted but unverified, WARN on every stream open (the `vstream_insecure_tls` precedent); `tls=preferred` → go-mysql has no try-TLS-then-plaintext mode, so the binlog stream keeps the skip-verify TLS half and REFUSES the plaintext-fallback half (a loud open failure against a no-TLS server, with the WARN saying so — silent plaintext is exactly the class this closes; set `tls=false` explicitly if plaintext is intended); an unregistered custom config name was already refused loudly by `ParseDSN` and is pinned so a driver change can't reopen the hole. Readers built without a DSN (the zero value) stay byte-identical to pre-fix behavior. Pinned by a unit mapping table through the real DSN parser plus the transport-WARN matrix, and ground-truthed by integration tests on real MySQL 8 with self-signed certs (skip-verify streams rows over TLS and warns; `tls=true` is refused on BOTH the query and the binlog connection itself, proving the syncer enforces the config; default plaintext still streams and warns).
- **An encrypted backup chain can no longer be silently extended with plaintext chunks when the chain-root manifest is transiently unreadable (silent-plaintext-extension on an error path; present since chain-extension encryption alignment landed in v0.22.1, through v0.99.200).** `ChainRootEncryption` returned nil on ANY store read error, and both consumers (`backup incremental` and `backup stream`'s encryption alignment) branch nil = "parent chain is plaintext". A transiently unreadable or corrupt root manifest therefore either (a) with no `--encrypt`: skipped the encrypted-chain guard and silently extended an encrypted chain with plaintext chunks at exit 0, or (b) with `--encrypt`: emitted the wrong "parent chain is plaintext" refusal, actively steering the operator to drop `--encrypt`. `ChainRootEncryption` now returns an error: a store read/parse failure propagates and both consumers abort with "cannot determine parent chain encryption state (refusing to assume plaintext)", while a genuinely ABSENT root manifest keeps the plaintext-chain semantics unchanged. Pinned at both levels — lineage unit tests (failing Exists, failing Get, parent-carried fast path, absent root, plaintext root) and consumer tests across {incremental, stream} × {`--encrypt`, plaintext} — with the encrypted-chain integration suite re-run green.
- **Argon2id KDF parameters read from a backup manifest are now bounds-checked before key derivation, closing a pre-auth memory/CPU bomb on restore (untrusted-manifest hardening; present since envelope encryption landed in v0.22.0, through v0.99.200).** The params were copied verbatim from `manifest.json` and fed to `argon2.IDKey` BEFORE anything about the manifest is authenticated (the KEK the auth tags depend on is itself derived from them), so a tampered manifest could ask for ~4 TiB of memory (`memory_kib: 4294967295`) or spin for hours on an unbounded iteration count — the classic unauthenticated-KDF-param bomb. The single chokepoint constructor now refuses params above named bounds — Memory ≤ 2 GiB, Iterations ≤ 64, Parallelism ≤ 32, Salt ≤ 1 KiB — naming the parameter, value, and bound. Legitimate sluice-written chains can never trip the bounds: the only construction site is the fixed default (64 MiB / 3 / 4 / 16-byte salt) and there is no flag to raise it. Pinned by at-bound and over-bound unit matrices proving the refusal fires before `argon2.IDKey` runs.
- **Webhook, Slack, and PlanetScale-telemetry URLs — which carry the credential in the URL itself — no longer leak into WARN logs through transport errors (credential-in-logs exposure; webhook/Slack since the metrics alerter landed in v0.99.108, the telemetry signed-URL legs since v0.99.95, through v0.99.200).** A webhook/Slack URL's path IS the credential (sluice's own redaction code documents exactly that), and both `url.Parse` (inside request building) and `client.Do` wrap failures in `*url.Error`, whose `Error()` embeds the full URL; the alerter WARN-logs the error verbatim, so any transport failure printed the secret. Fixed at the source: notify's shared `postJSON` now passes every request-build and transport error through `diagnose.SafeParseError` (the v0.99.197 helper — strips the `*url.Error` wrapper, keeps the underlying cause) and names the sink; the same-class sweep also covered the PlanetScale telemetry client, whose scrape leg carries a SIGNED URL (`sig` + `exp` query params — a bearer credential until expiry) and logged the same shape on both its legs. Pinned by tests that force a real transport error against URLs carrying secret bytes and assert the secret never appears in the returned error string, for webhook, Slack, the unparseable-URL build leg, and both telemetry legs.
- **Go toolchain bumped to 1.26.5 for GO-2026-5856 (`crypto/tls` standard-library fix).** govulncheck flagged the vulnerability as reachable from every TLS-using path in the shipped binary — the SMTP notifier, blob-store chunk IO, the metrics HTTP server, and (as of this release's binlog fix) the binlog CDC syncer. The release binaries are built on 1.26.5.

## Compatibility

**No breaking changes for correctly-configured setups; no CLI-flag, format, or data-path changes.** Five deliberate loud-where-it-was-silent behavior changes to check: (1) **`postgres-trigger` setup now refuses PostgreSQL < 13 at preflight** — previously a PG 12 source passed setup and then failed every write to every triggered table at first DML, so this makes an existing hard requirement loud and early, it does not drop anything that worked. (2) **New WARN messages on the native MySQL binlog CDC transport** — a plaintext stream (no `tls=`, or `tls=false`) and an unverified one (`tls=skip-verify`) now warn at stream open; informational, set a verifying `tls=` on the source DSN to silence. (3) **`tls=preferred` on the MySQL source DSN now refuses plaintext fallback on the binlog stream** — go-mysql has no try-TLS-then-plaintext mode, so against a server with TLS disabled the stream open now fails loudly instead of silently riding plaintext; set `tls=false` explicitly if plaintext is intended. (4) **Unknown `SLUICE_*` env vars now WARN loudly** (previously silently ignored — check automation for typos; kong bindings like `SLUICE_SOURCE`/`SLUICE_TARGET`/`SLUICE_NOTIFY_*` and operator-chosen `env:` secret holders are exempt), and previously-ignored underscore-keyed `SLUICE_*` vars now TAKE EFFECT — audit any that were set-but-dead, and note comma-separated values now split into slice fields. (5) **A `syncs.yaml` `schema-changes:` value outside {`forward`, `refuse`} now refuses at load** (previously silently forwarded DDL) — fix the value to the intended spelling. Additionally, a filtered `--include-table`/`--exclude-table` run whose subset excludes a standalone sequence's owner table now succeeds and drops that sequence with a WARN instead of failing with 42P01. Everything else — migrate/sync data paths, backup formats, VStream TLS, all engines — is byte-identical.

## Who needs this — action required

- **Anyone running `postgres-trigger` sync against a long-lived Postgres cluster: check the cluster's XID epoch, and if it has crossed 2^32, upgrade and re-verify.** Run `SELECT pg_current_xact_id()::text::bigint >= 4294967296 AS epoch_bumped;` on the source. If TRUE and you ran `postgres-trigger` sync on v0.85.0 through v0.99.200, changes from transactions that overlapped a poll or a cold-start handoff may have been silently skipped: upgrade, then re-verify the target against the source (`sluice verify`, row counts, or checksums) and re-run a cold start if discrepancies show. If FALSE — the overwhelmingly common case, including every cluster younger than ~4 billion lifetime transactions — the predicates behaved correctly and no data is at risk; upgrade for the future-proofing (epoch bumps arrive with cluster age, not with a warning).
- **Anyone running native `mysql`-flavor binlog CDC (`sync`) across an untrusted network: upgrade — until now the replication stream was plaintext even with `tls=true` on the DSN.** No data was lost or corrupted (this is transport confidentiality, not integrity), but every replicated row on prior releases rode cleartext between source and sluice: assess per your exposure policy (network path, data sensitivity, credential-rotation rules). After upgrading, `tls=true` covers the stream, and the new WARNs tell you exactly what the transport is. VStream/PlanetScale sources were always TLS and need nothing.
- **Fleet (`syncs.yaml`) operators using `schema-changes:`: grep your configs before upgrading.** Any value other than `forward`/`refuse` was silently forwarding source DDL; on v0.99.201 it refuses at load with the allowed set named. Fix the spelling to what you actually intended.
- **Operators extending encrypted backup chains: nothing to do in the common case.** The silent-plaintext-extension required a store read error at exactly the alignment moment — rare. If you ever saw an unexplained "parent chain is plaintext" refusal on an encrypted chain, or extended one during storage flakiness, run `backup verify` on the chain and inspect the incrementals' manifests for unencrypted chunks.
- **Anyone setting `SLUICE_*` env vars for underscore-keyed config (e.g. `SLUICE_KEYSET_SOURCE`, `SLUICE_INCLUDE_TABLES`): these now work.** They were silently dead before — meaning your effective config may CHANGE on upgrade to what you originally intended. Review any such vars in automation before upgrading.
- **Everyone else: drop-in upgrade, no action.** The remaining fixes (URL-leak redaction, Argon2id clamp, sequence-filter 42P01, PG < 13 preflight) either harden error paths or turn an existing failure loud-and-early.
