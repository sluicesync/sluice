# sluice v0.99.125

**HIGH fixes — restoring a backup chain with a large incremental no longer stalls (chain-restore now fans the incremental across the concurrent key-hash apply lanes, `restore --apply-concurrency W`), and a PostgreSQL index name that exceeds Postgres's 63-byte identifier limit is now refused loudly instead of being silently truncated (which could silently drop a distinct index). Drop-in: chain-restore is fast by default (auto:4, `--apply-concurrency 1` is the serial opt-out, exactly-once preserved); the only behavior change is the new loud refusal of a >63-byte PG index name that previously truncated silently.**

## Features

**Chain-restore fans an incremental's changes across W in-order PK-hash lanes (ADR-0105 Addendum 2).** `ChainRestore` — used by standalone `sluice restore --from <chain>` of a multi-incremental chain, and by the broker's `sync from-backup --reset-target-data` cold-start (which absorbs the chain before transitioning to polling) — replays each incremental's change chunks through its OWN applier, a path distinct from both the live streamer and the broker's polling applier. It had the identical gap the broker-polling path had before v0.99.123: it wired `applyMaxBufferBytes` / `applyTargetSchema` but never the apply-concurrency lane count, so chain-incremental replay ran single-stream serial through the ADR-0092 pipelined applier, RTT-bound on a cross-region link. The Track-C large-scale program caught it live: a cross-region MySQL→PlanetScale-Postgres cold-start restored the ~43 GB full backup fine (parallel COPY, ADR-0112) then wedged at ~1 change/s applying an 8 M-change incremental — the exact pre-fix broker-polling stall, but on the chain-restore leg. `ChainRestore` now resolves an `ApplyConcurrency` lane count through the same shared `resolveReplayApplyConcurrency` + `applyApplyConcurrency` helpers the broker and streamer use, so each incremental's changes fan across W concurrent in-order PK-hash lanes. The lane count is threaded from the broker cold-start (inheriting the broker's `--apply-concurrency`) and from a new `restore --apply-concurrency W` flag on the standalone path, following the ADR-0106 fast-by-default contract: `0`/unset → an adaptive default of 4, `1` → the explicit serial opt-out, `W>1` → honored verbatim. The full-restore COPY axis (`--table-parallelism` × `--bulk-parallelism`, ADR-0112) is unchanged and orthogonal — this closes only the incremental-apply leg. Exactly-once is preserved verbatim: every change carries the segment's chain-position token, so the lanes persist the identical resume position the serial path did, and the broker's idempotent re-replay-from-parent crash recovery is unchanged. The chain-restore + broker integration suites exercise the now-default concurrent apply under `-race` and assert convergence.

## Fixed

**HIGH (latent silent-loss) — a PostgreSQL-target index name exceeding Postgres's 63-byte identifier limit was emitted verbatim and silently truncated by the server, which could silently DROP a distinct index; it is now refused loudly (roadmap item 43).** PostgreSQL truncates any identifier longer than 63 bytes (NAMEDATALEN-1) at CREATE time, so two distinct source indexes whose names agree on their first 63 bytes both resolve to the same catalog identifier — and because sluice emits `CREATE INDEX IF NOT EXISTS`, the second create silently no-ops, leaving that index missing on the target. No error, no row-data loss, but a missing index is a silent schema-fidelity loss — exactly the silent-transform class the loud-failure tenet exists to prevent. The v0.49.0 fix (GitHub #26) already guarded the *generative* case where prepending `<table>_` would overflow (it falls back to emitting the source name verbatim), but a source name that itself exceeds 63 bytes — a MySQL source allows 64-byte identifiers, and a multibyte UTF-8 name can exceed 63 bytes while staying under 63 runes — still flowed through verbatim into the truncation. Flagged during the Bug #114 RCA as the `pgIndexName` >63-byte latent silent-collapse class. Per the "contain Postgres complexity, surface explicitly" tenet the fix is a loud refusal, not an auto-truncate or auto-rename (either would silently transform the operator's schema): a new `validatePGIndexName` byte-length preflight at the index-emit chokepoint (`emitCreateIndex` — through which the live serial, concurrent, and overlap index-build paths and the dry-run `schema preview` all route — plus the inline unique-key constraint promoted at CREATE TABLE time in `emitTableDef`) returns a clear, actionable error naming the index, its byte length, the offending table, and the source name, instructing the operator to shorten or alias it. The check is byte-based, not rune-based, matching how PostgreSQL counts NAMEDATALEN; names of 63 bytes or fewer are unaffected and emit byte-identically. The >63-byte refusal alone closes the whole collision class: two distinct ≤63-byte names cannot share a 63-byte prefix unless they are byte-identical (in which case the IR already carries a single index). Pinned by a value-fidelity matrix: exactly 63 bytes accepted, 64 bytes refused with the index name in the message, a multibyte name whose rune count is ≤63 but byte count >63 refused (proving the byte basis), and two names differing only after byte 63 both refused.

## Compatibility

No breaking changes for any healthy schema or any in-budget invocation; drop-in upgrade for the common case. Two changes touch behavior:

The chain-restore incremental-apply default flips from single-serial-stream to fast-by-default concurrent (auto:4). The new `--apply-concurrency W` flag on `sluice restore` is opt-in-fast-by-default: with no flag, chain restores (standalone CLI, broker `--reset-target-data` cold-start, and tests) resolve to an adaptive default of 4 lanes, `--apply-concurrency 1` restores the prior single-serial-stream behaviour exactly, and `W>1` is honored verbatim. Exactly-once is preserved exactly as before — every change carries the same chain-position token, the lanes persist the identical resume position the serial path did, and idempotent crash recovery is unchanged — so the concurrent path produces byte-identical results to the serial one. The full-restore COPY axis (`--table-parallelism` × `--bulk-parallelism`) is unchanged and orthogonal; only the incremental-apply leg is affected. Applies to both engine targets (MySQL via ADR-0104, Postgres via ADR-0105). The live `sync start` path and the `sync from-backup` broker-polling path already had concurrent apply (v0.99.123) and are unaffected.

The PG >63-byte index-name refusal is a behavior change for that one edge case: a PostgreSQL-target index whose effective name exceeds 63 bytes — which previously flowed through and was silently truncated by the server — is now rejected loudly at emit time with an actionable error, so a `migrate` / `restore` / `schema preview` that names such an index will now stop and ask the operator to shorten or alias it instead of proceeding to a silent truncation. Index names of 63 bytes or fewer are completely unaffected and emit byte-identically; only the >63-byte edge is rejected. This affects PostgreSQL targets only (MySQL targets, which allow 64-byte identifiers, are unaffected). There is no value-handling change and no change to any non-Postgres path.

## Who needs this — action required

Operators restoring a backup **chain that carries a large incremental** into a high-latency / cross-region target — via standalone `sluice restore --from <chain>` or a `sync from-backup --reset-target-data` cold-start — are who the chain-restore fix is for: it lifts the RTT-bound ~1-change/s stall. No action required beyond upgrading — the concurrent path is on by default (auto:4) and produces byte-identical, exactly-once results, so there is nothing to re-verify or re-run from prior releases; this is a throughput fix, not a correctness fix, and no earlier release silently lost or mis-applied data on this path (a restore that completed on a prior release completed correctly; one that hit the stall simply ran unbearably slowly). If you deliberately want the old single-serial-stream behaviour (a single-connection target, or to bound concurrency yourself), pass `--apply-concurrency 1`.

Operators migrating / restoring **into a PostgreSQL target from a source that can carry index names longer than 63 bytes** (a MySQL source with 64-byte identifiers, or any source with multibyte UTF-8 index names) must act if and only if such a name exists in their schema. **Affected releases for the silent-truncation class: every published release through v0.99.124 that emitted PG index DDL** — the verbatim-emit-then-server-truncate path predates v0.49.0's partial #26 guard (which only covered the generative `<table>_` prepend overflow, not a source name that itself exceeds 63 bytes), so the silent-collapse window is the full lifetime of PG index emission up to and including v0.99.124. If you ran a MySQL→Postgres (or any →Postgres) migration on a prior release **and your source schema has any index name longer than 63 bytes** (check the source's index catalog; for MySQL, names approaching the 64-byte limit, or multibyte names), re-verify the target index set against the source: a name sharing its first 63 bytes with another could have silently collapsed two indexes into one, leaving the second missing on the target with no error. After upgrading, such a name will be refused loudly at emit time and you shorten or alias it. If none of your PG-target index names exceed 63 bytes, you were never exposed and there is nothing to do. All other operators — anyone not restoring a large-incremental chain and not emitting a >63-byte PG index name — can upgrade with no action.
