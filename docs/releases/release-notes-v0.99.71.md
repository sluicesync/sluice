# sluice v0.99.71

**New: native (non-Vitess) MySQL concurrent cold-copy now fans EACH table across D plain-INSERT writer connections — taking the path from W × 1 to W × D and closing most of the remaining cross-region throughput gap toward Vitess parity (ADR-0102).** ADR-0101 (v0.99.70) made the native cold-copy copy W tables concurrently, but each table was still a single plain-INSERT stream; the live Track-D re-test measured that at ~10.4 MB/s (~15.7× over serial) against a cross-region PlanetScale-MySQL target, versus the Vitess W × D path's ~26 MB/s. This release composes ADR-0097's D-way per-table write fan-out onto that path so total write concurrency is W × D, and ships a CDC-apply resilience fix so a sync onto a loaded PlanetScale/Vitess target CONVERGES under the transaction killer instead of dying, plus the v0.99.70 follow-up that makes `--restart-from-scratch` / auto-resnapshot recovery actually clean a populated native-MySQL target before re-copying.

## Features

- **Native-MySQL concurrent cold-copy per-table write fan-out — W × D (ADR-0102).** ADR-0101 (v0.99.70) made the native cold-copy copy W tables CONCURRENTLY, but each table was still a single plain-INSERT stream (W × 1); the live Track-D re-test on v0.99.70 measured that at ~10.4 MB/s (~15.7× over serial) against a cross-region PlanetScale-MySQL target, versus the Vitess W × D path's ~26 MB/s. PlanetScale blocks `LOAD DATA LOCAL INFILE`, so each table's copy is a batched-`INSERT` stream on one pinned connection, and a single cross-region INSERT connection is round-trip-bound — so the remaining lever is fanning each table across D connections to overlap D round-trips, exactly as ADR-0097 already does for the idempotent VStream path. This release composes that D-way per-table write fan-out onto the native concurrent path: a new engine-neutral writer capability `ir.ParallelCopyWriter.WriteRowsParallel` (the plain-INSERT mirror of ADR-0097's `WriteRowsIdempotentParallel`, calling the same `writeBatchedConn` plain-INSERT core extracted byte-identically from `writeBatched`) is implemented by the MySQL target, and a new pipeline helper `copyTablePlainMaybeParallel` REUSES ADR-0097's `partitionRowsByPK` PK-hash row routing VERBATIM — only the per-worker write call differs (plain INSERT vs upsert). Total write concurrency is **W × D** (W disjoint tables concurrent, each fanned across D PK-hash-routed writers); the ADR-0100 consumer's native branch and the serial bulk-copy loop both route through it. Exactly-once is the product of two independently unit-pinned partitions — the disjoint table partition (W, ADR-0101/0099) × the PK-hash row routing (D, ADR-0097) — and plain INSERT is correct here because the snapshot is a gap-free read onto a FRESH target (the restart-from-scratch reset below guarantees no leftover rows), so there is no overlap to absorb and a stray double-write surfaces as a LOUD duplicate-key error rather than silent corruption. A no-PK table routes to the single-writer serial copy (fully copied, NOT refused — distinct from the idempotent keyless refusal). Composes the existing `--copy-fanout-degree` (D) and `copy_table_parallelism` (W = N) knobs with no new knob, and is zero-value-safe: D = 1 is byte-identical to the ADR-0101 W × 1 path, and W = 1 gains a 1 × D fan-out. The W × D connection budget is the operator's responsibility on this path (the native path has no target-slot prober) with an honest WARN naming W × D — no false auto-clamp claimed. A log-clarity fix rides along: the ADR-0079 cold-start gate no longer logs "using serial cold-start" when the source actually surfaced a concurrent-copy partition. The concurrency-chunk `-race` Integration gate is the merge gate (CGO is off locally).

## Fixed

- **CDC apply now CONVERGES under sustained PlanetScale/Vitess transaction-killer aborts instead of exhausting its retry budget and killing the stream (Error 1105 "code = Aborted … for tx killer rollback"; live Track-B finding).** On a loaded cross-region PlanetScale-MySQL target, a CDC apply batch could repeatedly trip Vitess's transaction killer — the commit takes longer than the target's tx-timeout window, Vitess aborts and rolls back the transaction, and sluice (correctly) classifies the 1105 abort as transient and retries. But the apply batch size never SHRANK in response, so each of the ADR-0038 retries re-submitted the same ~1000-row batch that was again too large to commit inside the window, exhausting the retry budget and killing the stream. Three causes compounded (confirmed by code-read, three-phase Phase A): a tx-killer abort fires at commit and propagates out of `runOnce` to the streamer-level retry loop (the batch loop never retries internally); the AIMD apply-batch controller (ADR-0052) only multiplicative-decreases after more than its retriable-error threshold accumulate inside a one-minute window, but a single tx-killer abort is fatal to the `runOnce` attempt so they never accumulate; and the controller is constructed PER-`runOnce`, so any shrink it did make was discarded when the streamer retry re-attached a fresh controller at the `ApplyBatchSize` ceiling. The fix introduces a narrow `ir.TransactionKilledError` surface (an optional extension of `ir.RetriableError`) that the MySQL classifier sets ONLY for the "tx killer" 1105 shape — never the other retriable 1105 codes that a same-size retry legitimately rides out; the controller treats a tx-killer abort as a STRONG, IMMEDIATE multiplicative decrease (bypassing the threshold-of-three), and a new `Config.OnShrink` hook persists the shrunk size into a `Streamer`-level atomic so the next `runOnce` attaches at the shrunk size instead of the ceiling — successive restarts converge (1000 → 500 → 250 → …) rather than dying at a fixed large size. Exactly-once is preserved with NO explicit batch-split path: the AIMD shrink plus the existing ADR-0010 warm-resume re-apply IS the split — a tx-killed batch was rolled back server-side (Vitess `Aborted` = rolled back) so the persisted position never advanced (the position write rides the batch transaction, ADR-0007), and the retry warm-resumes from the prior commit and re-applies the same source changes in source order regrouped into smaller batches, with per-row upsert idempotency unchanged. This was always a loud failure (stream death after budget exhaustion), never silent data loss; it affects CDC apply onto tx-killer-enforcing targets (PlanetScale/Vitess MySQL) under load.
- **`--restart-from-scratch` and the ADR-0093 auto-resnapshot recovery no longer dup-key-error (MySQL Error 1062) when re-copying onto a populated NATIVE-MySQL target — they now clean the target first instead of trusting an idempotency the native path never had.** Both paths force a fresh cold-start onto a non-dropped target while skipping the Bug-9 populated-target preflight, on the premise that "the idempotent copy absorbs the overlap" — but that premise only ever held for IDEMPOTENT snapshot readers (the VStream/PlanetScale UPSERT cold-copy and the Postgres cold-copy). The native MySQL binlog snapshot does NOT implement `ir.IdempotentCopyReader`, so its cold-copy runs PLAIN INSERT, which collided on the prior copy's leftover rows and errored loudly (`Error 1062`) — a broken, misleadingly-hinted recovery path for the common native-MySQL case (and ADR-0101's concurrent native cold-copy inherited it). The fix makes "from scratch" actually start from a clean target on the non-idempotent path: the cold-start gate detects a non-idempotent reader and DROPS + recreates the in-scope target tables (reusing the FK-safe `--reset-target-data` drop machinery — PG CASCADE / MySQL InnoDB referential drops) before the re-copy, so the plain-INSERT copy lands cleanly; the `sluice_cdc_state` row is preserved (only the position is discarded, distinguishing this from the fully-destructive `--reset-target-data`). The idempotent VStream/Postgres path is unchanged — it keeps its tables and genuinely absorbs the overlap via UPSERT. The misleading help text and the auto-resnapshot WARN/terminal-error wording were corrected to describe the actual per-source behavior. Affects native (binlog-flavor) MySQL sources on the multi-table single-database path and the multi-database fan-out (ADR-0074); no behavior change for VStream/PlanetScale or Postgres sources.

## Compatibility

- **No breaking changes; drop-in upgrade.** The W × D native write fan-out composes the EXISTING `--copy-fanout-degree` (D) and `copy_table_parallelism` (W = N) knobs — no new knob — and is zero-value-safe: D = 1 reproduces the ADR-0101 W × 1 path byte-for-byte, and absent both knobs the cold-copy is the serial single-snapshot path exactly as v0.99.70. The tx-killer convergence fix changes behavior only on the failure path (it shrinks and converges where the stream previously died) and is inert when no tx-killer abort fires. The restart-from-scratch fix changes behavior only for native-MySQL `--restart-from-scratch` / auto-resnapshot recovery onto a populated target (where it previously dup-key-errored).
- **W × D connection budget is the operator's responsibility on the native path.** The native cold-copy has no target connection-slot prober, so v1 does not auto-clamp W × D — sluice emits a WARN naming W × D at copy start. Keep `W × D` within the target's connection budget and `N ≤ source max_connections`; the FTWRL/RELOAD requirement and managed-MySQL serial fallback from ADR-0101 are unchanged.
- **Scope: native (binlog-flavor) MySQL.** The W × D fan-out and the restart-from-scratch fix affect the native MySQL cold-copy only. The VStream/Vitess/PlanetScale-source cold-copy, the PostgreSQL cold-copy, the ADR-0079 fast shareable-snapshot path, and the `sluice migrate` path are untouched. The tx-killer convergence fix affects CDC apply onto tx-killer-enforcing targets (PlanetScale/Vitess MySQL) under load.

## Who needs this — action required

- **Running `sluice sync` from a vanilla / native MySQL source into PlanetScale/MySQL over a large multi-table database and want the fastest cold-copy:** set BOTH `copy_table_parallelism=4` (W, source DSN) and `--copy-fanout-degree=4` (D) — this fans W tables concurrently AND each table across D writers (W × D), lifting the v0.99.70 W × 1 ceiling (~10.4 MB/s in the Track-D test) toward the Vitess W × D rate (~26 MB/s). The source user needs RELOAD (FTWRL); on managed MySQL without it the path degrades to serial with a loud WARN. Mind the W × D connection budget — v1 does not auto-clamp it (the WARN names W × D).
- **Running `sluice sync` onto a loaded PlanetScale/Vitess MySQL target:** the tx-killer convergence fix means a sync that previously died after repeated "for tx killer rollback" aborts now shrinks its apply batch and converges automatically — no flag, no action required; just upgrade.
- **No re-verification of prior copies is required.** The W × D change is a throughput optimization, the restart-from-scratch fix repairs a loud-erroring recovery path, and the tx-killer fix repairs a loud stream-death path — none of these produced wrong or missing data on a completed copy. Completed copies are unaffected; D = 1 reproduces the prior path byte-for-byte.
