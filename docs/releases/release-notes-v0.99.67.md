# sluice v0.99.67

**New: the `sluice sync` / VStream cold-copy can now open K independent concurrent COPY streams over disjoint subsets of a keyspace's tables, lifting the read-side throughput ceiling that ADR-0097's write-side fan-out alone couldn't (ADR-0099).** Opt-in and drop-in: set the source-DSN knob `vstream_copy_table_parallelism=K` (e.g. `4`) to enable it; absent / `0` / `1` resolves to K=1 and is byte-identical to the existing sequential auto-shard path (ADR-0095/ADR-0098), so existing syncs are unchanged. Connection-budget caveat to read first: K is **not** auto-clamped against the target connection budget in v1 — you are responsible for keeping `K × D ≤ --max-target-connections` (D = the ADR-0097 write fan-out degree); sluice emits a WARN at copy start naming K and stating that contract.

## Features

- **Cross-table VStream cold-copy concurrency — K independent read streams over a disjoint table partition (ADR-0099).** ADR-0095 auto-shards the VStream cold-copy by table (one single-table COPY at a time → bounded memory) but drains them *sequentially*, one after another over a single read stream; ADR-0097's write-side fan-out only reached ~1.4× on this path because the single vtgate read stream plus one dispatcher is the bottleneck (the write workers starve waiting on it). The lever that actually scales — confirmed ~4× near-linear in live Track-B measurement — is N *independent* read streams, each on a different table, each its own vtgate gRPC session. This release internalizes that: it opens K concurrent single-table COPY streams, each copying a disjoint subset of the in-scope tables, each running the ADR-0095 per-table auto-shard sub-sequence restricted to its group. The table→stream partition is a deterministic, shuffle-invariant pure function of the (sorted) table set and K (size-balanced when row-count estimates exist, round-robin as the floor), so cold-start and resume re-derive the identical assignment, and every in-scope table is copied by exactly one stream. Opt in via the source-DSN knob `vstream_copy_table_parallelism=K`; it is zero-value-safe (absent / 0 / 1 → K=1 sequential, byte-identical to ADR-0095/0098 — the v0.99.51 trap avoided by mapping the zero value to the safe sequential default), and it composes multiplicatively with ADR-0097 (K read streams × D write workers) and with the ADR-0098 auto-shard-aware resume (per stream). The K streams multiplex over a single gRPC `*grpc.ClientConn` (HTTP/2 stream multiplexing), so K logical streams cost one TCP/TLS connection to vtgate.
- **Memory stays bounded at the same total cap, via a per-stream byte sub-budget.** Each of the K streams gets its own `--max-buffer-bytes / K` sub-budget, so `K × (cap/K)` is still the same total in-flight cap — the read concurrency does not raise the memory ceiling, it splits it. The per-stream sub-budget is also what keeps a look-ahead stream (racing ahead on a table the serial consumer hasn't reached yet) from monopolizing the buffer and starving the stream that owns the table the consumer is actively draining: the consumer draining a table implies that stream's earlier tables already drained, so its sub-cap always has room to enqueue, and the other streams park on their own sub-caps, never on the one the consumer is waiting on. A single row larger than a sub-cap is admitted when that stream's own queue is empty (one-row-at-a-time degradation, never a wedge), exactly as the sequential path does.
- **Gapless, overlap-safe snapshot→CDC handoff stitched across the union of all K streams.** Each stream finishes its tables at their own per-table snapshot positions; the single CDC tail resumes from the per-shard GTID-set **minimum** across the union of every stream's every per-table snapshot — ADR-0095's `stitchSnapshotMin`, fed the concatenation of all streams' captures. It is parallelism-agnostic: the minimum is `⊆` every captured per-table set regardless of which stream produced it, so CDC replays the full `(P_start, P_i]` window for every table and no committed change after any table's snapshot can be skipped (the silent-loss direction — the *maximum* — is never constructed). The K streams are K independent vtgate sessions rather than one causally-ordered session, so where ADR-0095's single session guaranteed one captured set is always a subset of the others by construction, the K-stream case relies on the existing loud refusal: a genuinely disjoint per-shard set (which concurrent sessions reading the same monotonic GTID stream should still not produce) is refused loudly with the shard and divergent sets named, never guessed. The global CDC position is recorded only **after all K streams have completed all their tables** (ADR-0007 join-then-stitch ordering), so CDC never starts past an incompletely-copied table; any stream's error aborts the whole copy loudly with no position advance, the other K-1 streams cancel cleanly, and ctx-cancel joins all K with no leaked goroutines, gRPC streams, or connections. Pinned by partition coverage/disjointness and shuffle-invariant-determinism unit tests, the per-stream-cap no-deadlock pin (fails on a shared cap, passes on the sub-budget), set-min-across-all-streams and multi-shard-set-min stitches, a global-position-only-after-all-streams-complete pin, resume-composes-per-stream pins (including a seed table that is not its group's first sorted table), loud-stream-error and ctx-cancel-no-leak pins, and a `vstream`-tagged `-race` integration test that cold-copies a multi-table keyspace with K > 1 (per-table `COUNT(*)` + content checksum == source, clean handoff) and an interrupt+resume with K > 1.

## Compatibility

- **No breaking changes; drop-in upgrade.** Read-side concurrency is opt-in via the source-DSN knob `vstream_copy_table_parallelism=K`. Absent the knob (or `0` / `1`) the cold-copy behaves exactly as v0.99.62+ (sequential per-table auto-shard, ADR-0095/ADR-0098); K=1 is byte-identical to that path. The persisted handoff/position token shape is unchanged (no new field).
- **Connection budget: K is NOT folded into the target-connection preflight in v1 — operator responsibility.** During cold-copy the target sees `K × D` connections (K streams, each running its active table through a D-way ADR-0097 write fan-out), plus K source-side gRPC streams. K is a source-DSN knob parsed inside the mysql engine; the pipeline's target connection-budget preflight lives a layer up and only resolves D, so it cannot clamp the `K × D` product yet (a clean cross-layer follow-on). v1 ships honest surfacing instead: when the concurrent copy starts, the engine emits a **WARN naming K** and stating that **`K × D ≤ --max-target-connections` is the operator's responsibility**. The D-axis budget preflight still fires its loud refusal exactly as before; only the K multiplier is unguarded.
- **Resume must use the same K (operator contract).** The table→stream partition is a pure function of `(sorted tables, K)`, so a resume with a different K would re-derive a different partition and risk mis-assigning the in-progress table's cursor. v1 requires resuming with the same `vstream_copy_table_parallelism` the cold-start used (the DSN is operator-supplied and stable across a resume of the same job); a changed-K resume is documented as "restart the cold-start" rather than risked, and the position token shape is left unchanged (consistent with ADR-0098's no-new-field decision).
- **Scope: the `sluice sync` / VStream (Vitess/PlanetScale-source) cold-copy only.** The vanilla MySQL binlog and PostgreSQL cold-copy paths are untouched, as is the `sluice migrate` path (which has its own within-table parallel chunking — ADR-0096/ADR-0019). A single-table scope is already constant-memory single-stream and unaffected.

## Who needs this — action required

- **Anyone running `sluice sync` from a Vitess/PlanetScale source over a large multi-table keyspace into a cross-region target who wants a faster cold-copy:** set `vstream_copy_table_parallelism=4` (or higher) in the source DSN. Start at 4 and raise it for a known-large cross-region copy, but mind the connection budget — keep `K × D ≤ --max-target-connections` (D is your `--copy-fanout-degree`, default 4), because v1 does not auto-clamp K; the WARN at copy start names K and restates the contract. Memory stays bounded regardless of K (each stream gets `--max-buffer-bytes / K`). Resume the job with the same K you started it with.
- **No re-verification of prior syncs is required.** This is a throughput optimization to *how fast* the cold-copy runs, not a fix to a path that produced wrong or missing data. The previous sequential auto-shard copy (v0.99.62+) was correct and gapless; completed syncs on prior versions are unaffected and need no re-check, and K=1 (the default) reproduces that path byte-for-byte.
