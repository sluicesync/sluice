# sluice v0.99.253

**New: managed-provider preflight/advisory hardening from this week's live validations (Neon, Supabase, DigitalOcean Managed MySQL probes) + the ps-discovery comparison (roadmap items 68a/68b, 69a–g, 70a/70b).** Two new coded refusals (`SLUICE-E-CDC-POOLER-ENDPOINT`, `SLUICE-E-CONNECT-IPV6-ONLY`), pooler-host and DO-retention WARNs at run start, a provider-shaped `wal_level` remedy, and two loud-failure-tenet closures on the PG source path: FDW foreign tables now WARN-and-skip loudly instead of vanishing silently, and old-style INHERITS hierarchies now refuse loudly instead of silently duplicating child rows. Additive drop-in — no flag or default changes — but **if you ever migrated a Postgres source containing an old-style INHERITS parent, read the Who-needs-this section: child rows copied twice on every prior release.**

## Fixed

- **PG foreign tables (FDW) now WARN-and-skip loudly, naming each table and its foreign server (item 68a; the silent skip existed in every prior release with a Postgres source).** The PG schema reader filters on `table_type='BASE TABLE'`, so foreign tables never entered the migration — correctly (a foreign table holds no local rows; its data lives on the remote server the FDW points at, and materializing remote data would itself be a surprising side effect) — but with zero signal: a table the operator can SELECT from simply vanished from the plan. A new `relkind='f'` census (`SchemaReader.ForeignTables`, over `pg_foreign_table` + `pg_foreign_server`) feeds a preflight WARN naming every skipped foreign table, the server its data actually lives on, and the recovery paths (migrate the foreign server directly as its own sluice source, or recreate the FDW wiring on the target). Passing `--exclude-table` for a foreign table acknowledges the skip and silences the WARN for that table. No data was ever wrong — the wart was the silence, not the skip.

- **Old-style INHERITS hierarchies now refuse loudly at migrate and sync cold start (item 68b; the silent duplication affected every prior release — declarative partitions were guarded since v0.92.0's Bug-100 refusal, but the legacy twin never was).** Inheritance children are ordinary BASE tables the reader copies independently — but a SELECT on the parent (without `ONLY`) ALSO returns every child's rows, so an unguarded migration landed the child data twice: flattened into the parent's target table AND again in each child's. There is no `pg_partitioned_table` row to trip Bug 100's probe; the only catalog signal is a `pg_inherits` parent with `relkind='r'`, which the new `SchemaReader.InheritanceParents` census probes (disjoint from the declarative probe by construction). The refusal follows the Bug-100 gate/skip/in-scope shape exactly — filter-aware, so `--exclude-table=<parent>` opts you out — and names every offending parent plus three recovery paths: exclude the parents (children copy individually; first verify the parent stores no own rows via `SELECT count(*) FROM ONLY <parent>`), exclude the children to deliberately flatten, or scope via `--include-table`.

## Added

- **Coded `SLUICE-E-CDC-POOLER-ENDPOINT` — `sync start` through a connection pooler now refuses with the mechanism and the direct-endpoint remedy (item 69e, live-caught on Supabase).** A pooler (Supabase's Supavisor, pgbouncer) strips the `replication=database` startup parameter, so sluice's `CREATE_REPLICATION_SLOT` reaches a normal backend as plain SQL and dies with a bare SQLSTATE 42601 syntax error — previously a genuinely baffling failure. The classifier sits at the `createLogicalReplicationSlot` chokepoint and matches that exact live-probed signature (42601 on/near the slot command; anything else passes through), producing a coded refusal that names the pooler, the direct endpoint, and the Supabase-specific IPv4-add-on caveat. Logical replication cannot run through a pooler on any provider; the error now says so.

- **Coded `SLUICE-E-CONNECT-IPV6-ONLY` — AAAA-only resolve failures now explain themselves (item 69f, live-probed against Supabase).** Supabase free-tier direct endpoints carry only an AAAA record (IPv4 is a paid add-on), so from an IPv4-only network sluice failed in about a second with the platform resolver's cryptic no-data error. `WrapWithHint` gains a structural DNS classifier: on a no-data/not-found `net.DNSError` it probes the host for an AAAA record (bounded at 3s, and only on the already-terminal failure path — zero cost to healthy runs); when the host is resolvable-but-IPv6-only, the error gains the remedy — pooler endpoint for bulk migrate (it has an A record), IPv4 add-on or an IPv6-capable network for CDC (a pooler cannot proxy replication). Hosts with no AAAA fall through to the static hint registry unchanged.

- **Pooler-host WARN at migrate/sync/backup start (items 69a; live-validated against Neon and Supabase).** New optional `ir.SourceHostAdvisor` engine surface — the WARN-level sibling of `ir.DSNValidator`, derived from the source DSN's host alone with no connection — called at the top of migrate, sync, and the backup CDC paths. The PG engine WARNs when the source host matches a named pooler pattern (Supavisor `*.pooler.supabase.com`, the `-pooler` host label Neon uses, `pgbouncer`): bulk migrate through a pooler works (a full snapshot-pinned parallel migrate passed through Neon's pooler at validation scale), but parallel copy pins server connections inside long-lived snapshot transactions — pool-exhaustion risk at scale — and transaction-mode poolers break pgx's statement cache, silently dropping you to the single-reader path. The WARN recommends the direct endpoint; the run proceeds.

- **DigitalOcean Managed MySQL lying-retention advisory at sync/backup start (item 70a, live-probed on a fresh MySQL 8.4 cluster).** On DO defaults an out-of-band platform reaper purges every binlog roughly 10–15 minutes after creation — while `@@binlog_expire_logs_seconds` reads 3 days and no SQL-level preflight can see it. The variable lies, so the `*.db.ondigitalocean.com` host pattern is the only reliable signal: on CDC-anchoring runs (sync, backup — a plain migrate never returns to the binlog) the MySQL engine WARNs with the mechanism and the confirmed fix, DO's config-API knob `binlog_retention_period` (`PATCH /v2/databases/{id}/config`, seconds 600–86400, 86400 recommended for migrations; effective immediately, no restart). Without the advisory the hazard is loud only AFTER loss, and a >15-minute cold copy can livelock auto-resnapshot.

- **The `wal_level` refusal remedy is now provider-shaped (item 69b).** "Edit postgresql.conf and restart" is self-managed-only advice — on Neon the fix is the `enable_logical_replication` project toggle (irreversible, seconds, no visible downtime; validated live), on Supabase `logical` is already on, on RDS it's the `rds.logical_replication` parameter. The refusal now points at the new provider matrix in `docs/postgres-source-prep.md` (self-managed, Neon, Supabase, PlanetScale Postgres, RDS/Aurora, CloudSQL, Azure) rather than baking one provider's console flow into the error text.

- **Within-table-parallel bulk copies emit the table-level "bulk copy complete" line (item 69c).** The per-chunk tickers logged per-chunk completion (and the raw-copy lane only at DEBUG), so a within-table-parallel table previously finished silently at INFO — only verify proved the copy. The parallel path now emits the single-reader path's completion sibling with whole-table rows, chunk count, and duration, so log consumers see one completion vocabulary across both copy shapes.

- **`docs/managed-services.md` gains live-validated Neon, Supabase, and DigitalOcean Managed MySQL sections (items 69d/69g/70b) plus the ps-discovery acknowledgment.** Live-probed facts only: Neon (byte-identical fidelity incl. NaN-in-`numeric[]`/±Infinity/2-D-arrays-with-NULLs, the `wal_proposer_slot` internal-slot note, direct-vs-pooler), Supabase (session vs transaction pooler modes, the IPv6-only direct endpoint, platform schemas), DO MySQL (the lying retention window, private-CA TLS via `--source-tls-ca`, `ANSI` sql_mode and `sql_require_primary_key` gotchas).

## Compatibility

- **Additive; no flag, default, or copy-behavior changes.** The new surfaces are WARNs and refusals on paths that were previously silent or confusing. The foreign-table WARN formalizes existing behavior — foreign tables were always skipped, they are still skipped, the skip is just loud now. The two new error codes fire only where the old behavior was an uncoded, cryptic failure. The parallel-copy completion line is a new INFO log line.
- **One new refusal on a previously-proceeding shape:** a Postgres source containing an in-scope old-style INHERITS parent now refuses at preflight instead of silently duplicating child rows. If your runs relied on that shape, `--exclude-table=<parent>` (or excluding the children, or `--include-table` scoping) opts you back in deliberately — the refusal message walks through all three.

## Who needs this — action required

- **If you ever migrated (or cold-started sync on) a Postgres source containing an old-style INHERITS hierarchy — on any release through v0.99.252 — verify the target for duplicated child rows.** The parent's target table received every child's rows (the parent SELECT ran without `ONLY`) AND each child copied independently; compare `SELECT count(*) FROM ONLY <parent>` on the source against the parent's target count, and check child-key overlap in the parent's target table. Declaratively-partitioned parents are NOT affected (guarded since v0.92.0); this is only the legacy `INHERITS` shape. Completed migrations of schemas without INHERITS parents need nothing.
- **If a Postgres source of yours has FDW foreign tables**, know that they were never copied on any release — that behavior is unchanged and correct, but you now get a WARN naming them; if you assumed their data had migrated, it did not (migrate the foreign server directly, or recreate the FDW wiring on the target).
- **Everyone running managed sources** picks up better failures for free: Supabase/pgbouncer pooler CDC attempts and IPv6-only endpoints now fail coded with the fix in the message, DO MySQL sync/backup runs get the retention warning before the platform purges the binlog out from under the stream, and `wal_level` refusals point at your provider's actual knob.
