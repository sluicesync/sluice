# sluice v0.99.205

**CLI/automation contract hardening + an internal-libs small-fixes sweep (audit wave 4b — the LOW tail). An operator-declined `--reset-target-data` confirmation no longer reports success: under `--format json` it now emits a proper `{"status": "aborted"}` envelope with a nonzero exit instead of `{"status": "completed"}` with prompt text polluting the one-JSON-object stdout contract. Ctrl-C is no longer swallowable — the typed destructive-confirm prompt aborts on Ctrl-C, a second Ctrl-C force-exits (130) instead of hanging behind a wedged shutdown, and D1 `--stage-local` staging is cancellable and cleans its temp dir. `sync status --all` exits non-zero when EVERY configured target is unreachable. `fleettui` no longer renders `--connect` basic-auth passwords in its banner or fetch errors, and `file://` backup stores now create 0700 directories with a loud WARN about their file posture. Plus: diagnose bundles can no longer silently omit sections, migrate-state reads work under a read-only DB user, and an all-zero apply-sizing config refuses loudly. Drop-in for existing invocations, with four deliberate loud-over-silent exit/status changes listed under Compatibility.**

## Security

- **`fleettui` no longer displays `--connect` basic-auth credentials — the banner and both fetch-error shapes now render the endpoint through a redacted form (display-only exposure; fleettui's whole life, v0.99.137 through v0.99.204).** The banner rendered the raw connect address and the HTTP fetcher embedded the raw endpoint — userinfo included — in its transport and non-200 error text, so a `--connect https://user:pass@host` password appeared on screen and in any captured terminal output. The banner address is now stored pre-redacted at construction and fetch errors name the endpoint through its redacted display form, while the un-redacted endpoint still drives the fetch itself (the userinfo is functional basic auth); unparseable connect strings fall back to the post-`@` remainder so no credential bytes can survive any parse shape. Pinned by a redaction table, the banner-through-construction path, both fetch-error shapes, and a functional-userinfo-retention pin on the fetch itself.
- **`file://` backup URLs now create owner-only 0700 directories and WARN about the residual file posture (loose-permissions hardening; since blob-store URLs landed in v0.16.0, through v0.99.204).** A `file:///` `--backup-target` routes to gocloud's fileblob, whose defaults are 0666 files / 0777 directories — silently looser than the hardened 0600/0700 local store that `--output-dir` uses. The URL annotator now defaults fileblob's directory-mode option to 0700 unless the operator set their own, and store open WARNs that the FILES remain world-readable-minus-umask (fileblob exposes no per-file mode option), steering sensitive backups to `--output-dir`'s hardened store. The file posture is surfaced, not silently mutated; existing directories keep their modes.

## Fixed

- **The `--reset-target-data` destructive-confirm gate now has an honest abort contract across json and text modes, a Ctrl-C-able prompt, and a second-SIGINT escape hatch (the json envelope lie since the envelope work landed in v0.99.175, the abort-exits-0 and swallowed prompt Ctrl-C since the confirm gate landed in v0.6.0; through v0.99.204).** An aborted confirm returned nil, so `--format json` rendered `{"status": "completed"}` for a run that never happened — with the prompt text written to stdout, breaking the one-JSON-object contract any automation harness parses. The abort now returns a declined sentinel (generic exit 1), the envelope renders the new status `"aborted"`, and the prompt writes to stderr in json mode — applied at all three embedding sites: `migrate`, `sync start`, and `sync from-backup run` (text-only there; it has no `--format` flag). The prompt read also races the signal context, so Ctrl-C at the prompt aborts instead of being swallowed by the blocking stdin read; and the signal wiring arms a second-signal escape hatch — the first SIGINT/SIGTERM keeps the graceful drain, a second exits 130 (128+SIGINT) immediately, so an operator is never trapped behind a cancellation-deaf code path. Pinned by the full command-path matrix {migrate, sync start, from-backup} × {json where present, text} × {confirm, abort} through the real command `Run()`, the envelope `"aborted"` shape, prompt ctx-cancel in both orders, and the signal counter via direct invocation (Windows-safe, no real signal delivery).
- **`sync status --all` now exits non-zero when EVERY configured target is unreachable (the fleet roll-up's whole life, v0.99.132 through v0.99.204).** The roll-up's per-target failure isolation (report-and-skip a dead target) degenerated when the whole fleet was dark: an empty table plus exit 0 read as "no streams" to any script checking `$?`. Partial reachability keeps exit 0 with the per-target lines; total unreachability now returns a plain error (generic exit 1 — 2 stays config-load, 3 stays coded refusals) after the per-target diagnosis lines. Both shapes and the exit code are pinned through the real exit-code mapping.
- **D1 `--stage-local` staging honors Ctrl-C and no longer leaks its temp dir (since `--stage-local` landed in v0.99.167, through v0.99.204).** Staging ran on `context.Background()` because engine resolution predated the signal wiring — a Ctrl-C during the potentially minutes-long D1 replica neither stopped the HTTP paging nor released the `sluice-d1-stage-*` temp dir until process kill. Engine resolution now takes the signal-aware context and threads it into staging; the existing error-path cleanup removes the temp dir on cancellation. Pinned: a canceled ctx aborts before any network I/O and leaves no temp dir behind.
- **`migratestate.Store.Read` is now genuinely read-only — the legacy-format upgrade moves to the first write path (since the ADR-0082 per-table progress rework in v0.99.34, through v0.99.204).** Read performed the one-way legacy-blob upgrade — a whole write transaction — inside a nominally read-only call, so it failed under a read-only target user and locked older binaries out of a row the operator only inspected. Read now detects a legacy header, decodes the blob, and records a needs-upgrade note; the first write for that migration id runs the unchanged upgrade transaction exactly once under the store mutex, exploding the blob into rows BEFORE flipping the header sentinel so no progress can be lost, and a clear or an already-upgraded re-read drops the note so stale blob progress can never replay. Pinned by scripted-driver units (legacy Read issues only the header SELECT; the first write runs the byte-identical upgrade exactly once; failure rolls back and the next write retries) plus both engines' real-DB integration pins asserting Read succeeds on an ACTUALLY read-only server (PG `default_transaction_read_only`, MySQL `super_read_only`) and leaves the row untouched.
- **Diagnose bundles now write a `sections.json` accounting for every section — written, skipped (with reason), or failed (with reason) — so a bundle can no longer silently omit a section (since bundles landed in v0.75.0, through v0.99.204).** ~25 section writes in the bundle assembler discarded their errors, so a failed section vanished with no trace exactly when someone was debugging with the bundle. Per-section failures still never fail the whole bundle (only the manifest and `sections.json` themselves are structural), and the writer now encodes BEFORE creating the zip entry, so a failed encode no longer leaves a phantom empty file masquerading as a section. Additive file; the manifest version is unchanged. Pinned by forcing a real section-encode failure through the production path and asserting the bundle still assembles, the zip carries no phantom entry, and `sections.json` names the failure alongside its written/skipped peers.
- **`appliercontrol` refuses an all-zero sizing config loudly instead of silently degenerating to permanent 1-row apply batches (defensive; the clamp shape dates to the controller's introduction in v0.72.0, through v0.99.204 — no shipped production path could reach it).** The defaulting clamped an all-zero Floor/Ceiling/InitialSize trio to Floor=Ceiling=1 with no signal; since Ceiling has no sensible package default (it is the operator's `--apply-batch-size` cap), the constructor now refuses the all-zero shape naming the fields and the way out, while deliberate PARTIAL configs keep their documented clamps (pinned). A construction-site audit confirmed both production sites always pass Floor 1 / Ceiling > 1, so this is a tripwire for future callers, not a behavior change.
- **Sustained sqlite-trigger WAL-checkpoint failures now escalate DEBUG → WARN after 3 consecutive failures (since the cadenced checkpoint landed in v0.99.152, through v0.99.204).** A failing cadenced `wal_checkpoint(TRUNCATE)` was visible only at `--log-level=debug`, so an operator whose source WAL was silently growing unbounded had no default-visibility signal. The checkpoint now counts CONSECUTIVE failures and escalates to WARN from the third on, naming the streak; a single routine failure (a writer held the WAL lock this cycle) stays DEBUG, and the first success resets the streak. Still non-fatal — WAL management remains orthogonal to stream correctness. Pinned across the boundary in both directions.

## Internal

- **Lint headroom + hygiene:** the backup-codec envelope parser sat at exactly the gocyclo ceiling — the next tag added would have failed Lint — and is now split along tag-family seams with verbatim case bodies (byte-identical decode; the existing decode-differential, malformed-envelope, and Bug-74 family-matrix suites all pass unchanged). A dead-code trio (`triggerSetupExampleSchema`, `quoteForPrometheusLabelValue`, `LatencyWindow.Reset` — all self-admitted `//nolint:unused`) is deleted with its stranded imports; a stale-named mysql file is renamed to match what it holds; 25 verb-less `fmt.Errorf` calls become `errors.New` per the house rule (byte-identical messages); `waitForStopComplete`'s comment no longer promises an exit 2 the code never returned (exit 1 is the right shape — the drain continues in the background — and the exit code is now pinned so code and comment can't drift again); the crash-hook doc no longer names `sync from-backup run` as an embedder (it never was); and the envelope-status docs gain the `"aborted"` vocabulary.

## Compatibility

**No CLI-flag changes; drop-in for existing invocations, with four deliberate loud-over-silent behavior changes:** (1) a text-mode destructive-confirm abort now exits 1 (was 0) — a script that checked for 0 on abort must update; (2) the `--format json` envelope gains status `"aborted"` (nonzero exit; the prompt moves to stderr in json mode) — automation matching only `"completed"`/`"failed"` should add it; (3) `sync status --all` with a FULLY unreachable fleet now exits non-zero — cron/monitoring relying on an unconditional 0 must update (partial reachability keeps exit 0); (4) a second Ctrl-C now force-exits 130 instead of continuing to wait on the graceful drain. Everything else is shape-preserving: the fileblob change only tightens NEW directory modes (existing dirs keep theirs), the migrate-state upgrade still happens — just on the first write instead of the first read — and the codec split, rename, and `errors.New` sweep are byte-identical.

## Who needs this — action required

- **Automation/script authors around `migrate`/`sync start` confirms, `sync status --all`, or the JSON envelope: review the four Compatibility items before upgrading.** The changes are all in the loud direction — nothing that previously failed now succeeds — but exit-code and envelope matchers written against the old shapes need the one-line updates above.
- **`fleettui --connect` users who used basic-auth URLs: the password was rendered on screen (banner) and in fetch-error text — if terminal recordings, screenshots, or captured TUI output exist from v0.99.137–v0.99.204 sessions, treat the credential as exposed and rotate it.** sluice itself did not persist it anywhere.
- **`file://` backup-target users: new store directories are 0700 from this release, but the FILES remain world-readable-minus-umask (a gocloud fileblob limitation, now loudly WARNed) — tighten existing store trees by hand if the data is sensitive, or prefer `--output-dir`'s hardened 0600/0700 store.**
- **Operators who inspect migrate state with a read-only DB user: `Read` now works there — no more upgrade-transaction failure on a row you only wanted to look at.**
- **Everyone else: drop-in upgrade, no action.**
