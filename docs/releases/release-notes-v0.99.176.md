# sluice v0.99.176

**Fidelity closures + the first performance wave. Chain restore no longer strands identity sequences at the base full's max (loud 23505 aborts on constrained tables, silent duplicate ids on keyless ones ‚Äî live-reproduced pre-fix), SQLite‚ÜíMySQL temporals no longer silently truncate fractional seconds, temporal precision round-trips faithfully (bare stays bare, an explicit `(0)` stays `(0)`), and UNIQUE constraints land on Postgres targets as real constraints ‚Äî taking the PG‚ÜíPG restore-parity oracle's ledger to ZERO TRIAGE entries (every remaining divergence is a cited, documented decision). Plus roadmap item 54's first perf wave: MySQL sync cold-starts go parallel by default, migrate's parallel PG readers share one exported snapshot, the replay paths gain buffered channels and chunk read-ahead, and opt-in `--analyze-after` refreshes planner statistics post-migration. Drop-in for most paths; see Compatibility for the temporal catalog-shape notes and the MySQL cold-start default change.**

## Features

**MySQL sync cold-starts are now parallel by default (native binlog flavor).** A MySQL user's first `sync start` cold-copy previously ran ~4√ó slower than the `migrate` they had just run, because migrate defaults to cross-table auto:4 while the sync cold-copy defaulted serial. The native-binlog cold-copy now defaults to the same auto ‚Äî min(4, tables), clamped at 32, CLI override > DSN param > default ‚Äî and it is consistency-identical to serial by construction: the FTWRL-coordinated N-snapshot takes ONE global cut and ONE binlog position (no stitch), and the serial opener already attempted the same FTWRL on every cold start. `--copy-table-parallelism=1` (or DSN `copy_table_parallelism=1`) is the serial opt-out, and an FTWRL-denied source keeps the loud serial fallback. VStream deliberately stays serial-by-default: the stream count is not persisted in the position token, so a default flip would silently re-derive a different table‚Üístream partition for a resumed job across the upgrade boundary (a miss/double-copy silent-loss class) ‚Äî the sequential VStream cold-copy now emits an INFO naming `--vstream-copy-table-parallelism` instead. This is the first wave of the cross-mode perf-parity program (roadmap item 54), with each change cell-verified against the living parity matrix.

**migrate's parallel Postgres readers are now all pinned to one exported snapshot.** Previously each parallel bulk-copy reader observed its own MVCC view (fine on a quiesced source, a documented window on a live one); only the sync cold-start pinned its readers. When the source is PG, the exporting connection runs `pg_export_snapshot()` under REPEATABLE READ and every cross-table pool pair and within-table chunk reader imports that one snapshot ‚Äî a strict consistency upgrade on live sources, with no replication prerequisites (no wal_level change, no REPLICATION privilege, no slot). The pin is released the moment the copy ends ‚Äî never spanning the index/constraint phases, so it cannot bloat the source ‚Äî and an export failure (e.g. a hot standby, where `pg_export_snapshot()` cannot run) falls back LOUDLY to the previous independent-reader behavior rather than failing a previously-working migrate. Non-PG sources are byte-identical (capability-gated). Pinned by an integration test proving rows inserted after the export are invisible to every parallel reader and that zero idle-in-transaction sessions remain at release time.

**Replay-path throughput: buffered channels plus one chunk of read-ahead.** The three replay hops ‚Äî restore's chunk-group row channel, chain-restore's incremental change channel, and the broker's apply channel ‚Äî were unbuffered rendezvous points while migrate's discipline is a 64-slot bounded buffer everywhere; all three now carry the standard 64 buffer, and migrate's redact and shard-stamp tees (previously unbuffered relay hops re-introduced into the bulk-copy hot path whenever redaction or the Shape-A discriminator was engaged) get the same treatment. Chain replay additionally overlaps fetching chunk N+1 with applying chunk N via a single fetcher goroutine handing verified chunks over an unbuffered channel ‚Äî exactly one chunk of read-ahead, never an N-deep pipeline. Behavior-neutral except throughput: apply order is strictly preserved, the per-chunk fetch retry is untouched, back-pressure stays bounded, and durable positions/cursors only ever advance AFTER the consumer applies, so buffered-but-unapplied items can never advance them. Landed through the `-race` integration gate.

**`--analyze-after`: opt-in post-migration statistics refresh.** A freshly bulk-loaded table has stale planner statistics, so first post-cutover queries plan badly until autovacuum/background stats catch up (pgcopydb runs per-table VACUUM ANALYZE by default for the same reason). With `--analyze-after`, migrate runs a per-table statistics refresh after constraints and views complete ‚Äî PG `ANALYZE`, MySQL `ANALYZE TABLE` (queried, not exec'd, so error result-rows are caught), SQLite `ANALYZE` ‚Äî on all three target engine families. Advisory by contract: it runs strictly post-success, a per-table failure WARNs naming the table and never fails the run, and an engine without the surface gets one loud WARN. Default off; the dry-run plan carries `analyze_after`.

## Fixed

**Chain restore now re-syncs identity sequences at the chain tail (silent-loss class on keyless tables; affected v0.17.0‚Äìv0.99.175 ‚Äî every release with chain restore).** Identity-sequence sync ran only inside the base full's restore phase; nothing re-synced after incremental links applied higher ids, so a chain restore left every identity column's sequence at the base full's max ‚Äî the first post-restore insert hit a loud `23505 duplicate key` on constrained tables, or silently minted duplicate ids on keyless ones. Live-reproduced pre-fix with the exact 23505. `ChainRestore.Run` now calls an identity-sequence re-sync at the chain tail, beside the v0.99.175 standalone-sequence re-prime: the newest link's schema snapshot, retargeted and filtered exactly like the base full's restore, through the engine-neutral `SyncIdentitySequences` surface (real on PG, documented no-op on MySQL/SQLite), data-derived MAX+1, idempotent, and under the reparent retry. The broker's `--reset-target-data` reset leg inherits the fix through `ChainRestore.Run`. Pinned by an integration test (full at max id 2, links apply 3/4, first post-restore insert must draw 5) verified failing pre-fix.

**Temporal precision now round-trips faithfully ‚Äî bare stays bare, an explicit `(0)` stays `(0)`, and SQLite‚ÜíMySQL temporals stop silently truncating fractional seconds (restore-parity TRIAGE #3; the truncation shape affected v0.99.141‚Äìv0.99.175, the catalog shapes effectively every prior release).** The IR had no "precision unspecified" state for temporals, so three divergences hid in one gap. (1) A bare `timestamptz`/`timestamp`/`time` on a PG source materialized as `timestamp(6) with time zone` in the target catalog ‚Äî same runtime behavior, divergent DDL. (2) Worse, the emitter's old 0‚Üíbare collapse silently WIDENED an explicit `timestamptz(0)` (or a MySQL `datetime(0)`, genuinely precision 0) to the 6-behaving bare form ‚Äî the target accepted fractional seconds the source would have rejected. (3) Worst, SQLite temporals (no precision concept) landed on MySQL as bare `DATETIME` ‚Äî which MySQL defines as precision 0 ‚Äî so fraction-bearing values were silently truncated to whole seconds on the migrated rows themselves. The IR temporal types now carry a `PrecisionUnspecified` state mirroring the decimal `Unconstrained` precedent: the PG reader keys declaredness off `pg_attribute.atttypmod` (the ground truth ‚Äî information_schema cannot distinguish bare from an explicit `(6)`), including temporal ARRAY elements; unspecified emits bare on PG and materializes as `(6)` on MySQL (whose bare form would truncate); a KNOWN precision always emits, including `(0)`; and SQLite‚ÜíMySQL temporals now land `DATETIME(6)`/`TIMESTAMP(6)` with a value-level pin proving the read-back preserves source microseconds. The wire field is append-only with no format bump: old backups decode their materialized `(6)` as an explicit `(6)` and restore byte-identically to the old behavior. Pinned per the family√óshape discipline: read+emit matrices over {time, timetz, timestamp, timestamptz} √ó {bare, (0), mid, (6)} plus array elements, wire round-trips, and a PG‚ÜíPG integration matrix ground-truthed on `format_type`.

**UNIQUE constraints now land on Postgres targets as real constraints, not demoted unique indexes (restore-parity TRIAGE #4; affected effectively every prior release for PG targets).** A source `CONSTRAINT x UNIQUE (cols)` previously landed as `CREATE UNIQUE INDEX` ‚Äî near-equivalent for enforcement, but it broke `INSERT ... ON CONFLICT ON CONSTRAINT x` on the target (the named constraint didn't exist) and diverged the pg_dump catalog. The IR index model now records constraint-backed uniqueness (set only by the PG reader, from `pg_constraint`); the PG writer skips those entries in the index phase (live, preview, and the overlapped-build path) and adds them in the constraints phase via `ALTER TABLE ... ADD CONSTRAINT` ‚Äî ALL unique constraints before ANY FK, preserving the ordering guarantee FKs referencing another table's unique key relied on. Constraint names emit VERBATIM (PG source names are schema-unique; prefixing would break `ON CONFLICT ON CONSTRAINT`), resume is idempotent (detect-then-skip, like FKs), `INCLUDE` payloads carry, an expression entry refuses loudly (PG forbids it ‚Äî a sluice-bug condition), and the keyless-table inline promotion still works without a duplicate ADD. MySQL and SQLite targets keep landing a plain unique index ‚Äî MySQL UNIQUE is always an index under the hood and SQLite inline-UNIQUE auto-indexes are unnamed ‚Äî a documented asymmetry. The backup-wire carry is pinned so a future marshal change cannot silently reopen the demotion through the restore door. Pinned end-to-end by an integration test performing a real `ON CONFLICT ON CONSTRAINT` upsert on the migrated target. With this and the temporal closure, the restore-parity oracle's TRIAGE ledger is empty.

**The double-quoted-token refusal now carries its documented error code and exit 3 (Bug 176; affected v0.99.175 only).** v0.99.175's SEC-1 refusal of SQLite double-quoted tokens bound for MySQL fired loudly but exited 1 uncoded ‚Äî while the same release's `docs/operator/error-codes.md` declared `SLUICE-E-EXPR-BACKSLASH-LITERAL` covers that shape. Agents and scripts branching on the code or on exit 3 missed the refusal. Found same-day by the v0.99.175 regression cycle; the refusal now carries the code and the named-refusal exit 3, honoring the shipped machine contract, and the pin asserts the code rides the error.

## Compatibility

**Temporal catalog shapes (no value regression ‚Äî values are identical or MORE faithful, never less).** A target created by v0.99.176 can render temporal types differently than one created by v0.99.175: a bare source temporal now stays bare on PG (previously `(6)`), an explicit `(0)` stays `(0)` (previously widened to bare), and SQLite‚ÜíMySQL temporals land `DATETIME(6)`/`TIMESTAMP(6)` (previously bare). The backup wire change is append-only with no format bump ‚Äî old backups restore byte-identically on the new binary. Two edges: (a) an upgraded backup chain may log a one-time phantom `alter_table` delta on the first DDL-free window post-upgrade (the type comparison against the old materialized `(6)`); the replay is a no-op and self-heals after one segment. (b) Downgrade caveat: an OLD binary restoring a NEW manifest cross-engine to MySQL emits bare `DATETIME` for precision-unspecified temporals (silent fractional-second rounding) ‚Äî forward compatibility is not guaranteed, so upgrade restore-side binaries first.

**MySQL sync cold-start default change.** Native-binlog cold-copies now run parallel (auto min(4, tables)) instead of serial. Consistency is identical (one FTWRL cut, one position); operators who deliberately relied on a serial cold-start ‚Äî e.g. to bound source read pressure ‚Äî should set `--copy-table-parallelism=1` (or DSN `copy_table_parallelism=1`). VStream and the trigger flavors are unchanged.

**UNIQUE constraint names now exist on PG targets.** `INSERT ... ON CONFLICT ON CONSTRAINT <name>` now works on migrated targets. Scripts or tooling that referenced the previously created unique-index names should switch to the constraint names (the underlying index still exists, owned by the constraint). MySQL/SQLite targets are unchanged.

**Everything else is drop-in.** `--analyze-after` is opt-in and default off; the shared exported snapshot is capability-gated with a loud fallback and changes no output; the replay buffering is behavior-neutral except throughput; the Bug 176 change only adds the code/exit-3 that v0.99.175 already documented. No flags were removed or re-defaulted beyond the cold-start change above.

## Who needs this ‚Äî action required

- **Anyone who ran a chain restore (or broker `--reset-target-data`) to a Postgres target with identity columns on v0.17.0‚Äìv0.99.175, where the chain carried incremental links past the base full:** if the restored target then took inserts, constrained tables failed loudly with 23505 ‚Äî but **keyless identity tables may carry silent duplicate ids: check them** (`SELECT id, COUNT(*) ... GROUP BY id HAVING COUNT(*) > 1`). Re-run the chain restore on v0.99.176, or manually `setval` each identity sequence to MAX(id)+1.
- **SQLite/D1‚ÜíMySQL migrations run on v0.99.141‚Äìv0.99.175 whose temporal values carry fractional seconds:** the migrated values were silently truncated to whole seconds on the target. Re-verify those columns; re-migrating on v0.99.176 lands `DATETIME(6)`/`TIMESTAMP(6)` with microseconds preserved.
- **PG targets whose source declared explicit `timestamptz(0)`/`time(0)`-style precision (any prior release):** the target carries the bare form, which accepts fractional seconds the source would reject. Tighten in place (`ALTER TABLE ... TYPE timestamptz(0)`) or re-run the schema migration.
- **Applications using `INSERT ... ON CONFLICT ON CONSTRAINT <name>` against a migrated PG target:** prior targets carry a unique index, not the named constraint. Re-run the schema migration on v0.99.176, or convert in place with `ALTER TABLE ... ADD CONSTRAINT <name> UNIQUE USING INDEX <index>`.
- **Operators who relied on serial MySQL sync cold-starts:** set `--copy-table-parallelism=1` (or the DSN param) before upgrading the sync fleet.
- **Agents/scripts branching on `SLUICE-E-EXPR-BACKSLASH-LITERAL` or exit 3 for the double-quoted-token refusal:** v0.99.175 exited 1 uncoded on that shape ‚Äî upgrade to restore the documented contract.
- Everyone else: nothing ‚Äî data-path behavior is drop-in.
