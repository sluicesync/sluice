# sluice v0.99.175

**Fidelity batch + agent-ready CLI. A new PG→PG restore-parity oracle (sluice's output diffed against `pg_dump|psql`) surfaced — and this release closes — three silent-fidelity classes: standalone Postgres sequences now carry as first-class schema objects with live positions (backup format v4), per-column `COLLATE` now lands on Postgres targets, and SQLite expression DEFAULTs no longer flatten into wrong literal values. Plus JSON result envelopes, stable `SLUICE-E-*` error codes with an exit-code taxonomy, and trigger-CDC prune hardening. Drop-in for all data paths; see Compatibility for the backup-format and exit-code notes.**

## Features

**Standalone Postgres sequences are now first-class schema objects.** Previously the PG reader collapsed every `nextval()` column default into an identity column: explicit `CREATE SEQUENCE` objects, shared sequences (one sequence feeding several tables), re-optioned serials, and app-driven sequences were silently not carried — a shared sequence became independent per-table identities (breaking cross-table uniqueness for future inserts), sequence options (INCREMENT/MIN/MAX/CACHE/CYCLE) were lost, and an application calling `nextval('myseq')` on the target failed because the sequence didn't exist. Now the reader classifies `nextval()` defaults three ways — identity columns stay identity, the exact factory-serial shape keeps the documented serial→identity modernization, and everything else is read into `ir.Schema.Sequences` with full options, `OWNED BY` binding, and current position — and the PG writer creates each sequence and primes its position **atomically in one transaction** before tables, so post-migration `nextval()` continues exactly where the source would. Re-primes are forward-only and direction-aware (never a rewind; an already-existing target sequence is re-primed, not silently skipped), chain restore re-primes at the chain tail from the newest link's end-of-window positions, and `cutover`'s prime walk now covers standalone sequences with the same margin-disciplined decision tree as identity columns. Cross-engine targets (PG→MySQL-family, PG→SQLite/D1) **refuse loudly** naming the sequence and the remedy — sluice never silently drops a sequence again. Backups whose schema carries standalone sequences stamp the new backup **FormatVersion 4** so pre-v0.99.175 binaries refuse-before-touch instead of restoring sequence-less (see Compatibility). Pinned by value-level ground truth (the first post-migration insert continues at the source's next value, across migrate, resume, chain restore, and cutover) and the full option/ownership matrix. Affected: every release before v0.99.175, for any PG source using non-factory-shape sequences.

**JSON result envelopes and JSON dry-run plans (`--format json` on the primary verbs).** `migrate`, `sync start`, `backup full`, and `restore` can now emit exactly one terminal JSON envelope on stdout — status `completed`/`refused`/`failed`, elapsed time, engine names, per-table stats, a resume affordance, the error message, and next-step suggestions — and `migrate --dry-run --format json` / `sync start --dry-run --format json` emit the plan as structured JSON instead of text. Text mode is byte-identical to before; slog stays on stderr; every registered DSN is scrubbed from the rendered envelope. Built for scripts, pipelines, and AI agents driving the CLI (a new top-level `AGENTS.md` documents the full agent operating surface: command taxonomy, env-first credentials, the dry-run-first workflow, and the envelope/plan/log/error-code map).

**Stable error codes (`SLUICE-E-*`) and an exit-code taxonomy.** Every error class that already carries an operator hint now also carries a frozen machine-matchable code (`SLUICE-E-<DOMAIN>-<SLUG>`, e.g. `SLUICE-E-INDEX-STATEMENT-TIME-LIMIT`, `SLUICE-E-VALUE-ZERO-DATE`), surfaced as `code`/`hint` attributes on the terminal `--log-format json` ERROR record and lifted into the JSON envelopes. Exit codes gain a taxonomy: 0 success and 1 generic failure are unchanged; **2** = config-file load/parse error; **3** = named refusal (sluice declined and named the remedy — retrying without acting on the hint will fail identically); 80 = kong usage error, as before. The registry and taxonomy are documented in `docs/operator/error-codes.md` and test-enforced against the code in both directions.

**PG→PG restore-parity oracle (roadmap item 51).** A new integration harness migrates a kitchen-sink schema with sluice and restores the same schema with version-matched `pg_dump|psql`, then compares `pg_dump --schema-only` of both targets as statement sets keyed by object identity — every divergence must match a cited allowlist entry or fail the run. Its first run produced the sequence, COLLATE, and default-rendering findings this release fixes; it now runs in CI as a permanent latent-gap detector. Enum, domain, EXCLUDE, stored generated columns, FK actions, partial/expression indexes, and COMMENTs verified at full parity.

## Fixed

**SQLite/D1 expression DEFAULTs no longer flatten into silently wrong literal values (silent-loss class; affected v0.99.141–v0.99.174).** SQLite's `PRAGMA table_info` strips the outer parentheses off a parenthesised expression default, so `DEFAULT ('a' || 'b')` is reported as `'a' || 'b'` — text that merely starts and ends with a quote. The reader's endpoint check swallowed that as the literal string `a' || 'b`, which landed on any target as a mangled string default with exit 0, routing around every expression-boundary refusal. The reader now runs a well-formedness lexer over candidate string literals — anything malformed classifies as an expression, never a guessed literal — and the PG/MySQL writers route portable expression bodies through the v0.99.174 SQLite→canonical translator, so `DEFAULT ('a' || 'b')` now **evaluates** to `ab` on the target (integration-proven insert-and-read on real PG and MySQL). The independent review closed five sibling classes in the same pass: MySQL's verbatim-carry fallback no longer forwards parseable-but-divergent bodies (`upper('a') || 'b'` parsed as logical OR and landed the default as `0`; `7/2` landed `3.5` vs SQLite's `3`; `7.5 % 2` landed `1.5` vs `1`) — anything outside a small proven-faithful residue set is now warn-dropped with the table.column named; `0x…` hex literals are refused to MySQL (MySQL reads them as a binary string — bytes `\x1a` where SQLite stores `'26'`); the SQLite writer re-parenthesises expression defaults, fixing a loud syntax error on every functional default written to a SQLite/D1 **target** (affected v0.99.146–v0.99.174); bare `TRUE`/`FALSE` on a declared-boolean PG column now translates faithfully (on a non-boolean column it stays warn-drop, keeping the PG DEFAULT arm's never-abort contract); and the MySQL reserved-word requote no longer corrupts content inside a carried double-quoted residue (`DEFAULT "order"` previously got interior backticks injected inside the string — silent-wrong). Column DEFAULTs are non-data metadata: **migrated rows were never affected** (they carry explicit values); the wrong value applied only to rows inserted on the target after migration.

**Per-column `COLLATE` now carries to Postgres targets, and cross-engine collation handling is a named WARN policy (silent-fidelity class; affected effectively every prior release).** Per-column collation has been read into the IR since the v0.11 line, but the PG DDL emitter never rendered a `COLLATE` clause — so `region_code text COLLATE "C"` landed bare on a PG→PG migrate, silently changing sort/comparison order and any collation-dependent index or constraint behaviour. The PG emitter now renders `COLLATE "<name>"` for PG-dialect collations (only explicit non-default collations are stored by the reader, so ordinary columns stay clause-free, mirroring pg_dump), including on the schema-forward ALTER path and through domain-typed columns. Cross-engine, collation names do not translate — the policy is now explicit and loud: a foreign-dialect collation is **dropped with a WARN** naming the table, columns, and collations. This also fixes a latent PG→MySQL hard failure: a PG-dialect collation forwarded verbatim died mid-create-tables with raw `Error 1273 "Unknown collation: 'C'"` — now a named WARN-drop. Ground-truthed by an integration pin asserting the target's `attcollation` and that `ORDER BY` matches the source's byte order on a seed that sorts differently under `en_US`.

**MySQL DDL string emission is now backslash-safe, double-quoted SQLite tokens are refused to MySQL, and MySQL ENUM labels decode correctly at the read boundary (SEC-1; silent classes).** Three related closures. (1) `quoteSQLString` emitted backslash-bearing DEFAULT literals, ENUM/SET labels, and column/table COMMENTs undoubled, so MySQL's lexer silently decoded them — and a trailing backslash swallowed the closing quote (the DDL-position escape class); it now doubles backslashes keyed off the session `sql_mode` (skipped under `NO_BACKSLASH_ESCAPES`, where doubling would itself corrupt). This has been latent since the MySQL writer's inception and matters only for values actually containing backslashes. (2) Ground-truthing on real MySQL 8.0 revealed that ENUM/SET `COLUMN_TYPE` arrives **escaped** from information_schema, so the reader now decodes doubled backslashes at the read boundary — fixing a latent MySQL→PG doubled-enum-label corruption (MySQL→MySQL previously round-tripped only by accidental escape symmetry). (3) The SQLite→MySQL expression path now **refuses** backslash-bearing string literals and any double-quoted token (SQLite's string misfeature tokenizes as an identifier, sails to MySQL, and default sql_mode re-lexes `"…"` as a string literal with escape semantics — silent value corruption in generated-column position, a vacated predicate in CHECK position); both refusals name the construct and the remedy, and index expressions WARN-skip. Generated-column/CHECK carry positions were affected v0.99.144–v0.99.174; DEFAULT positions v0.99.141–v0.99.174.

**Trigger-CDC change-log pruning no longer runs one monolithic DELETE, and the D1 poll batch is clamped to the response cap.** A single `DELETE … WHERE id <= cut` over a large backlog stalls the single-writer SQLite source for its whole duration, and on D1 can blow the per-query CPU ceiling — after which the failed tick retried an even **bigger** delete next interval, so the v0.99.174 `--auto-prune-change-log` sidecar could fall behind permanently on D1 (affected: v0.99.174 for the sidecar; the operator-run `sluice trigger prune` CLI, v0.99.151–v0.99.174, shared the monolithic-DELETE stall class). Both trigger engines now reap in bounded keyset batches (20k local/PG, 2k on D1) with a 30-second per-tick time budget on the sidecar path (a partial tick resumes next cadence), the per-tick `COUNT(*)` full scan is replaced with rows-affected arithmetic plus an hourly true recount, and pgtrigger's sidecar reuses one pooled connection across ticks. The cut-watermark invariant is unchanged and now pinned across every batching state: no DELETE's upper bound ever exceeds the durably-applied frontier, so not-yet-applied rows survive every intermediate step. Separately, the `d1-trigger` poll batch is clamped from 10,000 to 1,000 rows (affected v0.99.149–v0.99.174): change rows carry full before/after images and a 10k catch-up poll could overflow Cloudflare's ~1 MB `/query` response cap; catch-up throughput is preserved via the full-batch fast-repoll keyed off the clamped size.

**Security and operations hygiene.** Crash/operator bundles (`sluice diagnose`) now mask the bare-secret CLI flag family — `--planetscale-metrics-token`/`-id`, `--notify-webhook`, `--notify-slack`, `--notify-smtp-password`, `--encryption-passphrase` — which previously survived verbatim into bundle args (DSN-bearing flags were already redacted). The D1 HTTP client path-escapes account/database ids into the query URL and bounds response reads at 8 MiB; SMTP notification config now warns loudly when plain/login credentials are configured with `--notify-smtp-tls=none`. The backup chain-compact orphan sweep now surfaces per-file delete failures in its WARN (previously discarded, so a persistently failing delete leaked backup-store disk with zero diagnostic trail). And a generalized CI run-filter coverage guard rescued four integration tests that compiled clean but ran in **no** workflow (including the purged-GTID reactive-cold-start pin) — a test-integrity fix, no user action.

## Compatibility

**Backup format v4 (sequence-bearing backups only).** A backup whose schema carries standalone sequences stamps `FormatVersion: 4`; a pre-v0.99.175 binary **refuses it loudly before touching the target** (the Bug-116 refuse-before-touch contract) rather than restoring sequence-less. Stamping is proportional: a backup without standalone sequences keeps its previous format version and remains fully readable by older binaries. If your fleet mixes binary versions, upgrade restore-side binaries before taking backups of sequence-bearing PG schemas.

**Exit codes.** Scripts checking `exit != 0` (including systemd `Restart=on-failure`) are unaffected — every failure class is still non-zero. Scripts that branch on `exit == 1` specifically should be updated: config-file errors and named refusals that previously exited 1 now exit 2 and 3 respectively. `verify`/`diff`/`sync-health` keep their long-standing per-command exit contracts unchanged.

**SQLite expression-default behavior change.** Expression defaults that previously landed as mangled literals now translate correctly, warn-drop, or refuse loudly. Backups taken with a pre-fix binary carry the mangled default **as captured at backup time** and cannot be retro-repaired; after upgrading, the corrected schema rendering can surface as a schema-hash change (drift → re-snapshot) — that is the fix landing, not corruption.

**New loud refusals (each with a named remedy).** PG→MySQL/SQLite migration of a schema with standalone sequences, which previously "succeeded" by silently collapsing them, now refuses naming the sequence. SQLite expressions bound for MySQL now refuse backslash-bearing string literals, double-quoted tokens, and `0x…` hex literals. PG-dialect collations on a MySQL target now WARN-drop instead of hard-failing with Error 1273 (loud→loud, but no longer aborts the run). `--format json`, the error codes, and the JSON envelopes are purely additive; text output is byte-identical.

## Who needs this — action required

- **SQLite/D1 migrations run on v0.99.141–v0.99.174 whose schemas carry parenthesised expression DEFAULTs:** inspect the target's column defaults — a `DEFAULT ('a' || 'b')`-shaped expression may have landed as a wrong literal string (your migrated **rows** are correct; only rows inserted on the target after migration used the wrong default). Fix the default in place or re-run the schema migration on v0.99.175. Pre-fix backups carry the mangled default as captured and cannot be retro-repaired.
- **PG-target migrations (any prior release) whose source uses explicit per-column `COLLATE`:** check the target's column collations (`attcollation`) — they landed as the database default, silently changing sort/comparison order. Re-apply the collation on the target (and rebuild any indexes on those columns), or re-run the schema migration.
- **PG sources (any prior release) using standalone, shared, or re-optioned sequences:** targets migrated earlier are missing those sequence objects (app `nextval()` calls fail loudly, and a shared sequence became independent per-table identities). Recreate and prime the sequences on the target, or re-migrate the schema on v0.99.175.
- **MySQL sources with backslashes in ENUM/SET labels, string DEFAULTs, or comments (especially MySQL→PG):** re-verify those schema strings on the target — labels may carry doubled backslashes and defaults/comments may have been silently backslash-decoded.
- **v0.99.174 users running `--auto-prune-change-log` against a D1 source:** upgrade — the monolithic prune DELETE could blow D1's CPU ceiling and fall behind permanently. No data-loss exposure (the prune cut never exceeded the applied frontier), but the change log stops shrinking until upgraded.
- **Anyone who shared a `sluice diagnose` bundle generated with a token/webhook/passphrase passed as a CLI flag:** those values were captured verbatim in the bundle's args — rotate them.
- **Script authors branching on `exit == 1`:** account for the new exit codes 2 (config error) and 3 (named refusal). `!= 0` checks need no change.
- Everyone else: nothing — data-path behavior is drop-in.
