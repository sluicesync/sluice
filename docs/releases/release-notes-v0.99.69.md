# sluice v0.99.69

**New: the `sluice sync` / VStream cold-copy now writes multiple tables to the target CONCURRENTLY — ADR-0099's K read streams become K end-to-end read→write pipelines, removing the serial-write consumer that was the missing throughput lever (ADR-0100).** Opt-in and drop-in: it engages on the existing source-DSN knob `vstream_copy_table_parallelism=K` (no new knob); absent / `0` / `1` resolves to K=1 and runs the existing serial table loop byte-identically, so existing syncs are unchanged. This is the write-side companion that makes the read-side concurrency (ADR-0099, v0.99.67) and the per-table write fan-out (ADR-0097, v0.99.65) actually multiply: each gave little alone (~1.0× / ~1.4×) because the orchestrator still drained one table at a time — this release makes the write side concurrent across tables, which combined with the per-table fan-out degree D is what enables multiplicative scaling (W×D total write concurrency) on a cross-region PlanetScale-MySQL cold-copy. Connection-budget caveat to read first: W = K is **not** auto-clamped against the target connection budget in v1 — you are responsible for keeping `K × D ≤ --max-target-connections` (D = the ADR-0097 `--copy-fanout-degree`); sluice emits a WARN at copy start naming K and stating that contract.

## Features

- **Cross-table VStream cold-copy WRITE concurrency — K end-to-end read→write pipelines (ADR-0100).** ADR-0099 (v0.99.67) made the VStream cold-copy READ side concurrent — K independent vtgate VStreams, each over a disjoint subset of the in-scope tables — but the orchestrator's bulk-copy loop (`runBulkCopyWithOpts`) still drained those tables ONE AT A TIME through a single serial consumer, so only one table was ever written at a time. Live Track-B measurement showed the target PROCESSLIST receiving rows for exactly one table at a time (28 of 28 polls over 90 s, never two concurrent) and the path held at the same ~1.4× ceiling ADR-0097's write fan-out alone reached — K (reads) and D (per-table fan-out) are orthogonal levers on a serial-write workload, so their product was capped by the serial consumer. This release turns ADR-0099's K read streams into **K end-to-end read→write pipelines**: the engine surfaces the EXACT disjoint table partition it gave the producers via a new optional reader capability (`ir.ConcurrentCopyPartitioner`), and the pipeline runs W = K consumer goroutines (one per group, joined by an errgroup), each looping its group's tables serially through the SAME per-table copy helper the serial path uses (`copyTableColdStartIdempotentMaybeParallel`) — so the ADR-0097 D-way write fan-out composes per table and the total write concurrency is **W × D** (W tables written concurrently, each through a D-way fan-out). Multiple tables now receive rows on the target at once instead of one-at-a-time, removing the serial-consumer ceiling that pinned ADR-0099 + ADR-0097 at ~1.4×; bounded by source/target/network capacity and the `W × D` connection budget, not by sluice's consumer. (We deliberately quote no headline speedup figure: the live Track-B re-measurement runs after this ships — the mechanism is the multiplicative write concurrency D and K could not deliver alone.) W = K reuses the existing `vstream_copy_table_parallelism` knob — **no new knob, no new default to mis-invert** (the v0.99.51 trap avoided by tying to the existing zero-value-safe knob), and the single shared `*sql.DB`-pooled writer is concurrency-safe for W × D pinned-connection callers, so no per-pipeline writer factory is introduced.
- **Correctness — four silent-loss-class invariants, each pinned and value-fidelity reviewed.** (a) *Exactly-once:* the partition is disjoint, so every in-scope table is written by EXACTLY ONE pipeline — none dropped (a silently un-copied table), none double-written; the consumer reads the SAME groups the producers used, so ADR-0099's unit-pinned coverage + disjointness is inherited, never re-derived, and a per-table write-count == 1 pin backs it. (b) *Position-after-all:* the global CDC position advances only after BOTH the W-way write join (the consumer errgroup) AND ADR-0099's K-way producer join (the read/stitch barrier) precede the streamer's `stream.Position` read — there is no per-pipeline position write, so CDC never starts past an incompletely-copied table (ADR-0007). (c) *Mid-COPY checkpoint disabled:* the durable-progress checkpoint stays OFF on the concurrent path by two independent code facts (the concurrent engine pump records no mid-COPY breadcrumb, and the pipeline does not wire the durable-progress reporter on this branch), so no breadcrumb can checkpoint past an un-written early row under concurrent writers (the ADR-0097 §3 silent-loss class) — unit-pinned that no checkpoint sink fires during a concurrent copy. (d) *Stable partition:* the partition is a deterministic, shuffle-invariant pure function of the (sorted) table set and K, so cold-start and resume re-derive the IDENTICAL groups — composing with the ADR-0098 auto-shard-aware resume per pipeline (the in-progress table resumes past its cursor, the rest re-copy idempotently). Any pipeline's error aborts the whole copy loudly (errgroup cancels its peers, no position advance, no partial silent success); ctx-cancel joins all W consumers with no leaked goroutines or pinned connections. Pinned by a dispatch-observer test (≥2 groups → W-goroutine path; nil / 1 group / not-implemented → serial loop byte-identically), the load-bearing write-overlap pin (a recording writer asserts ≥2 tables' write-windows OVERLAP — the exact thing the PROCESSLIST showed missing; it FAILS on the serial loop and PASSES on the pipeline path), the position-only-after-all-pipelines pin, the exactly-once per-table-write-count pin, the loud-abort and ctx-cancel-no-leak pins, the mid-COPY-checkpoint-disabled pin, and a `vstream`-tagged `-race` integration test plus a resume × concurrent-write pin (`TestVStream_AutoShardResume_ConcurrentWrite_*`) composing ADR-0098 resume with ADR-0100 W>1 write concurrency end-to-end.

## Compatibility

- **No breaking changes; drop-in upgrade.** Write-side cross-table concurrency engages on the existing source-DSN knob `vstream_copy_table_parallelism=K`. Absent the knob (or `0` / `1`) the cold-copy takes the serial table loop exactly as v0.99.67/v0.99.68 did — no new goroutine, no errgroup, the byte-identical pre-ADR-0100 path (the engine surfaces no concurrent groups at K=1, so the pipeline never enters the concurrent branch). There is no new flag and no change to the persisted handoff/position token shape (no new field).
- **Connection budget: W = K is NOT folded into the target-connection preflight in v1 — operator responsibility.** During the concurrent copy the target sees `W × D` connections (W = K group consumers, each driving a D-way ADR-0097 write fan-out on its active table), plus K source-side gRPC streams (one HTTP/2 connection) and the one CDC connection opened later. W = K is a *source*-DSN knob parsed inside the mysql engine, invisible to the pipeline-layer connection-budget preflight (which resolves only D), so v1 cannot clamp the `W × D` product yet — a clean cross-layer follow-on. Consistent with ADR-0099 §3's honesty fix, sluice emits a **WARN naming K** at copy start and states that **`K × D ≤ --max-target-connections` is the operator's responsibility**; the D-axis preflight still fires its loud refusal / `--max-target-connections` cap exactly as before. No clamp is claimed that isn't implemented (the loud-failure / honesty tenet); since W = K, the demand is the same `K × D` product ADR-0099 already named.
- **Resume must use the same K (operator contract).** The partition is a pure function of `(sorted tables, K)`, so a resume with a different K re-derives a different partition; v1 requires resuming with the same `vstream_copy_table_parallelism` the cold-start used (a changed-K resume is "restart the cold-start", not silently honored) — the same contract ADR-0099/ADR-0098 already document. The W-goroutine consumer itself is resume-agnostic: it reads the same surfaced groups, so resume exactly-once is structurally inherited from the deterministic partition plus the producer-side cursor seeding ADR-0098/0099 already cover.
- **Scope: the `sluice sync` / VStream (Vitess/PlanetScale-source) cold-copy only.** The vanilla MySQL binlog, PostgreSQL cold-copy, the ADR-0079 fast shareable-snapshot path, and the `sluice migrate` path (its own within-table parallel chunking, ADR-0096/ADR-0019) are all untouched. A single-table scope is already single-stream / single-consumer and unaffected. Readers that don't implement the new optional capability (Postgres snapshot, MySQL binlog snapshot, single-stream VStream) keep the serial cold-start loop.

## Who needs this — action required

- **Anyone running `sluice sync` from a Vitess/PlanetScale source over a large multi-table keyspace into a cross-region target who wants a faster cold-copy:** set `vstream_copy_table_parallelism=4` (or higher) in the source DSN. This is the lever that was missing — on its own, K (cross-table reads, v0.99.67) and D (per-table write fan-out, v0.99.65) each gave little because the write side was serial; with this release the write side is concurrent across tables, so K and D now multiply (W × D). Start at 4 and raise it for a known-large cross-region copy, and mind the connection budget — keep `K × D ≤ --max-target-connections` (D is your `--copy-fanout-degree`, default 4), because v1 does not auto-clamp K/W; the WARN at copy start names K and restates the contract. Resume the job with the same K you started it with.
- **No re-verification of prior syncs is required.** This is a throughput optimization to *how fast* the cold-copy runs, not a fix to a path that produced wrong or missing data. The serial-consumer cold-copy on v0.99.67/v0.99.68 was correct and gapless; completed syncs on prior versions are unaffected and need no re-check, and K=1 (the default) reproduces that serial path byte-for-byte.
