# sluice v0.99.178

**New: two WAN round-trip amortization wins (roadmap item 54). The MySQL bulk-load path now composes ~1 MiB INSERT statements (ADR-0150) — the pscale-dumper's battle-tested size — instead of flushing every 500 rows, cutting statements/round trips 10–20× on narrow rows into PlanetScale/Vitess (the flavor with no LOAD DATA, where batched INSERT IS the bulk path). And the live-D1 reader now prefetches page N+1 while page N streams (ADR-0151), hiding one HTTP round trip per 1000-row page on Cloudflare D1 imports. Values and requests are byte-identical in both; drop-in, no action required.**

## Features

**Byte-targeted ~1 MiB INSERT batching on the MySQL bulk-load path (ADR-0150).** The batched-INSERT bulk loops (plain + idempotent, shared by the parallel fan-outs) flushed on a fixed 500-row cap, producing 50–100 KB statements on narrow rows — and on Vitess/PlanetScale, which has no LOAD DATA, that path IS the bulk-load path, so WAN imports paid 10–20× more round trips than necessary. A shared batch composer now accumulates rows to an estimated ~1 MiB statement byte target (the pscale-dumper statement size, deliberately equal to the CDC applier's ADR-0139 coalescing constant), with the row cap demoted to a 10,000-row safety ceiling and a 60,000-placeholder bound protecting MySQL's 16-bit prepared-statement parameter limit; a single row whose estimate alone exceeds the target ships as a one-row statement (never split, never refused — `max_allowed_packet` stays the loud upper bound). Statement composition is the ONLY thing that changes: every value still binds to a `?` through the same `prepareValue` codec, so the wire encoding of each value is byte-identical (the Bug-74 safety property), and each flush remains one autocommit statement = one transaction, so Vitess's ~20 s transaction killer, the AIMD controllers, connection budgets, grow-gate, and reparent-retry are all untouched. The change reaches every mode that flows through the batched paths — migrate (PlanetScale/Vitess always; vanilla MySQL on the LOAD DATA fallback and all idempotent/resume/chunked writes), sync cold-start COPY, and the restore/chain-restore/broker full-restore legs — while LOAD DATA, CDC steady-state apply (already 1 MiB via ADR-0139), and the other engines' write paths are byte-identical. Companion hint: when the bulk path engages against a hosted-PlanetScale target, one INFO per writer notes that writes are tier-CPU-bound, not connection-bound (ADR-0116) — a larger tier or Metal is the lever, not more parallelism; self-hosted `vitess` is excluded. Pinned by unit tests (byte target primary, ceilings, placeholder clamp, oversize single row, flush-boundary exactness, ≥10× fewer statements vs the 500-row baseline) plus real-MySQL integration legs (value-matrix round-trip through >500-row composed statements; a >1 MiB row ships alone byte-exact).

**Single-page prefetch on the `d1` query-API reader (ADR-0151).** The live-D1 reader keyset-paginates over HTTP in 1000-row pages, and the loop was strictly sequential — fetch page N, decode, stream, THEN request N+1 — even though page N's last key (the only input N+1 needs) is known the moment N arrives, so every page paid one avoidable WAN round trip of dead time. A single fetcher goroutine now issues page N+1's request while page N's rows decode and stream, handing pages over an unbuffered channel: exactly one page of read-ahead (memory bounded at ≈ one extra page), never an N-deep pipeline. The contract is preserved and pinned: requests are byte-identical to the sequential loop (same SQL, same exact-text string bounds including the > 2^53 discipline), a failed prefetched page surfaces in sequence after the prior page's rows, a ctx-cancelled fetch reports the cause via `Err()` instead of a clean silently-truncated read (a hazard the naive handoff would have introduced — named and pinned), the fetcher is reaped on every return path (no goroutine leak), and the LIMIT/OFFSET fallback prefetches through the same loop. Reaches both D1-source read paths — `migrate --source-driver d1` and the `d1-trigger` flavor's snapshot leg. This is RTT-hiding only, not within-table chunking (that remains a deferred follow-up).

**Development/CI (dev-facing).** The weekly fuzz round-trip harness gains the four supported SQLite directions (sqlite→postgres/mysql, postgres/mysql→sqlite) with storage-class-aware value domains and a central expected-outcome shield for the documented ADR-0134 emit refusals, and its cross-engine oracle is tightened: a row-count skew in the no-faithful-columns regime now FAILS (previously uncompared for all cross-engine directions — the tightening benefits the existing PG↔MySQL directions too). Test-integrity only, no released-binary behavior change. The pre-commit hooks also self-heal golangci-lint's stale-worktree phantom-finding cache (clean + retry once).

## Fixed

**Latent >65,535-placeholder failure on very wide tables at the old 500-row cap (loud failure; latent in every release since v0.1.0, never reported in the field).** The batched-INSERT path bound rows × columns placeholders per statement with no parameter bound, so a full 500-row flush on a table with ≥132 non-generated columns exceeded MySQL's 16-bit prepared-statement parameter limit and the server refused the statement — a loud abort, not silent loss, and only reachable on the batched path (PlanetScale/Vitess bulk, MySQL idempotent/resume writes). The new `maxBulkInsertPlaceholders = 60,000` bound clamps the row ceiling per table so any column count now composes valid statements. Nothing to re-verify: the old behavior failed loudly with exit ≠ 0. Pinned by `TestInsertBatcher_PlaceholderBoundClampsWideTables`.

## Compatibility

**No breaking changes; drop-in.** No flags were added, removed, or re-defaulted. Two behavior notes: (1) MySQL-family targets now receive fewer, larger INSERT statements (~1 MiB, up to 10k rows / 60k placeholders each) — still one statement per autocommit transaction, so timeout budgets, the durable-progress watermark, and the ADR-0108 retry proof (a failed batch was always a single atomic multi-row INSERT) are unchanged, and a batch retry now simply carries more rows; (2) `--max-buffer-bytes` on the MySQL batched bulk path now only LOWERS the effective per-statement byte target — a value above 1 MiB leaves the target in charge (previously moot: the 500-row cap always bound first), and the applier-side ADR-0028 byte caps are untouched. The D1 reader holds at most one extra page (~1000 rows) in memory and its HTTP requests are byte-identical. PG binary COPY, vanilla-MySQL LOAD DATA, SQLite writes, CDC steady-state apply, and every non-D1 read path are byte-identical to v0.99.177.

## Who needs this — action required

- **No one must act.** No silent-loss fixes in this release — nothing to re-verify, no counts to re-check.
- **PlanetScale/Vitess-target imports (migrate, sync cold-start, restore/chain-restore):** automatic 10–20× round-trip amortization on narrow rows; watch for the once-per-run tier-CPU-bound INFO on hosted PlanetScale targets — if you were adding copy parallelism to speed up writes, a larger tier or Metal is the effective lever.
- **Live Cloudflare D1 imports (`--source-driver d1`, `d1-trigger` cold start):** one HTTP round trip per page is hidden automatically.
- **Operators with ≥132-column tables on MySQL-family batched writes:** flushes that previously aborted on the prepared-statement parameter limit now compose within the bound.
