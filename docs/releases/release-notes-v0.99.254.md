# sluice v0.99.254

**New: `sluice deploy-ddl` + `sluice control-tables ddl` (ADR-0165, roadmap item 66) — ship one verbatim DDL statement to a PlanetScale production branch through the governed deploy-request channel with the full ADR-0162 safety wrapper, and bootstrap sluice's own control tables on a safe-migrations branch as a documented per-statement one-liner.** The companion fixes make the bootstrap story true end to end: Error 1105 "direct DDL is disabled" at every mysql control-table DDL site is now the coded refusal `SLUICE-E-PS-DIRECT-DDL-BLOCKED` (echoing the exact refused statement, deploy-ddl-pasteable), and the sync applier's three control-table ensures are detect-then-create so a bootstrapped `sync` actually starts cleanly. Purely additive drop-in — no flag, default, or data-path changes.

## Features

- **`sluice deploy-ddl` — one verbatim `--ddl` statement, shipped safely as one command.** The flow is the full ADR-0162 wrapper: preflight (token/org/database/branch + the safe-migrations prerequisite, refused coded, never auto-enabled) → dev branch with the **stale-base freshness gate** (a fresh PlanetScale branch can silently propose *reverting* recent production schema; sluice's schema-compare + one-shot backup-rebase is currently the only implementation of that guard anywhere) → apply the DDL on the branch → deploy request → deploy → skip-revert finalize → always-cleanup. The dev-branch name is deterministic on the DDL alone (`sluice-ddl-<10-hex>`), preserving the refuse-on-leftover crash story; the tolerant deploy-request poller and the three ADR-0162 codes (`SLUICE-E-PS-{SAFE-MIGRATIONS-DISABLED,DEPLOY-REQUEST-FAILED,BRANCH-STALE-BASE}`) are reused unchanged — no new failure classes. `--dry-run` prints the plan with **zero control-plane calls** (pinned), and the command has no data plane at all, so a dry run touches nothing. This replaces five hand-driven pscale commands plus a hazard the operator can't see; it is deliberately NOT raw deploy-request CRUD (the pscale CLI owns that) — the differentiated piece is the safety wrapper.

- **`sluice control-tables ddl` — the control-table bootstrap printer.** Prints the five-table bootstrap set (`sluice_migrate_state`, `sluice_migrate_table_progress`, `sluice_cdc_state`, `sluice_cdc_schema_history`, `sluice_shard_consolidation_lease`) via the new optional `ir.ControlTableDDLProvider` engine surface, **single-sourced**: the engine returns the same strings its own Ensure\* paths execute, pinned byte-identical by test. Credential-free and read-only by design (a separate printer, not a deploy-ddl flag, so its output composes with any channel — deploy-ddl per statement, the PlanetScale UI, a reviewed migration file); `--engine` defaults to `planetscale` and the whole mysql family prints the same dialect. The safe-migrations bootstrap is now a documented three-command story: `sluice control-tables ddl` → `sluice deploy-ddl --ddl '<statement>'` per printed statement → run `backfill`/`sync`/`migrate` normally. See the new section in `docs/managed-services.md`.

- **Internals: the expand-contract deploy-leg machinery is extracted into a shared `legRunner`** (branch provisioning + freshness gate + rebase backup, the tolerant poller, skip-revert finalize, cleanup) that expand-contract's two legs and deploy-ddl's single leg compose — never fork. Behavior-preserving: all prior expand-contract pins pass unchanged.

## Fixed

- **Error 1105 at every mysql control-table DDL site is now the coded refusal `SLUICE-E-PS-DIRECT-DDL-BLOCKED` instead of a raw driver error (every prior release surfaced the raw 1105; loud-failure UX fix, zero data risk).** On a PlanetScale branch with safe migrations enabled, direct DDL is refused — including sluice's own control-table CREATEs and column-migration ALTERs — and until now that surfaced as an uncoded `Error 1105: direct DDL is disabled` with no named way out. Every site (the bootstrap CREATEs, the `sluice_keysets` and `sluice_target_metrics_history` CREATEs, every detect-then-ALTER column migration, and the item-65a LONGTEXT widen) now classifies it into the new ClassRefusal code whose message names safe migrations, points at the `control-tables ddl` → `deploy-ddl` bootstrap path, and echoes the exact refused statement (whitespace-collapsed, pasteable straight into `deploy-ddl --ddl`); `ErrSafeMigrationsBlocked` stays in the chain for `errors.Is` callers.

- **The sync applier's three control-table ensures (`sluice_cdc_state`, `sluice_cdc_schema_history`, `sluice_shard_consolidation_lease`) are now detect-then-create, so a bootstrapped `sync` actually starts on a safe-migrations branch.** They were bare `CREATE TABLE IF NOT EXISTS` — and PlanetScale refuses the *statement* whether or not the table exists, so even after shipping the tables via deploy-ddl, the next `sync` start would still 1105 on the no-op CREATE. All three now probe `information_schema` first and issue zero DDL when the tables are current (pinned per table), mirroring the migrate-state store's v0.99.248 gate. Affected releases, honestly stated: the bare-CREATE ensure shape predates this release by essentially the project's whole history (the cdc-state ensure dates to the earliest releases; the migrate-state side got the same gate in v0.99.248), but it was only ever reachable as a failure on a safe-migrations PlanetScale target branch — a loud startup error, never silent, and a workflow that only became supported with ADR-0162 (v0.99.248). Nothing to re-verify anywhere.

## Compatibility

- **Purely additive; no breaking changes.** Two new commands (`deploy-ddl`, `control-tables ddl`), one new coded refusal, one new optional engine surface; no flag, default, or behavior change to any existing command, and no data-path changes anywhere.
- **The detect-then-create conversion is behavior-preserving:** zero DDL when the tables are current, at the cost of up to three extra `information_schema` lookups per applier start (once per start). The `legRunner` extraction leaves expand-contract's public behavior unchanged (all prior pins pass; two operator-guidance strings were generalized from "the %s DDL" to "the DDL" to share the template).

## Who needs this — action required

Nothing to re-verify — this release contains no silent-loss fix; both fixes convert loud-but-cryptic failures into coded, remedy-bearing ones. Pick it up if you run sluice against a PlanetScale branch with safe migrations enabled: the first `backfill`/`sync`/`migrate` against such a branch previously died on a raw Error 1105 with no named way out, and now the refusal itself walks you through the bootstrap (`sluice control-tables ddl`, then `sluice deploy-ddl --ddl '<statement>'` per table, then run normally). It is also the general escape hatch for any ad-hoc schema change on a safe-migrations branch — the freshness gate, tolerant poller, and cleanup apply to whatever single statement you pass. Everyone else: nothing to do.
