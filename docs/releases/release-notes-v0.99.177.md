# sluice v0.99.177

**New: `backup full` now splits large tables into parallel PK-range reads (ADR-0149) — a Postgres-source backup dominated by one huge table no longer runs at single-connection speed. Every range reader is pinned to the SAME exported snapshot, so consistency is identical to the single-reader sweep; the chunk-file layout and manifest are byte-compatible with every prior release. On by default (auto), drop-in, `--bulk-parallelism=1` is the opt-out.**

## Features

**Within-table read chunking for the full-backup row sweep (perf-parity gap #1, roadmap item 54).** The backup row sweep got its cross-table axis in v0.86 (ADR-0084), but each table still streamed through one full-scan reader — so a corpus dominated by one huge table backed up at single-connection speed while `migrate` split the very same table into PK ranges. `backup full` now reuses migrate's chunk machinery verbatim inside the backup table pool: an eligible table's worker plans N disjoint half-open PK ranges pre-stream (single integer PK → MIN/MAX divide; other orderable/composite PKs → sampled keyset) and streams them concurrently, each range paged with a SQL-side inclusive upper bound (the ADR-0096 exactly-once contract — no row missed, no row doubled at range seams). Every additional range reader is minted against the SAME exported snapshot through the pool's existing importer factory, so the backup's point-in-time consistency is identical to the single-reader sweep by construction. Eligibility is deliberately narrow — shareable-snapshot source path (Postgres lazy-import), chunkable PK, estimated rows above threshold; MySQL's coordinated-FTWRL readers, the non-snapshot fallback, SQLite, and the PlanetScale/Vitess VStream-COPY path stay single-stream per table, with one loud INFO at run start naming the reason (each is a cited deliberate absence in the perf-parity matrix, not an accident).

**New flags: `--bulk-parallelism` and `--bulk-parallel-min-rows`, composed with `--table-parallelism` under one budget.** Both mirror migrate's flags: `--bulk-parallelism` 0 (default) = auto min(8, NumCPU) then budget-split, 1 = off; `--bulk-parallel-min-rows` 0 (default) = auto (base 80000, dialled down on many-table schemas, floored at 10000). The table × within-table product is bounded by the source's connection budget through one run-wide token gate with tail work-stealing — finished tables' tokens are stolen by a big table's surplus ranges, so the backup tail collapses to full-width reads on whatever remains. The budget split satisfies the cross-table axis FIRST (swapped relative to migrate), so the resolved `--table-parallelism` is byte-identical to prior releases and a single-huge-table corpus hands the within axis the entire remaining width. `--chunk-size` keeps its meaning — rows per chunk FILE (the roll boundary), orthogonal to how many readers produced the files.

**The never-ANALYZEd estimator trap is closed for the backup chunk decision (the v0.99.176 migrate-side class, backup edition).** The backup snapshot's pinned primary reader carried no estimator path, and a never-ANALYZEd source hits the `reltuples` sentinel that pinned readers decline — either way the size estimate reports 0 and chunking would silently never engage. The primary reader now runs an exact `COUNT(*)` on a fresh off-snapshot connection for the chunk decision (a size estimate with no consistency requirement; chunk bounds and row streams stay snapshot-pinned), and importer-minted range readers opt in through the new `ir.ExactCountEstimateOptIn` surface. Sync cold-start reader minting deliberately does not share this path (ADR-0079 v1.1 unchanged, pinned by the exporter test). Prior releases had no backup chunk decision, so this ships as part of the feature — no published release carried it as a bug.

**Layout, manifest, and resume posture are unchanged.** Chunk-file naming is byte-identical (`chunkFilePath`); indexes are allocated from a per-table atomic shared by the range workers — collision-free and gapless — and the manifest's chunk list accrues in arrival order, which is safe because every downstream consumer (restore, verify, compact, prune, chain) treats the list as a set, verified. A chunked table may carry up to ranges−1 extra partial-size chunk files (each range flushes its own tail). Bug-135 resume posture is untouched: partial tables re-stream from scratch, whole-table complete reuse stays, and chunking never moves the manifest's `EndPosition`/anchor. Pinned by unit tests over every eligibility-fallback clause, the budget split, and concurrent index allocation, plus PG integration legs (int-PK, collation-sensitive keyset, composite PK, per-chunk-CEK encryption, cancel-mid-table + resume) all run against never-ANALYZEd sources so each doubles as the estimator-trap end-to-end.

## Compatibility

**No breaking changes; drop-in.** No flags were removed or re-defaulted; the two new flags default to auto. The one default-behavior note: eligible large tables on Postgres-source backups now read in parallel ranges automatically, which raises concurrent read pressure on the source during the sweep — operators who deliberately sized for one reader per table should set `--bulk-parallelism=1` to keep the previous shape. Backup artifacts are fully interchangeable across the boundary: chunk naming, manifest schema, and encryption plumbing are byte-compatible in both directions (no format bump), old binaries restore new backups and vice versa, and MySQL, SQLite, VStream, and non-snapshot-fallback backup paths are byte-identical to v0.99.176.

## Who needs this — action required

- **No one must act.** This release contains no bug fixes to shipped behavior — nothing to re-verify, no counts to re-check.
- **Postgres-source backups dominated by one or a few large tables:** you get the speedup automatically on upgrade; watch for the `within-table read chunking enabled (ADR-0149)` INFO and per-table `ranges` counts.
- **Operators who bound source read pressure during backups:** set `--bulk-parallelism=1` (or raise `--bulk-parallel-min-rows`) if you want the previous single-reader-per-table sweep.
