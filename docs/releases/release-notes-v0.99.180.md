# sluice v0.99.180

**Maintainability & robustness release — one operator-visible behavior addition (per-sync engine options so a fleet `sync run` no longer shares one process-global sql_mode / zero-date / copy-parallelism knob across every sync), one new advisory WARN (a MySQL DSN `sql_mode` that disagrees with `--mysql-sql-mode` on NO_BACKSLASH_ESCAPES), and a stage-local D1 read speedup. Everything else is internal/test/CI hardening. No bug fixes to shipped data behavior, no format change, drop-in — nobody needs to re-verify anything.**

## Features

- **Per-instance engine options replace the process-wide mutable engine globals (audit A-4).** The CLI used to set the MySQL session `sql_mode`, the zero-date policy, the VStream/native copy-table parallelism overrides, the VStream preserve-skew override, and the SQLite default date-encoding as process-global mutable state — so a fleet `sync run` that hosts multiple syncs in one process could only carry ONE value of each knob across all of them. These are now plumbed as per-engine-instance options applied at engine resolution (the `ir.ConnectionLabeler` template): every value-carrying command and the fleet `buildStreamerFromSpec` set them per sync via structural optional interfaces, keeping engine-specifics out of the neutral `ir` package. Load-bearing internals: `sql_mode` injection moved from DSN parsing into the `openDB` chokepoint, the DDL-emit backslash policy now rides a per-writer `mysqlEmitter`, and the reader zero-date fold happens at construction. Every option's zero value reproduces today's default, so the DSN-per-source override precedence (CLI flag > DSN param > default) and the unset-flag common case stay byte-identical. Pinned by per-option precedence unit tests, a whole-emit-tree backslash pin across both sql_modes, and `applyEngineOptions` CLI tests; the MySQL-engine integration suite (zero-date, sql_mode, LOAD-DATA-warn, copy-parallelism, schema-writer, backup) is green. One deliberate deferral is flagged in-code: the sqlite-trigger / d1-trigger CDC change-decode path does not yet receive the engine's `--sqlite-date-encoding` default (its decoder folds an unset DSN to ISO — a loud refusal on mismatch, never silent-wrong; the DSN-param path is unaffected).
- **`--stage-local` D1 imports now overlap the next page's HTTP fetch with the current page's local insert (ADR-0151, audit P-2 residual).** The bulk-read D1 path already prefetched a single page ahead (v0.99.178); the stage-local materializer's page loop was still strictly sequential — fetch page N over the Cloudflare API, decode, insert locally, THEN request N+1 — paying one avoidable WAN round trip of dead time per 1000-row page. `stageD1Table` now drives the same single fetcher goroutine over an unbuffered handoff (exactly one page of read-ahead, memory bounded at one extra page), hiding one RTT per page behind the local insert. Contract preserved and pinned at the staging seam: request order and exact-text string bounds byte-identical (incl. the > 2^53 bound), a failed prefetched page surfaces in sequence, a cancelled fetch reaps the goroutine and surfaces loudly (never a silently truncated staged file), and overlap is proven deterministically by holding the staging file's write lock while observing the page-2 request.

## Changed

- **New advisory WARN when a MySQL DSN `sql_mode` param disagrees with `--mysql-sql-mode` on NO_BACKSLASH_ESCAPES.** sluice's DDL emitters decide whether to double string-literal backslashes off the configured session sql_mode, but a DSN-level `sql_mode` param wins on the actual wire (the two-tier override); when the two disagree specifically on the NO_BACKSLASH_ESCAPES bit, the emitter would escape against the wrong mode. At connect, sluice now WARNs once — naming both values and the NBE bit on each side — when an operator-supplied DSN `sql_mode` disagrees with the session mode on that bit. It never blocks (the DSN override is legitimate; the remedy is to align `--mysql-sql-mode`), fires only when the operator set the DSN param (when sluice injects the session mode itself the two agree by construction), and is silent for both-agree and no-DSN-param cases. Pinned by unit tests over both disagreement directions, both agreeing regimes, the no-param case, and the NBE classifier over quoted / mixed-case / comma-list forms.

## Compatibility

**No breaking changes; drop-in; no format bump.** No flags added, removed, or changed; no default behavior changes. The per-sync engine-options refactor is behavior-preserving by construction — every option's zero value is today's default and the CLI-flag > DSN-param > default precedence is byte-identical. The only new runtime output is the NBE-disagreement WARN above, which fires only for the specific misconfiguration it names and never blocks. All engines, directions, and paths are unaffected. The reverse-direction MySQL-source-backslash surface (MySQL literals carried to PG/SQLite targets) was audited this release and found faithful by construction — PG pins `standard_conforming_strings=on` and SQLite has no backslash escapes, so neither target doubles backslashes — and is now locked down by regression pins; this is an audit-clean confirmation, not a fix, so there is no behavior change and no affected-releases range.

## Who needs this — action required

- **No one must act.** There are no data-behavior fixes in this release, nothing to re-verify, and no format change — older binaries and this one interoperate exactly as before.
- **Operators running a fleet `sync run` with multiple syncs in one process** can now carry distinct `sql_mode` / zero-date / copy-parallelism / preserve-skew / date-encoding values per sync instead of one process-wide value; existing single-sync and unset-flag invocations are byte-identical.
- **Anyone who sets a MySQL DSN `sql_mode` param AND `--mysql-sql-mode`** may see one new connect-time WARN if the two disagree on NO_BACKSLASH_ESCAPES; align `--mysql-sql-mode` with the DSN param to silence it. No action required otherwise.
