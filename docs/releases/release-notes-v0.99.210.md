# sluice v0.99.210

**New (opt-in Security) — KMS-backed manifest signing for backups (ADR-0154 Phase 3a, AWS KMS first). The signing private key stays in the cloud HSM: a signing operation is an IAM-granted, audit-trailed AWS KMS Sign call, so no raw key material ever lives on the signing host — and a compact/prune maintenance cron can now re-sign a chain WITHOUT holding any private key (the keyless-cron property). Verification stays pure local crypto — a DR/CI host verifies a KMS-signed chain with ONLY a public key (offline via an exported PEM, or online via a KMS GetPublicKey of the operator-named key), never a signing secret. Layered on Phase 2's asymmetric verify path and fully opt-in and drop-in: nothing changes unless you pass `--sign-key kms://…`, and every v0.99.208 (Phase-1 HMAC) and v0.99.209 (Phase-2 Ed25519) signed backup verifies and restores UNCHANGED (no canonicalization bump). GCP KMS and Azure Key Vault signing are a Phase-3b fast-follow.**

## Security

- **KMS-backed manifest signing — HSM-custody signing keys, AWS KMS first (ADR-0154 Phase 3a; opt-in via `--sign-key kms://…` / `--verify-key`).** Phase 1 (v0.99.208, HMAC-off-KEK) and Phase 2 (v0.99.209, Ed25519 keypair) both required a *raw* signing secret on the host that writes backups — the passphrase-derived MAC key, or the Ed25519 private key file. Phase 3a adds a third signature-scheme family, `kms`, where the private key never leaves the cloud HSM: `--sign-key kms://aws/<key-ref>` on `backup full`/`incremental` signs the manifest via an AWS KMS Sign call (ECDSA P-256/P-384/P-521 or RSA-PSS-256), which is IAM-granted and CloudTrail-logged, so you get a real audit trail of who signed what and no raw key material ever sits on the signing host. The signing key is independent of the encryption keystore — it may differ from the `--kms-key-arn` encryption key (signing ≠ encryption custody) — reusing Phase 2's whole verify-policy, freshness, and compact/prune machinery; only the signer adapter and public-key resolution are new.
- **The operable property: a keyless re-signing maintenance cron (ADR §7-Q4).** A signed chain refuses to compact or prune without the ability to re-sign (never silently emit an unsigned successor to a signed chain). Before 3a, that meant the maintenance host had to hold the raw private key. Now a compact/prune/incremental cron re-signs the restructured survivor set via IAM-granted KMS Sign — `--sign-key kms://…` — WITHOUT ever holding raw key material. Pinned by `buildKMSSignedChain`'s ResignLineage-via-KMS.
- **The algorithm is authoritatively bound — composite scheme token, no canon bump.** The `kms` scheme records its concrete algorithm as a COMPOSITE `kms/<algorithm>` token (e.g. `kms/ecdsa-p256`, `kms/rsa-pss-256`) in the same `scheme` field canon v3 already folds into the SIGNED canonical bytes. The family (before `/`) drives verifier selection; the algorithm (after `/`) is bound by being inside the signed token — an adversary relabeling the algorithm (e.g. `ecdsa-p384`→`ecdsa-p256`) changes both the signed bytes AND the verifier's chosen primitive, so it fails closed. The token stays a single length-prefixed string, so injectivity holds and the `hmac-kek` / `ed25519` goldens are byte-frozen (`ManifestCanonVersion` stays v3; the canonical-bytes functions are unchanged — callers just pass a longer scheme for kms). This is what makes Phase-1 HMAC and Phase-2 Ed25519 signatures byte-unchanged.
- **Verification is pure local crypto, anchored on the OPERATOR's key — never a store-supplied ref.** `--verify-key` accepts BOTH a local SPKI PEM public key (offline DR verification, done locally against the recorded algorithm — the Phase-2 parser generalized to Ed25519 / ECDSA / RSA) AND a `kms://aws/<key-ref>` URI (online KMS GetPublicKey of the operator-NAMED key). The verify path performs no KMS Sign call and, for a PEM, no KMS call at all — only the optional public-key fetch. The `.sig` gains advisory `algorithm` + `key_ref` fields (`omitempty`, so `hmac-kek`/`ed25519` `.sig` bytes are unchanged) for rotation/audit/online-fetch, but they are NEVER trusted for verification: the verifier anchors on the operator's supplied trusted key, so a store adversary rewriting the advisory `key_ref` cannot redirect trust. Pinned by `TestVerifyManifest_KMS_KeyRefIsAdvisory`.
- **Adversarially reviewed, SHIP verdict — the 3-way scheme + algorithm binding.** The scheme family is now `{hmac-kek, ed25519, kms}`. The 3-way scheme confusion (a `kms` `.sig` relabeled to/from `ed25519` or `hmac-kek`, and vice versa), intra-`kms` algorithm downgrade, malformed composite schemes, and DER-vs-raw-`r‖s` encoding confusion were all verified to fail closed — via the signed composite scheme token, the operator-anchored verify key, and the digest/key-type/DER binding — with no no-op path, no weaker-primitive fallback, and no panics. AWS KMS returns ECDSA signatures as ASN.1-DER (RFC 3279) and the pure verifier uses `ecdsa.VerifyASN1`, pinned by `TestAWSKMSSigner_ECDSA_IsDER_NotRawConcat` (the same signature in raw `r‖s` / IEEE-P1363 form does NOT verify). A KMS chain presented with only a KEK takes the unverifiable warn/refuse path, never a spurious HMAC verify. Ground-truthed on real localstack KMS (asymmetric key, live Sign/GetPublicKey): a KMS-signed + KMS-encrypted full+incremental chain verifies, restores byte-exact, and refuses a signature-specific store-tamper; Phase-1/Phase-2/pre-v6 chains verify and restore unchanged (no regression).

## Compatibility

**No breaking changes for existing invocations; fully opt-in; every v0.99.208 (Phase-1 HMAC) and v0.99.209 (Phase-2 Ed25519) signed backup verifies and restores unchanged.** Three things to know:

1. **Opt-in.** Nothing about KMS signing engages unless you pass `--sign-key kms://…` (write) or a `kms://` verify key (read). Migrate, sync, verify, plaintext and encrypted backup/restore, and every Phase-1 and Phase-2 signing path are byte-identical. New CLI surface: `--sign-key kms://aws/<key-ref>`, and `--verify-key` now also accepts a `kms://aws/<key-ref>` URI (in addition to a PEM path).
2. **No canonicalization bump — Phase-1 and Phase-2 chains are byte-unchanged.** The `kms` scheme rides canon v3 via the composite `kms/<algorithm>` token, so `ManifestCanonVersion` / `LineageCatalogCanonVersion` stay v3 and the `hmac-kek` / `ed25519` signed-byte goldens are frozen. A v0.99.208 HMAC chain and a v0.99.209 Ed25519 chain both verify and restore exactly as before; the dual-version verifier from Phase 2 is unaffected.
3. **KMS-signed chains are canon v3, the same as Ed25519.** Older binaries that don't know the `kms` scheme refuse with an upgrade-not-tamper message — but in practice pre-v0.99.208 binaries have no signing at all and already refuse a v6 backup at the FormatVersion preflight, so there is no new incompatibility rung beyond the one Phase 1 documented. Signing does not change the FormatVersion story: a signed backup is stamped v6; unsigned backups keep their versions and remain readable by older binaries.

`--sign` (Phase-1 HMAC) on a KMS-ENCRYPTED chain still refuses (the KEK never leaves the HSM) — `--sign-key kms://…` is the KMS-signing path. A chain keeps its scheme across extension and maintenance (you cannot mix schemes within one chain).

## Who needs this — action required

- **Operators who want HSM-custody signing keys — an IAM-audited signing trail with no raw key material on the signing host: opt in on AWS by pointing `--sign-key kms://aws/<key-ref>` at a KMS asymmetric SIGN_VERIFY key (ECDSA P-256/P-384/P-521 or RSA-PSS-256) on your `backup full`/`incremental`, then verify on restore/`backup verify` with `--verify-key <exported-PEM | kms://aws/<key-ref>>`.** Distribute only the PUBLIC key (an exported SPKI PEM) to your verify/restore/CI hosts for offline verification; nothing on those hosts needs KMS access unless you prefer the online GetPublicKey path. The signing key may differ from your `--kms-key-arn` encryption key — signing and encryption are separate custody.
- **Operators who want a keyless-signing maintenance cron on AWS: run compact/prune/incremental with `--sign-key kms://aws/<key-ref>`.** The cron re-signs the restructured chain via IAM-granted KMS Sign and never holds raw key material — the case Phases 1 and 2 could not serve without putting the private key on the maintenance host.
- **Operators already using Phase-1 `--sign` (HMAC) or Phase-2 `--sign-key <pem>` (Ed25519): nothing to do.** Your existing signed backups verify and restore unchanged on v0.99.210, and their chains continue to extend with the same scheme (a chain keeps its scheme; you cannot mix HMAC, Ed25519, and KMS links). No re-signing, no re-verification, no migration.
- **Everyone else: nothing to do — this is a purely additive, opt-in feature.** No existing backup, restore, migrate, sync, or verify behavior changes; no data is retroactively affected; unsigned and pre-v6 backups restore exactly as before. There is no silent-loss fix here to re-verify against.

## Coming (Phase 3b)

GCP KMS (including its Ed25519 signing) + Azure Key Vault signer adapters. The pure verifier already covers ecdsa/rsa/ed25519, so 3b is signer-adapter-only — each normalizing its own SDK Sign request shape, signature encoding (Azure returns raw `r‖s`, not DER — a real divergence the adapter must handle), and public-key export (GCP PEM, Azure JWK) into the `crypto.KMSSigner` seam. GCP and Azure `kms://` scheme-keys refuse loudly with a "Phase 3b" message today.
