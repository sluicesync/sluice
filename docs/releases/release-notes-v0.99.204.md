# sluice v0.99.204

**Source-write diet for the trigger-CDC engine + two config/DR fixes (audit wave 4a) — (1) `postgres-trigger` capture is now ~39% lighter on the SOURCE database (200k-insert churn under capture: 1850 ms → 1133 ms; change-log index footprint 16 MB → 4.4 MB): setup no longer creates two indexes no query ever read, and the capture trigger no longer re-derives the PK column list from the catalogs on every fired row — exactly the write amplification the engine imposes on the managed (Heroku-class) sources it exists for; existing installs converge by re-running trigger setup. (2) Fleet `syncs.yaml` explicit `0` values now actually disable (or refuse) the way the matching `sync start` flags document, instead of silently becoming 60s/60s/64MiB (fleet releases v0.99.132 through v0.99.203). (3) `backup verify --rebuild-catalog` now sniffs the chain's real codec from chunk magic bytes instead of stamping zstd unconditionally — a gzip/none-era chain whose lineage.json was lost no longer gets a rebuilt catalog that lies and a restore that dies with a bare zstd decode error pointing back at the rebuild tool (v0.67.0 through v0.99.203). Plus: the MySQL schema phases' per-object catalog probes are batched (hundreds of serial vtgate round trips removed at DDL/verify time on many-index PlanetScale schemas) and the VStream cold-copy snapshot drain — the engines' last unbuffered row channel — is buffered. Drop-in for existing invocations, with three deliberate behavior changes listed under Compatibility.**

## Performance

- **The `postgres-trigger` change-log is on an index diet and the capture trigger's PK list is baked at setup time — ~39% faster insert churn on the source under capture (1850 ms → 1133 ms per 200k rows), change-log index footprint 16 MB → 4.4 MB (audit N-16; both write shapes date to the engine's introduction in v0.85.0).** Setup created two indexes on `sluice_change_log` that no engine query has ever read — `sluice_change_log_id_idx ON (id)`, an exact duplicate of the BIGSERIAL PK's implicit index, and `sluice_change_log_table_idx ON (schema_name, table_name, id)`, re-verified unused against the full query inventory (poll, anchor, the v0.99.202 settle-clamp, prune/auto-prune batches, stats/recount, and teardown all key on `id` and `txid` alone) — both maintained on EVERY captured source DML for zero read benefit, on precisely the managed tiers where the operator pays for source IOPS. Setup no longer creates them and emits idempotent `DROP INDEX IF EXISTS` for both names, so any pre-existing install converges on its next setup re-run (no migration machinery — pre-1.0 simplification). The capture trigger also re-derived the table's PK column list from a `pg_constraint`/`pg_attribute` catalog join per fired row while ADR-0066 §3's own TG_ARGV bake sat vestigial; the bake is now real — setup loads each table's PK columns (conkey order, composite PKs unchanged) and `CREATE TRIGGER` carries them as a JSON array in `TG_ARGV[0]` — with a loud stale-bake guard: if a baked PK column stops matching the row image (e.g. after a post-setup ALTER of the PK), the trigger refuses the write naming the remedy (re-run trigger setup), never rows keyed on the wrong columns. Pinned by DDL-shape unit pins plus integration pins for fresh setup and pre-v0.99.204 convergence, composite-PK end-to-end across all three payload modes, and the stale-bake loud refusal.
- **The MySQL schema phases no longer pay one catalog round trip per index/FK — the three per-object information_schema probes are chunked IN-list batch reads (audit V-1).** Index verification (the `SLUICE-E-INDEX-MISSING` net), the detect-then-skip probe before index builds, and the per-FK constraint check now share one `table_name IN (…) AND <name> IN (…)` catalog read per ~400 wanted objects with a Go-side set compare, instead of one serial round trip per object. On a vtgate/PlanetScale target every probe fans out through the cluster, so a hundreds-of-index schema paid hundreds of serial RTTs per DDL/verify phase — a 200-index schema now verifies in one metadata query. Refusal shapes are byte-identical (`SLUICE-E-INDEX-MISSING` still names each missing table.index; the existing pins pass unchanged), detect-then-skip still probes at attempt time so the ADR-0108 reparent-retry replay never double-creates, and every value rides a parameter (or the flavor's ADR-0153 interpolated equivalent) — never string-built SQL. One named wart is pinned: the Go-side compare case-folds both sides, matching the case-insensitive comparison the old DB-side `=` probes did under information_schema's collation and `lower_case_table_names` targets. Ground-truthed on real MySQL and real vtgate, green and red directions.
- **The VStream cold-copy snapshot drain — the engines' last unbuffered `ir.Row` channel — is now buffered (64) with batched lock pops (audit P′-1).** The drain popped one row per mutex acquisition, broadcasting the cond to every producer/consumer per row before an unbuffered rendezvous with the writer — a client-side ceiling on faster PlanetScale tiers. It now pops up to 16 rows per acquisition with one broadcast per batch, and the buffer lets dequeue overlap the downstream write. Byte-budget backpressure is unchanged by construction: the debit still happens at pop time under the same lock, so the bounded-memory cap-wait and the cross-table loud refusal are untouched (both pins pass unchanged); the buffer merely holds ≤ 64+16 already-debited rows in flight — the same bounded-overshoot class every buffered tee carries.

## Fixed

- **`backup verify --rebuild-catalog` no longer stamps zstd on the rebuilt lineage record regardless of the chunks' actual compression — it sniffs the codec from chunk magic bytes and refuses loudly when it cannot prove one (wrong-heal on the DR path; every release carrying the rebuild tool, v0.67.0 through v0.99.203).** A gzip- or none-compressed chain whose `lineage.json` was lost got a rebuilt catalog that LIED — restore trusts the record, so restore/prune/compact died on a bare zstd decode error whose remediation hint pointed back at the very rebuild tool that wrote the wrong record. The rebuilt record's codec is now sniffed from one chunk per manifest and verified to agree across the chain: the signatures are disjoint and deterministic (zstd `28 B5 2F FD`, gzip `1F 8B`, uncompressed `{` — the codec input always begins with the header line's `{`), and mixed probes, unknown magic, and truncated chunks all refuse loudly — the durable-record path never guesses. Encrypted chains shaped the fix: the codec sits INSIDE the encryption envelope (chunks are compressed then encrypted), so on-disk encrypted bytes carry no codec magic — a keyless rebuild of an encrypted chain now refuses with a steer to `--encrypt`, and a keyed rebuild decrypts one chunk through the same ADR-0152 unwrap/AAD chokepoints restore uses and sniffs the plaintext. The read-only synthetic-root path (verify/prune/broker over a chain without `lineage.json`) sniffs best-effort and WARNs naming the default-codec assumption when it cannot probe, so keyless flows like SHA-only verify keep working; restore itself still never sniffs — only the record's (re)creation does. Pinned per the family discipline across all three codecs (magic table, real-writer bytes, encrypted layout), the refusal shapes, and the DR scenario end-to-end per codec family — backup, delete `lineage.json`, rebuild, restore — including the encrypted leg.
- **Fleet `syncs.yaml` explicit zeros now mean what the flags document (silent config inversion, fleet layer only; the sync command center's whole life, v0.99.132 through v0.99.203).** An explicit `apply-exec-timeout: 0` / `heartbeat-interval: 0` / `max-buffer-bytes: 0` was coerced to the 60s/60s/64MiB defaults, while the matching `sync start` flags document "0 disables" (and `--max-buffer-bytes=0` = no orchestrator cap) and pass an explicit 0 through verbatim — "unset" and "explicit 0" were conflated on the fleet layer only. And an explicit `apply-retry-attempts: 0` (or either backoff key at 0) was silently absorbed into the ADR-0038 default, where the CLI refuses the same value out-of-range. The three "0 disables" knobs are now pointer-typed in the spec (key omitted → the `sync start` flag default; a present value, INCLUDING 0, passes through verbatim; YAML key names unchanged), the retry trio reaches the CLI's validator verbatim and refuses with the identical out-of-range message, and the SIGHUP hot-reload fingerprint distinguishes explicit-0 from omitted, so flipping between them restarts the sync. Pinned through the REAL parsers (the Bug-180 lesson): the full {omitted, explicit 0, non-zero} matrix through `loadFleetConfig`, the retry-refusal through the load path with the CLI-computed message, and a kong-parity pin proving `sync start` explicit-0/default runs produce the same Streamer values as the fleet keys.

## Internal

- **Testing-guard hardening + a real kmsverify suite (audit N-17).** The run-filter coverage guard now auto-discovers the build-tag axis set from `*_test.go` `//go:build` lines and fails CI Lint on any tag with neither a manifest axis nor a documented exemption — a future `integration && newtag` suite can no longer type-check, pass both coverage guards, and run in no workflow — and cross-checks every manifest regex verbatim against the workflow file its leg names, so a `-run` filter edited on one side fails loudly. The shard guard's matching now mirrors ci.yml's actual non-recursive package listing (bare entries no longer cover subtrees), pinned by an always-run self-test whose load-bearing case asserts `internal/pipeline` does NOT cover `internal/pipeline/migcore` — previously an integration test under a pipeline subpackage could pass the guard while running in no shard. And the kmsverify scaffolding went from all-skip to four real tests against a localstack KMS container + real PG (round-trip, chain extension, missing-key and wrong-key loud refusals) on a weekly + dispatch extended-suites leg under `-race`; only the IAM access-denied case remains an honest skip (localstack community edition doesn't enforce IAM; that branch stays unit-pinned). Both guard fixes negative-proven with staged fake escapees.

## Compatibility

**No CLI-flag changes; drop-in for existing invocations, with three deliberate behavior changes:** (1) a `postgres-trigger` setup re-run drops the two legacy change-log indexes (intended; no data effect), and an ALTER of a captured table's PK now requires a trigger-setup re-run — the refusal is loud and names it (previously the per-row catalog lookup tracked the ALTER silently on the polled-fingerprint tier, but the event-trigger tier already refused the stream at the ALTER, so no working flow regresses). (2) Fleet `syncs.yaml` explicit-0 semantics change as described — anyone who wrote `apply-exec-timeout: 0` EXPECTING the 60s default (unlikely; it contradicted the flag docs) now gets an unbounded exec timeout, and explicit `apply-retry-*: 0` now refuses loudly at load instead of silently running with the default; remove the key to keep a default. (3) `--rebuild-catalog` over an ENCRYPTED chain now requires `--encrypt` + key material (previously a keyless rebuild wrote a wrong codec record); keyless read-only flows such as SHA-only verify keep working through the WARN-named best-effort path.

## Who needs this — action required

- **`postgres-trigger` users: re-run trigger setup once after upgrading to collect the diet.** Existing triggers keep capturing correctly until then (the old per-row lookup path still functions); the re-run drops the two write-amplifying indexes and bakes the PK lists into the triggers. From then on, PK-altering DDL on a captured table needs another setup re-run — the trigger's refusal is loud and names exactly that remedy.
- **Anyone who ran `--rebuild-catalog` over a gzip- or none-compressed chain on v0.67.0–v0.99.203 (or plans to rebuild any chain): re-run the rebuild on v0.99.204.** A prior rebuild over a non-zstd chain wrote a wrong codec record; nothing was lost silently (restore fails loudly), but the record stays wrong until rebuilt — re-running `backup verify --rebuild-catalog` on this release writes the truthful record and restores work again. Encrypted chains now need `--encrypt` on the rebuild.
- **Fleet operators with explicit `0` values in `syncs.yaml`: confirm intent before upgrading.** A `0` written to disable now actually disables; a `0` on the `apply-retry-*` keys now refuses at load with the CLI's out-of-range message — remove the key to get the default.
- **PlanetScale / Vitess users: nothing to do — you get the speedups on upgrade.** DDL/verify phases on many-index schemas drop from hundreds of serial round trips to a handful of batched reads, and the cold-copy drain's client-side ceiling rises.
- **Everyone else: drop-in upgrade, no action.**
