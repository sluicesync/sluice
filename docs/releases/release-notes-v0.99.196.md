# sluice v0.99.196

**CRITICAL fix — a PlanetScale/Vitess source with MULTIPLE keyspaces no longer risks silently copying an incomplete shard set when `SHOW VITESS_SHARDS` under-reports a fully-SERVING sibling sharded keyspace. Live-confirmed on real PlanetScale: `SHOW VITESS_SHARDS` can persistently drop an entire sharded keyspace's shards while `SHOW VITESS_TABLETS` and `SHOW KEYSPACES` report it complete — and because the database-level result is incomplete-not-empty, the v0.99.194 retry-on-empty never fired and a cold-copy / CDC / backup against the omitted keyspace would have silently missed every one of its shards. Shard discovery now runs both `SHOW VITESS_SHARDS` and `SHOW VITESS_TABLETS` and returns the per-keyspace UNION of serving shards, never dropping a shard either source reports. Drop-in: on the common single-keyspace source the union equals the old `SHOW VITESS_SHARDS` result byte-for-byte; the only behavioral change is that a multi-keyspace source whose sharded keyspace `SHOW VITESS_SHARDS` omitted now discovers all its shards.**

## Fixed

- **A PlanetScale/Vitess source no longer risks silently missing an entire sharded keyspace's shards when `SHOW VITESS_SHARDS` under-reports (silent-partial-copy risk on a multi-keyspace source; present on the default discovery path since auto-discovery became the default in v0.99.191, and reachable via explicit `vstream_auto_discover_shards=true` back to when VStream multi-shard discovery landed in v0.1.0).** sluice's Vitess shard discovery relied solely on `SHOW VITESS_SHARDS`, and that query — live-confirmed on real PlanetScale — can PERSISTENTLY omit a fully-SERVING sharded keyspace's shards: a database with a default unsharded keyspace plus two 4-shard keyspaces had `SHOW VITESS_SHARDS` drop one of them entirely for 6+ minutes (never converging) while `SHOW VITESS_TABLETS` and `SHOW KEYSPACES` listed all three complete throughout. Per keyspace the omission is all-or-nothing (0 or all N shards), but at the database level the result is incomplete-not-empty — the connected keyspace can be present-and-complete while a sibling keyspace is silently dropped from the inventory. The v0.99.194 retry-on-empty only re-runs discovery when the *connected/target* keyspace comes back empty, so a present target with a dropped sibling never triggered it, and a cold-copy / CDC / backup that drew its shard set from that incomplete inventory would have silently copied zero of the omitted keyspace's shards — exit 0, missing data. The fix: `discoverAllShards` now runs BOTH `SHOW VITESS_SHARDS` and `SHOW VITESS_TABLETS` and returns the per-keyspace UNION of serving shards (distinct `(keyspace, shard)` where `State=SERVING`, accepting any serving tablet type so a momentary reparent gap doesn't drop a shard), never dropping a shard either source reports, and WARNs loudly naming any keyspace where the two disagree — either a shard recovered-via-tablets that `SHOW VITESS_SHARDS` omitted, or a `SHOW VITESS_SHARDS` shard with no serving tablet (kept in the set, flagged possibly-unstreamable). The cross-check is additive safety, not a new hard dependency: if `SHOW VITESS_TABLETS` is unavailable (older vtgate, insufficient permissions) discovery falls back to the `SHOW VITESS_SHARDS`-only result with a WARN — byte-identical to before. Both consumers inherit the fix: shard enumeration (cold-copy + CDC, via `discoverShards` / `resolveVStreamShards`) and the `--control-keyspace` sharded-target auto-detect (`discoverUnshardedKeyspaces`). Why it wasn't caught earlier: it requires a source with multiple keyspaces where a *secondary* sharded keyspace is the one omitted; a single-keyspace sharded source — the common case, and what every prior sharded test used — is unaffected (targeting the omitted keyspace directly still fails loud with "returned no shards", never silently). Column parsing is by name (reorder-safe, and a missing expected `Keyspace`/`Shard`/`State` column is a loud error, never a silent mis-parse). Pinned by pure reconcile/parser unit tests seeded with the exact observed omission plus a vttestserver integration test proving the union equals the `SHOW VITESS_SHARDS` result on a healthy cluster with no spurious WARN.

## Compatibility

**No breaking changes; drop-in. No format or CLI-flag change.** On the common single-keyspace source — sharded or unsharded — the per-keyspace union equals the old `SHOW VITESS_SHARDS`-only result byte-for-byte, so migrate, sync, backup, and restore are identical. The only behavioral change is on a multi-keyspace PlanetScale/Vitess source whose sharded keyspace `SHOW VITESS_SHARDS` omitted: discovery now returns all of that keyspace's serving shards (previously it silently under-reported the inventory, or — if that keyspace was the connect target — failed loud with "returned no shards"). `vstream_auto_discover_shards` and `vstream_shards` keep their meanings; an explicit `vstream_shards=…` still pins and skips discovery entirely. The reach is MySQL/PlanetScale/Vitess shard discovery only; Postgres, SQLite/D1, and vanilla single-shard MySQL never run this path and are byte-identical. When `SHOW VITESS_TABLETS` is unavailable, discovery degrades to exactly the pre-fix `SHOW VITESS_SHARDS`-only behavior.

## Who needs this — action required

- **PlanetScale/Vitess operators whose source database has MULTIPLE keyspaces (a secondary sharded keyspace beside others): re-verify any migrate / sync / backup you ran against such a source on v0.99.191–v0.99.195 (or earlier with `vstream_auto_discover_shards=true`).** If `SHOW VITESS_SHARDS` happened to omit one of the sharded keyspaces during discovery, a copy of that keyspace could have silently missed every one of its shards while exiting 0. Re-run on v0.99.196 and compare row/shard counts for the affected keyspace; the new WARN ("SHOW VITESS_SHARDS under-reported keyspace … — recovered N serving shard(s) via SHOW VITESS_TABLETS") fires whenever the cross-check catches a live omission, so watch for it as the signal that this was actually biting you.
- **PlanetScale/Vitess operators with a single-keyspace source (the common case): no action.** The union equals the old result; a single-keyspace sharded source was never affected (a directly-targeted omitted keyspace failed loud, not silently), and nothing changes for you.
- **Everyone else: no action.** Postgres, SQLite/D1, and vanilla MySQL never run Vitess shard discovery; nothing to re-verify, nothing to re-run.
