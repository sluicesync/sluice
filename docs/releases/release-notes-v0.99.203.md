# sluice v0.99.203

**HIGH fix + PlanetScale/Vitess write-path speedup — (1) MySQL-source FLOAT (single-precision) columns read on pages WITHOUT bind parameters — full-table scans (small/keyless tables, backup and binlog-snapshot reads) and the FIRST unbounded page of every cursor-paged read — were silently display-rounded by MySQL's non-round-tripping FLOAT→text conversion before sluice ever saw them (exit 0, zero WARN; every release, v0.1.0 through v0.99.202; DOUBLE was never affected). Now read bit-exact through a version-universal DOUBLE promotion; anyone who ever migrated FLOAT-bearing MySQL tables should consider re-verifying those columns. (2) ADR-0153: PlanetScale/Vitess MySQL flavors now default to client-side statement interpolation on write-path connections — one round trip per statement instead of two — measured on real PlanetScale PS-10 at ~100 ms RTT: bulk copy −33%, CDC burst drain −26%, per-flush p95 −105 ms (almost exactly one RTT), trickle flush ceiling ~35% higher. Byte-exactness is pinned across the full value-family matrix under BOTH protocols, explicit DSN `interpolateParams=` always wins, and vanilla MySQL keeps the binary protocol by default (documented high-RTT opt-in). Drop-in upgrade for all existing invocations.**

## Fixed

- **MySQL-source FLOAT columns are no longer silently precision-rounded on read pages that travel the text protocol (HIGH; silent precision loss — exit 0, zero WARN; MySQL, PlanetScale, and Vitess sources; every release, v0.1.0 through v0.99.202; discovered by ADR-0153's fidelity matrix).** Any read page issued WITHOUT bind parameters travels MySQL's text protocol (COM_QUERY), and MySQL's FLOAT→text conversion does NOT round-trip single precision — ground-truthed on 8.0.46: a stored float32 `8388608`, verifiably exact (`fl = 8388608` is true), renders as `"8388610"` — so sluice received an already-rounded value and faithfully landed the wrong number on the target. The affected pages, precisely: the arg-less full-table scan `ReadRows` (small and keyless tables, plus the backup and binlog-snapshot row readers) and the first unbounded page of every cursor-paged read (its LIMIT is inlined; only the subsequent bound pages went prepared/binary and were always exact). DOUBLE was never affected — MySQL prints DOUBLE shortest-round-trip, now pinned. The fix: single-precision FLOAT columns are read through the projection `(col * 1E0) AS col` — an exact, sign-preserving promotion to DOUBLE whose text form round-trips — applied at the single projection seam every SQL row reader shares (`selectColumnExpr`, the same seam as the temporal CAST), so full-scan, chunked, backup-snapshot, and binlog-snapshot readers all inherit it. The form is deliberate: NOT `CAST(col AS DOUBLE)` (8.0.17+ syntax; sluice has no floor excluding 5.7-compatible sources like Aurora MySQL 2.x) and NOT `col + 0E0` (IEEE −0.0 + 0.0 = +0.0 silently drops the zero's sign; multiplication preserves it). The alias keeps positional and name-based consumers unchanged, NULL propagates, and predicates still reference the real column, so pagination is unaffected. Pinned failing-first (the raw pre-fix SELECT provably returns the rounded value — which doubles as a retire-the-fix canary if a future MySQL fixes FLOAT printing) and mutation-verified: a float32 torture matrix (2^23, 2^24−1, ±MaxFloat32, denormals, −0.0, >6-significant-digit values) bit-exact (`math.Float64bits`) through the real full-scan reader, cross-checked bit-identical against the binary-protocol chunked reader, DOUBLE and NULL legs on both paths, and a vtgate leg proving the promotion parses and evaluates through vtgate with the −0 sign intact.

## Performance

- **PlanetScale/Vitess MySQL flavors now default to client-side statement interpolation (`interpolateParams=true` semantics) on write-path connections, collapsing the 2 round trips every parameterized statement paid (COM_STMT_PREPARE + COM_STMT_EXECUTE, prepared fresh per call) to 1 COM_QUERY (ADR-0153).** Measured A/B on real PlanetScale PS-10 at ~100 ms client RTT (released v0.99.202 binaries, runs alternated): bulk copy −33% (bulk_copy phase, 200k×4 + 2 indexes), CDC burst drain −26% (20k-row transaction), CDC steady-state per-flush p95 −105 ms ≈ 1.04× the median RTT — the smoking gun that the entire gap is the hidden PREPARE — and a ~35% higher per-lane trickle flush ceiling (~2.4 → ~3.3 flushes/s), so sustained-lag territory arrives later. The absolute seconds scale with RTT (datacenter-adjacent runners see proportionally less; the per-statement RTT tax is structural). The decision is made once at the DSN parse choke point, so every derived connection — the concurrent applier lanes, shard-discovery and purged-GTID probes, snapshot pools, resnapshot re-parses — inherits it with no per-site logic, and one INFO line announces the resolved protocol. An explicit `interpolateParams=` in the DSN always wins, true or false (explicitness is detected on the raw DSN, since the driver's parser collapses "explicitly false" with "absent"), so `interpolateParams=false` restores the old protocol verbatim. Vanilla MySQL deliberately keeps the binary protocol by default — typical LAN RTTs gain little — with `interpolateParams=true` in the DSN as the documented high-RTT opt-in (docs/throughput-tuning.md), and every guard below keys on the RESOLVED interpolation state, never the flavor, so a vanilla opt-in gets identical protections. Row-data READ sessions deliberately keep the prepared/binary protocol on all flavors — the FLOAT fix above is exactly why: the prepared path's fidelity never depends on the server's text formatter.
- **The fidelity evidence (the Bug-74 family discipline, pinned before the flip).** The interpolation encoder is a second, independent value encoder, so the full driver-value family matrix — microsecond temporals, DECIMAL(20,6)/(65,30) extremes, DOUBLE extremes and denormals, VARBINARY/BLOB across all 256 octets, 4-byte UTF-8 with quotes/backslashes/NUL, BIGINT (UNSIGNED) extremes, BIT(1/8/64), SET, ENUM, JSON, GEOMETRY (SRID-prefixed WKB), zero-date substitutes, and per-column NULLs — is pinned byte-exact under BOTH protocols on both write paths (bulk batched INSERT and the CDC coalesced multirow apply) on real vanilla MySQL, plus a third-codec leg through vtgate's OWN parser/normalizer on a real vttestserver (an independent literal-handling layer the plain-MySQL matrix cannot vouch for). The matrix forced two real encoder fixes, shipped on both protocols and all flavors: float64 −0.0 now encodes as the string `'-0'` — the interpolated literal `-0` parses as unary-minus-on-unsigned-zero, plain +0, silently dropping the IEEE sign bit the binary protocol preserves — pinned with ABSOLUTE stored-text assertions on every leg (bulk, CDC, vtgate) against a raw binary-protocol control, including the applier's type-free cache-miss path; and NaN/±Inf now refuse loudly with `SLUICE-E-VALUE-UNREPRESENTABLE`, naming the column, before the driver ever sees the value — previously an interpolated NaN drew an Error-1054 shape the schema-drift self-healer retried for ~30 minutes instead of failing fast (the data was never writable either way).
- **Guards, all keyed on the resolved interpolation state.** Unsafe collations AND charsets (the big5/cp932/gb2312/gbk/sjis/gb18030 families — a trailing-byte 0x5C injection hazard): the flavor DEFAULT steps aside with a WARN and keeps the binary protocol, scanning EVERY `charset=` occurrence with any unsafe entry disqualifying (the driver's own check reads only the collation and misses `charset=` entirely — sluice polices both shapes), because a perf default must never turn a previously-working DSN into a connect failure; operator-EXPLICIT `interpolateParams=true` + an unsafe collation or charset is a loud parse-time refusal, and the collation list is pinned two-sided against the driver's so a future driver addition fails loudly at build, never silently. `NO_BACKSLASH_ESCAPES` needs no guard — ground-truthed correct: the driver switches its escaper on the server-reported session status flag, pinned byte-exact on a backslash/quote/NUL-heavy corpus under a real NBE session (if that pin ever fails on a driver upgrade, the posture flips to a loud refusal). Packet headroom is pinned at both legs: the ~1 MiB statement byte targets sit far under every `max_allowed_packet`, and at the true boundary the driver returns `ErrSkip` and database/sql transparently falls back to the prepared path for that statement — no silent loss, no spurious packet error. The end-to-end wire pin observes Com_stmt_prepare = 0 on the default path at a real server, through the engine registry.

## Compatibility

**No CLI-flag changes; no breaking changes for existing invocations; drop-in upgrade.** Four deliberate behavior changes to know about: (1) **PlanetScale/Vitess MySQL targets change write protocol by default** (client-side interpolation, 1 RTT per statement) — byte-exactness is pinned exhaustively across the full value-family matrix under both protocols including through vtgate, and `interpolateParams=false` in the DSN restores the old binary protocol verbatim; vanilla MySQL, Postgres, and SQLite/D1 paths are unchanged by default. (2) **A PS/Vitess DSN carrying an unsafe charset or collation (big5/cp932/gb2312/gbk/sjis/gb18030 families) keeps the binary protocol with a WARN**; explicitly combining `interpolateParams=true` with one is now a loud parse-time refusal (previously the charset shape sailed through the driver un-policed). (3) **NaN/±Inf float values now refuse loudly with `SLUICE-E-VALUE-UNREPRESENTABLE`, naming the column, on all flavors and both protocols** — previously they failed with a confusing server error (and under interpolation would have drawn a drift-shaped error that was retried for ~30 minutes); the data was never writable either way, so nothing that worked is lost. (4) **MySQL-source FLOAT values read on arg-less pages may CHANGE on upgrade — to the CORRECT values**: previous releases landed MySQL's display-rounded rendering, v0.99.203 lands the stored value bit-exact, so a verify comparing an old target against a fresh read can now surface the old copy's rounding.

## Who needs this — action required

- **Anyone who ever migrated, synced, or restored MySQL-source tables with FLOAT (single-precision) columns, on any release through v0.99.202: upgrade, then consider re-verifying those columns.** The loss is silent (exit 0) and affects FLOAT values read on pages without bind parameters — full-table scans (small/keyless tables, backup and binlog-snapshot reads) and the first unbounded page of every cursor-paged read — so a FLOAT-bearing table copied on a prior release may hold display-rounded values (e.g. a stored 8388608 landed as 8388610). Run `sluice verify` (or column checksums) against the source on v0.99.203; if discrepancies show, re-copy the affected tables. DOUBLE columns were never affected, bound (non-first) cursor pages were always exact, and non-MySQL sources are untouched. This is the same re-verify guidance shape as prior silent-loss fixes.
- **PlanetScale / self-hosted Vitess users: nothing to do — you get the speedup on upgrade.** Bulk copy, CDC drain, and per-flush latency all improve, scaling with your RTT to the target. If you want the old protocol back (A/B comparison, or an abundance of caution), set `interpolateParams=false` in the target DSN — that is the complete escape hatch. DSNs carrying unsafe charsets/collations are stepped around automatically with a WARN.
- **Vanilla-MySQL users with a high-RTT (WAN) target: consider the documented opt-in.** Add `interpolateParams=true` to the target DSN to collapse the per-statement PREPARE round trip; the benefit scales with RTT and the guards apply identically. See docs/throughput-tuning.md. At LAN RTTs the default binary protocol is fine and nothing changes.
- **Everyone else: drop-in upgrade, no action.** Postgres and SQLite/D1 paths are byte-identical; the only universally-visible change is that NaN/±Inf floats now fail fast with a named error instead of a confusing one.
