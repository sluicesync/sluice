# sluice v0.99.208

**New (opt-in Security) — detached cryptographic signing of encrypted backup manifests (ADR-0154 Phase 1, manifest FormatVersion 6). Pass `--sign` alongside `--encrypt` on a passphrase-encrypted `backup full`/`incremental` and sluice writes a detached `manifest.sig` + `lineage.json.sig` over a deterministic canonical serialization of the whole manifest; restore/verify then refuse loudly on a tampered, rolled-back, or truncated chain. This closes the ADR-0152 honest residual — R1 whole-manifest rollback (substituting a complete older signed backup) and R2 change-list tail truncation — against a store-WRITE adversary who has bucket write access but NOT the encryption key. Signing is the SECOND-order defense, layered on invariants sluice already holds: it does NOT substitute for "the manifest is data, never instructions" (the Vitess manifest-RCE class has no analog here — the codec is a closed enum) or for path-sanitisation (the traversal class is already defended by absPath/sanitiseBlobKey). Fully opt-in and drop-in: nothing changes unless you pass `--sign`, and pre-v6 backups restore unchanged — but a v6 signed backup is refused (loudly) by pre-v0.99.208 binaries; see Compatibility.**

## Security

- **Optional detached signing of encrypted backup manifests (ADR-0154 Phase 1; manifest FormatVersion 6; opt-in via `--sign`).** ADR-0152 (v0.99.202) bound every encrypted *chunk* to its manifest identity and recorded position, closing chunk splice/replay/reorder/substitution — but it deliberately left the manifest itself UNSIGNED, so a store-write adversary could still present a complete, internally-consistent, genuinely-older backup (R1 whole-manifest rollback, including dropping a chain's newest links) or truncate the tail of a change-chunk list (R2) and the restore exited 0 with authentic-but-stale or short data. Phase 1 adds whole-manifest authenticity on top: with `--sign` on a passphrase-encrypted chain, sluice writes a detached `manifest.sig` per manifest and a `lineage.json.sig` over the chain-tip link enumeration (never inside the manifest — detached keeps the signed bytes byte-stable and lets an old reader ignore them), and restore/verify recompute and check them. The signature closes R2 and intra-chain rollback (dropped-newest-link) outright; the honest R1 residual — a whole-chain swap to an older, fully-signed chain — is documented below and needs an external high-water anchor (ADR option b, not in Phase 1).
- **Phase 1 key mechanism is zero-new-key-management: HMAC-SHA-256 keyed off the chain KEK (Option A, "HMAC-off-KEK").** The signing key is HKDF-derived from the same passphrase-derived KEK that already protects the chain, under a distinct label (`sluice-manifest-sig/v1`) so it is cryptographically separated from the encryption use — anyone who can DECRYPT the backup can VERIFY it, and anyone with the passphrase can WRITE a signed backup. There is no new keypair to generate, distribute, or lose. Because Phase 1 keys off the passphrase KEK, it signs ONLY passphrase-encrypted chains: `--sign` on a KMS-encrypted chain refuses loudly (the KEK never leaves the HSM — KMS Sign is Phase 3), and `--sign` on a plaintext backup refuses (it has no KEK to key off — Ed25519 for plaintext is Phase 2). The detached signature records a non-secret `key_id` fingerprint plus the scheme tag `hmac-kek`.
- **What is signed: a deterministic, provably-injective canonical serialization covering the whole security-relevant manifest (`sluice-manifest-canon/v2`).** The signed bytes are a length-prefixed token stream over format version, identity, parent pointer, `SchemaHash`, the `ChainEncryption` descriptor (including `WrappedCEK` + Argon2id params), the per-table row-count map (schema and name as separate tokens), every row and change chunk (path + SHA + ordinal), `SchemaDelta` (restore drives DDL from it — folded as each before/after table's round-trip-stable fingerprint) and `SchemaHistory`, the resume start/end positions, and the freshness anchors (a monotonic sequence number + chunk count). Change-list truncation and intra-chain rollback are caught by the signed sequence + chunk-count on each link plus the `lineage.json.sig` over the link enumeration (the per-manifest signatures alone can't see a dropped-newest-link; the lineage signature can). This also makes ADR-0152's `SchemaHash` verification genuinely tamper-proof rather than corruption-detecting, because the hash is now under a signature.
- **Verification policy (ratified in ADR-0154 §4/§7): strict where intent is expressed, forgiving where a hard-fail would break a disaster restore — and signedness is decided from the PRESENCE of `.sig` artifacts, never the tamperable FormatVersion field.** A signed chain restored with a KEK-holding envelope (an encrypted restore always has the KEK) verifies strictly and refuses on a missing, invalid, rolled-back, truncated, or downgraded manifest with `SLUICE-E-BACKUP-SIGNATURE-INVALID` / `SLUICE-E-BACKUP-SIGNATURE-MISSING` (exit 3). With no verification key configured, restore WARNs that a signature is present-but-unverified and proceeds — a forgotten verify key must never fail a DR restore — unless `--require-signature` is set, which asserts the chain SHOULD be signed and refuses a missing signature. Pre-v6 (unsigned) backups always restore, forever — the FormatVersion gate means "predates signing," not "untrusted." `backup verify` reports each manifest's signature status (valid / invalid / unsigned) plus the lineage status, and an invalid signature is a verify failure (non-zero exit). Signed chains also propagate through maintenance: `compact`/`prune` refuse to restructure a signed chain without the key (never emit an unsigned successor to a signed chain), and with `--encrypt` supplied they re-sign the whole restructured survivor set at the new positions.
- **The implementation went through two adversarial review rounds before shipping.** The first round caught a CRITICAL FormatVersion-downgrade bypass — the restore-side "is this chain signed?" decision originally trusted the MAC-covered `FormatVersion` field, so flipping `v6→v5` (v5/v6 decrypt identically) made both verify paths skip every check; fixed by deciding signedness from the presence of `.sig` objects (with the sigs present, `format_version` is inside the signed bytes, so a downgrade now still verifies and refuses). It also caught a canonicalization-injectivity forgery vector — raw `key=value` concatenation embedded unsanitised source-derived table names and chunk paths, so an embedded newline/`.`/`:` could let two distinct manifests collide under one MAC; fixed with the length-prefixed framing (canon bumped `v1→v2`). Both were caught and closed before any release carried the feature.

## Compatibility

**No breaking changes for existing invocations; fully opt-in.** Nothing about signing engages unless you pass `--sign` — migrate, sync, verify, and every unsigned backup/restore path (plaintext and encrypted alike) are byte-identical. Three things to know: (1) **a signed backup is stamped manifest FormatVersion 6** and is refused (loudly, at the existing unknown-FormatVersion preflight) by pre-v0.99.208 binaries — this is the expected forward-compat behavior, not a regression; unsigned encrypted backups stay on FormatVersion 5 and remain readable by older binaries (the Bug-116 proportionality discipline). (2) **Pre-v6 backups restore unchanged on v0.99.208** — no retroactive signature requirement. (3) **Two new coded errors** — `SLUICE-E-BACKUP-SIGNATURE-INVALID` and `SLUICE-E-BACKUP-SIGNATURE-MISSING` (both refusals, exit 3) — fire only on a *signed* chain that fails verification (or, for `-MISSING`, under `--require-signature`). Phase 1 signs only passphrase-encrypted chains: `--sign` on a KMS or plaintext backup refuses loudly rather than silently no-op. `backup stream` refuses to extend a signed chain in Phase 1 (rollover signing needs its own `-race` gating — use `backup incremental --sign`), rather than emitting an un-restorable unsigned tail.

## Who needs this — action required

- **Operators storing encrypted backups on shared or untrusted object storage who want tamper-evidence against manifest substitution or truncation: opt in by adding `--sign` to your `backup full` / `backup incremental` (alongside the passphrase `--encrypt` you already use).** New backups then carry the detached signatures, and a store-write adversary who rolls the chain back to an older signed backup, drops the newest links, or truncates the change-list tail is refused loudly at restore/verify instead of exiting 0. For strict-always verification (refuse a signed backup you cannot verify), add `--require-signature` on restore/verify — this also closes the strip-all-signatures case. There is no re-verification of existing backups to do and no key to manage: Phase 1 keys off the passphrase you already hold.
- **Everyone else: nothing to do — this is a purely additive, opt-in feature.** No existing backup, restore, migrate, sync, or verify behavior changes; no data is retroactively affected; unsigned and pre-v6 backups restore exactly as before. There is no silent-loss fix here to re-verify against — the security-review findings were caught in adversarial review before the feature ever shipped, so no released version carried them.

## Honest residual

The manifest signature closes R2 (change-list tail truncation) and intra-chain rollback (dropped-newest-link) outright. Full R1 — a whole-chain swap to an *older, fully-signed* chain — still requires an external high-water anchor (ADR-0154 option b, opt-in, not built in Phase 1): an adversary who strips ALL `.sig` objects AND downgrades the FormatVersion AND the operator did not pass `--require-signature` evades verification, because you cannot detect a whole-backup deletion-to-unsigned purely from artifacts the attacker controls. `--require-signature` closes the strip case (a chain asserted signed with no signature present refuses). This is the inherent external-anchor gap, documented — not a hole.

## Coming (Phases 2–3)

Ed25519 keypair signing (`sluice backup keygen` / bring-your-own — covers plaintext backups and key-separated verification where a CI/restore host holds only the public key) is Phase 2; KMS-backed Sign/Verify (AWS/GCP/Azure, private key never leaving the HSM, for chains already in KMS mode) is Phase 3, layered on Phase 2's verify path. The external high-water anchor for full R1 coverage is a documented opt-in for a later increment. None of these ship in v0.99.208.
