# sluice v0.99.194

**PlanetScale sharded/fleet robustness batch — five related fixes and additions, validated live on a real 2-shard PlanetScale target. Two loud-failure fixes matter most: a freshly-provisioned sharded keyspace that transiently reports 0 shards no longer aborts the sync (it retries the fresh-keyspace window instead of forcing an operator relaunch), and a healthy-but-idle PlanetScale source over the public vstream endpoint no longer gives up after ~6 minutes (idle reconnects are now exempt from the retry give-up budget). The `--control-keyspace` sharded-target support from v0.99.193 now works in the fleet config too, a sharded→sharded fleet move can finally opt into the cross-shard collision guard, and DSN passwords no longer leak into fleet plan/WARN/status output. Drop-in with one exception: a `syncs.yaml` carrying a previously-silently-ignored unknown/typo'd key now fails to load loudly (see Compatibility).**

## Fixed

- **A freshly-created sharded PlanetScale/Vitess keyspace that transiently reports 0 shards no longer aborts the sync (loud failure, no data risk; the default-path transient is present in v0.99.191–v0.99.193, and reachable via explicit `vstream_auto_discover_shards=true` back to when multi-shard auto-discovery landed).** `SHOW VITESS_SHARDS` against a just-provisioned keyspace can succeed yet return zero shards for the target keyspace before the vtgate's serving graph settles — the query works, the keyspace just isn't in the result yet — and sluice treated that first empty read as fatal (`shard auto-discovery for keyspace … returned no shards`), forcing operators to relaunch until the graph caught up. Discovery now retries on an empty result *for the target keyspace only* through a shared `discoverAllShardsForKeyspace` wrapper — bounded (4 attempts) with a short capped exponential backoff (500ms base, 2s cap) and ctx-aware so a cancel aborts promptly. The non-empty first read returns immediately (fast path, no added latency in the common case), and a keyspace that is genuinely absent stays empty past the budget so the caller's existing loud refusal fires unchanged — the retry only buys the transient case a few chances, it never masks a real absence. Both shard consumers ride it identically: the reader's `resolveVStreamShards` enumeration and the v0.99.193 `--control-keyspace` auto-detect (`discoverUnshardedKeyspaces`, which classifies the target as unsharded when its shard set is empty). Pinned by a `retryShardDiscoveryOnEmpty` unit matrix over a fake discover func (empty-then-nonempty rides the retry; always-empty returns the last empty result to the caller's existing handling; a discover error returns immediately without retrying; ctx-cancel during backoff aborts) plus real-2-shard-PlanetScale integration coverage.
- **A healthy-but-idle PlanetScale source no longer gives up after ~6 minutes (loud failure, no data effect — an idle source warm-resumes cleanly either way; present since the two-phase liveness watchdog landed in v0.99.10, on idle/low-traffic sources over an endpoint that sends no idle heartbeats, i.e. the public PlanetScale vstream endpoint).** An idle source hits the ~45s Phase-2 progress timeout each reconnect cycle, and the streamer's bounded-retry give-up counter (`--apply-retry-attempts`, default 8; ADR-0038) resets *only* when the persisted CDC position advances — which an idle source, processing zero events, never does — so the counter climbed to the cap and the sync gave up after roughly six minutes even though nothing was wrong. A new optional engine surface `ir.LivenessProgressTimeoutError` (engine-neutral, checked via `errors.As`, mirroring `RetriableError`/`TransactionKilledError`) marks *only* the established-then-idle Phase-2 timeout (`vstreamProgressTimeoutError` — a stream that proved a serving tablet and then went quiet), never the Phase-1 never-established liveness timeout (`vstreamLivenessTimeoutError`) or any open/connection error. `runWithRetry` neither increments nor resets the give-up counter for a marked idle timeout, so a truly idle source survives indefinitely while a stream that can *never* establish still accumulates toward the budget and fails loudly — a dead / no-tablet stream must never loop forever, and a real failure interspersed with benign idle timeouts keeps its accumulated count. The same exemption is applied to the backup rollover loop. Pinned by streamer-retry unit tests asserting the marked idle timeout leaves the counter untouched while a Phase-1 / connection error still exhausts the budget, plus the mysql-side classifier test that only the Phase-2 constructor sets the marker.
- **`sync run --dry-run`, the shared-target-budget WARN, and `sync status --all`'s unreachable-target line no longer print DSN passwords (credential-in-logs leak, no data effect; present for the go-sql-driver DSN form since the fleet sync command center landed in v0.99.132).** The dry-run plan (`printFleetPlan`) now routes source and target DSNs through `diagnose.RedactDSN`, and `dsnEndpoint` — the coarse host:port extractor that feeds the shared-target-budget WARN and the `status --all` unreachable line — now parses the go-sql-driver form (`user:pass@tcp(host)/db`, the common MySQL/PlanetScale shape) down to a bare `host:port` (strip the userinfo, pull the addr out of `net(addr)`, drop the db) instead of falling through to `return dsn` and printing the raw connection string with its password. A genuinely-unparseable DSN is now redacted (`<redacted-dsn>`) rather than returned raw, so this logged value can never carry a credential regardless of shape. Pinned by `TestDSNEndpoint` over the exact PlanetScale go-sql-driver form.

## Added

- **`--control-keyspace` is now a per-entry `control-keyspace` field in the `syncs.yaml` fleet config, closing the v0.99.193 known limitation.** v0.99.193 added `--control-keyspace` (and its auto-detect) to the single-target `sync` commands but not the fleet paths; it is now a per-entry field threaded through `sync run` and `sync status --all` via the same `applyControlKeyspace` chain, with identical behavior — auto-detect the unsharded sidecar keyspace when the field is unset and the target is sharded, override when set. The `status --all` de-dup key includes the control keyspace so distinct sidecars don't collapse into one bucket. A fleet with a sharded PlanetScale/Vitess target can now be driven end-to-end from `syncs.yaml`.
- **The fleet config gains per-entry `allow-cross-shard-merge` and `inject-shard-column` fields, so a sharded→sharded fleet move can opt into the cross-shard collision guard instead of refusal-looping (this path never worked in the fleet before — the flags simply had no fleet equivalent since the sync command center landed in v0.99.132).** These mirror `sync start`'s `--allow-cross-shard-merge` / `--inject-shard-column` (ADR-0048 / Bug 152) and are threaded onto the streamer exactly as the single-command flags are (reusing the shared `parseInjectShardColumn`): `inject-shard-column` is the structural NAME=VALUE fix (a per-shard discriminator column + composite PK so cross-shard rows land disjoint), `allow-cross-shard-merge` is the "my keys are globally unique across shards" override. A load-time `validateCrossShard` guard refuses a per-entry entry that sets *both* (they are mutually exclusive in effect — the discriminator makes the override inert — so a curated config surface refuses the contradiction loudly, naming the stream-id, rather than let an operator believe an inert override took effect) and validates the `inject-shard-column` NAME=VALUE shape at config-load instead of at first cold-start. Pinned by fleet cross-shard unit coverage over both-set refusal, malformed-value refusal, and the each-set-alone threading.
- **The fleet loader now rejects an unknown or typo'd `syncs.yaml` key with a loud load failure instead of silently dropping it.** The koanf unmarshal sets `ErrorUnused`, so a misspelled `allow-cross-shard-merge`, or a `sync start` flag that isn't part of the curated fleet subset, now fails the config load loudly rather than being ignored with no effect. Every key documented in the `syncs.yaml` samples (ADR-0122 / ADR-0126, the docs, and the `fleet-operator` skill) was verified to be a real `SyncSpec` / `RestartConfig` field first, so this only rejects genuinely-unsupported keys. See Compatibility — this is the one behavior change for existing configs.

## Compatibility

**Drop-in for the data path; one config-load behavior change.** No data-path change, no backup format-version change, and no CLI flag semantics change for existing invocations — single-command `sync start` and every migrate / backup / restore path are byte-identical. The engine × direction reach is MySQL/PlanetScale/Vitess sync and the fleet only; Postgres, SQLite/D1, and every non-fleet command are untouched. **The single behavior change:** a `syncs.yaml` that carried an unknown or typo'd key was previously ignored (silently dropped); it now fails the fleet config load loudly. Any fleet config that *loads today* and uses only documented keys keeps loading and behaving identically — the only configs that break are ones already silently mis-configured (a knob that was never taking effect). The shard-discovery retry adds latency only in the transient fresh-keyspace case (a healthy keyspace returns on the first read); the idle-source exemption changes only *when* a stream gives up (an idle source no longer does, a never-established stream still fails loudly on the same budget); and the DSN redaction changes only logged/plan output, never a connection.

## Who needs this — action required

- **PlanetScale/Vitess fleet operators: before upgrading, check your `syncs.yaml` for unknown or misspelled keys.** After v0.99.194 an unsupported key fails the load loudly instead of being silently ignored. If your fleet loads cleanly on v0.99.194 you are fine; a load failure names a key that was never taking effect on prior releases — fix or remove it (this surfaces a latent mis-configuration, it does not change any correct config's behavior).
- **PlanetScale/Vitess operators on idle or low-traffic sources: no action, but note the behavior change.** A healthy idle source that previously gave up after ~6 minutes now stays up; if you had a workaround (raising `--apply-retry-attempts`, external restart loops) it is no longer needed. No data was ever lost — an idle source warm-resumes cleanly — so there is nothing to re-verify; the sync simply no longer stops.
- **Operators syncing into a freshly-provisioned sharded keyspace: no action.** The transient 0-shard abort that forced relaunches is handled automatically now; nothing to re-run.
- **Operators moving a sharded source into a sharded target via a fleet: you can now do this from `syncs.yaml`** by setting `control-keyspace` (or leaving it unset to auto-detect) plus exactly one of `inject-shard-column` / `allow-cross-shard-merge`.
- **Everyone else: no action.** Postgres, SQLite/D1, single-command `sync start`, and every migrate/backup/restore path are byte-identical; nothing to re-verify, nothing to re-run.
