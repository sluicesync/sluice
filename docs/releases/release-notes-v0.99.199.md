# sluice v0.99.199

**CRITICAL fix ‚Äî a `migrate` or `sync` cold-start into a PlanetScale / self-hosted Vitess MySQL target silently created NO secondary indexes (only PRIMARY keys), exit 0 with zero warnings, on v0.99.30 through v0.99.198. Every plain, unique, composite, and FK-backing secondary index was silently dropped, degrading every query that relied on them and defeating the v0.99.198 `--skip-foreign-keys` "keep the columns indexed" guarantee. Upgrade and check the secondary indexes on any Vitess/PlanetScale MySQL target you migrated or cold-started on those releases ‚Äî they are very likely missing. Drop-in: no flag, default, or data-path change beyond building the indexes that should always have been built. Non-Vitess targets (plain MySQL, Postgres, SQLite/D1) were never affected.**

## Fixed

- **A migrate or sync cold-start into a PlanetScale / Vitess MySQL target now builds every secondary index instead of silently building none (CRITICAL; silent schema loss ‚Äî exit 0, zero WARN; regression present in v0.99.30 through v0.99.198).** v0.99.30's ADR-0080 extended the overlapped (during-copy) index build to MySQL targets. That path correctly DECLINES to run concurrent `ALTER ‚Ä¶ ADD INDEX` against a Vitess/PlanetScale flavor ‚Äî concurrent ALTERs fight vtgate's online-DDL / Safe-Migrations queue ‚Äî but the branch it declined into was a no-op (`drainTablesFiringCallback`) that drained the completed-tables channel and marked every table `IndexesBuilt=true` WITHOUT building anything, and a false code comment claimed the post-copy whole-schema `CreateIndexes` ran as a fallback when on the overlapped branch that fall-through was unreachable (it existed only in the orchestrator's non-IIB else branch). So on a Vitess/PlanetScale target every secondary index ‚Äî plain, unique, composite, and FK-backing ‚Äî was silently dropped; the target came out with PRIMARY keys only, and it also defeated the v0.99.198 `--skip-foreign-keys` guarantee (the referencing columns it "keeps indexed" vanished), turning joins into full scans. Before v0.99.30, MySQL didn't take the overlapped path, so a Vitess migrate built its indexes via the sequential `CreateIndexes` and they landed ‚Äî this is a regression, not a long-standing gap. The fix, on both migrate and the sync cold-start (they share `BuildTableIndexesFromChannel`): the Vitess/PlanetScale gate now drains the completed-tables channel to completion (never backpressuring the copy producer) and then builds each drained table's indexes SERIALLY post-copy ‚Äî one table at a time, so there are never concurrent ALTERs against vtgate, and with byte-identical SQL to `CreateIndexes` (same inline-skip carve-out, alphabetical order, one combined `ALTER TABLE ‚Ä¶ ADD INDEX a, ADD INDEX b` per table = a single InnoDB scan). It is build-then-mark: a table's `IndexesBuilt` fires only AFTER its build succeeds, so a mid-build failure leaves the unbuilt tables unmarked and a `--resume` rebuilds them rather than stranding them marked-done. This restores the pre-v0.99.30 behavior for Vitess, which never had the overlap (that was plain-MySQL-only) ‚Äî so there is no throughput regression versus how Vitess always behaved ‚Äî and the vanilla-MySQL overlapped path is byte-identical. Pinned by unit tests (the Vitess path now emits an `ADD INDEX` per table and observes build-then-mark ordering) plus a vttestserver Vitess-target integration test asserting every index family lands for both migrate and sync cold-start (ran green on real Vitess).
- **New loud-failure safety net `SLUICE-E-INDEX-MISSING` (ClassRefusal, exit 3) ‚Äî flavor-agnostic, all MySQL flavors ‚Äî so this silent-loss class can never recur unnoticed on any engine.** A new optional `ir.IndexVerifier` / `SchemaWriter.VerifyIndexes` surface is wired into the orchestrator immediately after the index phase, on both the migrate + fast-parallel cold-start path (`runBulkCopyPhases`) and the serial cold-start path (`runBulkCopyWithOpts`). After the build, sluice probes `information_schema` for exactly the set the build targeted (reusing the same `indexBuildJobsForTables` eligibility, so the inline-emitted keys a `CREATE TABLE` already carries are never false-flagged) and refuses loudly ‚Äî naming the missing `table.index` list ‚Äî if any expected secondary index is absent. It is cheap (one catalog read per expected index) and general: it runs for every target that implements the surface, so no future refactor can re-break the build path silently. Engines without the surface are a no-op. Pinned by a `VerifyIndexes` unit matrix that raises the code when an index is missing.

## Compatibility

**No breaking changes; drop-in. No CLI-flag, default, format, or data-path change** ‚Äî the only behavioral change is that a Vitess/PlanetScale MySQL target now actually gets the secondary indexes it should always have gotten. The vanilla-MySQL overlapped index build is byte-identical, and Postgres, SQLite/D1, and all row/CDC data paths are untouched. The new `SLUICE-E-INDEX-MISSING` refusal fires only when an expected secondary index is genuinely absent on the target after the build phase; a run that builds its indexes correctly (every prior release did on non-Vitess targets, and every run does now) never sees it. There is no throughput regression on Vitess: it never had the during-copy overlap in the first place, so building its indexes serially post-copy is exactly how a Vitess migrate behaved before v0.99.30.

## Who needs this ‚Äî action required

- **Anyone who ran `migrate` or `sync` cold-start into a PlanetScale or self-hosted Vitess MySQL target on v0.99.30 through v0.99.198: upgrade and check that target's secondary indexes now.** They are very likely missing ‚Äî only PRIMARY keys will be present ‚Äî which silently degrades every query that relied on a secondary, unique, composite, or FK-backing index (and, if you used `--skip-foreign-keys`, leaves the referencing columns unindexed so joins full-scan). Verify with `SHOW INDEX FROM <table>` (or a schema diff against the source) on the affected tables, then rebuild the missing indexes: the simplest path is a fresh `migrate` onto a clean target on v0.99.199 (which now builds them and verifies them via `SLUICE-E-INDEX-MISSING`); alternatively, add the missing indexes manually with `ALTER TABLE ‚Ä¶ ADD INDEX`. Data rows were copied faithfully ‚Äî this is a schema-completeness loss, not a row-loss ‚Äî so re-verifying counts is not required, but re-verifying index presence is.
- **Anyone who migrated or cold-started into a non-Vitess target (plain MySQL, Postgres, SQLite/D1): no action.** Those paths never took the overlapped-decline branch, so their secondary indexes were always built. Upgrading is still recommended so the new `SLUICE-E-INDEX-MISSING` net guards any future run.
