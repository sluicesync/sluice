# sluice v0.99.255

**New: migrate now recovers automatically when a deferred post-copy `ADD INDEX` on a PlanetScale target hits the ~900s statement-time wall (errno 3024) or the safe-migrations direct-DDL block (errno 1105) — the ADR-0148 deploy-request index-build fallback, revived on the ADR-0162/0165 foundations (roadmap item 67).** With `--planetscale-org` plus a service token (ambient `PLANETSCALE_SERVICE_TOKEN_ID`/`_TOKEN` env works) and safe migrations already ON, the table's still-missing indexes build through a dev branch + deploy request — VReplication, async, unbounded by the wall — on the already-copied data, with no re-copy and no operator action. Purely additive drop-in: unarmed or unavailable, the index phase behaves exactly as before.

## Features

- **The automatic fallback (ADR-0148).** When a deferred index build on a `planetscale` target fails with errno 3024 (the ~900s statement-time wall) or errno 1105 "direct DDL is disabled" (the safe-migrations block) — on either deferred-build call site (the whole-schema post-copy build that resume/restore also take, and the VStream serial build-as-copied overlap) — sluice creates a dev branch off the target branch, applies that table's still-pending index DDL, opens a deploy request, deploys, skip-revert finalizes, and cleans up the branch. The writer re-probes the catalog first, so indexes that landed before the wall are never re-sent, and the DDL shipped is byte-identical to what the direct path would have run. It engages **only** when safe migrations is already ON — **sluice never toggles it** (the enable/disable propagation lag and the change it forces on every future schema change made any toggle-around design a non-starter; increasingly, safe-migrations-on is the assumed PlanetScale posture, and on such branches the direct deferred `ALTER` isn't just walled — it's refused outright with 1105, so until now those migrations ended with a hint instead of an index).
- **Arming is opportunistic, one deploy request per table, guarded against stale branches.** Six new optional `migrate` flags: `--planetscale-org` (env `PLANETSCALE_ORG`), `--planetscale-service-token-id`/`--planetscale-service-token` (env `PLANETSCALE_SERVICE_TOKEN_ID`/`_TOKEN`, the pscale-CLI convention), `--planetscale-database` (defaults from the `--target` DSN's database), `--planetscale-branch` (default `main`), `--planetscale-deploy-timeout` (default 1h). Anything missing logs a WARN at most — never a refusal. Batching is one dev branch + one deploy request per TABLE carrying every still-pending index statement (a multi-index table's second-and-later ALTERs — FULLTEXT/SPATIAL must be separate statements — ride the staged-DDL hook on the same branch before the single DR opens), keeping the diff, the narration, and a mid-run crash's blast radius one table wide. Every branch passes the ADR-0162 stale-base freshness gate (schema-compare + one-shot backup-rebase, coded `SLUICE-E-PS-BRANCH-STALE-BASE`) — without it a stale index branch's deploy request would silently propose reverting newer production schema. Branch names are deterministic (`sluice-index-<10-hex>`), leftovers from a crashed run refuse by name, and `--resume` converges regardless: the index phase re-probes and rebuilds only what is still missing.
- **A conservative `DATA_LENGTH` probe skips the doomed direct attempt on clearly-huge tables.** When armed, a cheap per-table `information_schema.TABLES` read routes tables past **64 GiB** straight to the deploy request instead of burning ~900s on a direct `ALTER` that cannot finish. The threshold is deliberately conservative (the sole ground truth is a 49 GB table dying at ~901s on a small tier; larger tiers build faster), and a mis-route is harmless in both directions — below it the wall failure still engages the fallback after the burn; above it the DR builds the same index correctly. **Speed honesty:** the fallback's value is UNBLOCKING, not speed — when the probe doesn't skip it, the direct attempt still burns up to ~900s before the fallback engages (on a 1105-refused branch it fails instantly, no burn), and the VReplication deploy is real wall-clock on large tables, polled with `--planetscale-deploy-timeout` and resume guidance.
- **Internals: the fallback composes the shared ADR-0165 `legRunner`** — one leg machine (branch provision + freshness gate + DR lifecycle + cleanup), three consumers: expand-contract's legs, `sluice deploy-ddl`, and now the index fallback. The chunk initially landed with its own deliberately collision-free duplicate and converged onto the shared extraction in the same release, as planned at both chunks' inception; plumbing only, all fallback pins pass unmodified.

## Compatibility

- **Purely additive; no breaking changes.** Six new optional `migrate` flags, one new opaque engine surface (`ir.IndexBuildFallback`), and **no new error codes** — the fallback reuses the ADR-0162 set (`SLUICE-E-PS-DEPLOY-REQUEST-FAILED`, `SLUICE-E-PS-BRANCH-STALE-BASE`) plus the existing index codes (`SLUICE-E-INDEX-STATEMENT-TIME-LIMIT`).
- **The never-worse contract is the load-bearing promise.** sluice never enables or disables safe migrations. Unarmed (no org/token, or a non-planetscale target — the env vars are ambient in pscale-shaped shells, so a stray token never changes behavior elsewhere) or unavailable (safe migrations off, bad token), the ORIGINAL direct error surfaces and the index phase ends exactly where it did before: at the existing `--upfront-indexes`/`--resume` hints, which now also name the fallback as the third option. The fallback only ever ADDS a recovery.
- Non-PlanetScale targets, all non-migrate commands, and every data path are untouched.

## Who needs this — action required

Nothing to re-verify — this release contains no silent-loss fix; the feature converts a loud dead-end into an automatic recovery. Pick it up if you migrate large tables onto PlanetScale: on a safe-migrations branch the deferred post-copy `ADD INDEX` was refused outright (errno 1105) and on any branch a big-enough table hit the ~900s wall (errno 3024) — both previously ended the index phase at a hint. Arm the fallback with `--planetscale-org` + a service token (branch + deploy-request scopes) and the run finishes its indexes through deploy requests instead. Per ADR-0148's own recommendation, do a live `psverify`-style run against a real PlanetScale database before relying on the fallback in anger — the client verbs, poller, and freshness gate are exactly the ADR-0162-hardened ones, but the fallback path's live validation so far is the unit/fake harness. Everyone else: nothing to do.
