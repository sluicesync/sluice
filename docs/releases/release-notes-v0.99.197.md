# sluice v0.99.197

**Security hardening ‚Äî an INVALID-URI DSN no longer leaks its password (or any userinfo token) into sluice's error output or logs. Go's `net/url.Parse` embeds the entire raw input ‚Äî including a DSN's `user:PASSWORD` userinfo ‚Äî verbatim in its `*url.Error` string, and sluice wrapped that error with `%w` at several DSN-parse sites, so a DSN that FAILS to parse as a URI (e.g. a password carrying an un-encoded control or special character) printed its credential on any surface that showed the error: a `sync status --all` unreachable line, a `migrate`/`sync`/`verify` failure, a log. Valid DSNs already parsed cleanly and were redacted (including the v0.99.194 `dsnEndpoint` work); this closes the parse-ERROR door of the same credential-in-logs class. Drop-in: no behavior change for valid DSNs or any data path ‚Äî the only change is that an invalid-DSN error now shows the failure reason without the DSN.**

## Security

- **An invalid-URI DSN no longer leaks its credential into sluice's error output or logs (credential-in-logs exposure; LOW likelihood ‚Äî only an INVALID-URI DSN triggers it ‚Äî but a real password disclosure; long-standing, present at the postgres DSN-parse site since v0.1.0, at pgtrigger since v0.85.0, and at fleettui since v0.99.137, all the way through v0.99.196).** Go's `net/url.Parse` returns a `*url.Error` whose `.URL` field is the verbatim argument, so for a DSN that fails to parse ‚Äî for example `postgres://user:PASSWORD@host‚Ä¶` whose password carries an un-encoded control character ‚Äî the error string is `parse "postgres://user:PASSWORD@host‚Ä¶": net/url: invalid control character in URL`, password included. sluice wrapped that error with `%w` at several DSN-parse sites and let it propagate, so any surface that printed the error ‚Äî the `sync status --all` unreachable line, a `migrate`/`sync`/`verify` failure, a captured log ‚Äî disclosed the credential. It is narrow: a VALID DSN parses cleanly and was already redacted (userinfo and query string stripped, including the v0.99.194 `dsnEndpoint` redaction), so only a malformed DSN reaches this path ‚Äî but a malformed DSN can still carry a real secret, so it is the same credential-in-logs class the v0.99.194 DSN-redaction work targeted, reached through the parse-error door. The fix: a new `diagnose.SafeParseError` helper unwraps the `*url.Error` (via `errors.As`, so it reaches the error even after a caller wraps it further with `%w`) and returns its `.Err` field ‚Äî the credential-free failure reason (e.g. "net/url: invalid control character in URL"), which is the useful part ‚Äî dropping the `.URL` field that carries the DSN. It is applied at every DSN-parse site that propagates the error: `internal/engines/postgres` (`parseURIDSN`, `withReplicationParam`, `withSchemaURI`), `internal/engines/pgtrigger` (`parseURIDSN`), and `internal/fleettui` (`NormalizeURL`, which also stopped echoing the raw connect address ‚Äî that address can carry basic-auth userinfo). A whole-tree grep confirmed no other in-class sites: the `err == nil`-guarded parses never touch the error, blobcodec's query-string echoes carry a documented no-secret invariant, and the Azure KMS key ID is a public resource locator. Valid-DSN handling is byte-identical ‚Äî only the parse-ERROR branch is sanitized. Pinned at three layers: a `diagnose` unit test that ground-truths the raw `url.Parse` error DOES contain the secret and then asserts the sanitized error does not, while preserving the reason; a postgres engine-site test at `parseURIDSN`; and a `fleettui` `NormalizeURL` test.

## Compatibility

**No breaking changes; drop-in. No CLI-flag, default, format, or data-path change.** Valid DSNs ‚Äî the overwhelming common case ‚Äî parse and connect exactly as before, and every migrate, sync, verify, backup, and restore path is byte-identical. The only observable change is on the error path: when a malformed DSN fails `url.Parse`, the resulting message now shows the failure reason (e.g. "net/url: invalid control character in URL") without echoing the DSN string. This touches every engine that accepts a URI DSN (Postgres and the Postgres-trigger CDC variant) plus the fleet TUI's connect-address parsing; MySQL/PlanetScale/Vitess DSN parsing, SQLite/D1, and all row/CDC data paths are unaffected.

## Who needs this ‚Äî action required

- **Anyone who ran sluice in CI, shared terminals, or captured logs where a DSN-parse error could have been recorded: if you saw a `migrate`/`sync`/`verify` failure or a `sync status --all` unreachable line that echoed a malformed `postgres://‚Ä¶`/`postgresql://‚Ä¶` DSN, treat any password in that DSN as exposed and rotate it.** The exposure required an INVALID-URI DSN (a valid one was always redacted), so it is low-likelihood, but where it fired the credential was printed in full. Upgrade to v0.99.197 to close the leak; there is nothing to re-run or re-verify for data correctness ‚Äî no copy, sync, or backup was ever affected, only the text of an error message.
- **Everyone whose DSNs are well-formed: no action.** A valid DSN never reached the leaking path; its userinfo was already stripped from error output. Upgrading is still recommended so a future malformed DSN can't disclose a credential.
