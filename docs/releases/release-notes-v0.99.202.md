# sluice v0.99.202

**HIGH fix + encrypted-backup integrity — (1) the `postgres-trigger` cold-start handoff could silently miss a transaction that was still open when the snapshot anchor ran (exit 0, no warning; every release of the engine, v0.85.0 through v0.99.201; epoch-INDEPENDENT and distinct from v0.99.201's xid8 fix; live-confirmed 2026-07-08), now closed with a bounded settle-wait plus an MVCC visibility clamp. (2) Encrypted backups gain real integrity via manifest FormatVersion 5: every encrypted chunk's AES-GCM ciphertext is now bound to its backup identity and recorded position, chain-CEK KMS unwraps are identity-bound (AWS EncryptionContext / GCP AAD / passphrase AAD; Azure documented as unable), Azure key auto-rotation no longer breaks restores of newly-written chains, and the chunk-header and SchemaHash checks two in-code comments falsely claimed now actually run. (3) PlanetScale/Vitess migrates now ride out a storage-grow reparent landing mid-index-build instead of aborting. Drop-in upgrade for all data paths; pre-v5 encrypted chains restore byte-identically — but note that encrypted backups WRITTEN by v0.99.202+ are refused (loudly) by older sluice binaries; see Compatibility.**

## Fixed

- **The `postgres-trigger` snapshot→CDC cold-start handoff no longer silently loses the changes of a transaction that was still open when the anchor query ran (HIGH; silent loss class — exit 0, zero WARN; present in every release of the engine, v0.85.0 through v0.99.201; live-confirmed 2026-07-08).** A change-log row INSERTed by a still-open transaction is MVCC-invisible to the anchor query, so the anchor's `MIN(id)-1` hold-down arm could land ABOVE that transaction's already-allocated lower id — its changes were then in neither the bulk copy (the transaction hadn't committed at the snapshot) nor the `id > anchor` CDC tail. This is the epoch-INDEPENDENT residual the v0.99.201 notes documented honestly and deliberately did not bundle: it affects the same range but through pure MVCC visibility, on any cluster, at any epoch (the v0.86.1 Bug-94 anchor rework and v0.99.201's xid8 fix each narrowed the cold-start gap class but could not close this hole). The fix, in `OpenSnapshotStream` after the anchor query on the pinned snapshot connection: export the snapshot's visibility horizon (`pg_current_snapshot()::text`), assign a fresh txid upper bound on the pool (`pg_current_xact_id()` — a freshly ASSIGNED xid, the same own-xid bound logical-decoding slot creation waits on), poll fresh snapshots until every pre-bound transaction settles (5-minute budget, then a loud refusal naming the stuck txids and the `pg_stat_activity` remedy), then clamp the anchor below `MIN(id)` of change-log rows not `pg_visible_in_snapshot` in the copy's horizon. A quiescent source exits on the first poll — no added latency in the no-overlap case. Stated honestly: the naive `pg_snapshot_xip`-only wait the roadmap sketched was proven UNSOUND during implementation — snapshot xmax is latestCompletedXid+1, so the only running transaction can sit at/above xmax outside every xip list (live-probed: snapshot `744:744:` with `backend_xid=744` idle-in-transaction); the shipped wait unions snapshot xmin (the oldest still-running txid) with xip under the assigned bound, and a pin freezes the unsound shape out. Pinned Phase-A failing-first and mutation-verified (both loss pins fail with the clamp disabled): a real-handoff engine test, a real-Streamer cold-start test reproducing the missing row on the target, the wedged-writer loud-refusal test at the at-/above-xmax shape, and the quiescent-source first-poll-exit test.
- **A PlanetScale storage-grow reparent landing mid-`ALTER … ADD INDEX` no longer aborts the whole migrate — both MySQL index-overlap paths now ride the transient out (loud failure before, `--resume` recovered, no data risk; the vanilla concurrent-worker path since the overlap landed in v0.99.30 through v0.99.201, the Vitess/PlanetScale serial build path since it landed in v0.99.199 through v0.99.201; the Postgres overlap has carried this ride-out since v0.99.120).** Both paths called `buildTableIndexes` bare, so a reparent mid-ALTER killed the migrate on exactly the platform where reparents happen, while the writer's own transient classifier went unused there. The fix mirrors the PG shape in-engine: a pure retry policy reusing the ADR-0108 cold-copy reparent envelope (same attempt/backoff variables, wall-clock bound, loud exhaustion naming the budget) and the same error classifier; the concurrent-worker path re-acquires a fresh pinned `*sql.Conn` after a transient, the serial Vitess path rides the pooled `*sql.DB`'s implicit reacquire. Replay is double-create-safe because `buildTableIndexes` re-probes detect-then-skip (Bug 131) — a partially-applied or committed-but-unacked combined ALTER is skipped, never re-emitted — and build-then-mark is unchanged, so `IndexesBuilt` still fires only after a table's build truly succeeds. Pinned by the PG-mirror policy set (rides-transient / terminal-unchanged / reacquire-rides-budget / loud-exhaustion / ctx-cancel) plus wiring pins through `BuildTableIndexesFromChannel` on both paths and both VStream flavors, including the no-double-create-after-mid-ALTER-retry pin; a new pipeline-level integration test also drives a real migrate end-to-end through the overlap wiring against a live vttestserver Vitess target (the flavor routing where the v0.99.199 silent-index-loss bug lived), asserting overlap engagement, zero index loss, and no back-pressure under a deliberately slow builder. Live-validated on real PlanetScale post-merge: a migrate carrying unique, plain, and composite index families landed every index (`information_schema` ground truth + the `SLUICE-E-INDEX-MISSING` verification green) with all rows counted.

## Security

- **Encrypted backup chunks are now cryptographically bound to their backup identity and recorded position, closing chunk splice/replay/reorder/substitution by a store-level adversary (ADR-0152, manifest FormatVersion 5; encrypted backups provided confidentiality ONLY since envelope encryption landed in v0.22.0, through v0.99.201 — while SECURITY.md's shared-storage framing and two in-code comments implied integrity that did not exist).** Chunk encryption was AES-GCM with nil AAD, the manifest's SHA-256 entries are adversary-writable, and per-chain mode (the default) shares one CEK across the chain — so any valid ciphertext chunk decrypted cleanly wherever a rewritten manifest pointed it, and the restore exited 0. Under a FormatVersion-5 manifest every chunk now carries GCM AAD binding the ciphertext to the manifest's identity prefix (CreatedAt/engine/kind — deliberately NOT the BackupID, which hashes the EndPosition unknown at chunk-write time) plus the chunk's recorded path, and change chunks additionally bind their list ordinal (replay order is semantic — without the ordinal, swapping two self-consistent manifest entries would replay events out of source order; row chunks deliberately don't bind it, since a table's rows are a set and the range workers legitimately append out of index order). A tampered, spliced, reordered, or cross-backup-replayed chunk now fails the auth check at open, before any row or event is emitted, with an error naming the class. The chain CEK's wrap is likewise identity-bound through the new `BoundEnvelope` surface — AWS KMS `EncryptionContext` (service-authenticated, CloudTrail-visible, key-policy enforceable), GCP KMS AAD, and GCM AAD on the passphrase wrap; Azure Key Vault deliberately does NOT implement it because the KeyWrap API has no context parameter and faking one client-side would claim integrity the service cannot provide (pinned as a non-implementation; Azure chains' splice resistance rests on the chunk AAD, which is the load-bearing defense everywhere). Version gating lives in one function per shape and is never try-both (try-both would convert the integrity check into a silent downgrade oracle). Honest boundary, documented in ADR-0152 and SECURITY.md: the manifest itself is UNSIGNED, so a store adversary can still present a complete self-consistent older manifest+chunks pair (whole-backup rollback) or truncate the tail of a change-chunk list while fixing up what the same manifest records — closing that requires a signed manifest and is explicitly out of scope. Pinned by golden derivation pins (the AAD strings are on-disk contract), an e2e tamper matrix (content swap with SHAs fixed up, entry reorder, same-CEK cross-backup replay — all refuse; untampered restores), the BoundEnvelope accept/refuse matrix over all three implementing families with the test fakes upgraded to enforce context/AAD equality like the real services, and a pre-v5 compat restore proving old chains restore byte-identically.
- **Azure Key Vault chains no longer break on restore when key auto-rotation is enabled: the wrap-time key version is now pinned at wrap and recorded in the manifest's KEKRef (audit N-9; Azure KMS chains since the envelope landed in v0.34.0, through v0.99.201).** The unwrap targeted the vault's LATEST key version and discarded the WrapKey response `KID`, while the code comment claimed Azure recovers the wrap-time version from blob metadata — that is AWS/GCP behavior (their ciphertexts self-identify the wrapping key version); Azure key-wrap ciphertext carries none. Enabling rotation — a common vault policy — made restores of older chains fail with a misleading "wrong key or tampered". Now the GetKey preflight resolves the operator's unversioned key URL to a concrete version, WrapKey responses reinforce the pin, `ResolvedKEKRef` records the VERSIONED key URL in `ChainEncryption.KEKRef` (per-chain and per-chunk modes), and the unwrap retargets the envelope at the recorded wrap-time version — advisory by contract, since the ref comes from an unauthenticated manifest: a malformed or foreign ref WARNs and keeps the current target, whose unwrap then fails loudly if wrong. Chains recorded before v0.99.202 carry an unversioned KEKRef and keep the latest-version behavior, plus a once-per-envelope WARN naming the rotation hazard and the recovery (pass the wrap-time version explicitly). Pinned by an Azure version-pin/rebind matrix including a simulated auto-rotation, with the fake reproducing Azure's real no-version-metadata semantics.
- **Restore now actually performs the chunk-header and SchemaHash checks two in-code comments claimed existed (hardening; the checks were absent in all prior releases).** `Restore` validates every chunk header's column set against the manifest schema's non-generated columns (set equality, source-shaped, pre-retarget; the first non-test caller of `ChunkReader.Header()`) and refuses with the table, chunk, and both column deltas before any row is emitted — the chunk-format doc's promised renamed-column check now exists, on plaintext and encrypted chunks alike, for single-full restore, every chain segment, and the broker's reset leg. Chain-restore recomputes and compares every link's recorded `SchemaHash` before anything lands on the target — honestly a corruption check (bit-rot, truncated rewrite), not tamper-proofing, since the hash lives in the same unsigned manifest as the schema, and the field's comment now says so. Write-side companions: fulls now record `SchemaHash` (previously only incrementals/streams did), and the no-DDL sequence-refresh swap re-stamps the hash over the swapped schema, closing the wart the old comment predicted a hash-verifying walker would trip on; one named carve-out WARNs and proceeds for pre-v5 manifests carrying standalone sequences (old writers legitimately swapped sequence state without re-hashing — refusing legitimate old chains on the DR path would be worse than the narrow miss). Pinned by the renamed-column header refusal (SHA fixed up), the SchemaHash mismatch refusal firing before any target phase, and the carve-out pin.

## Compatibility

**No breaking changes for existing invocations; no CLI-flag or data-path changes; migrate/sync/verify and all plaintext backup paths are byte-identical.** Three deliberate behavior changes to know about: (1) **Encrypted backups written by v0.99.202+ are stamped manifest FormatVersion 5** — OLDER sluice binaries refuse to restore them loudly at the existing unknown-FormatVersion preflight ("newer than this build supports; upgrade sluice") instead of failing with a misleading GCM error; pre-v5 encrypted chains restore byte-identically on v0.99.202 through the version-gated legacy path; plaintext backups keep their existing versions and stay readable by older binaries; an encrypted incremental extending a pre-v5 chain is itself stamped v5 (each link's chunks are gated by its OWN version, so mixed chains read correctly per link); and resuming a pre-v5 encrypted in-progress full deliberately keeps the prior version and unbound chunk shape, since its kept chunks were written unbound. (2) **A `postgres-trigger` cold start now waits (bounded, 5 minutes) for transactions that were open at the snapshot to settle** — a wedged writer transaction now surfaces as a loud refusal naming the stuck txids and the `pg_stat_activity` remedy instead of a silent data gap; quiescent sources exit on the first poll with no added latency. (3) **`backup compact` now refuses to merge Azure KMS segments wrapped under different key versions** (loud and correct — their unwrap targets genuinely differ; previously the merged chain would have failed later at restore). Additionally, a PlanetScale/Vitess reparent landing mid-index-build now retries within a bounded budget instead of aborting the migrate (strictly better; exhaustion is still loud), and two new WARNs exist: unwrapping an Azure chain with a pre-v0.99.202 unversioned KEKRef warns once about the rotation hazard, and a pre-v5 chain-restore link carrying standalone sequences may warn on the SchemaHash carve-out.

## Who needs this — action required

- **`postgres-trigger` sync operators whose source had ACTIVE WRITERS during any cold start (the initial `sync start`, or any resnapshot) on v0.85.0 through v0.99.201: upgrade, then re-verify the target against the source.** The window is narrow — a transaction that had already written to a captured table when the handoff anchor ran and committed afterwards — but the loss is silent (exit 0, no WARN) and this class was live-confirmed. Run `sluice verify` (or row counts/checksums) on synced tables; if discrepancies show, re-run a cold start on v0.99.202, where the settle-wait + clamp closes the hole. If your cold starts ran against a quiescent source (no concurrent writers), no data is at risk — upgrade for the future-proofing. This is the same verify-guidance shape as v0.99.201's epoch item, for the epoch-INDEPENDENT sibling hole.
- **Anyone storing encrypted backups on shared or third-party storage: upgrade — backups written from v0.99.202 carry chunk-position and CEK bindings, so a store-level splice/replay/reorder/substitution fails loudly at restore instead of exiting 0.** Chunks written by earlier releases remain unbound (their integrity cannot be retrofitted); if the threat model warrants it, cut a fresh encrypted full on v0.99.202 so the chain's protection starts now. Read the honest boundary in SECURITY.md: the manifest is unsigned, so whole-backup rollback and tail truncation by a store adversary remain out of scope until a signed-manifest design lands.
- **Azure Key Vault users with key auto-rotation enabled (or planned): upgrade before the next rotation.** Chains written by v0.99.202 pin and record the wrap-time key version and survive rotation; chains written earlier still unwrap against the latest version — if a rotation has already broken such a restore, pass the wrap-time key version explicitly in the key URL (the new WARN names this remedy).
- **PlanetScale/Vitess migrate operators: nothing to do.** A storage-grow reparent landing mid-index-build now rides out automatically; previously it was a loud abort that `--resume` recovered, so no past data is affected.
- **Everyone else: drop-in upgrade, no action.** The chunk-header and SchemaHash checks harden the restore path without changing any legitimate restore, and the two test-only additions (the VStream overlap e2e pin, the live refutation pin of the audit's NO_BACKSLASH_ESCAPES decoder hypothesis) change no behavior.
