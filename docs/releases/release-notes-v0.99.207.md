# sluice v0.99.207

**Silent-precision fix ‚Äî a PlanetScale / self-hosted Vitess VStream cold-start COPY (the `sync` cold-start snapshot and `backup full` on the VStream path) silently display-rounded single-precision `FLOAT`/`REAL` columns to 6 significant digits (`8388608` ‚Üí `8388610`), exit 0, on every release from v0.1.0 (cold-start) / v0.44.0 (backup) through v0.99.206 ‚Äî the exact MED-HIGH known-issue v0.99.206 disclosed. This release fixes it: single-precision FLOAT columns are now re-read EXACTLY from the source over SQL and the landed rows patched, EXACT by default on both paths. Drop-in for existing invocations (a data-correctness change ‚Äî affected FLOAT values now land correct rather than rounded); no CLI-flag change is required, and the three new opt-out/opt-in flags all default to exact. Scope is PlanetScale/Vitess sources only; every other engine and path is untouched. This CLOSES the v0.99.206 Known-issue.**

## Fixed

- **A VStream cold-start COPY on a PlanetScale / self-hosted Vitess source now lands single-precision `FLOAT`/`REAL` columns EXACTLY instead of silently display-rounding them to 6 significant digits (MED-HIGH; silent-precision class ‚Äî exit 0, zero WARN; present on the cold-start COPY since the VStream snapshot path landed in v0.1.0, and on the `backup full`-on-VStream leg since v0.44.0, through v0.99.206).** The mechanism, ground-truthed on vttestserver mysql80: vttablet's rowstreamer issues a bare-column `SELECT` over MySQL's text protocol, so the wire bytes are mysqld's own 6-significant-digit FLOAT rendering ‚Äî the exact bug class ADR-0153's `(col * 1E0)` DOUBLE-promotion projection fixed on the SQL reader, but that `SELECT` is built inside vttablet and is unreachable from the client, so the projection cannot be injected and the reader stays rounded by design. Every consumer of `openVStreamSnapshotStream` ‚Äî the `sync` cold-start snapshot and `backup full` on the VStream path ‚Äî inherited the rounding; `8388608` landed as `8388610`, `-123456.789` as `-123457`, float32-max as a different float32. The fix moves the repair one layer up and is EXACT by default on both consumers, gated by a new `ir.LossyFloatCopyReader` (the VStream cold-start reader returns true) plus an `ir.FloatRepairWriter` target surface (MySQL + PG). On **`sync` cold-start**, after the bulk-copy completes and BEFORE the CDC anchor is persisted, sluice re-reads each repairable table's single-precision FLOAT columns exactly from the source over a separate SQL connection (`Engine.OpenRowReader` ‚Üí the `(col * 1E0)` projection, float32-exact and sign-preserving) and UPDATEs the target rows by PK; the ordering is load-bearing ‚Äî CDC then replays from the copy anchor forward, so any value that changed inside the re-read window is re-applied to its final value (exact + eventually-consistent), and a crash mid-repair leaves no anchor so the run re-cold-starts. The re-read is bounded-memory: a keyset-paginated live UPDATE that streams its cursor rather than buffering the table. On **`backup full`**, a per-table exact re-read builds a PK‚Üíexact-floats map and patches each archived COPY row's FLOAT columns before it is written; the VGTID/EndPosition is untouched. `DOUBLE`/double-precision columns (exact on both legs before and after), the CDC/binlog steady-state leg (float32-exact ‚Äî vttablet re-encodes the binlog float32 bits shortest-round-trip), float `-0.0` sign, and every non-PlanetScale/Vitess source (vanilla MySQL, Postgres including PlanetScale Postgres, SQLite/D1) were never affected and are unchanged. Pinned by converting `TestVStream_FloatCarrierParity`'s previously-skipped COPY sub-test into a reader-still-rounds ground-truth pin (the reader is unfixable by design), with target exactness pinned at the pipeline layer (`TestStreamer_FloatRepair_*_VStream_PG`), the MySQL writer (family √ó shape), and the backup patch reader / plan / key. The upstream root-cause fix would live in Vitess's rowstreamer (long lead, out of scope) and is noted.
- **The `backup full` exact FLOAT re-read is bounded-memory: capped at `--float-reread-max-rows` (default 2,000,000 rows) with a LOUD fallback, never a silent OOM.** The exact-patch reader buffers a per-table PK‚Üífloat map; on a large FLOAT-bearing table an unbounded buffer would OOM by default, violating the ADR-0071 bounded-memory tenet. The map build now stops at the cap (default ‚âà a few hundred MB worst case, and the serial VStream sweep bounds it to one table at a time) and reports over-cap instead of buffering the whole table. An over-cap table falls back LOUDLY ‚Äî by default a WARN plus that one table archived ROUNDED (every under-cap table stays exact), or under `--strict-float` a refusal (`SLUICE-E-VSTREAM-FLOAT-LOSSY`, exit 3). A bounded streaming merge-join was ruled infeasible on ground truth: the VStream COPY stream is NOT PK-ordered ‚Äî Vitess scans by a cheaper unique key than the table's PK and re-emits already-scanned rows during binlog catch-up (out of order, with duplicates) ‚Äî so there is no monotonic key on the COPY side to merge against the strictly-ordered exact re-read; the capped buffer with loud fallback is the bounded-memory answer. The `sync` cold-start path needs no cap ‚Äî it already streams the re-read cursor-paginated.

## Changed

- **FLOAT columns copied or backed up from a PlanetScale/Vitess source are now exact by default ‚Äî a data-CORRECTNESS change.** Values that previously landed rounded to 6 significant digits now land at their true float32 value. Anyone who (inadvertently) relied on the rounded values will see different ‚Äî correct ‚Äî numbers; that is the fix, not a regression.
- **`backup full` of a very large FLOAT-bearing table may now emit a WARN and fall back to a rounded archive for that table** when it exceeds `--float-reread-max-rows` (default 2,000,000). Raise the cap if you have the memory headroom and want that table exact, or accept the rounded fallback (the WARN names the table). Every table under the cap is archived exact.
- **New flags (all opt-out/opt-in named, so the Go zero value keeps exact-by-default):** `--no-float-exact-reread` (`sync` + `backup`) keeps the old rounded-but-perfectly-consistent behavior ‚Äî for operators who value within-row snapshot consistency over sub-6-digit FLOAT precision; `--strict-float` (`backup` only) refuses loudly (`SLUICE-E-VSTREAM-FLOAT-LOSSY`, exit 3) rather than archive anything it cannot make exact, keeping the exact archive for every repairable under-cap table and refusing only the un-exactable ones (keyless / float-PK-only, refused upfront; over-cap, refused when reached); `--float-reread-max-rows` (`backup` only, default `0` ‚Üí 2,000,000) tunes the memory cap. All three are inert on non-VStream sources (native MySQL, Postgres ‚Äî already float-exact) and when a table has no single-precision FLOAT columns.

## Compatibility

**No breaking changes; drop-in for existing invocations.** No existing flag, default, or format changes meaning, and the fix requires no flag to engage ‚Äî the exact re-read is on by default. Three deliberate, well-bounded behavior changes, all in the correctness / loud-over-silent direction:

1. **Data correctness:** single-precision FLOAT columns from a PlanetScale/Vitess cold-copy or `backup full` now carry their exact float32 value instead of the mysqld-rounded one. Nothing that was correct becomes wrong; values that were silently lossy become exact. Anyone who compared targets byte-for-byte against the old rounded output will see the FLOAT columns change ‚Äî to the correct values.
2. **Backup of a large FLOAT table** may WARN and fall back to a rounded archive for that table when it exceeds `--float-reread-max-rows`. This is loud (never silent), affects only single tables over the cap, and is tunable (`--float-reread-max-rows`) or acceptable as-is.
3. **`--strict-float`** (opt-in only) can refuse a `backup full` that previously succeeded ‚Äî by design, it demands exact-or-fail and refuses keyless / float-PK-only or over-cap FLOAT tables (`SLUICE-E-VSTREAM-FLOAT-LOSSY`, exit 3). It never fires unless you pass it.

Scope is PlanetScale MySQL (Vitess-backed) and self-hosted Vitess sources ONLY, on the cold-start COPY and `backup full` legs only. Vanilla MySQL, Postgres (including PlanetScale Postgres), and SQLite/D1 sources do not use the VStream COPY path and are byte-identical to prior releases. The CDC/binlog steady-state leg and `DOUBLE` columns were always exact and are unchanged. The one honest cost, `backup full` only: the exact re-read introduces a bounded within-row temporal skew (the FLOAT reflects a read instant slightly after the snapshot VGTID) ‚Äî ZERO on a quiescent source, SELF-HEALING on a chain restore (incrementals replay from the full's EndPosition forward), and persisting only for a standalone-full restore of a source with concurrent FLOAT writes; `--no-float-exact-reread` restores the old rounded-but-perfectly-consistent snapshot for operators who prefer that trade.

## Who needs this ‚Äî action required

- **PlanetScale-MySQL / self-hosted-Vitess operators who ran a `sync` cold-start or a `backup full` that copied single-precision `FLOAT`/`REAL` columns on any release from v0.1.0 (cold-start) / v0.44.0 (backup) through v0.99.206: upgrade, and re-verify precision-sensitive FLOAT data on those targets and in those backups.** The copied/archived values were display-rounded to 6 significant digits and may differ from the source past that precision (exit was 0, so nothing flagged it ‚Äî this is the v0.99.206 Known-issue). On v0.99.207 a fresh `sync` cold-start or `backup full` lands those columns exact by default; to correct an already-copied target, re-run the cold copy (or, if it's a live sync, the exact re-read + CDC replay heal it on a fresh cold-start). `DOUBLE` columns, the CDC/binlog steady-state path, and all non-PlanetScale/Vitess sources were never affected ‚Äî you only need to check single-precision FLOAT columns that went through a VStream cold-start COPY or a VStream `backup full`.
- **PlanetScale/Vitess operators backing up very large FLOAT tables:** confirm your `backup full` runs don't hit the 2,000,000-row default cap on a FLOAT-bearing table ‚Äî if they do, that one table is archived rounded with a WARN. Raise `--float-reread-max-rows` (with the memory headroom) for an exact archive, or pass `--strict-float` to refuse rather than archive anything lossy.
- **Operators who value within-row snapshot consistency over FLOAT precision:** `--no-float-exact-reread` keeps the prior rounded-but-perfectly-consistent behavior on both paths.
- **Everyone else (vanilla MySQL, Postgres, PlanetScale Postgres, SQLite/D1; anyone not copying FLOAT from a Vitess source): drop-in upgrade, no action.**
