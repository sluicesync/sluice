# sluice v0.99.188

**Fixed — two more MySQL `information_schema` lossy-read fidelity fixes, the sibling sweep after the v0.99.186/187 BINARY-`DEFAULT` thread: an ENUM/SET label that carries a control byte (e.g. a NUL) is now decoded byte-exact instead of landing as a wrong allowed-value set, and a NUL-truncated table `COMMENT` is recovered faithfully. Both are schema-metadata fidelity fixes (not data-value fixes), both are uncommon in practice, and the same audit *cleared* the high-severity suspects — generated-column expressions, string/`VARCHAR` defaults, `CHECK` clauses, and view definitions are all faithful in `information_schema`, so there is no "wrong computed values" class and v0.99.187's binary-only scoping is confirmed correct. Drop-in: no flag, default, or backup-format change.**

## Fixed

**MySQL ENUM/SET labels are now decoded with MySQL's full string-escape set, so a label containing a control byte lands byte-exact instead of as a wrong allowed-value set (medium-low; silent schema corruption; pre-existing since the MySQL engine's first release, v0.1.0).** `parseEnumOrSet` reads the allowed-value list out of `information_schema.columns.COLUMN_TYPE`, but it decoded only two escapes — a doubled quote (`''`→`'`) and a doubled backslash (`\\`→`\`) — while `COLUMN_TYPE` actually re-escapes control bytes inside a label with MySQL's *full* single-backslash string-escape set (`\0 \b \t \n \r \\` plus the doubled quote). The two-escape shortcut passed every other escape through literally, so an enum label whose real value contains a control byte — e.g. `ENUM('nul_x\0y')`, where the label is `nul_x` + NUL + `y` — was carried into the IR as the literal 8-character string `nul_x\0y` (backslash + `0`), i.e. a *wrong* set of allowed values on the target enum/domain. That is silent schema corruption (exit 0, no error); it is rare because control characters in enum/set labels are uncommon, but nothing in `COLUMN_TYPE` distinguishes it. Each label is now decoded with the same MySQL string decoder v0.99.187's binary-default recovery already uses (`decodeMySQLQuotedString`), refactored onto a shared escape-aware scanner (`scanMySQLQuotedString`) that also makes the comma/quote *split* escape-aware — a label containing an escaped comma, an escaped quote, or a `\0` can no longer mis-terminate and mis-split the list. A well-behaved ASCII enum decodes byte-identically to before. Cross-engine, a genuinely NUL-bearing label round-trips byte-exact MySQL→MySQL (MySQL accepts a raw NUL in an enum label) and *loud-fails* at apply MySQL→Postgres (PG `text` cannot hold a NUL) — correct loud-failure, not silent loss. Pinned across the family (Bug-74 discipline): a unit matrix over every escape (`\0 \b \t \n \r \\`, doubled and backslashed quote) × ENUM and SET × comma-in-label × empty label × a NUL label mid-list × an unchanged plain-ASCII enum, plus MySQL→MySQL integration round-trips for escaped labels and a byte-exact NUL label.

**MySQL table `COMMENT`s that contain a NUL byte are now recovered faithfully from `SHOW CREATE TABLE` instead of the NUL-truncated `information_schema` value (cosmetic; low; pre-existing since v0.1.0).** `information_schema.tables.TABLE_COMMENT` C-string-truncates a table comment at its first `0x00` byte, exactly as `COLUMN_DEFAULT` does and unlike `COLUMN_COMMENT` (which does not truncate — so column comments are left alone). A comment carrying a NUL was therefore read short (or, for a leading-NUL comment, empty). The reader now recovers a non-empty comment from the authoritative `COMMENT='…'` clause of `SHOW CREATE TABLE`, decoded with the same escape-aware decoder. This is the same silent-truncation class as the binary-`DEFAULT` fix, but table comments are cosmetic metadata, so the severity is low. The extra read is amortized onto v0.99.187's existing per-table `SHOW CREATE` recovery pass: a table that needs *either* a binary-default recovery or a comment recovery is fetched exactly once, and a table needing neither (the common case — no binary default and an empty comment) pays no extra query. A `SHOW CREATE` that cannot be re-read keeps the (possibly-truncated) comment with a WARN, since a short comment is cosmetic, not corrupting. Pinned by a clause-parse unit matrix (NUL-bearing, doubled-quote, newline/tab escapes, no-comment, and the two false-match traps — a column-level `COMMENT '…'` must not be picked up, and a table comment whose text itself contains `COMMENT=` still resolves the real option), the amortization-gate unit test, and a MySQL→MySQL integration round-trip over a NUL-bearing / plain / no-comment trio.

## Compatibility

**No breaking changes; drop-in.** No CLI flag, default, or backup format-version change affects any existing invocation. Both fixes are additive metadata-fidelity corrections: an enum/set label with a control byte, or a table comment with a NUL, that previously landed wrong now translates faithfully; every other schema-read path — every well-behaved ASCII enum/set label and every non-NUL table comment — is byte-identical. All four MySQL↔Postgres directions plus SQLite/D1 targets are covered; non-MySQL sources are untouched, and all sync / backup / restore runtime paths are unchanged. The only new observable behaviors are (a) a correct loud-failure at apply for a genuinely NUL-bearing enum label targeting Postgres `text`/`enum` (PG cannot hold a NUL — this was already a loud-failure, now with the byte-correct value driving it), and (b) a WARN when a table comment cannot be re-read from `SHOW CREATE`. Internally, the release also stops auto-pushing the winget manifest during `release.yml` (winget is submitted manually and selectively, and its fork auto-push intermittently flaked on the GitHub API and could fail the whole release after the binaries, GHCR image, and homebrew/scoop taps had already published) — no effect on the shipped binary or on brew/scoop/container installs.

## Who needs this — action required

- **Anyone who migrated a MySQL schema containing a control-character ENUM/SET label — a label whose real value carries a NUL or other control byte (`\0 \b \t \n \r`) — to MySQL, Postgres, or SQLite/D1 on any release through v0.99.187.** The target enum/domain got a **wrong allowed-value set** silently (exit 0, no error) — the literal backslash-escaped text (`nul_x\0y`) instead of the real bytes. Re-run the schema migration on v0.99.188, or re-verify the affected columns' allowed values, rather than trusting the earlier result. This shape is uncommon (control chars in enum labels are rare), so most schemas are unaffected.
- **Anyone who migrated a MySQL schema with a NUL-bearing table `COMMENT`.** The comment was truncated at the first NUL (cosmetic only — no data or schema-structure impact); re-run on v0.99.188 to carry it faithfully if the full comment matters to you.
- **Everyone else: no action.** Plain ASCII enum/set labels, non-NUL table comments, column comments (never truncated), generated-column expressions, string/`CHECK`/view definitions (all confirmed faithful this audit), non-MySQL sources, and all sync / backup / restore paths are unchanged and byte-identical.
