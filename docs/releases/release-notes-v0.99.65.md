# sluice v0.99.65

**New: the `sluice sync` / VStream cold-start copy now fans the snapshot out to N concurrent batched-INSERT writer workers, closing the cross-region PlanetScale-MySQL throughput gap (ADR-0097).** Opt-in and drop-in: add `--copy-fanout-degree N` on `sync` to enable it (the new flag defaults to degree 4 when set; a single-stream sync is unchanged otherwise). This is the write-side analogue of the within-table read chunking — for the VStream path where read-side chunking is impossible — and it does not touch the `sluice migrate` path, which has its own chunking (ADR-0096).

## Features

- **Write-side parallel writer fan-out on the VStream/CDC cold-start copy (ADR-0097).** A `sluice sync` cold-copy from a Vitess/PlanetScale source previously drained the snapshot through a single reader into a single writer. Against a cross-region PlanetScale-MySQL target that one connection is round-trip-bound — PlanetScale blocks `LOAD DATA LOCAL INFILE`, so the copy falls back to one pinned-connection batched-`INSERT` stream measured at ~13 GB/h. You cannot PK-range-chunk the read side here (vtgate streams the snapshot down one logical channel; there is no arbitrary range-`SELECT` to split — this is exactly what makes the path differ from `sluice migrate`), so the only lever is write-side fan-out. This release fans the single incoming snapshot row-stream out to N concurrent batched-INSERT writer workers, each pinning its own target connection with its own ~1 MB batch buffer, via a new engine-neutral writer capability (`ParallelIdempotentCopyWriter`) implemented by the MySQL target. Rows are routed by a deterministic PK-hash (`worker = fnv1a(pk_bytes) % degree`), not round-robin, so every emission of a given primary key always goes to the **same** worker — re-emitted rows during a VStream COPY (the Bug-125 idempotent path) serialize cleanly within one worker's batch stream and never race two upserts on the same row across workers. A live experiment showed near-linear scaling: 3–4 workers already beat PostgreSQL's single-stream COPY. Opt in via `--copy-fanout-degree N` on `sync`; the degree is a zero-value-safe `int` (0 → the default of 4, 1 → the existing serial path, the v0.99.51 trap avoided by naming the field for the safe default rather than an `Enable…` bool that inverts for non-CLI callers), and the connection-budget preflight accounts for the fan-out degree so the loud refusal still fires if the target has no slots.
- **Exactly-once landing, with durability pinned at the whole-table join.** The fan-out is correctness-load-bearing: the reader sends each row to exactly one worker channel (no drop, no dup, no fan-to-many), and order-independence across workers is already guaranteed by the idempotent upsert (ADR-0010), so cross-worker interleaving is irrelevant to the final state. No snapshot position advances until **every** worker has drained its channel, flushed its final batch, and had that batch commit — the whole-table join at `COPY_COMPLETED` is the sole durability guarantee on a fanned-out table, and the orchestrator persists the COPY_COMPLETED position only after that join returns. To keep resume gapless the ADR-0072 mid-COPY durable-progress watermark is disabled on fan-out tables (a flat flushed-row count summed across workers is not order-equivalent to the reader's enqueue-order frontier, so a mid-COPY breadcrumb could otherwise checkpoint past a row a lagging worker has not yet flushed); serial (non-fan-out) tables in the same run keep their mid-COPY checkpoint unchanged. Any worker error aborts the table copy loudly (first error cancels a shared child context, the others unwind, the call returns the error wrapped, and the orchestrator fails the phase with no position advance and no partial silent success), ctx-cancel leaks no goroutines or pinned connections, and the reader stream-error gate (Bug 68) still surfaces a mid-stream scan/decode abort loudly. Pinned by exactly-once routing tests (including re-emitted PKs and same-PK→same-worker), a flush-before-position guard, a mid-COPY-checkpoint-inert-under-a-lagging-worker test, loud-worker-error and ctx-cancel-no-leak tests, the zero-value-safe degree test, and an integration test on a real MySQL container that cold-copies a sizable table with fan-out and asserts `COUNT(*)` + a content checksum match the source exactly.

## Compatibility

- **No breaking changes; drop-in upgrade.** The fan-out is opt-in via the new `--copy-fanout-degree` flag on `sync`. Without the flag a sync behaves exactly as before (single-stream serial copy); existing invocations are unaffected, and the persisted position/handoff token shape is unchanged.
- **Scope: the `sluice sync` / VStream (Vitess/PlanetScale-source) cold-start snapshot copy only.** This is where the single-connection cross-region tail was most visible (PlanetScale-MySQL target, `LOAD DATA` blocked). The `sluice migrate` path is untouched — it already has its own within-table parallel chunking (ADR-0096/ADR-0019). PostgreSQL targets stay serial by default: the eligible PG cold-copy uses the fast raw-COPY / parallel snapshot path, and PG's writer does not implement the parallel capability, so the default differs by writer capability rather than by flag.
- **Fan-out applies to PK'd tables only; everything else falls back safely.** PK-hash routing needs a partition key, so a table with no usable primary key (the Bug-125 no-PK case) routes through the existing serial cold-start path unchanged — the no-PK loud-refusal / unique-key contract is never weakened. A degree of 1, a non-parallel-capable writer, or a no-PK table all fall through to today's serial copy; the fan-out never silently does nothing.
- **Resume of a fanned-out table stays single-stream (a documented v1 limitation, consistent with ADR-0095, not a correctness gap).** A process-restart resume runs the serial resume path; because the fan-out copy never persisted a mid-COPY cursor, a resume either restarts the whole table's COPY from the last fully-durable position (the idempotent upsert absorbs the re-copied overlap) or skips a completed table.

## Who needs this — action required

- **Anyone running `sluice sync` from a Vitess/PlanetScale source into a PlanetScale-MySQL target who wants faster cold-start throughput:** add `--copy-fanout-degree 4` (or higher) to your `sync` invocation. The default-on behavior is unchanged, so the speed-up is opt-in — start at 4 (already beats PostgreSQL's single-stream COPY in the experiment) and raise it for a known-large cross-region copy, bounded by your PlanetScale connection budget (the preflight will refuse loudly if there are no slots).
- **No re-verification of prior syncs is required.** This is a performance optimization to *how fast* the cold-start copy runs, not a fix to a path that was producing wrong or missing data — the previous single-stream copy was correct, just slow, and completed syncs on prior versions are unaffected and need no re-check.
