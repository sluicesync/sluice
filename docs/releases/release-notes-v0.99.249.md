# sluice v0.99.249

**Hardening: the MySQL→PG schema advisories now fire for every MySQL-family source (mydumper and vitess included), one unsupported-charset table no longer blocks migrating the rest of a mydumper/`pscale database dump` directory, sluice's own control tables get schema hygiene (LONGTEXT position columns on MySQL + the full 11-table reader-exclusion roster), and MySQL `TIME` duration columns get a preflight advisory naming the lossless `interval` override.** Everything in this release is advisory or loud-failure class — zero silent-loss classes, in this release or in the gaps it closes — and it is a drop-in upgrade: no flag, default, or CLI change.

## Added

- **MySQL `TIME` → PG range advisory — the default mapping is unchanged (Bug 187's improvable half).** MySQL `TIME` is semantically a *duration* spanning `-838:59:59`…`838:59:59`; PostgreSQL `time` is a time-of-day (00:00:00–24:00:00, never negative), so a stored duration outside that window has no representation on the target. That case already refused loudly at copy time (pgx declines to encode; coded `SLUICE-E-BULKCOPY-TABLE-FAILED`, zero rows on the table, zero silent loss) — but as a raw driver error, mid-copy, after earlier tables had already landed, with no sluice-framed hint. Now `schema preview` emits a per-column hint on every MySQL-family → PG `TIME` column and `migrate`/`sync` preflight WARNs with the same wording before any DDL or data moves, both naming the copy-pasteable lossless remedy: `--type-override TABLE.COL=interval` (PG `interval` holds the full MySQL `TIME` range, and the override is first-class on both migrate and continuous sync). The scan runs on the post-override schema, so an already-overridden column is not re-nagged. The default `TIME` → `time` mapping deliberately stays — time-of-day is the overwhelmingly common usage, and an out-of-range value still refuses loudly, never clamps or wraps. `docs/type-mapping.md` gains the `TIME(p)` caveat row and a dedicated edge-case section.

- **Control-table position columns widened `TEXT` → `LONGTEXT` on MySQL (roadmap item 65a).** The engine-opaque position token is a GTID/VGTID set with no upper bound sluice controls, and `sluice_cdc_schema_history.anchor_position` has been LONGTEXT from the start for exactly this reason — but `sluice_cdc_state.source_position` and `sluice_shard_consolidation_lease.anchor_position` were 64 KB `TEXT`, so a very large set (≈1000 server-UUID entries, or a very heavily sharded VGTID) would have aborted the position write loudly mid-stream (sluice pins `STRICT_TRANS_TABLES`, so never a silent truncation — an avoidable stream abort, never observed in the field). Fresh CREATEs now declare LONGTEXT directly; tables created by an earlier binary widen exactly once via a new detect-then-ALTER migration (information_schema `DATA_TYPE` probe, `MODIFY` only when the column is still `text`). Detect-first is load-bearing, not an optimization: on a PlanetScale safe-migrations production branch every direct DDL is refused (Error 1105) regardless of effect, so the already-widened path issues no DDL at all — and when a genuinely needed widen ALTER trips that block, the failure is loud and remedy-bearing (wraps `ErrSafeMigrationsBlocked` and carries the exact ALTER to ship via a deploy request), never a silent skip. Postgres is unaffected (its TEXT is unbounded). Pinned by exactly-once/missing-column/1105-refusal units plus real-MySQL integration: legacy-table widen with data preserved, fresh-CREATE LONGTEXT, idempotent re-ensure, and a >64 KB token round-trip through the position write.

## Fixed

- **Bug 186 — the MySQL→PG advisories now fire for EVERY MySQL-family source, mydumper and vitess included, via one registry-pinned family gate.** The unsigned-bigint range-narrowing advisory (the Bug-11/v0.68.2 notice that names each affected `table.column`, states the `2^63-1` ceiling, and gives the `--type-override TABLE.COL=decimal(20,0)` remedy at both `schema preview` and `migrate` preflight) keyed on a two-name engine list (`mysql`, `planetscale`), so the identical schema that WARNed via the live driver was silent via `mydumper` or `vitess` — and the sibling expression-gap scanners (`ScanMySQLToPGGaps`, generated-column and CHECK-literal gap scans, PG-validity) keyed on `"mysql"` alone. Data was never at risk: actual over-range values always aborted the copy loudly (coded, zero rows on target), and the documented override always worked on every driver — the gap was the documented-loudness contract, i.e. a schema-only migration silently narrowed the column *type* with no advisory anywhere, and an operator who hit the overflow abort was never shown the remedy the live path prints. Class fix, not an instance fix: one shared `translate.IsMySQLFamily` helper now gates all seven MySQL↔PG notice/gap scanners (the PG→MySQL-family target notices fold onto it too), pinned in BOTH directions against the engine registry's declared DDL dialect — registering a new MySQL-dialect engine without covering the notices now fails CI — plus a per-family-source pin that the same schema produces the same advisory through every family door. Affected releases, by leg: mydumper sources v0.99.247–v0.99.248 (the engine's entire published life); vitess-flavor sources missed the unsigned-bigint advisory from the flavor's introduction (v0.99.15) through v0.99.248; the `"mysql"`-only sibling scanners skipped planetscale/vitess/mydumper sources since each scanner shipped (translator-gap preflight since v0.39.0). Advisory-only in every case — nothing to re-verify.

- **Bug 188 — one unsupported-charset table no longer blocks migrating the rest of a mydumper/pscale dump; `--exclude-table` now routes around it (affected: v0.99.247–v0.99.248).** The ADR-0161 §5 unsupported-charset refusal (verbatim bytes + a non-UTF-8 charset = silent-mojibake risk; the refusal itself is correct and its wording is unchanged) fired inside the mydumper engine's `ReadSchema`, BEFORE the pipeline's include/exclude table filter — so one legacy latin1 table made an entire dump unmigratable, and the documented `--exclude-table` mechanism could not rescue the run (the only workaround was moving the table's schema+data files out of the dump directory). The refusal is now deferred, not weakened: the parse records the violation on the parsed table, and the new optional `ir.TableReadPreflighter` surface — consulted by the pipeline at all five post-filter sites (migrate, multi-database migrate, preview, diff, sync cold-start) — refuses loudly for any REMAINING doomed table up front, before any DDL or data moves, naming every violation and the `--exclude-table` remedy in one pass; the row reader re-checks at `ReadRows` as defense in depth for callers that skip the preflight. Integration-pinned on a real dump with an injected latin1 table: refuses pre-DDL (zero tables landed) when the table is included, migrates the other five tables cleanly when it's excluded. Affected: v0.99.247 and v0.99.248 (the refusal shipped with the mydumper engine in v0.99.247). Loud-failure UX fix — no data to re-verify.

- **Schema readers now exclude the FULL 11-table control-table roster (roadmap item 65b — the cutover-flow fix).** The mysql and postgres schema readers excluded only 3 and 5 of the 11 sluice-owned control tables from user-table enumeration, so a promoted ex-target read later as a migration source (the cutover flow: yesterday's sync target becomes today's source) carried sluice bookkeeping — `sluice_cdc_schema_history`, `sluice_shard_consolidation_lease`, `sluice_target_metrics_history`, `sluice_keysets`, and friends — along as user tables (loud and visible on the target, never silent). Both hand-maintained lists are replaced by one shared roster, `appliershared.ControlTableNames` (all 11 `sluice_*` tables including the heartbeat table and the trigger-CDC change-log trio), and a both-directions source-scan test pins every `sluice_*` table-name constant under `internal/` against the roster, so a future control table cannot be forgotten by either reader. SQLite/D1 readers already excluded their trio and carry no other control tables — unchanged. One documented edge: `sluice_heartbeat` is excluded under its default name only; an operator-renamed heartbeat table (`--source-heartbeat-table-name`) still needs an explicit `--exclude-table`. Cosmetic/loud class — data already migrated was never wrong, the extra tables were simply visible.

## Compatibility

- **Drop-in; no breaking changes.** No flag, default, or CLI behavior change to any existing invocation. The behavior deltas are all additive or advisory: (1) MySQL-family → PG runs may now emit WARNs and `schema preview` sections that were previously (wrongly) silent — the Bug 186 advisories on planetscale/vitess/mydumper sources, plus the new TIME advisory on any MySQL-family source with `TIME` columns. (2) Fresh MySQL control tables declare LONGTEXT position columns, and legacy ones widen exactly once via `MODIFY` on next use — behavior-preserving; on a PlanetScale safe-migrations branch an already-widened table issues zero DDL, and a genuinely needed widen that hits the 1105 direct-DDL block refuses loudly carrying the exact ALTER to ship via a deploy request. (3) A migration sourced from an ex-sync-target no longer enumerates the sluice bookkeeping tables as user tables; the excluded names are all `sluice_*`-reserved, so no user schema is affected. (4) The Bug 188 charset refusal fires earlier (at preflight, all violations named in one pass) and only for tables that survive the filter — its wording and posture are unchanged. Postgres control tables, and the SQLite/D1 readers, are untouched.

## Who needs this — action required

Nothing to re-verify anywhere: every class in this release is advisory or loud-failure, and no published release ever silently lost or altered data on these paths. Pick it up and act if any of these fit: **(a)** you migrate from a mydumper or `pscale database dump` directory that a legacy-charset table was blocking on v0.99.247/.248 — upgrade and add `--exclude-table <table>` instead of moving files out of the dump directory (an *included* unsupported-charset table still refuses, now up front before any DDL). **(b)** You ran a schema-only MySQL→PG migration or `schema preview` from a mydumper, vitess, or (for the gap scanners) planetscale source on an affected release — re-run `schema preview` on this version to see any advisories you weren't shown (unsigned-bigint narrowing, translator gaps, and now TIME); already-migrated data is correct either way, since over-range values were always a loud abort. **(c)** Your MySQL schema stores durations (elapsed time, offsets) in `TIME` columns — read the new advisory and apply `--type-override TABLE.COL=interval` before migrating. **(d)** You use the cutover flow (a promoted ex-target as a migration source) — the sluice bookkeeping tables no longer come along. Operators with very large GTID/VGTID position tokens need to do nothing: the LONGTEXT widen applies itself on next use.
