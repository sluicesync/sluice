# sluice v0.99.124

**HIGH fix — a single transient object-storage read (a truncated / flaky S3 body) could abort an entire restore, `sync from-backup` broker replay, or `backup verify`, even though the backup was perfectly intact; chunk reads now retry before emitting any row (ADR-0117). Drop-in: no flag or behavior change on healthy reads, and genuine at-rest corruption still fails loudly.**

## Fixed

**HIGH — one flaky object-store chunk read no longer wastes a multi-hour restore (ADR-0117).** The restore path streamed each backup chunk straight from object storage into the target's bulk `COPY`/`WriteRows`, opening the chunk with a single `Store.Get` and emitting rows as it decoded, with no retry around the read. A short / truncated GET or a mid-stream connection reset therefore surfaced deep in the decode path as `chunk reader: row decode: unexpected end of JSON input`, and because rows from the partially-read chunk had already been pushed into the open `COPY`, the read could not be safely re-streamed (that would duplicate the emitted rows) — so one transport blip aborted the whole run. The Track-C large-scale program caught it live: a cross-engine MySQL→PlanetScale-Postgres cold-start restore of a 3220-chunk (~43 GB) corpus died ~6 tables in on a single chunk, and a `sluice backup verify` of the same chain immediately afterward reported all 3220 chunks SHA-OK — proving the chunk was intact at rest and the failure was a transient short read. Across thousands of chunks the odds of hitting at least one such blip per restore are high; this is the read-phase counterpart of the DDL/index-phase retry gap closed in v0.99.118 (ADR-0114). The fix reads each content chunk fully into memory through a shared `fetchChunkVerified` helper with a bounded transient-retry (4 attempts, 200/400/800 ms backoff) and verifies its SHA-256 against the manifest **before** emitting any row — so the retry is safe (no partial emit to duplicate) and idempotent (chunks are content-addressed), and because the manifest hash is over the raw object bytes a complete object matches for both the plaintext and encrypted read paths. A persistent mismatch — genuine at-rest corruption, which re-fetching identical bad bytes cannot fix — still surfaces loudly as a chunk SHA-256 mismatch (`ErrChunkHashMismatch`) once the attempts are exhausted, so the loud-failure tenet is preserved and the failure is never silent. Applied to the restore table-data read, the chain-restore and broker incremental-replay change-chunk reads, and `sluice backup verify` (which no longer reports a false mismatch on a transient short read). Pinned by unit tests covering truncated-then-full (retried to success), happy-path (exactly one GET, no added latency), persistent-corruption (loud after all attempts exhausted), transient open-error (retried then succeeds), and context-cancel (prompt stop, not all attempts burned).

## Compatibility

No breaking changes, no flag changes, no API changes; drop-in upgrade. There is **no behavior change on healthy reads** — a clean object is fetched in one GET with no added latency, exactly as before. The only change is on the failure edge: a transient short / truncated read that previously aborted the run is now retried, while a persistent at-rest corruption still fails loudly with a chunk SHA-256 mismatch after the bounded attempts. Each plaintext content chunk is now buffered whole in memory (compressed size, one per concurrent worker) before decode rather than streamed; this was already the behaviour for encrypted backups, and it is bounded by the existing chunk-parallelism budget. Applies to all three paths that read content chunks from object storage — `restore` table data, chain-restore and `sync from-backup` broker incremental replay, and `backup verify` — across both engine targets and all storage backends. No change to value handling or to any non-restore command.

## Who needs this — action required

Anyone who runs `sluice restore`, `sync from-backup`, or `sluice backup verify` against an object store — especially S3-compatible storage (e.g. Vultr / MinIO and other non-AWS S3 endpoints) where a transient short / truncated body is more likely. No action required beyond upgrading: this is a robustness fix, not a correctness fix — no data was ever at risk on prior releases (a transient read aborted the operation **loudly**; it never silently lost or mis-applied rows), so there is nothing to re-verify, re-count, or re-run from earlier versions. There is no "affected releases" re-check to perform here: a restore that *completed* on any prior release completed correctly, and one that hit this blip aborted visibly with a decode error rather than producing a bad result. After upgrading, a restore that previously died on a flaky chunk will ride the blip and complete. Genuine at-rest corruption still fails loudly — so if a chunk SHA-256 mismatch persists after the retries, that is a real corrupt backup and must be investigated, not retried away.
