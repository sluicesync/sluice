# sluice v0.99.200

**Performance: on a PlanetScale / self-hosted Vitess MySQL target, the secondary-index build now OVERLAPS the bulk copy instead of running entirely after it ‚Äî each table's indexes build the moment that table's copy completes, hiding most of the index-build time under the copy of the remaining tables. Correctness is identical to v0.99.199 (every secondary index still lands, build-then-mark preserved, `SLUICE-E-INDEX-MISSING` verify net unchanged), it is still exactly ONE `ALTER ‚Ä¶ ADD INDEX` in flight at a time (no concurrent-ALTER-vs-vtgate concern), and vanilla MySQL / Postgres / SQLite/D1 are byte-identical. Drop-in: no flag, default, format, or data-path change ‚Äî a pure timing win. On real PlanetScale PS-10 (6 tables √ó 200k rows, 4 secondary indexes each) it hid ~10s of index-build work under the copy for a ~3.6% faster total migrate; the win grows on faster tiers and larger / more-heavily-indexed datasets.**

## Performance

- **Vitess/PlanetScale secondary-index build overlaps the copy instead of running as a serial post-copy batch (pure timing improvement; correctness-identical to v0.99.199; Vitess/PlanetScale MySQL targets only).** v0.99.199 fixed the silent secondary-index loss on Vitess/PlanetScale by draining the completed-tables channel to completion and then building every table's indexes serially AFTER all copies finished (drain-all-then-build). v0.99.200 keeps that exact correctness but moves the work earlier: the index builder (`buildEachAsCopiedSerial`) ‚Äî already running concurrently with the copy pool under one errgroup, consuming the completed-tables channel buffered to the table count ‚Äî now builds each table's secondary indexes AS that table's copy completes, so the index build of the already-copied tables overlaps the copy of the tables still copying instead of firing in one serial batch at the very end. It remains a single-loop consumer, so there is still exactly ONE `ALTER ‚Ä¶ ADD INDEX` in flight at any moment ‚Äî no concurrent ALTERs against vtgate (concurrent ALTERs fighting vtgate's online-DDL / Safe-Migrations queue, not ALTER-during-copy, was why ADR-0080 declined the overlap for Vitess), and because the channel is buffered to the table count the copy producer is never back-pressured by a slow build. The build reuses the same `indexBuildJobsForTables` + `buildTableIndexes` machinery as `CreateIndexes`, so the emitted SQL (inline-skip set, alphabetical order, one combined `ALTER TABLE ‚Ä¶ ADD INDEX a, ADD INDEX b` per table, idempotent detect-then-skip) is byte-identical, and it stays build-then-mark: a table's `IndexesBuilt` fires only AFTER its own build succeeds, so a mid-build failure leaves the later tables unmarked and a `--resume` rebuilds them. The `SLUICE-E-INDEX-MISSING` verify net is unchanged, and the vanilla-MySQL concurrent index path is byte-identical (this only changes the Vitess/PlanetScale branch). Pinned by a new unit test that asserts the overlap property (a table's indexes build before the channel closes ‚Äî which times out against the old drain-all behavior) plus the existing vttestserver Vitess integration coverage confirming every index family still lands for both migrate and sync cold-start.
- **Measured win, stated honestly.** On a real PlanetScale PS-10 with 6 tables √ó 200k rows and 4 secondary indexes per table, the overlap hid roughly 10 seconds of index-build work under the copy ‚Äî about 11.6 seconds / ~3.6% off the total migrate at that scale ‚Äî because each table's ~2.5s build fired immediately after that table's own copy instead of in a ~13s serial batch at the end, leaving only the last table's build unhideable. At this scale the copy dominates (~95% of the total on a CPU-bound PS-10), so the index build is only a ~4% slice and that bounds the win. The speedup grows on faster tiers (where the copy shrinks and the index build is a larger fraction of the total) and on larger or more-heavily-indexed datasets (more index-build time to hide under the copy). It is the same work as v0.99.199, just started earlier ‚Äî there is no throughput regression.

## Compatibility

**No breaking changes; drop-in. No CLI-flag, default, format, or data-path change** ‚Äî the only difference from v0.99.199 is WHEN the Vitess/PlanetScale secondary indexes build (overlapped with the copy rather than in a post-copy batch), not WHICH indexes build or the SQL that builds them. Correctness is identical to v0.99.199: every secondary index still lands, build-then-mark is preserved, and the `SLUICE-E-INDEX-MISSING` refusal is unchanged. The vanilla-MySQL overlapped index path, Postgres, SQLite/D1, and all row/CDC data paths are byte-identical. Only the Vitess/PlanetScale MySQL target's index-build timing is affected.

## Who needs this ‚Äî action required

- **Anyone migrating or sync-cold-starting into a PlanetScale / self-hosted Vitess MySQL target: no action ‚Äî you just get a modest, free speedup on upgrade.** The total migrate/cold-start finishes sooner because most of the secondary-index build now hides under the bulk copy; the amount hidden scales with dataset size, index count per table, and target tier (faster tiers see a larger relative win). Nothing to re-run, re-verify, or re-configure ‚Äî correctness is identical to v0.99.199.
- **Everyone else (vanilla MySQL, Postgres, SQLite/D1): no action, nothing changes.** This is a Vitess/PlanetScale-only timing change; every other path is byte-identical to v0.99.199.
