## Fixed

**MySQL `BINARY`/`VARBINARY` column DEFAULTs that contain a NUL byte are now recovered byte-exact from `SHOW CREATE TABLE` instead of the NUL-truncated `information_schema` value (Finding C; pre-existing since v0.1.0; silent data corruption on Postgres and SQLite/D1, loud only on the same-engine MySQL CREATE-reject subset).** MySQL's `information_schema.COLUMN_DEFAULT` is a text column and MySQL C-string-truncates a binary column's literal default at its first `0x00` byte before storing it there, so the metadata read is lossy for *any* binary default containing a NUL: `BINARY(1) DEFAULT 0x00` and `BINARY(4) DEFAULT 0x00FF00FF` both come back as the bare `0x` (empty), while `BINARY(2) DEFAULT 0x2700` comes back as the well-formed but short `0x27`. v0.99.186's `hexLiteralDefault` carried those as the emitted default: the bare-`0x` cases fell through as a wrong `'0x'` literal (Postgres stored `\x3078` â€” the two ASCII bytes of "0x" â€” a corrupted default), and the truncated-but-well-formed cases were the more dangerous ones, because `hexLiteralDefault` *accepted* them and every emitter rendered a value silently short of the real default. Nothing in `COLUMN_DEFAULT` alone distinguishes a truncation from a genuine short value, so the class cannot be detected from the metadata â€” the only safe move is to re-read the true bytes. The MySQL reader now flags every binary-family column whose default `information_schema` renders as a hex literal, and after the column scan re-reads its authoritative default from `SHOW CREATE TABLE` (once per affected table, cached), which reports the byte-exact stored default in one of the two forms MySQL emits there â€” a `0x<hex>` literal when any byte is â‰Ą 0x80, else a single-quoted escaped string (escape set ground-truthed on MySQL 8.0: `\0 \b \t \n \r \\` plus a doubled `''`, every other byte < 0x80 raw) â€” both decoded to exact bytes and re-encoded to the same `hexLiteralDialect` `DefaultExpression` v0.99.186's emitters already consume (MySQL bare `0xâ€¦`, Postgres `'\xâ€¦'::bytea`, SQLite/D1 `X'â€¦'`, with the existing fixed-width `BINARY(N)` NUL-padding, which is an idempotent no-op on the already-width-padded `SHOW CREATE` value). A default that `SHOW CREATE TABLE` is unavailable for, or whose clause is neither of the two forms, is DROPPED with a loud WARN naming the column (a DEFAULT-less target column is safe; a wrong-bytes default is not) â€” never silently carried. A trailing-NUL default on a *fixed* `BINARY(N)` was accidentally saved before by width-padding coinciding with the stored value; the genuinely-corrupted shapes are leading-NUL, `VARBINARY`-with-NUL, and mid-NUL-with-trailing-data. Pinned across the family (Bug-74 discipline): reader unit pins over the hex and escaped-string decoders (every emitted escape form incl. tab and literal backslash) and the per-column `SHOW CREATE` extraction (prefix-collision + `COMMENT`-safety), plus MySQLâ†’MySQL / MySQLâ†’Postgres / MySQLâ†’SQLite integration round-trips over leading/mid/trailing-NUL, mid-NUL-with-data, `VARBINARY`-mid-NUL, NUL-padded-narrower, and the v0.99.186 well-formed no-regression case â€” each verified red without the fix (reproducing the `\x3078` corruption) and green with it.

**A pathological column literally named `` `x DEFAULT 0xAA` `` can no longer mislocate the DEFAULT keyword inside its own name (value-fidelity review follow-up).** The `SHOW CREATE` per-column extractor now searches for ` DEFAULT ` only *past* the matched backtick-quoted column name, so a column name that itself contains the substring ` DEFAULT ` can't cause bytes to be parsed out of the name instead of the real default clause. Pinned end-to-end, together with the tab/backslash escape-decode path.

## Compatibility

**No breaking changes; drop-in.** No CLI flag, default, or backup format-version change affects any existing invocation. The fix is additive: a NUL-bearing binary-column literal default that previously mis-stored or blanked now translates byte-exact; every other default path â€” including the no-NUL binary defaults that v0.99.186 already fixed, and any genuine string column whose default text merely looks like hex (`VARCHAR DEFAULT '0x1234'`, still gated to a quoted string on the `BINARY`/`VARBINARY` column type) â€” is byte-identical. All four MySQLâ†”Postgres directions plus SQLite/D1 targets are covered; `BLOB` is unaffected (MySQL forbids a `DEFAULT` on `BLOB`, so this literal never arises there). Non-MySQL sources are untouched. The only new observable behavior is the loud WARN + dropped DEFAULT for a default `SHOW CREATE TABLE` cannot yield â€” chosen over carrying wrong bytes.
