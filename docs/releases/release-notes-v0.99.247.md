# sluice v0.99.247

**New: `--source-driver mydumper` — migrate straight from a mydumper or `pscale database dump` directory, no live source database needed.** This is sluice's first flat-file source engine (ADR-0161, roadmap item 55 Phase 1; research in `docs/research/flat-file-sources.md`) and a purely additive drop-in upgrade: a new engine name, no change to any existing command, flag, or default.

## Added

- **The `mydumper` source engine: `sluice migrate --source-driver mydumper --source /path/to/dumpdir --target-driver postgres --target '<dsn>'` (ADR-0161).** Reads the per-table flat-file layout mydumper emits — a `metadata` file (binlog position / GTID), one `<db>.<table>-schema.sql` per table containing exactly one CREATE TABLE, and `<db>.<table>.<NNNNN>.sql` extended-INSERT data chunks, transparently gzip- or zstd-compressed — and migrates it into any sluice target. **PlanetScale's own `pscale database dump` writes exactly this format** (verified against the pscale CLI source), so this is the provider-export migration path: providers that only give you a dump, and air-gapped moves where the source database can't be reached, no longer need a scratch MySQL server to load through. It is source-only, the same capability shape as d1: `migrate` and `verify` from a dump directory work; every writer/CDC/snapshot surface is a loud `ErrNotImplemented` refusal. `verify --depth count` is implemented via chunk re-scan (tuple count, no value decode); sample depth is a documented refusal — deterministic sampling needs cheap row addressing flat files don't have.

- **A deliberately bounded schema parse that does not fork the type mapping.** The IR-first tenet forbids grammar over DDL strings, but a flat file has no catalog, so the engine carries a contained, ADR-blessed exception on the ADR-0133 precedent (SQLite's `sqlite_master` DDL is likewise that engine's only schema surface): each schema file must hold exactly one CREATE TABLE (plus comments/SET), and anything else — a second statement, an ALTER, an unrecognised column attribute — refuses loudly naming the file and token; the parser never skips what it doesn't understand. Parsed column metadata folds into the **live MySQL engine's own type translator** through a narrow exported shim, so a dump of a MySQL database maps through the SAME validated code the live reader uses — a fix to either path is a fix to both.

- **Value fidelity ground-truthed against real mydumper v1.0.3 dumps, with the live database as the oracle.** Data chunks decode through a faithful MySQL string-literal lexer — the full backslash escape set, doubled quotes, `_binary`/charset introducers, double-quoted strings, `0x…` hex-blob AND PlanetScale's no-hex-blob backslash-escaped binary (binary fidelity there rides entirely on the escape decoder), `b'…'` bit literals, and mydumper's `CONVERT(… USING UTF8MB4)` JSON wrappers — then through the live engine's value decoder, so every value lands in the `ir.Row` contract byte-identical to a live read. Integers parse straight from decimal text, never through a float, so BIGINT values past 2^53 and uint64-max round-trip exactly; DECIMAL stays text. The integration suite migrates REAL mydumper-generated dumps (default-escape, hex-blob, gzip, and zstd legs) into MySQL and Postgres targets and compares row-by-row against the live-`mysql`-engine migration of the same source, across every value family — >2^53 and MinInt64 ints, uint64 max, DECIMAL, FLOAT/DOUBLE, temporal with fractional seconds, every escape shape, ENUM/SET, BIT, JSON, Boolean, and geometry with SRID+WKB (the Bug-74 pin-the-class discipline). That oracle earned its keep pre-release: it caught the BIT(1)-as-Boolean quoted-byte shape and mydumper's FLOAT display-rounding (below) before they could ship.

- **Loud failure everywhere the format could lie.** Torn dumps (`metadata.partial*`), unrecognised files, and multi-database dump dirs refuse at open; non-UTF-8 charsets refuse at three gates (`SET NAMES`, value introducers, declared column charsets — a latin1 table refuses at ReadSchema with the convert-or-migrate-live remedy) rather than transcoding silently; a `TIME_ZONE` header other than UTC refuses in every spelling MySQL accepts (`SESSION`/`GLOBAL`/`LOCAL`-qualified, `@@time_zone`, `@@session.time_zone`, …) so no qualified form slips the gate; zero/partial dates, BIT values overflowing their declared width, ambiguous Boolean bytes (held to exactly one byte ≤ 0x01 — a lenient branch would read the TEXT digit `'0'` as true), and any literal-kind × type pairing outside the matrix all refuse naming the column — never a silent coercion.

- **Two named honesty warts, WARNed rather than hidden (ADR-0161 §4/§5).** First: mydumper itself display-rounds single-precision FLOAT to ~6 significant digits at dump time — 8388608 dumps as `8.38861e6`; the loss is IN THE FILE and no reader can undo it — so the engine WARNs once per table naming the FLOAT columns, with the remedy of migrating from the live database if FLOAT precision matters. DOUBLE is proven unaffected, not assumed: ground-truthed against v1.0.3, `3.141592653589793`, `0.1`, and `1.7976931348623157e308` all dump at full precision in the same run where FLOAT rounds, and the integration corpus pins the sibling family. Second: mydumper v1.0.3 always converts TIMESTAMPs to UTC and stamps every file with a UTC `TIME_ZONE` header, but a dump from another producer with NO header could carry server-local instants the reader cannot detect — so a TIMESTAMP-bearing table whose chunks declared no time zone WARNs once, naming the columns. The dump's binlog position / GTID metadata is parsed and logged at open as the future dump→CDC-handoff hook — recorded, not built.

- **Non-goals, so expectations are exact:** plain mysqldump `.sql` single-file dumps and anything pg_dump are refused (Phase 3 adds signature-detection with a recipe-bearing refusal); CSV/NDJSON is Phase 2 and Parquet Phase 4; no myloader-side writing, no multi-database dump dirs, no CDC handoff yet.

## Compatibility

- **Purely additive; no breaking changes.** One new engine name (`mydumper`), no flag, default, or behavior change to any existing command or engine path. One shared-decoder note: the MySQL package's quoted-string decoder now keeps the backslash for `\%` / `\_` per MySQL's string-literal grammar (LIKE-pattern escapes) — load-bearing for dump decode, unreachable in practice on the live information_schema paths because MySQL's own literal printing never emits those escapes, so no live-source migration changes behavior. The new engine is source-only: `migrate` and `verify --depth count` work from a dump directory; using it as a target, a CDC source, or with sample-depth verify is a loud refusal, not a silent no-op.

## Who needs this

No action required for anyone — this release contains no fixes to shipped behavior, so there is nothing to re-verify or re-run. Pick it up if your migration starts from a dump directory instead of a live DSN: PlanetScale users migrating from a `pscale database dump` export are the first-class case, along with anyone holding a mydumper directory from an air-gapped or decommissioned source. One caveat to read before relying on it: the FLOAT display-rounding wart above — if single-precision FLOAT columns matter to you, migrate from the live database; the dump itself has already rounded them.
