# sluice v0.99.250

> **Correction (2026-07-15):** the Compatibility line "refused loudly on any non-flat-file source, so they cannot silently mis-apply" was wrong at release: in v0.99.250 and v0.99.251 the four `--csv-*` flags were silently IGNORED on non-flat-file sources (Bug 189) — a `migrate --source-driver mysql --csv-null=NULL …` ran as if the flags weren't there, exit 0. The promised refusal shipped in v0.99.252. No data was affected (the flags never mis-applied; they simply didn't apply), but the contract stated here did not hold on this version.

**New: CSV, TSV, and NDJSON files are now migrate sources — `--source-driver csv|tsv|ndjson` (ADR-0163, roadmap item 55 Phases 2+3).** With Phase 1's mydumper engine (v0.99.247), the flat-file arc's main line is complete: dump directories, spreadsheets, and log pipelines all migrate through the standard pipeline, and the formats sluice deliberately does not parse (plain mysqldump/pg_dump `.sql`, `pg_dump -Fc`) now refuse with a copy-pasteable scratch-server recipe instead of a confusing mid-parse error. Purely additive drop-in: three new source drivers, four new `--csv-*` flags inert everywhere else, four new refusal codes, no change to any existing invocation. One fix in this release does reach published versions: the SQLite/D1 ISO temporal validator/decoder alignment (loud wrong-refusal on live `sqlite`/`d1` `--infer-types` runs, v0.99.166–v0.99.249 — details in Fixed; nothing to re-verify, the class was always a loud abort).

## Added

- **`--source-driver csv` / `tsv` / `ndjson` — schema-less flat files as migrate sources, staged into SQLite with validated type inference (ADR-0163 §1–§2).** Each open materializes the file into a fresh temp all-TEXT SQLite database (one table named from the file's basename) read through new sqlite staged-reader shims (`OpenStagedSchemaReader`/`OpenStagedRowReader`, the ADR-0130 tempPath ownership rules), and `migrate` auto-engages `--infer-types` (ADR-0144) so name-hinted timestamp/timestamptz/jsonb/uuid columns are promoted only after every non-NULL value validates. TEXT staging is the fidelity decision, not a shortcut: SQLite TEXT affinity stores text verbatim, so `007.1500`, `9007199254740993`, and `-0.000` all land byte-exact. Numbers deliberately stay TEXT (ADR-0144 has no numeric promotion — conservative, lossless; `--type-override table.col=bigint` places a specific column and always wins over inference). Source-only, the d1/mydumper capability shape: never a target, no CDC, every write surface a loud `ErrNotImplemented`. Operator guide: `docs/operator/flat-file-sources.md`.

- **CSV producer conventions are declared, never sniffed (ADR-0163 §3–§5).** RFC 4180 encodes neither NULL nor header presence, so sluice requires both declared: `--csv-null=REPR` names the unquoted field text that means SQL NULL (`--csv-null=''` adopts the PostgreSQL COPY CSV convention; `--csv-null='\N'`/`NULL` declare that literal, and unquoted-empty is then the empty string) — a quoted field is ALWAYS data (`"NULL"` is the four-character string, `""` the empty string), and an unquoted empty field with no declaration refuses loudly with the new `SLUICE-E-CSV-NULL-AMBIGUOUS` naming record and column (NULL-vs-empty-string is the #1 CSV silent-loss class). `--csv-header`/`--csv-no-header` declare header presence, else `SLUICE-E-CSV-HEADER-UNDECLARED` (a wrong guess silently eats a data row or turns data into column names). `--csv-delimiter` takes a single ASCII char or `\t`/`tab`; `tsv` is the same engine registered with the delimiter fixed to TAB (the MySQL-flavor pattern), and extension cross-checks catch the wrong-dialect trap but yield to an explicit `--csv-delimiter`. The package carries its own strict RFC 4180 lexer because the contract requires the quoted-empty vs unquoted-empty distinction that `encoding/csv` collapses; malformed quoting, ragged records, lone CRs, NUL bytes, and non-UTF-8 refuse loudly naming the record (UTF-8 BOM stripped with a WARN; UTF-16 refused with a transcode hint).

- **NDJSON value posture: numbers as raw source text, refusals at the offending line (ADR-0163 §6).** One JSON object per line; keys become columns in first-seen order and the set may grow mid-file (earlier rows read NULL for later keys). Numbers are carried as their RAW source text end-to-end — never through a float64 — so integers > 2^53, snowflake IDs, and arbitrary-precision decimals land exact (the D1 2^53 lesson); nested objects/arrays land as raw JSON text and a name-hinted column promotes to jsonb. One named collapse, documented: an absent key and an explicit `null` both land as SQL NULL. Duplicate keys within an object (encoding/json's silent last-wins is a silent-loss shape), non-object lines, trailing content, empty object keys, and case-colliding keys all refuse loudly at the offending line; a single JSON array document refuses naming the `jq -c '.[]'` conversion.

- **Foreign-dump and wrong-driver refusals with recipes (ADR-0163 §7, roadmap 55 Phase 3).** A new content-signature detector (`internal/engines/internal/dumpsig`) runs at open on the flat-file drivers, the sqlite driver (extending the ADR-0130 sniff — previously a mysqldump `.sql` died mid-materialize on a confusing SQL error), and the mydumper driver's file branch. Plain mysqldump (`-- MySQL dump`), plain pg_dump, and `PGDMP` (`pg_dump -Fc`) refuse with `SLUICE-E-SOURCE-FOREIGN-DUMP` carrying the exact scratch-server-replay recipe (docker a scratch server → native restore tool → `sluice migrate` from it) — sluice deliberately parses no full-dialect SQL dumps (the IR-first tenet; pgloader draws the same line). Cross-driver misuse — a mydumper directory to `csv`/`sqlite`, a CSV/NDJSON file to `mydumper`/`sqlite`, a binary SQLite `.db` to `csv`, gzip/zstd (decompress first, or for a mydumper dump: point at the directory, its chunks decompress automatically), UTF-16 (transcode first) — refuses with `SLUICE-E-SOURCE-WRONG-DRIVER` naming the right driver or preparation step.

- **`sluice verify --depth count` for flat-file AND plain sqlite endpoints.** The sqlite file `SchemaReader` now implements `ir.Verifier.ExactRowCount` (authoritative `COUNT(*)` over the staged/opened database), which lights up count-depth verify both for the new flat-file sources (each verify open re-stages the file — the same re-scan cost class as mydumper's chunk re-scan) and for plain sqlite `.db`/dump endpoints, which previously had no count surface. Sample depth stays a documented refusal (a staged keyless table has no deterministic sampling key).

## Fixed

- **Pre-land value-fidelity review (the mydumper P1 discipline): a silent NULL-row drop in one-column CSVs, caught and fixed BEFORE release — never in any published version.** In a ONE-column CSV the blank-line skip consumed a legitimate one-empty-field record before the NULL contract saw it: under `--csv-null=''` a NULL row was silently dropped, and without a declaration the promised `SLUICE-E-CSV-NULL-AMBIGUOUS` refusal was silently bypassed. The lexer now carries the established record width — at width 1 an empty line is a real record routed through the NULL contract; blank-line skip applies only at width > 1 or before the first record — pinned across {undeclared, `''`, `\N`} × {interior blank, trailing newline, trailing blank}. The same review also moved the NDJSON empty-key and case-colliding-key refusals (staged SQLite column names are case-insensitive) and the CSV header collision refusal to the offending line with named messages, instead of leaking the staging database's zero-length-identifier/duplicate-column errors. These defects existed only between this release's two commits on `main` — no published release ever carried them.

- **SQLite/D1 ISO temporal decode now accepts every shape the `--infer-types` validator promotes — the validator/decoder mismatch was a loud wrong-refusal on live sqlite/d1 sources (affected: v0.99.166–v0.99.249).** Since the ADR-0144 inference shipped (v0.99.166), the validator's temporal GLOBs blessed layouts the value decoder then refused at copy time: a T-separated naive datetime (`2024-01-02T03:04:05`) and — surfaced by this release's flat-file integration corpus, which carries a real PG-COPY timestamptz column — the space-separated zoned and colon-less `±hhmm`/`±hh` offset spellings (Postgres COPY CSV renders timestamptz as `... 08:09:10.123456+00`). A hinted column whose values used those shapes was promoted, then aborted loudly mid-copy with a raw decode error. Loud-failure class throughout: exit nonzero, zero rows silently wrong or missing, data already migrated on affected releases is correct. The decoder gains the missing layouts; the validator GLOBs gain the matching `±hhmm` and ANCHORED `±hh` spellings (anchored after seconds or a fraction, because a bare DATE's `-02` tail is exactly the naive `±hh` shape — the invented-zone false positive the anchoring avoids); and every separator × zone × fraction cell is now pinned on BOTH sides (`TestDecodeTemporal_ISOSeparatorZoneMatrix`, `TestValidateInferredType_ZoneSpellingMatrix`) plus instant-checked end-to-end on real PG and MySQL targets, so the two surfaces cannot drift apart silently again. The decode-refusal hint now also scopes `--sqlite-date-encoding` to sqlite/d1 sources (csv/tsv/ndjson: `--type-override` only).

## Compatibility

- **Purely additive; no breaking changes.** Three new source-only driver names, four new `--csv-*` flags (refused loudly on any non-flat-file source, so they cannot silently mis-apply), and four new coded refusals; no flag, default, or behavior change to any existing command or engine. The temporal alignment is acceptance-widening only: the decoder now decodes values the validator was already promoting (previously a loud abort — conforming data that copied cleanly before is byte-identical), and the validator's new `±hhmm`/anchored-`±hh` zone spellings mean a live sqlite/d1 `--infer-types` run may now promote a hinted timestamptz column that previously stayed at its declared type — inference doing its documented job on more conforming spellings, with the same every-non-NULL-value-validates gate and `--type-override` still winning. `verify --depth count` on plain sqlite endpoints is new capability where there was previously a refusal. The mydumper driver's only change is the new wrong-driver/foreign-dump refusals on inputs that previously failed with unframed errors.

## Who needs this — action required

Nothing to re-verify anywhere: the one fix reaching published releases was always a loud abort, and the silent-loss CSV defect never shipped. Pick this up if: **(a)** you have CSVs, TSVs, or NDJSON/JSONL files to migrate — `sluice migrate --source-driver csv --source ./file.csv --csv-header --target-driver postgres --target '<dsn>'` is the shape; read `docs/operator/flat-file-sources.md` for the NULL-representation contract before your first run. **(b)** You ran `migrate --infer-types` from a live `sqlite` or `d1` source on v0.99.166–v0.99.249 and hit a temporal decode abort mid-copy on a column inference had promoted (T-separated naive datetimes, or space-separated/`+00`-style zoned timestamps) — upgrade and re-run; the promoted column now copies. If you worked around it with `--type-override col=text`, you can drop the override. Completed runs on those releases need no re-verification — the failure mode was a refusal, never wrong data. **(c)** You have a plain mysqldump/pg_dump `.sql` or `-Fc` archive — sluice still does not parse these, but the refusal now hands you the exact scratch-server recipe instead of a mid-parse error.
