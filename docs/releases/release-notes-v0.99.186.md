# sluice v0.99.186

**Fixed: MySQL BINARY/VARBINARY column DEFAULTs now round-trip correctly to MySQL, Postgres, and SQLite/D1. Before this release, migrating any table whose binary column carried a literal `DEFAULT` failed loudly at CREATE on same-engine MySQL‚ÜíMySQL (`Error 1067 ‚Äî Invalid default value`) but **silently stored a corrupted default** (the ASCII text of the `0x‚Ä¶` literal instead of the intended bytes) on MySQL‚ÜíPostgres and MySQL‚ÜíSQLite/D1 ‚Äî verified against the shipped v0.99.185 binary. A real-world shape (e.g. MediaWiki's `BINARY(14)` timestamp columns). The defect is pre-existing since the MySQL engine's first release (v0.1.0); every published release through v0.99.185 carries it. Drop-in: no flag, default, or backup-format changes.**

## Fixed

**MySQL BINARY/VARBINARY column DEFAULTs are now classified as hex-bytes defaults and translated byte-exact per target (pre-existing since v0.1.0; loud failure on MySQL, silent data corruption on Postgres and SQLite/D1).** MySQL 8 reports a binary column's literal default in `information_schema.COLUMN_DEFAULT` as a bare hex literal ‚Äî `BINARY(14) NOT NULL DEFAULT '19700101000000'` comes back as `0x3139373030313031303030303030` (with `EXTRA` empty, i.e. a literal, not a generated expression). sluice's MySQL reader carried that verbatim as a plain string default, so the writer re-quoted it as the *string* `'0x3139‚Ä¶'` ‚Äî a 30-character value on a 14-byte column. Same-engine MySQL‚ÜíMySQL rejected the re-emitted `CREATE TABLE` loudly with `Error 1067 (Invalid default value)` ‚Äî nothing corrupt landed there. But the two cross-engine directions failed **silently**: MySQL‚ÜíPostgres and MySQL‚ÜíSQLite/D1 exited 0 and stored the ASCII text of the `0x‚Ä¶` literal as the default instead of the intended bytes (confirmed on the shipped v0.99.185 binary ‚Äî a `BINARY(8) DEFAULT 'hi'` landed the six bytes of the string `0x6869` rather than `hi` + 6 NUL bytes). The net effect was that a table with a binary-column literal default could not be migrated at all to MySQL, and was silently given wrong default values on Postgres and SQLite/D1. The reader now recognizes a well-formed `0x<even-hex>` literal *only when the column is a binary-family type* (`BINARY`/`VARBINARY`) and tags it as a hex-bytes default; the gate on column type is deliberate, because a genuine `VARCHAR DEFAULT '0x1234'` reports the identical surface form in `information_schema` and must stay a quoted string. Each writer then renders the tagged default in its own dialect: MySQL bare `0x‚Ä¶`, Postgres `'\x‚Ä¶'::bytea`, SQLite/D1 `X'‚Ä¶'`. Fixed-width `BINARY(N)` fidelity is preserved across the width-agnostic BYTEA/BLOB targets: MySQL reports the *un-padded* literal but stores it right-padded to the column width and re-pads on INSERT, while BYTEA/BLOB cannot re-pad, so the Postgres and SQLite writers now zero-fill the payload to the declared fixed-BINARY width (`VARBINARY` is never padded) ‚Äî without this the default-applied row would land, e.g., 2 bytes on Postgres/SQLite versus 8 on MySQL, a silent divergence. Pinned across the family (Bug-74 discipline): reader unit pins over `BINARY`/`VARBINARY` at multiple widths, NUL-bearing and uppercase-hex values, plus the `VARCHAR '0x‚Ä¶'` stays-a-string and truncated/odd-length fall-through cases; writer unit pins on each emitter's bare/`::bytea`/`X'‚Ä¶'` arm; and MySQL‚ÜíMySQL, MySQL‚ÜíPostgres, and MySQL‚ÜíSQLite integration round-trips that create the table (the former failure site) and assert the default-applied bytes are byte-identical to the MySQL source across exact-width, VARBINARY, and NUL-padded shapes. Each pin was verified to fail at baseline and pass with the fix. Known residue, documented and out of scope: a non-UTF8/leading-NUL binary default that MySQL itself mangles in `information_schema` (reporting a truncated `0x` with no digits) is unreadable from that path ‚Äî a MySQL limitation, not sluice's ‚Äî and falls through to the verbatim literal.

## Also in this release (internal quality / test infrastructure ‚Äî no shipped-binary behavior change)

These are trust-and-coverage investments; none change migrate/sync/backup runtime behavior. The MySQL `mysqldump` restore-parity oracle (roadmap item 51 Phase 2) round-trips a migrated MySQL target and diffs it against `mysqldump` output ‚Äî it is what surfaced the BINARY/VARBINARY defect above. A real-world corpus suite (item 52) adds dump-parity and sample-row verify legs and a scheduled weekly corpus job. The cookbook is now backed by executable tests (item 53) that run the documented recipes against the released binary; that harness caught several stale-recipe doc bugs (the bidirectional-cutover zero-downtime model, the redaction-keyset YAML shape, and a fabricated quickstart dry-run banner), all fixed in this release. The skills‚ÜîCLI doc-sync CI guard (item 56) was extended to catch subcommand drift, not just top-level flags.

## Compatibility

**No breaking changes; drop-in.** No CLI flag, default, or behavior change affects any existing invocation, and there is no backup format-version change. The fix is additive: binary-column literal defaults that previously blocked or corrupted a migration now translate correctly; every other default path is byte-identical. A genuine string column whose default text merely looks like hex (`VARCHAR DEFAULT '0x1234'`) is explicitly unaffected ‚Äî it still emits as a quoted string, because the new classification is gated on the column being a `BINARY`/`VARBINARY` type. All four MySQL‚ÜîPostgres directions plus SQLite/D1 targets are covered; `BLOB` columns are not affected (MySQL forbids a `DEFAULT` on `BLOB`, so this literal never arises there). Non-MySQL sources are untouched.

## Who needs this ‚Äî action required

- **Anyone migrating a MySQL schema that uses `BINARY`/`VARBINARY` column DEFAULTs ‚Äî to MySQL, Postgres, or SQLite/D1.** Upgrade to v0.99.186 before running that migration. On a **MySQL target** the previous behavior was a loud abort at `CREATE TABLE` (`Error 1067`), so nothing corrupt shipped ‚Äî just re-run the migration on the new binary. But on **Postgres and SQLite/D1 targets the pre-fix path silently stored a wrong default value** (exit 0, no error), so if you previously migrated a MySQL schema with binary-column defaults to Postgres or SQLite/D1 on any release through v0.99.185, **re-verify those columns' defaults** (or re-run the schema migration on v0.99.186) rather than trusting the earlier result.
- **Everyone else: no action.** Migrations without binary-column defaults, non-MySQL sources, string/other-typed defaults, and all sync / backup / restore paths are unchanged and byte-identical.
