# sluice v0.99.195

**Two sharded-PlanetScale backup fixes. The one that matters most: a `backup full` of a PlanetScale/Vitess (VStream) source now records its exact COPY-completion position as the chain anchor, so a following `backup incremental` chains off the full's real position instead of restarting from CDC's "current" â€” closing a bounded, WARN-loud change-loss window that sat on every VStream chain root since v0.99.3. Also new: `restore --control-keyspace` unblocks a chain restore *into* a sharded PlanetScale/Vitess target (the v0.99.193 sidecar-keyspace escape hatch, now on the restore path too). Drop-in: no on-disk backup format change â€” existing manifests and chains still restore; the anchor fix only affects newly-created VStream backups, and the new flag is optional and inert everywhere except a sharded Vitess chain restore.**

## Fixed

- **A `backup full` of a PlanetScale/Vitess (VStream) source now anchors the chain at the real COPY-completion position instead of an empty one, so a subsequent `backup incremental` chains off the full's exact position rather than starting from CDC's "current" (bounded, WARN-loud change-loss window; regression present in v0.99.3 through v0.99.194 on VStream backup chain roots; sharded or not).** The VStream backup snapshot's `Position` is captured at open time, where it is the zero value: ADR-0071 (first shipped v0.99.3) made the COPY pump a concurrent goroutine that only finalizes the terminal VGTID after the backup row sweep drains every table, so reading `Position` at snapshot construction recorded an EMPTY `EndPosition`. A `backup incremental` then could not chain off that empty anchor and fell back to CDC's current position â€” meaning any source write between the full backup finishing and the incremental opening its stream fell into an uncaptured gap. This is the identical race `#243` fixed on the cold-startâ†’CDC handoff (at v0.99.68), overlooked on the backup path, which carried it continuously. Fix: `irbackup.Snapshot` gains an optional `FinalizePositionFn`; the VStream snapshot sets it to join the copy-completion barrier (`WaitCopyComplete`) after the row sweep and read the finalized VGTID, and the backup orchestrator records that as `EndPosition` instead of the zero-at-open value. General to all VStream backup chains, not sharded-specific. Postgres and vanilla MySQL leave the finalizer nil and are byte-identical â€” their open-time position (Postgres exported-snapshot LSN, in-transaction GTID) is authoritative and recorded unchanged. It was hidden because the only prior `recordEndPosition` coverage mocked a non-empty position; pinned now by `recordEndPosition` unit tests plus vstream/vttestserver integration on both an unsharded (`finishCopy`) and a sharded (`finishCopyAutoShard` stitched-min) chain root, each proving open-time-empty vs finalized-non-empty and a WARN-free incremental resume.

## Added

- **`restore --control-keyspace` routes a chain restore's CDC control tables to an unsharded sidecar keyspace, unblocking a chain restore into a *sharded* PlanetScale/Vitess target (new capability; a sharded-target chain restore previously aborted mid-replay with `Error 1105 â€¦ does not have a primary vindex`; loud failure, no data risk).** A chain restore (a full backup plus one or more incrementals) replays CDC change chunks, and its incremental leg calls `ChainRestore`'s `EnsureControlTable`, which creates the three internal control tables (`sluice_cdc_state`, `sluice_cdc_schema_history`, `sluice_shard_consolidation_lease`) in the target's data keyspace. Vitess requires a primary vindex on every table in a sharded keyspace, and those three have none, so a sharded target rejected them during incremental replay and `restore` had no way to relocate them. The new `--control-keyspace <name>` mirrors `sync start --control-keyspace` (v0.99.193): it routes the control tables to a separate unsharded keyspace â€” auto-detected when the target is sharded and the flag is unset (a loud refusal that names the candidates if zero or several unsharded keyspaces exist), or explicit to override the auto-pick. It reuses the existing `applyControlKeyspace` / `ResolveControlKeyspace` / `WithControlKeyspace` stack as a fifth caller alongside the `sync start` / `sync stop` / `sync status` / `sync health` family, so the control-keyspace-configured target engine is the exact one that reaches `ChainRestore`'s `OpenChangeApplier`. A single FULL restore writes no control tables, so the flag is inert there; an unsharded target, a non-Vitess engine (Postgres, SQLite/D1), or an unset flag is byte-identical to before. This completes the v0.99.193 sharded-target story for the backup/restore path â€” `sync` got the sidecar treatment then, `restore` gets it now. Pinned by unit coverage over the resolve-and-apply path (auto-detect, explicit override, inert on a single-full / non-Vitess target).

## Compatibility

**No breaking changes; drop-in. No on-disk backup format change â€” every existing manifest and chain still restores.** The `EndPosition` fix affects only *newly-created* VStream backups (they now carry a real chain anchor instead of an empty one); a chain rooted on a pre-fix VStream full backup keeps its empty-`EndPosition` WARN and start-from-current behavior on incremental â€” re-take the full on v0.99.195 to get precise chaining. `restore --control-keyspace` is a new optional flag: omitting it preserves today's behavior for every non-sharded case, and the whole mechanism engages only for a chain restore into a sharded MySQL/PlanetScale/Vitess target. Postgres, SQLite/D1, vanilla MySQL, single full restores, and every migrate / sync path are byte-identical â€” the backup finalizer is nil off the VStream path and the restore flag is inert everywhere except a sharded Vitess chain restore.

## Who needs this â€” action required

- **PlanetScale/Vitess operators who took `backup full` chain roots on v0.99.3 through v0.99.194: your VStream full backups recorded an empty chain anchor, so any incremental chained off one started from CDC's current position and any source write in the window between the full finishing and the incremental opening its stream was not captured by that chain.** The condition was WARN-loud (the empty-`EndPosition` warning fired and the incremental logged start-from-current), so check whether those WARNs appeared on your VStream chains. To get a gap-free chain going forward, re-take the `backup full` on v0.99.195 (the new full anchors precisely) and chain incrementals off it; there is no on-disk format change, so older backups still restore, they simply retain the imprecise anchor. Postgres and vanilla-MySQL backup chains were never affected.
- **Operators restoring a backup chain into a sharded PlanetScale/Vitess keyspace: you can now do this.** On `restore` you usually need do nothing â€” sluice auto-detects the unsharded sidecar keyspace when the target is sharded; pass `--control-keyspace <name>` only to override the auto-pick or when auto-detect refuses because zero or several unsharded keyspaces exist (the refusal names them). A single full restore and any unsharded / non-Vitess target need no flag.
- **Everyone else: no action.** Postgres, SQLite/D1, vanilla MySQL, single full restores, and every migrate / sync path are byte-identical; nothing to re-verify, nothing to re-run.
