# sluice v0.99.183

**An internal-structure release that completes the pipeline decomposition from the July repository audit, with zero user-facing or on-the-wire change. The backup/restore/chain domain is carved out of the flat `internal/pipeline` package into two new sub-packages (`lineage` and `backup`) on top of last release's `migcore` core, and the first lint-ceiling ratchet turn splits the fourteen functions that exceeded the tighter function-length and complexity limits. All behavior is byte-identical; every existing pin passes unchanged, and the `-race` integration gate is green.**

## Internal

**Backup/restore/chain carved out of the flat pipeline root (audit 3.7b, completing the A-1 decomposition).** Building on v0.99.182's `migcore` core, the roughly 9,800-line backup / restore / chain-lineage cluster has been decomposed out of the flat `internal/pipeline` package in three cycle-safe steps. First, the shared substrate both the cluster and the root's streaming and broker orchestrators reached into bidirectionally — manifest I/O, the lineage catalog, and the backup-encryption envelope — moved *down* into a new `internal/pipeline/lineage` package; that mutual reach (the cluster calling the root's manifest reader while the root called the cluster's lineage-catalog loader and encryption type) was the exact import cycle that had blocked splitting the package, and pulling the substrate below both sides breaks it. Second, the remaining shared helpers the cluster still borrowed from the root — the table filter, the target-schema configurators, the DDL-phase and views-phase runners, row redaction and primary-key helpers, the apply-concurrency resolvers, the copy and reader helpers, the reparent observer, the reconcile-round bound, and the chain-resume preflight — moved into `migcore`. Third, the now import-clean cluster (the `Backup`, `Restore`, and `ChainRestore` orchestrators with their table pools, chain compaction and pruning, and the dump-parity oracle) was lifted into a new `internal/pipeline/backup` package, with `cmd/sluice` and the root repointed onto it. Every new edge is one-directional and verified acyclic: `backup` imports `lineage`, `migcore`, and `blobcodec` but never the pipeline root, and neither `lineage` nor `migcore` imports the root. The flat root shrank about 8,600 non-test lines (from roughly 46,900 to 38,400). It is a pure move — every backup, restore, chain-restore, and broker round-trip is byte-identical, proven across the same-engine and cross-engine integration suites, with `git mv` history preserved where whole files moved.

**First function-length and complexity lint-ceiling ratchet turn — 210→180 lines, 60→45 (audit 3.5).** With the decomposition landed, the fourteen functions that then exceeded the tighter ceilings were split by extracting cohesive, named sub-operations rather than arbitrary line chunks. `BackupStream.Run`'s 355-line rotation loop became a linear setup helper plus explicit empty-rollover and commit paths, retiring its standing function-length suppression; `captureWindow`'s in-flight chunk-writer state moved onto a small `changeChunkBuffer` value; the Postgres `translateType` type-family switch split into per-family dispatchers; and the schema-reader column and index populators, the two `cmd` command-composition roots, and the remaining long backup, migrate, and applier functions were each broken into named steps. The ceilings drop to 180 lines and complexity 45 with a single re-justified pre-existing suppression (the flat ~60-field `Streamer{}` CLI composition literal, which is a one-to-one flag-to-field mapping that indirection would only obscure). It is behavior-preserving: the concurrency-adjacent splits (`BackupStream.Run`, `captureWindow`, `copyChunk`, `RunOneBatch`) keep their exact goroutine, channel, defer, and return semantics — resource-cleanup defers stay in the function that owns the resource — and the full unit suite plus the `-race` integration gate are green.

## Compatibility

**No breaking changes; drop-in; nothing to re-verify.** No flags, CLI surface, position-token format, chunk or manifest format, or on-disk layout changed. Both items are pure internal moves and behavior-preserving refactors, validated by the full existing unit and integration suites plus the `-race` gate; there is no operator-visible behavior difference.

## Who needs this — action required

- **No one.** This release contains no bug fixes, no feature changes, and no format changes — only internal refactoring that reduces future maintenance risk. Upgrade at leisure.
