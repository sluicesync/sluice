# sluice v0.99.190

**Fixed — two PlanetScale (Vitess/VStream) fixes surfaced by a live PlanetScale MySQL→MySQL cross-region migration test. The significant one unblocks `migrate` and `sync start` on ordinary *unsharded* PlanetScale databases: shard auto-discovery ran `SHOW VITESS_SHARDS LIKE ?`, whose bind parameter reached vtgate as an unparseable `:v1` placeholder, so discovery failed and sluice refused every PS→PS run unless the operator passed `--allow-cross-shard-merge` — a flag that had become a required workaround for a query bug on databases that aren't sharded at all. The minor one stops `sync stop --wait` from reporting a false drain timeout on a VStream source. Drop-in: no data-path, format, or default change. Both loud-only, no data-fidelity impact.**

## Fixed

- **PS→PS `migrate` / `sync start` on an *unsharded* PlanetScale database no longer refuses with a spurious shard-discovery failure (significant; loud refusal, no data risk; pre-existing since VStream multi-shard auto-discovery landed in v0.1.0).** Shard auto-discovery ran `SHOW VITESS_SHARDS LIKE ?` with a bind parameter, but that parameter reached vtgate as a `:v1` placeholder it cannot parse (`syntax error … near ':v1'`), so `discoverShards` errored. The cross-shard-collision preflight treats an un-determinable shard layout as "refuse rather than risk a silent cross-shard overwrite," so on an ordinary unsharded PlanetScale database — where there is no sharding and nothing to merge — sluice refused the very first command of any migrate or sync unless the operator passed `--allow-cross-shard-merge`, which short-circuits the preflight before discovery runs. That turned the flag into a de-facto required workaround for a query bug, on exactly the databases (unsharded) where the flag's real purpose (opting into a genuine cross-shard merge) does not apply. Fixed by running a plain, parameterless `SHOW VITESS_SHARDS` and filtering to the target keyspace client-side: there is no bind variable for vtgate to choke on, and an unsharded keyspace returns a single `<keyspace>/-` row, so the preflight sees one shard and proceeds normally. `--allow-cross-shard-merge` is no longer needed on unsharded databases and now applies only to genuinely sharded sources — its intended purpose. The client-side keyspace filter is also now the primary filter (a self-hosted vtgate can serve several keyspaces), not a defensive backstop. Pinned by the reader-error classification unit tests; the `-race` Integration gate, including the vstream suite, was green on the main commit before the tag.
- **`sync stop --wait` no longer reports a false drain timeout when tearing down a VStream source (minor; no data effect).** Stopping a VStream stream on `sync stop` (or Ctrl-C / outer-context cancel) surfaces from the gRPC `Recv` as a gRPC **Canceled** status (or **DeadlineExceeded**), which `errors.Is(context.Canceled)` does not match — so the engine-neutral streamer treated the clean teardown as a terminal error, skipped clearing `stop_requested_at`, and `sync stop --wait` reported `did not complete drain within …` even though the stream had already stopped cleanly. `classifyReaderError` now normalizes a gRPC Canceled/DeadlineExceeded status to the corresponding standard `context` sentinel (the raw gRPC status stays on the error chain via `%w` for diagnostics), so the streamer recognizes the clean stop and completes the drain handshake. There is no data effect — the stream always stopped; only the `--wait` completion signal was wrong. Present on VStream sources since the `sync stop --wait` graceful-drain feature (v0.9.0). Pinned by a unit matrix asserting both Canceled and DeadlineExceeded normalize to their `context` sentinels, are not classified retriable (a cancel is intentional, not transient), and keep the underlying gRPC status reachable.

## Compatibility

**No breaking changes; drop-in.** No CLI flag semantics change, no default changes, no backup format-version change; every migrate / sync / backup / restore data path is byte-identical and every engine and direction is unaffected. The only observable change is corrective: on an unsharded PlanetScale (Vitess/VStream) source, `migrate` / `sync start` that previously refused now proceeds without `--allow-cross-shard-merge`, and `sync stop --wait` on a VStream source now completes instead of printing a false drain timeout. `--allow-cross-shard-merge` still behaves identically for genuinely sharded sources (its real purpose). Non-PlanetScale sources (vanilla MySQL, Postgres, SQLite/D1) are entirely unaffected — the changed paths are VStream-only.

## Who needs this — action required

- **PlanetScale MySQL users running `migrate` or `sync start`, especially PlanetScale→PlanetScale (e.g. region moves).** If you were passing `--allow-cross-shard-merge` on an *unsharded* PlanetScale database only to get past the shard-discovery refusal, you can drop it on v0.99.190 — discovery now succeeds on its own. Keep the flag only if your source is genuinely sharded *and* you have verified the key is globally unique across shards (its intended use); on a sharded source the collision preflight still refuses without it, correctly. This is a blocker-removal, not a data-fidelity fix: no re-verification of already-migrated data is warranted.
- **PlanetScale users who script `sync stop --wait` around a VStream sync.** A stop that previously reported a false `did not complete drain` timeout now completes cleanly; any wrapper that treated that timeout as a failure will stop mis-flagging a clean stop. The stream always stopped either way — nothing was left running and no data was affected.
- **Everyone else: no action.** Non-PlanetScale engines and directions are unaffected, and no migrated data needs re-verification.
