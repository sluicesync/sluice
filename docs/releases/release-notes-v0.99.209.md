# sluice v0.99.209

**New (opt-in Security) — Ed25519 asymmetric signing for backup manifests (ADR-0154 Phase 2), adding the two things Phase 1's HMAC-off-KEK structurally could not do: signing PLAINTEXT backups, and key-separated verification where a CI/restore host holds only the PUBLIC key and never a signing secret. `sluice backup keygen` mints an interoperable Ed25519 keypair; `--sign-key` on `backup full`/`incremental` signs with the private key (plaintext or encrypted, KMS included); `--verify-key` on restore/verify checks with the public key. Fully opt-in and drop-in — nothing changes unless you pass `--sign-key`, and every v0.99.208 (Phase-1, HMAC) signed backup verifies and restores UNCHANGED: the verifier is dual-version and checks each signature at its own recorded canonicalization version, so a mixed v2/v3 chain restores link-by-link. The signature scheme is now folded into the signed bytes so an HMAC signature can't be relabeled as Ed25519 (or vice versa) to force a weaker verify path.**

## Security

- **Ed25519 asymmetric manifest signing alongside Phase 1's HMAC-off-KEK (ADR-0154 Phase 2; opt-in via `--sign-key` / `--verify-key`).** Phase 1 (v0.99.208) keyed the signature off the chain KEK — excellent zero-key-management UX for encrypted chains, but structurally limited to encrypted backups (a plaintext backup has no KEK to key off) and symmetric (a verifier must hold the same signing secret). Phase 2 adds an asymmetric scheme that reuses Phase 1's entire canonical-serialization, verification-policy, freshness, and compact/prune machinery — only the signing primitive and key loading are new — and unlocks the two cases Option A could not reach: (1) **signing PLAINTEXT backups** (a plaintext Ed25519 backup is stamped FormatVersion 6, the plaintext-signing enablement Phase 1 lacked), and (2) **key-separated verification** — a CI or DR-runbook host verifies with ONLY the public key and never holds a signing secret.
- **`sluice backup keygen` generates an interoperable Ed25519 keypair.** It writes a PKCS#8 PEM private key (`0600`) and an SPKI PEM public key — exactly what `openssl genpkey -algorithm ed25519` / `ssh-keygen -m PKCS8` / Go's `x509` marshallers speak, with no third-party dependency — to `--out-dir` (as `sluice-sign-key.pem` + `sluice-verify-key.pem`) or explicit `--priv`/`--pub`, and refuses to clobber an existing private key without `--force` (replacing a private key strands any chain it already signed). The non-secret `key_id` is a domain-separated SHA-256 fingerprint of the PUBLIC key, so signer and verifier compute the identical id; the keypair round-trip is golden-pinned.
- **`--sign-key <path|env:>` signs; `--verify-key <path|env:>` verifies.** `--sign-key` on `backup full`/`incremental` signs with the Ed25519 private key and EXPLICITLY selects the Ed25519 scheme over the `--sign` HMAC-off-KEK default (the two are mutually exclusive). It works on BOTH plaintext AND encrypted chains, including KMS — the keypair is independent of the encryption keystore (Ed25519 signs the manifest; KMS wraps the CEK — orthogonal), so the Phase-1 KMS-`--sign` refusal does NOT apply to `--sign-key`. `--verify-key` on restore/`backup verify` supplies the public key and is REQUIRED to verify an Ed25519-signed chain: the KEK does NOT verify an Ed25519 signature even on an encrypted chain. Absent a verify key, an Ed25519-signed chain takes the same DR-safe path Phase 1 established — WARN present-but-unverified and proceed — unless `--require-signature`, which refuses `SLUICE-E-BACKUP-SIGNATURE-MISSING`. An Ed25519 chain presented with only a KEK hits the no-matching-material warn/refuse path, never a spurious "verify as HMAC" downgrade.
- **The signature scheme is now bound into the signed canonical bytes (the security-critical change).** The scheme (`hmac-kek` | `ed25519`) is folded in as a dedicated length-prefixed token, bumping the manifest and lineage canonicalization to `sluice-manifest-canon/v3` / `sluice-catalog-canon/v3`. The verifier is chosen from the operator's supplied verify MATERIAL against the chain's CLAIMED scheme, and it recomputes the canonical bytes with the claimed scheme AND runs the scheme-specific primitive — so an HMAC signature relabeled `ed25519` (checked with a real Ed25519 key), or an Ed25519 signature relabeled `hmac-kek` (checked with the KEK), both fail, and an explicit scheme-mismatch check gives a precise error ahead of the MAC. A relabel cannot force a weaker verification path. This went through an adversarial security review with a SHIP verdict; the accompanying chunk-position-binding change (a plaintext v6 backup has no CEK, so the write side now gates the ADR-0152 GCM AAD on the per-chunk CEK's presence rather than on the FormatVersion proxy) was verified to preserve the anti-splice binding across all encryption cases, and scheme-confusion refuses in every combination.

## Compatibility

**No breaking changes for existing invocations; fully opt-in; every v0.99.208-signed backup verifies and restores unchanged.** Three things to know:

1. **Opt-in.** Nothing about Ed25519 signing engages unless you pass `--sign-key` (write) or `--verify-key` (read). Migrate, sync, verify, plaintext and encrypted backup/restore, and every Phase-1 HMAC path are byte-identical. New CLI surface: `backup keygen`, `--sign-key`, `--verify-key`.
2. **v0.99.208 (Phase-1, HMAC) signed chains verify and restore unchanged — the verifier is DUAL-VERSION.** The canon bump would otherwise reject every Phase-1-signed chain as "tamper," a DR-availability regression violating "newer sluice always reads older." Instead the writer emits the newest version (v3) but the verifier recomputes each signature at its OWN recorded canonicalization version: a `.sig` recording v2 is rendered WITHOUT the scheme token, byte-identical to what v0.99.208 signed (the v2 rendering is preserved verbatim as on-disk contract and golden-pinned against the original SHA, which must never move), and a v2 signature's scheme is implicitly `hmac-kek`. A mixed chain — a v0.99.208 v2 full extended by a v0.99.209 v3 incremental — verifies AND restores, each link checked at its own version; existing v2 signatures are NOT rewritten on extend. A signature recording a canonicalization version NEWER than this build refuses with an "upgrade sluice" message, NEVER a false `SIGNATURE-INVALID`.
3. **New signed manifests are canon v3.** Older sluice binaries that don't know v3 refuse with an upgrade-not-tamper message — but in practice pre-v0.99.208 binaries have no signing at all and already refuse a v6 backup at the FormatVersion preflight, so there is no new incompatibility rung beyond the one Phase 1 documented. Signing does not change the FormatVersion story: a plaintext Ed25519 backup is stamped v6 (the Phase-1 v6 discipline extended to plaintext); unsigned backups (plaintext and encrypted) keep their versions and remain readable by older binaries.

There is no data-loss fix here and no released version carried the Phase-2 canon break — the dual-version back-compat guarantee was landed within the Phase-2 arc, before Phase 2 shipped, so it fixes an unreleased break, not a shipped one.

## Who needs this — action required

- **Operators who want to sign PLAINTEXT backups, or who want backup verification on a host (CI pipeline, DR runbook) that should never hold a signing secret: opt in by generating a keypair (`sluice backup keygen`) and adding `--sign-key <priv>` to your `backup full`/`incremental`, then `--verify-key <pub>` on restore/verify.** This is the case Phase 1's HMAC-off-KEK could not serve — it signed only encrypted chains and required the verifier to hold the same passphrase. Distribute only the PUBLIC key to your verify/restore/CI hosts; the private key lives wherever backups are written and is never logged (the flag value is a path or an env-var NAME, safe in bundles). For strict-always verification (refuse a signed chain you cannot verify), add `--require-signature`.
- **Operators already using Phase-1 `--sign` (HMAC-off-KEK) on encrypted chains: nothing to do — your v0.99.208-signed backups verify and restore unchanged on v0.99.209, and `backup incremental --sign` continues to extend them (a chain keeps its scheme; you cannot mix HMAC and Ed25519 links).** There is no re-signing, no re-verification of existing backups, and no migration.
- **Everyone else: nothing to do — this is a purely additive, opt-in feature.** No existing backup, restore, migrate, sync, or verify behavior changes; no data is retroactively affected; unsigned and pre-v6 backups restore exactly as before. There is no silent-loss fix here to re-verify against.
