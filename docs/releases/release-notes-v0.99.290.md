# sluice v0.99.290

**CRITICAL fix for filtered Postgres sync — a spurious target DELETE on TOASTed rows — plus the complete ADR-0176 arc: per-stream publications and PG 15+ publication row-filter push-down for `sync --where`.** In every release with filtered PG sync (v0.99.276–v0.99.289), an UPDATE that touched a *different* column of a row whose `--where` column is large enough to TOAST out-of-line was mis-classified as a move-OUT and DELETEd the row from the target — silently, with `sync status` green. This release fixes that, ships the server-side row-filter push-down whose review surfaced it, and lands the full remediation of the 2026-07-23 blind audit — including two more released-code fixes: a MySQL cold-copy batch that could be silently skipped when a terminal duplicate-key error's message happened to quote transient wording, and a v0.99.288–289 retry-loop path where a target outage could latch a destructive forced re-snapshot instead of a warm resume. Drop-in upgrade, no breaking changes.

If you run filtered PG sync with a predicate on a large text/JSON column, re-verify those tables after upgrading — see "Who needs this" below.

## Features

**Postgres publication row-filter push-down for `sync --where` on PG 15+ sources (ADR-0176 — the PG analogue of the VStream filter push-down).** A filtered PG-source sync now pushes its `--where` predicate into the stream's publication as a per-table `WHERE (<predicate>)` row filter, so out-of-scope changes are never decoded or sent — PG's own filter transformation implements the same move-IN → INSERT / move-OUT → DELETE truth table sluice's client evaluator uses, and the predicate text is single-sourced with the snapshot SELECT and verify COUNT renderings so the three server-side evaluation sites cannot drift. Because pushing down makes the server authoritative for delivery, the feature ships inside a proven envelope, gated three ways: a conservative eligibility classifier (integer, numeric, boolean, date, timestamp-without-tz, and text/varchar under the default or `"C"` collation; everything else streams unfiltered server-side and keeps today's client-side filtering, logged at INFO) whose family list is a change-detector pin that can only widen together with the oracle; a real-PG-16 per-cell equivalence oracle (~41 predicate-shape × value-family cells, all four row-move transitions plus NULLs in every workload) asserting the delivered row set under push-down is identical to client-side-only evaluation; and the client-side evaluator staying ON as a continuous equivalence belt. Below PG 15 the emitted DDL is byte-identical to before (client-side-only, no refusal, no flag — now announced at INFO). The ADR-0175 scope guard composes with it: a second stream's bare rescope that would clear a row filter still refuses, and a filtered stream's own cold restart is resolved by a transactional no-op probe (ALTER in a transaction, re-read, commit only when PG itself proves the definition unchanged), so restarts never false-refuse while peer streams exist.

**Per-stream Postgres publications with a control-state ratchet (the ADR-0176 prerequisite).** The stream's effective publication name is now recorded in its `sluice_cdc_state` row (a new `publication_name` column, `slot_name`'s exact sibling, on both target control stores), and `sync start` ratchets onto it: an explicit `--publication-name` wins (WARNing when it differs from the record), a recorded name is reused on warm resume without re-passing the flag, and only a genuinely NEW stream with a `--where` row filter derives a per-stream default (`sluice_<stream-id>`, normalized to a safe 63-byte PG identifier) — every other stream keeps the shared `sluice_pub` default byte-identically, so nothing changes on upgrade. Cleanup parity comes with it: the pre-anchor teardowns that drop a stream's just-created replication slot now drop its per-stream publication too (never the shared default), so abandoned cold starts don't accumulate dead publications on the source. `sluice schema add-table` now extends the stream's RECORDED publication rather than always extending `sluice_pub` (which against a per-stream-publication stream would loudly fail — or, when another stream's `sluice_pub` existed, silently leave the new table outside the stream's scope after bulk-copy). The ADR-0175 scope-conflict guard also widens from the member *set* to the full publication *definition*: a rescope that would silently clear a surviving member's row filter or column list now refuses with the same `SLUICE-E-CDC-PUBLICATION-SCOPE-CONFLICT`.

## Fixed

**CRITICAL (silent loss): filtered PG sync no longer emits a spurious DELETE for an in-scope row whose filtered column is unchanged out-of-line TOAST (affected: v0.99.276–v0.99.289, every release with filtered PG sync).** pgoutput sends an out-of-line TOASTed column as `unchanged-toast-datum` in the UPDATE's NEW tuple whenever a different column changed — even under `REPLICA IDENTITY FULL` — so the decoded after-image omitted the column, the `--where` row-move evaluation read it as UNKNOWN→false, and an in-scope sibling-column UPDATE was mis-classified as a move-OUT: sluice DELETEd a row the source still holds, silently, exit 0, status green. The trigger is ordinary: a predicate referencing a text/varchar/JSON column whose values exceed ~2KB post-compression, plus any UPDATE to another column of that row. The Postgres reader now backfills the omitted columns from the complete RI-FULL before-image on filtered tables (exact by PG's unchanged ⇒ old == new guarantee), and the router gains an after-image completeness belt — `SLUICE-E-WHERE-CDC-AFTER-IMAGE`, the before-image guard's sibling — that stops the stream loudly if a partial after-image ever slips past the reader again. Reproduced live before fixing (a 9.6KB `STORAGE EXTERNAL` column + sibling-column UPDATE deleted the target row); pinned by a new end-to-end integration gate plus a no-touch workload arc in the push-down oracle so every family cell now also exercises an UPDATE that never sets the filtered column.

**Terminal-code shield in both applier classifiers: a structured server error can no longer be flipped retriable by transient wording quoted in its message (affected: the MySQL silent batch-skip chain, v0.99.92–v0.99.289; the loud retry-delay class on both engines dates to the v0.42.0 classifier).** Both engines' error classifiers ended with text-fallback legs that scanned the whole error string for transport-transient wording (`reparent`, `not serving`, `connection refused`, `disk full`, ...) — and those legs were reachable even when the error carried a structured MySQL errno or PG SQLSTATE already judged terminal, because a server error's message routinely echoes row data, key values, and table names. The MySQL consequence was the worst class sluice has: a plain-INSERT cold-copy batch failing duplicate-key (1062) on a table named e.g. `reparent_history` — or with a colliding value like `planned-reparent-2026-07` — classified RETRIABLE, the byte-identical retry hit 1062 again, and the tolerate-1062-on-retry path (whose safety argument assumes a first-attempt 1062 stays terminal) swallowed it as "the rows already landed": both INSERTs had rolled back, so the entire batch was silently absent while the migration reported success. On Postgres the blast radius was smaller (PgError messages don't echo row data): a terminal error quoting transient-looking stored text retried a deterministically-failing batch through the full retry budget instead of failing loudly at once. Now a matched structured driver error is classified by its code ALONE — the text legs run only when no structured error is present — with the documented structured-code+message exceptions preserved. Pinned by a cross-product matrix on both engines: every structurally-terminal code × every transient substring must classify non-retriable, and every previously-pinned transient shape stays retriable.

**A failed between-attempt position read no longer latches a destructive forced re-snapshot (affected: v0.99.288–v0.99.289; the latch predates them but was unreachable — a terminal connect exit meant a fresh process always warm-resumed correctly).** The sync retry loop reads the persisted CDC position between attempts to decide cold-copy-phase (force a clean re-establishment) vs CDC-phase (warm resume) — but it discarded the read's error, and both engines report found=false on a failed read, so "the target is down and I could not read the row" was indistinguishable from "no anchor row exists". A routine target restart — the exact scenario the v0.99.288/289 connect-phase retry now survives — could therefore latch a restart-from-scratch mid-outage, and the first successful reconnect executed a forced fresh cold start ignoring the valid persisted position: in-scope tables dropped and re-copied on native MySQL targets, and on idempotent sources (Postgres/VStream) a re-snapshot whose slot is recreated at NOW — which never replays source DELETEs committed before the new snapshot, silent divergence where a warm resume would have replayed them from retained WAL/binlog. The discriminator now only trusts a SUCCESSFUL read: a failed read leaves the restart latch at its prior value (defaulting to warm resume whenever any successful read observed the anchor this run), and only a successful read proving "no anchor row" may force the re-establishment. Pinned by unit tests on the retry seam in both directions.

**Warm resume now reconciles the pushed publication row filter with the current `--where` flags (never released — this was the audit's ADR-0176 release blocker, closed before the arc ships in this release).** The push-down makes `--where` durable source-side catalog state, and warm resume deliberately never re-ensures the publication — so restarting with a widened, changed, or removed `--where` would have left the SERVER filtering on the stale predicate, with rows the new flags admit never decoded or sent, unobservable client-side by construction. The pushed subset's canonical hash is now recorded in `sluice_cdc_state` as `row_filter_hash` (`publication_name`'s exact sibling on both engine control stores) and compared on every warm resume of a publication-capable source, refusing drift with the new `SLUICE-E-WHERE-PUSHDOWN-DRIFT` naming the escapes — re-pass the original `--where`, or `--restart-from-scratch` to re-snapshot under the new predicate (required for a widened filter anyway) — both of which are themselves exempt from the comparison so the refusal can never block its own remedies. Pinned end to end on real PG: widened and removed `--where` both refuse (red before the fix — both silently resumed), same-filter resume stays byte-identical, and the restart-from-scratch escape re-establishes under the new predicate and re-records the hash.

**Temporal literal-granularity mismatches are excluded from the push-down envelope (never released — closed within the arc).** Postgres resolves a finer-than-column temporal literal by silently truncating or rounding it inside a publication row filter — a time-bearing literal against a DATE column stores as `(d < '2026-01-15'::date)`, and a >6-fractional-digit literal against a microsecond timestamp rounds — while sluice's client evaluator compares the literal at full precision, so the pushed filter and the equivalence belt provably disagree on the boundary (ground-truthed on real PG 16). Such predicates now classify out of the envelope: those tables stream unfiltered server-side and keep client-side filtering, the known-correct fallback under both of the audit's adjudications of the finding. The real-PG oracle also gains its missing op×family cells plus a permanent granularity-fallback gate that asserts both the live-PG divergence and the classifier exclusion.

**A dial-time connect failure on the APPLY path now retries instead of exiting with zero retries, on both engines (Bug 200, found by the v0.99.289 post-release regression cycle; always a loud, zero-loss exit — an availability gap, not a data risk).** The v0.99.288/289 connect-phase hardening covered the pipeline's setup/reopen sites — but the engine appliers dial too: when a target restart severs the pool, the next apply's pool acquire (Postgres) or `begin tx` (MySQL) dials fresh into the refused window, and neither applier classifier had a dial-shape leg — so the first apply during a routine target restart exited terminally with zero retries. Both classifiers now match the positive dial-transient shapes (POSIX `connection refused`, the Windows `connectex:`/`actively refused` wordings, `connection timed out`), and the Postgres concurrent lane's begin site — which returned errors raw, bypassing the classifier entirely — now classifies. Default-deny intact and pinned both ways: a typo'd hostname and an auth failure inside a connect error stay terminal on both engines.

## Also in this release

Audit quick-wins: the fleet YAML sync spec gains the `publication-name` key (the scope-conflict refusal's primary documented escape was previously unexpressible on the fleet surface most likely to trigger it); the error-codes registry and staged-wave guide now state the guard's v0.99.289 existence semantics instead of the stale active-slot wording, and the staged-wave recipe gains an explicit decommission step (drop the finished wave's slot and per-stream publication — a leftover slot pins WAL and blocks later differently-scoped cold starts); the below-PG-15 version-gate filter drop now logs at INFO so an operator never believes WAL delivery is server-filtered when it is not; and the scope-conflict refusal now names the always-valid own-publication escape.

## Compatibility

No breaking changes; no flag or default changes to existing invocations; drop-in upgrade. Existing streams keep their publication byte-identically on upgrade — the recorded/ratcheted name only diverges from the shared `sluice_pub` default for genuinely NEW filtered PG streams (or an explicit `--publication-name`).

On PG 15+ a filtered PG-source sync now pushes its predicate into the publication, so the server stops sending out-of-scope changes — a wire-traffic reduction, with the delivered row set proven identical to client-side-only evaluation by the per-cell oracle and the client evaluator still running as a belt. Below PG 15, and for every predicate shape outside the conservative envelope, behavior is byte-identical to before (client-side filtering), now with an INFO log naming the unfiltered tables. One new refusal is deliberate: restarting a filtered PG stream with a changed or removed `--where` now refuses with `SLUICE-E-WHERE-PUSHDOWN-DRIFT` (re-pass the original predicate, or `--restart-from-scratch` to re-snapshot under the new one) — the alternative was a silently narrowed stream.

The classifier changes only move errors between loud outcomes or into the existing retry budget: structurally-terminal server errors now fail loudly at once (previously, in the worst MySQL case, silently skipped — see Fixed), dial-time transients on the apply path now retry, and every terminal class (auth, DSN, unknown shapes, `no such host`) stays terminal. `migrate` is affected only by the MySQL cold-copy classification fix, which makes it stricter/louder. MySQL, PlanetScale, and Vitess sources have no publication and are untouched by the publication work.

## Who needs this — action required

- **Anyone running filtered PG sync (`sync --where` with a Postgres source) on v0.99.276–v0.99.289 whose predicate references a large text/varchar/JSON column** (values big enough to TOAST out-of-line, roughly ≳2KB after compression) — upgrade, then re-verify the filtered tables (`sluice verify`): any UPDATE to a *different* column of such a row may have deleted that row from the target. If verification finds missing rows, `--restart-from-scratch` (or a targeted backfill) restores them. Predicates on small always-inline columns (ints, short strings, dates) were not exposed.
- **Anyone who ran cold copies into native MySQL/MariaDB targets on v0.99.92–v0.99.289** — a narrow but real silent-skip: it required a first-attempt duplicate-key failure (e.g. string keys colliding under the target collation) AND transient wording (`reparent`, `connection refused`, `disk full`, ...) appearing in the table name or echoed key value. If both could describe your data, re-verify row counts on those tables (`sluice verify --depth count`). If neither does, no action.
- **Anyone whose v0.99.288–v0.99.289 sync survived a target restart and then unexpectedly re-copied from scratch** — that was the position-read latch. On Postgres/VStream sources, re-verify the target for rows deleted on the source during the outage (the re-snapshot never replays pre-snapshot DELETEs). Upgrading prevents recurrence.
- **Filtered PG stream operators upgrading onto the push-down:** nothing to do at upgrade — but know that changing `--where` on a running stream now requires `--restart-from-scratch` (correctness requires the re-snapshot anyway) and the refusal will tell you exactly that.
- Everyone else: no action needed beyond upgrading.
