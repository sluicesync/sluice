# sluice v0.99.63

**New: a full multi-table Vitess/PlanetScale keyspace now cold-copies in ONE `sluice migrate`/`sync` command at bounded memory — no per-table `--include-table` workaround (ADR-0095).** Drop-in: nothing to change in your invocation, and the snapshot→CDC handoff stays exactly-once across the seam. Also a HIGH fix: a PostgreSQL cold-start with `ENUM` columns no longer crash-loops on resume.

## Features

- **Auto-shard-by-table VStream cold-copy (ADR-0095).** A keyspace-wide VStream COPY (`Match:"/.*/"`) makes vtgate interleave every table down one stream, and sluice's orchestrator drains one table at a time — so every not-yet-active table buffered in memory, RAM ≈ Σ(all-but-active table sizes). On a large keyspace that hit the ADR-0071 loud-refusal floor (or OOM), and the only way through was driving the copy table-by-table by hand with repeated `--include-table` runs. The engine now auto-shards the COPY internally: it opens, drains, and completes one single-table VStream COPY (`Match:<table>`, constant memory, no interleave) at a time on the same gRPC connection, then opens the keyspace-wide CDC tail. You still run one command; the per-table loop is invisible above the `SnapshotStream` boundary. Engaged by default on a fresh cold-start with more than one table; opt out via `vstream_copy_single_stream=true` in the source DSN (restores the legacy single interleaved stream and its ADR-0071 refusal floor). Pinned by a set-min stitch unit matrix (single-uuid / multi-uuid / multi-shard × in-order / reverse / from-beginning / disjoint-refuse / missing-shard) plus a never-selects-max silent-loss guard, an auto-shard pump unit test driving N sequential per-table streams and the keyspace-wide CDC handoff, a bounded-memory unit test proving the loud refusal is gone, and a `vstream`-tagged integration test copying two cap-busting tables with no refusal and a clean CDC handoff.
- **Exactly-once across the per-table snapshot→CDC seam.** Each table finishes its COPY at a different per-table VGTID `P_i`, so there is no single global snapshot point to resume from. The CDC tail resumes from the per-shard GTID-set intersection `P_start = ⋂ P_i` (the set-minimum, never the maximum): gapless because `P_start` is a subset of every `P_i`, so no committed change after any table's snapshot can be skipped (the silent-loss class), and overlap-safe because the re-delivered window `(P_start, P_i]` is the same at-least-once seam every cold-start→CDC handoff already absorbs via the ADR-0010 idempotent upsert and the Bug-125 idempotent-copy writer. The stitch reuses the engine's existing GTID superset primitive (`Contain`/`gtidAtOrAfter`) to *select* the set-min among the observed per-table positions rather than synthesizing GTID interval arithmetic; a genuinely-disjoint per-shard set (which one causally-ordered VStream session should never produce) refuses loudly rather than guess a position that might gap.

## Fixed

- **PostgreSQL cold-start `ENUM` `CREATE TYPE` is now idempotent (Bug 154, HIGH).** The PG cold-start schema phase (`CreateTablesWithoutConstraints`) emitted a bare `CREATE TYPE ... AS ENUM` with no existence guard — `CREATE TABLE` on the same path is already `IF NOT EXISTS`, so only the enum type was non-idempotent. Once the first cold-start created the type, any mid-COPY interruption followed by a restart re-entered the schema phase and hard-failed with `SQLSTATE 42710 type already exists`, turning every restart into a crash-loop with zero progress. The ADR-0072 resume path makes this worse by design: it deliberately skips the populated-target preflight on the (until now false) assumption that `CreateTablesWithoutConstraints` is fully idempotent. PostgreSQL has no `CREATE TYPE IF NOT EXISTS`, so the cold-start create now routes through the same DO-block `duplicate_object` guard the CDC enum-forward path already used (`ensureEnumType`), extracted into a shared `guardedCreateEnumType` helper. Pinned by a unit test on the guard-wrapper shape and an integration test that a resumed cold-start with enum columns no longer 42710s and completes. This was always a loud failure (crash-loop, no progress), never silent data loss.
- **`--reset-target-data` now drops synthesized enum TYPES, including a PG-source enum carried by its original name (Bug 154).** `DropSchemaTypes` dropped only the synthesized `<table>_<col>_enum` name and missed a PostgreSQL-source enum type carried by its original `TypeName` (Bug 19c) — leaving that type orphaned to re-trigger 42710 on the next fresh cold-start. The drop now mirrors exactly what cold-start creates (`resolveEnumTypeName`, skip generated columns, dedup shared types), so a reset leaves no orphan and a fresh cold-start after a reset succeeds. Pinned by an integration test that reset drops both the synthesized and the PG-source-named types.

## Compatibility

- **No breaking changes; drop-in upgrade.** ADR-0095 is on by default and requires no change to your `migrate`/`sync` invocation. `--include-table` is preserved exactly (it scopes which tables the per-table loop iterates), and the persisted handoff position token shape is unchanged (`P_start` is itself a valid per-shard GTID array).
- **Scope: the Vitess/PlanetScale (VStream) cold-copy path only.** Auto-shard engages on a *fresh* cold-start with more than one table in scope. A single-table scope is already constant-memory single-stream, so auto-shard adds nothing there. Vanilla MySQL binlog and Postgres cold-copy paths are untouched.
- **Resume stays single-stream (documented v1 limitation, not a correctness gap).** A process-restart resume (a position carrying the ADR-0072 per-table `TablePKs` cursor) stays on the legacy single-stream resume path; resume is still correct (the idempotent COPY writer absorbs any overlap and `--include-table` scope is honored), but a resume of a very large multi-table copy can re-introduce the interleave for not-yet-finished tables and so re-hit the ADR-0071 refusal under a tight `--max-buffer-bytes`. The fresh cold-start — the dominant large-keyspace case — is fully covered.
- **Opt-out:** `vstream_copy_single_stream=true` in the source DSN restores the legacy single keyspace-wide interleaved COPY (and its ADR-0071 multi-table loud-refusal floor) for an operator who wants the global single-position consistency on a small keyspace or is debugging.
- The Bug 154 fix is a strict reliability improvement — no flag or behavior change to a successful migration.

## Who needs this — action required

- **Anyone cold-copying a full multi-table Vitess/PlanetScale keyspace in one command:** no action required, and the per-table `--include-table` workaround is no longer needed — just run `sluice migrate`/`sync` against the whole keyspace. No re-verification of prior migrations is needed: this is a new capability, not a fix to a path that was silently losing data (the old path *refused loudly* or OOM'd rather than completing with missing rows).
- **Anyone who has hit (or could hit) a `SQLSTATE 42710 type already exists` crash-loop when restarting a PostgreSQL-target cold-start that has enum columns:** upgrade to v0.99.63. The crash-loop is fixed and resume now completes. If a previous attempt left an orphaned enum type behind, `--reset-target-data` now cleans it up so a fresh cold-start succeeds. No data was ever silently lost by this bug — it was always a loud, zero-progress failure — so there are no completed migrations to re-verify on its account.
