# sluice v0.99.129

**Behavior change (headline) — multi-shard VStream CDC now relaxes vtgate's `MinimizeSkew` BY DEFAULT, fixing a cross-region catch-up FREEZE (ADR-0120, roadmap item 27).** Previously the steady-state multi-shard Vitess/PlanetScale CDC stream was opened with `MinimizeSkew=true`, so vtgate held the *ahead* shard back to keep the merged stream commit-time ordered; under an apply-deficit backlog (CDC catch-up after downtime, a large write burst, or a reshard) a real cross-region A/B showed this didn't just slow down — it **froze the lagging cross-region shard's stream entirely** (a liveness wedge). The default is now relaxed: both shards stream and drain concurrently. It is correctness-safe under range-sharding and exactly-once held across every gated harness; the new opt-out `--vstream-preserve-skew` (DSN `vstream_preserve_skew=true`) restores the old hold. **Affects multi-shard Vitess/PlanetScale CDC ONLY — inert on PostgreSQL / native-MySQL sources and on single-shard keyspaces.** Also new: an engine-neutral `sluice_sync_lag_seconds` metric + ungated alert, and an email/SMTP notification sink. Fully drop-in over v0.99.128 for everyone not running multi-shard VStream CDC.

## Features

**`sluice_sync_lag_seconds` — an engine-neutral "seconds behind source" sync-lag metric + an ungated threshold alert on it (roadmap item 45).** sluice already exported `sluice_target_replica_lag_seconds`, but that is the TARGET's internal replica lag read from PlanetScale control-plane telemetry (ADR-0107) — PS-gated, and not sluice's own apply lag. This adds the missing engine-agnostic signal: how far the target trails the SOURCE's latest applied commit, derived purely from each IR change's source commit timestamp (`ir.Change.SourceCommitTime`), which the CDC readers now carry from the binlog event header (native MySQL), the VStream event time (Vitess/PlanetScale), the pgoutput Begin/Commit commit time (PostgreSQL pgoutput), and the change-log `committed_at` (postgres-trigger). A streamer-side interceptor observes every change on its way to the applier — covering the batched, per-change, and concurrent-lane apply paths uniformly, and on a merged multi-shard VStream naturally surfacing the slowest shard's lag — and `/metrics` exposes `sluice_sync_lag_seconds{stream_id=…}`. The reading is honest when idle: the lag is FROZEN at the instant each change is applied (so it does not climb with wall-clock between changes), and a sustained absence of applied work retires a stale reading to 0 (caught up) rather than letting it grow — the #1 false-alarm class; a change with no source commit timestamp leaves the metric omitted (unknown), never reported as 0. The new `--notify-sync-lag-seconds SECONDS` flag wires a threshold alert on it through the existing notify sinks with the same edge-trigger + per-rule cooldown + hysteresis + advisory + failure-isolated semantics as the ADR-0107 rules — but UNGATED from PlanetScale telemetry, so it works on MySQL and PostgreSQL alike with no `--planetscale-org`. It is deliberately distinct from `--notify-lag-seconds` (the PS target-internal replica lag) and complements `sluice_seconds_since_last_apply` (which ages, to catch a stuck stream): sync lag answers "falling behind while flowing." Off by default (0 = disabled; no `/metrics` endpoint ⇒ no interceptor, no goroutine, default apply path byte-identical), so the zero value is the safe off default for every construction.

**Email / SMTP notification sink — completing the notify-sink set (roadmap item 48, an ADR-0107 amendment).** ADR-0107's item 36 shipped the generic-webhook + Slack alert sinks and deferred email; this adds an `SMTPNotifier` implementing the same `notify.Notifier` interface with identical advisory + failure-isolated semantics — a dead relay surfaces as a returned error the fan-out joins and the alerter logs-and-swallows, NEVER affecting the sync. A single SMTP sink covers every transactional provider (SendGrid, Mailgun, SES, Postmark are all SMTP) AND self-hosted/corporate relays, without N provider-specific HTTP integrations. It is wired into the SAME threshold-alerter path the other sinks use, so EVERY alert — the ADR-0107 storage/CPU/mem/lag/growth rules AND item 45's `--notify-sync-lag-seconds` — can be delivered by email, on both the sync-path alerter and the standalone `metrics-watch` daemon. New flags (mirroring the webhook/Slack patterns) on `sync start` and `metrics-watch`: `--notify-smtp-host` / `--notify-smtp-port` / `--notify-smtp-from` / `--notify-smtp-to` (repeatable), with `--notify-smtp-tls` (starttls | implicit | none), `--notify-smtp-auth` (none | plain | login), and `--notify-smtp-username`; the credential is supplied via the `SLUICE_NOTIFY_SMTP_PASSWORD` env var ONLY, never on the command line. Opt-in and zero-value-off: the sink is inert unless `--notify-smtp-host` is set, and a half-configured sink (host without from/to, or plain/login without a username) is refused loudly at startup naming the missing flag rather than silently dropping every alert. The email is a tasteful, minimal ops alert — a clean subject (`[sluice] <metric> threshold alert: <stream>`), a level-coloured HTML card stating the breached value vs threshold plus the rule/stream/timestamp facts, and a plaintext alternative for text-only clients. Built on `github.com/wneessen/go-mail` (context-aware, MIT). Rendered sample emails for a storage-utilization, a sync-lag, and a CPU alert are committed under `docs/operator/email-samples/`; the sink is covered by unit tests (config/validation, HTML+text rendering, zero-value-off, failure isolation against a dead relay) plus a hermetic integration test that sends through a Mailpit container and asserts the captured subject/recipient/bodies. This closes the notify-sink set — email was the last channel demand-deferred at item 36.

## Fixed

**The cross-region multi-shard VStream CDC catch-up FREEZE (ADR-0120, roadmap item 27).** On a multi-shard Vitess/PlanetScale source the CDC stream was opened with `MinimizeSkew=true`, so vtgate held the *ahead* shard's delivery back to keep the merged stream commit-time ordered. A real cross-region A/B (two regions ~82 ms apart, shards split across them) showed that under an apply-deficit backlog — the normal state during CDC catch-up after downtime, a large write burst, or a reshard — the old default did not merely slow down: it **froze the lagging cross-region shard's stream entirely** (~485k changes backlogged, never converging, reproduced four times). With the relaxation the same 1M-change backlog drained in ~153 s, exactly-once (source == target, no gap/dup/corruption). This is fixed by flipping the steady-state default to `MinimizeSkew=false` (both shards stream and drain concurrently); see Compatibility for the default change and how to restore the old behaviour. The relaxation is correctness-safe under range-sharding: a given `(table, PK)` lives in exactly one shard, the concurrent key-hash apply lanes serialize same-key changes within a shard, and `StopOnReshard=true` closes the only window where a key could transiently span two shards — so cross-shard commit-time ordering buys nothing (a consumer audit confirmed every multi-shard consumer — position orderer, lane apply, frontier checkpoint, snapshot stitch, stall watchdog, resume — is already per-shard-independent). Exactly-once held across four gated A/B harnesses + the reshard-mid-stream seam + the cross-region run + the CI `-race` integration suite. With uncongested apply there is no difference between relaxed and held, so the new default has **no steady-state cost** — it is pure upside in the catch-up regime. When relaxed, the per-shard stall WARN drops the now-impossible "MinimizeSkew catch-up hold" cause so it points only at a genuine stall.

## Compatibility

**One default changed — multi-shard Vitess/PlanetScale CDC only.** The steady-state multi-shard VStream CDC stream now opens with `MinimizeSkew=false` (relaxed) instead of `true`. The new opt-out `--vstream-preserve-skew` (DSN: `vstream_preserve_skew=true`) restores the old `MinimizeSkew=true` hold for anyone who specifically needs strict cross-shard commit-time delivery and accepts the catch-up wedge risk; it is opt-out-named, so the relaxed default is the Go zero value and no non-CLI construction can silently invert it (the v0.99.51 zero-value trap). This change is **inert on PostgreSQL and native-MySQL sources and on single-shard Vitess/PlanetScale keyspaces** — those have no merged multi-shard stream and behave byte-identically to v0.99.128. No other default moves; the two new features (`--notify-sync-lag-seconds` / `sluice_sync_lag_seconds` and the `--notify-smtp-*` sink) are both off-by-default and opt-in, so a config that sets neither sees no change. No read/write value path changes. Otherwise fully drop-in over v0.99.128.

## Who needs this — action required

**Operators running CDC from a multi-shard Vitess/PlanetScale source must read this — the default changed for you.** No data re-verification is needed (exactly-once is preserved either way; this is a liveness/throughput change, not a silent-loss fix, and no prior release silently lost data on this path), but the steady-state stream now relaxes `MinimizeSkew` by default. If you were affected by — or at risk of — the cross-region catch-up freeze, you want this default and need do nothing on upgrade. If you specifically depend on strict cross-shard commit-time delivery of the merged stream and accept the catch-up wedge risk, set `--vstream-preserve-skew` (or `vstream_preserve_skew=true` in the source DSN) to restore the old behaviour. Operators on **PostgreSQL, native-MySQL, or single-shard** sources need not act — the change is inert for you. Everyone else: the two new opt-in observability features (the engine-neutral `sluice_sync_lag_seconds` metric + `--notify-sync-lag-seconds` alert, and the `--notify-smtp-*` email sink) are available if you want them and require no action otherwise.
